“You can’t manage what you can’t measure.” — often attributed to Peter Drucker (and frequently misquoted; the truer Drucker line is “What gets measured gets managed — even when it’s pointless to measure and manage it, and even if it harms the purpose of the organization to do so.”). Both quotes belong in every analyst’s mental toolkit: measurement is necessary, but measurement is also dangerous when divorced from purpose.

This single document is designed as exhaustive preparation material for the Workera ELO / SAGE assessment of Business Analytics Fundamentals. It is opinionated, dense, and structured so that you can read it cover-to-cover and emerge able to (a) define every term, (b) derive every formula, (c) implement every method in code, and (d) reason about every trade-off in real organizations.

PART 1: Building a Data Foundation

MODULE 1: Preparing Business Data

1.1 Why data preparation dominates analytics work

Surveys from CrowdFlower, Anaconda, and Kaggle consistently show practitioners spending 60–80% of their time wrangling data rather than modeling. This is not a sign of immaturity — it is the job. Models are commoditized; trustworthy, well-shaped data is the durable asset. Every assessment item about “preparing data” is really asking: do you understand the chain of transformations that make raw operational records fit for analytical questions?

A useful mental model is the CRISP-DM loop (Cross-Industry Standard Process for Data Mining):

  1. Business Understanding → frame the decision and value at stake.
  2. Data Understanding → inventory sources, profile fields, surface anomalies.
  3. Data Preparation → clean, transform, enrich, validate.
  4. Modeling → select methods (statistical, ML, rules) appropriate to the question.
  5. Evaluation → confirm both technical performance and business fit.
  6. Deployment → operationalize, monitor, and feed learnings back.

Each iteration through CRISP-DM tightens the loop between question, data, and decision.

1.2 Data types — measurement scales

A single mistake in identifying the measurement scale of a field invalidates downstream statistics. Stevens’ (1946) taxonomy is the canonical reference.

Scale Definition Allowed Operations Example Center Spread
Nominal Labels with no order =, ≠ gender, country, SKU mode n/a
Ordinal Ordered categories, unequal gaps =, ≠, <, > NPS bucket, T-shirt size, Likert median IQR, range
Interval Equal gaps, no true zero +, − Celsius, calendar year mean std, var
Ratio Equal gaps, true zero exists +, −, ×, ÷ revenue, count, duration geometric mean possible CV

Common pitfall: computing the mean of a Likert-scale field (ordinal) and treating it as interval. Defensible in business reporting if you state the assumption explicitly; indefensible in formal inference without sensitivity checks.

Structured vs semi-structured vs unstructured:

  • Structured — rows and typed columns; sits in RDBMS or warehouse tables. SQL is the native language.
  • Semi-structured — has schema cues but no fixed columns: JSON, XML, Avro, Parquet with nested fields, log lines with key/value pairs.
  • Unstructured — free text, images, audio, video; requires feature extraction (NLP/CV) before classical analytics applies.

1.3 Profiling: the first hour with any new dataset

Before touching any model, profile. Profiling answers six questions:

  1. What is here? — schema, types, row count, file size.
  2. What’s missing? — null patterns, blanks, sentinel values (-1, 9999, "N/A").
  3. What’s unique? — primary keys, cardinality, duplicates.
  4. What’s the shape? — histograms, ranges, percentiles, skewness.
  5. What’s related? — correlations, joins, foreign keys.
  6. What’s anomalous? — outliers, type mismatches, encoding errors.

A minimal Python profiling skeleton:

import pandas as pd
import numpy as np

df = pd.read_csv("orders.csv")

print(df.shape)
print(df.dtypes)
print(df.head())
print(df.describe(include="all").T)
print(df.isna().sum().sort_values(ascending=False))
print(df.nunique().sort_values(ascending=False))

# Quick anomaly screen on numerics
for col in df.select_dtypes(include=np.number).columns:
    q1, q3 = df[col].quantile([0.25, 0.75])
    iqr = q3 - q1
    lo, hi = q1 - 1.5 * iqr, q3 + 1.5 * iqr
    n_out = ((df[col] < lo) | (df[col] > hi)).sum()
    print(f"{col}: {n_out} potential outliers (IQR rule)")

For a richer first pass, ydata-profiling (formerly pandas-profiling), sweetviz, dataprep.eda, or Great Expectations give you HTML reports and assertions in one shot.

1.4 Missing data: mechanism matters more than method

Rubin (1976) introduced the three canonical missingness mechanisms. Choose imputation based on which one is plausible.

  • MCAR — Missing Completely At Random. Missingness is independent of any observed or unobserved value. Example: a sensor randomly drops 0.1% of pings. Listwise deletion is unbiased, just inefficient.
  • MAR — Missing At Random. Missingness depends only on observed values. Example: older users skip the “income” field, but you observe age. Listwise deletion is biased; well-specified imputation (MICE, regression imputation) is unbiased.
  • MNAR — Missing Not At Random. Missingness depends on the unobserved value itself. Example: high earners refuse to disclose income. No imputation method can fix this without external information (proxy, instrumental variable, model of nonresponse).
Method When to use Caveats
Listwise deletion MCAR; small fraction missing Loses statistical power; biased under MAR/MNAR
Mean / median imputation Quick sanity baseline Shrinks variance; distorts correlations
Mode imputation Categorical fields Inflates dominant category
Forward / backward fill Time series with persistence Wrong if the gap spans regime change
KNN imputation Continuous, low-d Slow on large data; requires scaling
MICE / chained equations MAR, multiple variables Iterative; report multiple imputed datasets and pool
Model-based (regression, RF) When predictors are strong Hides uncertainty; consider multiple imputation
Indicator + impute Always pair: keep “was_missing” flag Lets the model learn that missingness itself carries signal 
from sklearn.experimental import enable_iterative_imputer  # noqa
from sklearn.impute import IterativeImputer, KNNImputer

mice = IterativeImputer(max_iter=10, random_state=0)
X_mice = mice.fit_transform(X)

knn = KNNImputer(n_neighbors=5)
X_knn = knn.fit_transform(X)

Golden rule: never silently drop or impute without first creating <col>_was_missing indicator features and quantifying how the choice changes downstream estimates.

1.5 Outliers — detection, judgment, and treatment

Outliers are observations inconsistent with the bulk of the data. They can be (a) errors, (b) rare-but-real, or (c) the very signal you care about (fraud, churn, defects). Always investigate before removing.

Univariate detection:

  • IQR rule (Tukey): flag values below \(Q_1 - 1.5 \cdot \text{IQR}\) or above \(Q_3 + 1.5 \cdot \text{IQR}\), where \(\text{IQR} = Q_3 - Q_1\). Use \(3 \cdot \text{IQR}\) for “extreme” outliers.
  • Z-score: \(z_i = \frac{x_i - \bar x}{s}\); flag \(|z| > 3\). Assumes normality and is itself sensitive to outliers (the mean and std are inflated by them).
  • Modified z-score (MAD-based): \(M_i = 0.6745 \cdot \frac{x_i - \tilde x}{\text{MAD}}\) where \(\text{MAD} = \text{median}(|x_i - \tilde x|)\). Robust because median and MAD are not pulled by outliers. Flag \(|M_i| > 3.5\).

Multivariate detection:

  • Mahalanobis distance with the covariance matrix \(\Sigma\): \(D^2(x) = (x - \mu)^T \Sigma^{-1} (x - \mu)\); compare against \(\chi^2_p\).
  • Isolation Forest (Liu, Ting, Zhou 2008) — anomalies are easier to isolate by random partitioning, so they have shorter average path lengths in random trees.
  • Local Outlier Factor (LOF) — measures the local density deviation of a point relative to its neighbors.
  • One-Class SVM — learns a boundary around the majority class.
  • DBSCAN — points not assigned to any density cluster are noise.
from sklearn.ensemble import IsolationForest

iso = IsolationForest(contamination=0.01, random_state=0)
df["anomaly_score"] = iso.fit_predict(df[features])

Treatment options:

  1. Investigate first. A \(999{,}999\) order might be a real B2B deal or a data-entry slip.
  2. Cap / winsorize at the 1st/99th percentile.
  3. Transform (log, Box-Cox, Yeo-Johnson) to compress tails.
  4. Model robustly — Huber, RANSAC, quantile regression.
  5. Segment — separate “whales” from the long tail and model both.

1.6 Feature engineering and transformation

The right transformation often matters more than the right model. Core toolbox:

  • Scaling
    • Min-max: \(x' = \frac{x - x_{\min}}{x_{\max} - x_{\min}}\).
    • Standardization: \(x' = \frac{x - \mu}{\sigma}\).
    • Robust (median/IQR): resistant to outliers.
  • Distribution shaping
    • \(\log(1+x)\) — for right-skewed counts/revenue.
    • Box-Cox: \(x' = \frac{x^\lambda - 1}{\lambda}\) for \(x > 0\).
    • Yeo-Johnson — generalizes Box-Cox to \(x \le 0\).
  • Discretization / binning
    • Equal-width, equal-frequency (quantile), or supervised (decision-tree based).
    • Useful for non-linear effects, regulatory reporting, and credit-scoring scorecards.
  • Encoding categorical variables
    • Label encoding — integer per category; OK for tree models, dangerous for linear models because it implies an order.
    • One-hot encoding — k or k−1 dummies; explodes dimensionality for high-cardinality fields.
    • Target / mean encoding — replace category with the mean of the target; powerful but leaks unless you apply out-of-fold and add smoothing \(\frac{n \bar y_c + m \bar y}{n + m}\).
    • Frequency encoding, hashing trick for very high cardinality.
    • Embeddings for ML pipelines (entity embeddings — Guo & Berkhahn 2016).
  • Feature crosses — multiplicative or pairwise interactions (price × discount, hour × day).
  • Date/time features — year, month, day-of-week, week-of-year, is_holiday, days_since_signup, cyclic encoding via \(\sin / \cos\) of \(\frac{2\pi t}{T}\).

Validation is the contract between data producers and consumers. Codify expectations early and check them on every load.

Embed validation in pipelines — fail loud, fail early. The cost of bad data scales exponentially as it moves downstream.

MODULE 2: Documenting Data Lineage

2.1 What lineage is and why it matters

Data lineage is the documented map of how data moves from source systems → ingestion → transformation → consumption. It answers: Where did this number come from? What changes if I modify this column?

Three levels of lineage granularity:

  • System-level — “warehouse ← Stripe + Salesforce + product events”.
  • Table-level — fct_revenue ← stg_stripe_charges + stg_refunds.
  • Column-level — fct_revenue.net_amount ← stg_stripe_charges.amount − stg_refunds.amount.

Lineage drives five concrete capabilities:

  1. Impact analysis — “If I rename this column, what breaks downstream?”
  2. Root-cause analysis — “Why did the executive dashboard tick up 12% last night?”
  3. Compliance & audit — GDPR Article 30 (records of processing), SOX, HIPAA, BCBS 239 demand traceability.
  4. Trust & adoption — analysts trust numbers they can trace.
  5. Cost optimization — find unused tables and dead pipelines.

Bare minimum lineage hygiene for a small team:

  • Use a transformation framework that produces lineage as a side effect of writing the transforms (dbt, SQLMesh, Dagster).
  • Tag every dashboard with the upstream models it depends on.
  • Make ownership a column in your catalog: every model has a human accountable.

2.3 Provenance and audit trails

Provenance is lineage + the story of how data came to be: who ran what code, when, on what version of the source. Reproducibility demands provenance.

A robust provenance record includes:

  • Source system + extraction timestamp + watermark / cursor.
  • Transformation code + git SHA.
  • Run id, executor, runtime metrics.
  • Output checksum / row count / data quality test results.
  • Schema version of every input and output.

OpenLineage’s RunEvent model is the de-facto open standard for emitting these records.

MODULE 3: Integrating and Validating Data Sources

3.1 Schema mapping and integration patterns

When you join sources, you must reconcile differences in: structure (schemas), semantics (definitions), granularity (one row per?), keys (natural vs surrogate), time (event time vs ingestion time), and units (USD vs cents vs local currency).

Patterns:

  • ETL — Extract → Transform → Load. Transformation happens before landing in the warehouse. Historically necessary when target stores were slow. Pentaho, Informatica, classic SSIS.
  • ELT — Extract → Load → Transform. Land raw data in cheap, scalable warehouse; transform with SQL/dbt. The modern default since Snowflake/BigQuery/Databricks made compute cheap.
  • Streaming / CDC — Change Data Capture from operational DBs (Debezium, Fivetran, Estuary) into Kafka → warehouse or lakehouse.
  • Reverse ETL — push curated metrics back into operational tools (HubSpot, Salesforce, Braze) using Census/Hightouch.

3.2 Master Data Management (MDM) and entity resolution

A single customer can appear as Jane Doe, J. Doe, Jane D. with different emails, phone numbers, and addresses across 6 systems. Entity resolution (a.k.a. record linkage) reconciles them into a single golden record.

Fellegi–Sunter (1969) probabilistic record linkage:

For each candidate pair, compute a match weight as the sum of agreement weights across fields:

\[ w_{ij} = \log_2 \frac{m_i}{u_i} \cdot \mathbb{1}[\text{agree}] + \log_2 \frac{1-m_i}{1-u_i} \cdot \mathbb{1}[\text{disagree}] \]

where \(m_i\) is the probability of agreement on field \(i\) given a true match, and \(u_i\) given a non-match. Sum the weights, threshold to decide match vs non-match vs review.

Practical pipeline:

  1. Standardization / normalization — lowercase, strip accents, parse names, geocode addresses.
  2. Blocking — partition records into candidate blocks (e.g., by zip code or by first letter of last name) so you don’t compare \(N^2\) pairs.
  3. Comparison — compute similarity per field: Jaccard for token sets, Jaro-Winkler for short strings (names), Levenshtein edit distance, Cosine over TF-IDF for long strings, phonetic codes (Soundex, Metaphone).
  4. Scoring / classification — Fellegi–Sunter, logistic regression, or ML.
  5. Clustering — connected components / hierarchical clustering across pairwise matches.
  6. Survivorship rules — pick the surviving golden record’s attributes (most recent, most complete, source-of-truth precedence).

Open-source tools: dedupe (Python), Splink (UK ONS), Zingg, RecordLinkage.

# Tiny dedupe example
import dedupe

fields = [
    {"field": "name",   "type": "String"},
    {"field": "address","type": "String", "has missing": True},

3.3 Slowly Changing Dimensions (SCDs)

Kimball’s SCD types describe how to track changes to dimension attributes (e.g., a customer’s address).

Type Behavior Use case
Type 0 Retain original; never change Birth date, original signup channel
Type 1 Overwrite; lose history Correcting typos; no analytical interest in past values
Type 2 Add new row per change with effective dates and current flag Most common; full history preserved
Type 3 Keep limited prior values as extra columns (prev_region) When only one back-version matters
Type 4 Move historical attributes to a separate history table Large dimensions where most queries hit current state
Type 6 Hybrid 1+2+3 — overwrite current, retain history, surface previous Advanced analytics needing both views

SCD Type 2 SQL pattern:

MERGE INTO dim_customer AS tgt
USING staging_customer AS src
ON  tgt.customer_id = src.customer_id
AND tgt.is_current = TRUE
WHEN MATCHED AND (tgt.address <> src.address) THEN
  UPDATE SET tgt.is_current = FALSE, tgt.valid_to = CURRENT_DATE
WHEN NOT MATCHED THEN
  INSERT (customer_id, address, valid_from, valid_to, is_current)
  VALUES (src.customer_id, src.address, CURRENT_DATE, NULL, TRUE);

3.4 DAMA-DMBOK data-quality dimensions

The DAMA DMBOK 2 recognizes six core dimensions, sometimes expanded to 10+. Memorize at least these six:

Dimension Question Example metric
Accuracy Do values reflect reality? % addresses verified against postal service
Completeness Are required fields populated? % null for mandatory columns
Consistency Do values agree across systems / over time? mismatches between CRM and billing
Timeliness Is data available when needed? lag between event and warehouse load
Validity Do values conform to defined rules / domains? % rows passing range/regex checks

A data quality SLA for any business-critical table should specify thresholds on each dimension, the owner, the alert channel, and the response runbook.

PART 2: Statistical Methods & Root Cause Analysis

MODULE 4: Descriptive Statistics

4.1 Center, spread, shape

For a numeric variable \(x_1, \dots, x_n\):

  • Mean: \(\bar x = \frac{1}{n}\sum_i x_i\) — sensitive to outliers.
  • Median: value at the 50th percentile — robust.
  • Mode: most frequent value — useful for categorical and discrete.
  • Variance: \(s^2 = \frac{1}{n-1}\sum_i (x_i - \bar x)^2\). The \(n-1\) (Bessel’s correction) gives an unbiased estimate.
  • Standard deviation: \(s = \sqrt{s^2}\) — same units as \(x\).
  • Coefficient of variation (CV): \(\text{CV} = s / \bar x\) — unitless, compare dispersion across series.
  • Skewness: \(\frac{1}{n}\sum_i \left(\frac{x_i - \bar x}{s}\right)^3\). >0 right-tailed, <0 left-tailed.
  • Kurtosis: \(\frac{1}{n}\sum_i \left(\frac{x_i - \bar x}{s}\right)^4 - 3\) (excess). >0 heavy-tailed, <0 light-tailed.

Five-number summary: min, \(Q_1\), median, \(Q_3\), max — basis of the box plot.

4.2 Robust statistics

When data contain outliers or non-normal tails, robust statistics behave better:

  • Median Absolute Deviation (MAD): \(\text{MAD} = \text{median}(|x_i - \tilde x|)\); \(\sigma \approx 1.4826 \cdot \text{MAD}\) for Gaussian data.
  • Trimmed mean: drop the top and bottom \(\alpha\%\) before averaging.
  • Winsorized mean: cap (not drop) extreme values before averaging.
  • Hodges–Lehmann estimator: median of all pairwise averages.

5.2 Central Limit Theorem (CLT) and the standard error

For an i.i.d. sample of size \(n\) from a distribution with finite mean \(\mu\) and variance \(\sigma^2\):

\[ \frac{\bar X_n - \mu}{\sigma / \sqrt{n}} \xrightarrow{d} \mathcal{N}(0, 1) \quad \text{as } n \to \infty. \]

Practical implication: the sampling distribution of the mean is approximately normal for “large enough” \(n\) (rule of thumb \(n \ge 30\), though heavy tails or strong skew demand more), even if the underlying data are not. The standard error of the mean is \(\text{SE} = \sigma / \sqrt{n}\) (estimated as \(s / \sqrt{n}\)).

5.3 Confidence intervals

A 95% confidence interval for the mean (with known \(\sigma\), large \(n\)):

\[ \bar x \pm z_{0.975} \cdot \frac{\sigma}{\sqrt{n}}, \quad z_{0.975} \approx 1.96. \]

With unknown \(\sigma\), swap in the Student-\(t\) critical value \(t_{n-1, \,1-\alpha/2}\) and the sample SD. Interpretation: if we were to repeat the experiment many times, 95% of constructed intervals would contain the true mean. It is not “a 95% probability that \(\mu\) lies in this interval” (frequentist vs Bayesian distinction).

5.4 Hypothesis testing framework

Five-step framework:

  1. State \(H_0\) (null) and \(H_1\) (alternative).
  2. Choose significance level \(\alpha\) (commonly 0.05).
  3. Pick the test statistic appropriate to the data and design.
  4. Compute the p-value: \(P(\text{statistic at least as extreme} \mid H_0)\).
  5. Decide: reject \(H_0\) if \(p < \alpha\).

Errors:

  • Type I (false positive) — reject true \(H_0\). Probability \(\alpha\).
  • Type II (false negative) — fail to reject false \(H_0\). Probability \(\beta\).
  • Power \(= 1 - \beta\) — probability of detecting a true effect.

P-value cautions: a p-value is not the probability the null is true, not the probability the result happened by chance, and not a measure of effect size (ASA Statement on p-values, 2016). Always report effect sizes and confidence intervals alongside.

5.6 Power and sample size

For a two-sample mean comparison with significance \(\alpha\), power \(1-\beta\), common SD \(\sigma\), and minimum detectable effect \(\delta\):

\[ n \approx \frac{2 (z_{1-\alpha/2} + z_{1-\beta})^2 \sigma^2}{\delta^2}. \]

For two proportions:

\[ n \approx \frac{(z_{1-\alpha/2} + z_{1-\beta})^2 \cdot [p_1(1-p_1) + p_2(1-p_2)]}{(p_1 - p_2)^2}. \]

Power analysis is the single most important tool to prevent both wasted experiments (underpowered) and wasted traffic (overpowered).

MODULE 6: Correlation, Regression, GLMs

6.1 Correlation

  • Pearson \(r\) — linear association between two continuous variables, \(-1 \le r \le 1\). \[ r = \frac{\sum (x_i - \bar x)(y_i - \bar y)}{\sqrt{\sum (x_i - \bar x)^2 \sum (y_i - \bar y)^2}}. \]
  • Spearman \(\rho\) — Pearson on ranks; captures monotonic, not necessarily linear.
  • Kendall \(\tau\) — based on concordant vs discordant pairs; robust to ties and outliers.

Correlation ≠ causation. Anscombe’s quartet (1973) and the Datasaurus Dozen (Matejka & Fitzmaurice 2017) drive this home: very different datasets can share the same \(r\), \(\mu\), \(\sigma\).

6.2 Ordinary Least Squares regression

For \(y = X\beta + \varepsilon\), the OLS estimator minimizes \(\|y - X\beta\|_2^2\) and has closed form:

\[ \hat \beta = (X^T X)^{-1} X^T y. \]

Gauss–Markov: under linearity, exogeneity (\(E[\varepsilon | X] = 0\)), no perfect multicollinearity, homoscedasticity, and no autocorrelation, \(\hat\beta\) is BLUE (Best Linear Unbiased Estimator).

LINE assumptions (mnemonic):

  • Linearity — relationship between \(X\) and \(E[y]\) is linear in parameters.
  • Independence — residuals are independent (no autocorrelation).
  • Normality — residuals are normally distributed (matters mostly for small-sample inference).
  • Equal variance — homoscedasticity.

Diagnostics:

  • Residual vs fitted plot — checks linearity and homoscedasticity.
  • Q-Q plot — residual normality.
  • Scale–location — variance pattern.
  • Cook’s distance, leverage — influential observations.
  • Variance Inflation Factor: \(\text{VIF}_j = 1/(1 - R_j^2)\), where \(R_j^2\) is the \(R^2\) of regressing \(x_j\) on the rest. \(\text{VIF} > 5\) (some say 10) → multicollinearity concern.
  • Durbin–Watson for autocorrelation in time series residuals.

Goodness-of-fit:

  • \(R^2 = 1 - \frac{\text{SSE}}{\text{SST}}\) — share of variance explained.
  • Adjusted \(R^2 = 1 - (1 - R^2)\frac{n-1}{n-p-1}\) — penalizes added predictors.
  • \(F\)-test for overall significance; \(t\)-tests for individual coefficients.

6.3 Logistic regression and GLMs

For binary outcomes, model the log-odds linearly:

\[ \log \frac{p}{1-p} = X\beta, \quad p = \sigma(X\beta) = \frac{1}{1 + e^{-X\beta}}. \]

Coefficients are interpreted as log-odds-ratios; \(e^{\beta_j}\) is the multiplicative odds change for a unit change in \(x_j\), holding others fixed. Parameters fit via Maximum Likelihood.

Generalized Linear Models (Nelder & Wedderburn 1972) generalize the OLS framework with three components:

  1. Random component — distribution from the exponential family (Normal, Binomial, Poisson, Negative Binomial, Gamma).
  2. Systematic component — linear predictor \(\eta = X\beta\).
  3. Link function — \(g(\mu) = \eta\).

6.4 Regularization

When \(p\) is large or predictors are correlated, regularize:

  • Ridge (L2): \(\min \|y - X\beta\|^2 + \lambda \|\beta\|^2\) — shrinks coefficients, keeps all.
  • Lasso (L1): \(\min \|y - X\beta\|^2 + \lambda \|\beta\|_1\) — produces sparse solutions; performs feature selection.
  • Elastic Net: mixture; useful when groups of correlated features.

Choose \(\lambda\) by cross-validation. Always scale features before applying.

MODULE 7: Forecasting

7.1 Time-series components

A classical decomposition:

  • Additive: \(y_t = T_t + S_t + C_t + I_t\) — trend + seasonal + cycle + irregular. Use when amplitude does not grow with level.
  • Multiplicative: \(y_t = T_t \cdot S_t \cdot C_t \cdot I_t\). Use when seasonal swings scale with level. Equivalent to additive on \(\log y_t\).
  • STL (Seasonal-Trend decomposition using Loess) — robust, handles multiple seasonalities.

7.2 Stationarity and differencing

Stationarity means moments do not depend on time. Most ARIMA-family models require weak stationarity (constant mean, constant variance, autocovariance depending only on lag).

  • Augmented Dickey–Fuller (ADF): \(H_0\) = unit root (non-stationary).
  • KPSS: \(H_0\) = stationary.

Apply differencing \(\nabla y_t = y_t - y_{t-1}\) until stationary; seasonal differencing \(\nabla_s y_t = y_t - y_{t-s}\) removes seasonality.

7.3 ARIMA / SARIMA

\(ARIMA(p, d, q)\):

\[ \phi(B) (1-B)^d y_t = \theta(B) \varepsilon_t, \]

where \(B\) is the lag operator, \(\phi(B)\) the AR polynomial of order \(p\), \(\theta(B)\) the MA polynomial of order \(q\), and \(d\) the differencing order. SARIMA adds seasonal terms \((P, D, Q)_s\).

Box–Jenkins method:

  1. Plot, transform (log), difference.
  2. Inspect ACF/PACF to choose \(p, q\).
  3. Fit and check residual diagnostics (Ljung–Box).
  4. Forecast and assess on holdout.

7.4 Exponential smoothing

  • SES (Simple Exponential Smoothing): \(\hat y_{t+1} = \alpha y_t + (1-\alpha) \hat y_t\).
  • Holt’s linear: adds a trend term.
  • Holt–Winters: adds seasonality (additive or multiplicative).
  • ETS state-space — unifies them with maximum likelihood.

7.5 Modern tools

  • Facebook Prophet — decomposable model robust to missing data and outliers; great default for business forecasts.
  • statsmodels — ARIMA, ETS, SARIMAX.
  • sktime / Darts / Nixtla statsforecast — modern unified APIs.
  • Croston / TSB / SBA — intermittent demand (e.g., spare parts).
  • DeepAR, N-BEATS, Temporal Fusion Transformer, TimesFM — global deep models for many series.

MODULE 8: Root Cause Analysis (RCA)

8.1 RCA as a discipline

RCA is the structured pursuit of the underlying causes of a problem, not just its symptoms. The Toyota Production System (TPS) elevated RCA from craft to method. Three meta-principles:

  1. Symptoms vs causes — the headache is not the brain tumor.
  2. Necessary vs sufficient — distinguish causes that had to be present from those that triggered.
  3. Stop at actionable — drill until the next “why” yields a controllable change.

8.2 5 Whys

Originated at Toyota (Sakichi Toyoda). Ask “why” until you reach a root cause you can act on. Example:

  1. Why did revenue drop? — fewer checkouts completed.
  2. Why? — checkout button errors on mobile.
  3. Why? — payment API returned 500s.
  4. Why? — TLS certificate rotation broke a downstream service.
  5. Why? — renewal automation lacked monitoring.

Action: monitor cert expiry; do not “fix the button”.

Limit: linear; misses multi-cause problems.

8.3 Ishikawa (fishbone, cause-and-effect)

Categorize potential causes along branches. The classic 6 Ms for manufacturing:

Manpower, Machines, Methods, Materials, Measurement, Mother Nature.

For services, the 4 Ps: People, Process, Policies, Plant.

Build the diagram with the team — it generates hypotheses; combine with data to test them.

8.4 Pareto analysis (80/20)

Vilfredo Pareto observed 80% of land was owned by 20% of Italians. In quality, ~80% of defects come from ~20% of causes. Plot causes by frequency or impact; act on the vital few. Pair with cost-weighted analysis (impact × cost-to-fix).

8.5 Fault Tree Analysis (FTA)

Top-down deductive analysis used in safety-critical industries. The undesired top event decomposes via AND/OR gates into basic events. Quantify with reliability probabilities.

8.6 Failure Mode and Effects Analysis (FMEA)

Forward-looking, exhaustive enumeration of failure modes for each step. For each: Severity (S), Occurrence (O), Detection (D), each 1–10. Compute Risk Priority Number \(\text{RPN} = S \times O \times D\). Prioritize highest RPN for mitigation.

8.7 A3 problem-solving

One-page report (A3 paper) capturing the entire problem-solving cycle: background, current condition, goals, root-cause analysis, countermeasures, plan, follow-up. Used at Toyota to enforce both rigor and brevity.

8.8 Kepner–Tregoe and Apollo RCA

  • Kepner–Tregoe — separates Problem Analysis (what, where, when, extent; what is and is not), Decision Analysis, and Potential Problem Analysis.
  • Apollo RCA — every effect has at least two causes (action + condition); chart them recursively.

8.9 Causal diagrams and counterfactuals

Modern RCA borrows from causal inference (Pearl). A Directed Acyclic Graph (DAG) encodes assumed cause–effect relationships. Counterfactual reasoning asks: what would \(Y\) have been if \(X\) had been different, holding \(Z\) fixed? When randomized experiments are impossible, lean on:

  • Difference-in-Differences (DiD) — compare pre/post trends in treated vs control.
  • Regression Discontinuity (RDD) — exploit a threshold.
  • Instrumental Variables (IV) — find variation in \(X\) unrelated to confounders.
  • Propensity Score Matching / IPW — re-weight to balance covariates.
  • Synthetic Control — construct a weighted “synthetic” control unit from non-treated peers.

8.10 End-to-end RCA worked example: a 12% conversion drop

  1. Frame. Conversion fell from 4.5% to 4.0% week-over-week on the signup flow.
  2. Triangulate. Decompose: visitors steady, signups down → conversion really fell. Check freshness of data: yes, ETL ran on time.
  3. Segment. By device: mobile −18%, desktop flat. By geo: US only. By browser: iOS Safari.
  4. Hypothesize. Recent release shipped a new TLS lib affecting iOS 17.
  5. Test. Sentry shows spike of TLS errors on iOS 17 builds; logs confirm.
  6. Counterfactual. Roll back the lib in a canary cell; conversion returns to 4.5%.
  7. Permanent fix. Pin compatible lib; add iOS-version smoke test; alert on conversion + error correlation.
  8. Document. A3 + postmortem; share to broader team.

PART 3: Strategy → Metrics, KPI Frameworks, North Star, BSC, OKRs

MODULE 9: Translating Business Goals to Metrics

9.1 Strategy maps and MECE decomposition

Kaplan & Norton’s strategy map visualizes how learning & growth investments enable internal processes, which deliver customer outcomes, which produce financial results. Each link is a testable hypothesis. Translating strategy to measurement means turning each box into a metric.

Barbara Minto’s MECE principle (Mutually Exclusive, Collectively Exhaustive) ensures decompositions cover the space without overlap. A revenue decomposition might look like:

\[ \text{Revenue} = \text{Users} \times \text{Conversion} \times \text{ARPU} \]

then ARPU \(= \text{Sessions per User} \times \text{Conversion per Session} \times \text{AOV}\), and so on. Each leaf is owned, measured, and movable.

9.2 Jobs-to-be-done (JTBD)

Clayton Christensen’s lens: customers “hire” products to make progress on a job. JTBD reframes metrics around progress rather than features. Example: a CRM’s “job” is “close the deal by Friday”; the metric should track that, not “logins per week”.

9.3 Hypothesis-driven analytics

Frame every analysis as: “We believe X will move Y because Z, and we will know we are right when we see W.” This forces falsifiability and avoids unfocused dashboards.

MODULE 10: KPI Frameworks

10.1 SMART criteria

A KPI worth its name is:

  • Specific — unambiguous numerator and denominator.
  • Measurable — instrumented end-to-end.
  • Achievable — within reach of plausible interventions.
  • Relevant — ties to a strategic outcome.
  • Time-bound — has a target date or cadence.

10.2 Lead vs lag metrics

  • Lagging indicators measure outcomes (revenue, churn, NPS). They confirm whether strategy worked.
  • Leading indicators measure inputs and intermediate behaviors (sign-ups, activations, support contact rate). They predict outcomes and provide steering capability.

Pair them: every lagging KPI should have one or more leading KPIs that move first.

10.3 Inputs → Outputs → Outcomes → Impact

A four-tier KPI hierarchy from logic-model program evaluation:

  • Inputs — resources committed (dollars, headcount).
  • Outputs — what we produce (features shipped, emails sent).
  • Outcomes — behavior change (activation, retention).
  • Impact — long-term, often multi-causal (revenue, market share, NPS).

Vanity metrics live at the Output layer. Real impact lives further down.

10.4 North Star Metric (NSM)

Coined by Sean Ellis and popularized by Reforge/Andrew Chen. The NSM:

  1. Captures the core value customers get.
  2. Reflects product-market fit and growth simultaneously.
  3. Is leading — predictive of long-term revenue.

Examples:

Company Plausible NSM
Airbnb nights booked
Spotify time spent listening
Slack weekly active teams sending ≥ 2000 messages
Duolingo DAU
Facebook (early) 7 friends in 10 days (activation proxy)

NSM should be one number, with a small set of input metrics under it (the “constellation”).

10.5 AARRR / Pirate metrics

Dave McClure’s funnel for product analytics:

Stage Question Sample metric
Acquisition How do users find us? sessions, signups, CPA
Activation Do they get value fast? % completing onboarding, time-to-value
Retention Do they come back? D1/D7/D30 retention, churn
Referral Do they invite others? viral coefficient, NPS
Revenue Do they pay? ARPU, LTV, MRR

10.6 Google’s HEART framework

For user experience: Happiness (sat surveys), Engagement (DAU/session length), Adoption (new users), Retention (returning), Task success (completion, error rates). Pair each with Goals, Signals, Metrics (GSM).

10.7 OKRs (Doerr, Measure What Matters)

Andy Grove → John Doerr → Google. An OKR:

  • Objective — qualitative, ambitious, time-bound.
  • Key Results — 3–5 quantitative, measurable outcomes.

Cadence: quarterly; review weekly; score 0–1; aim for ~0.7 on stretch.

Pitfalls:

  • Confusing KRs with tasks (“launch onboarding redesign” — that’s an initiative, not a KR).
  • Cascading rigidly instead of aligning.
  • Tying directly to compensation — breeds sandbagging.

10.8 Balanced Scorecard (Kaplan & Norton)

Four perspectives, each with objectives, measures, targets, initiatives:

  1. Financial — revenue, margin, ROI.
  2. Customer — satisfaction, retention, share of wallet.
  3. Internal Processes — cycle time, defect rate, throughput.
  4. Learning & Growth — skills, culture, information capital.

The BSC pioneered the idea that financials alone are lagging and insufficient. Strategy maps link the four perspectives causally.

10.9 KPI / Driver trees

Decompose a top-level KPI multiplicatively/additively. Example:

Revenue
├── New customer revenue
│   ├── Traffic
│   ├── Conversion rate
│   └── First order value
└── Existing customer revenue
    ├── Active customers
    ├── Orders per active customer
    └── AOV

Tools like Causal, Mosaic, Pigment, Anaplan, and Causal Lens specialize in driver trees with scenario modeling.

10.10 Vanity vs actionable metrics

Eric Ries’s distinction. Vanity metrics (cumulative signups, page views) only go up and tell you nothing. Actionable metrics are:

  • Comparable (across cohorts, channels, time)
  • Tied to behavior change
  • Per-capita rather than total

10.11 Goodhart’s and Campbell’s law

“When a measure becomes a target, it ceases to be a good measure.” — Marilyn Strathern’s restatement of Goodhart.

“The more any quantitative social indicator is used for social decision-making, the more apt it will be to distort and corrupt the social processes it is intended to monitor.” — Donald T. Campbell.

Implications:

  • Metric gaming is the default response, not the exception.
  • Pair every “drive-up” metric with a “do-no-harm” guardrail (e.g., DAU paired with retention; revenue paired with refunds).
  • Rotate or refresh metrics when gaming behavior emerges.

PART 4: Performance Trends, Driver Analysis, Forecasting Reality

MODULE 11: Performance Trends

11.1 Period-over-period framing

  • YoY (Year-over-Year): \(\frac{V_t - V_{t-12}}{V_{t-12}}\) — controls for seasonality.
  • MoM, WoW, QoQ: sensitive to seasonality; pair with YoY.
  • Trailing-Twelve-Months (TTM) or Last-Twelve-Months (LTM): rolling 12-month sum; smooths noise.
  • Year-To-Date (YTD) vs Prior-Year-To-Date (PYTD).

11.2 CAGR — Compound Annual Growth Rate

\[ \text{CAGR} = \left( \frac{V_f}{V_0} \right)^{1/n} - 1. \]

CAGR assumes smooth compounding — it does not describe the path, only endpoints. Pair with a variance-aware view (e.g., volatility).

11.3 Moving averages and smoothing

  • SMA (Simple Moving Average) — equal weight over \(k\) periods.
  • WMA (Weighted Moving Average) — declining weights.
  • EWMA / EMA (Exponential) — \(\hat y_t = \alpha y_t + (1-\alpha) \hat y_{t-1}\); controls memory via \(\alpha\).
  • LOESS / LOWESS — local regression smoothing for exploratory plots.

11.4 Indexing and base periods

Convert raw series into indices (Base 100) to compare differently scaled metrics on one chart. Use rebased views to compare since-launch performance regardless of absolute scale.

MODULE 12: Driver Analysis

12.1 Decompositional drivers

Use identities to attribute change:

\[ \Delta \text{Revenue} \approx \frac{\partial R}{\partial U} \Delta U + \frac{\partial R}{\partial \text{ARPU}} \Delta \text{ARPU}. \]

The price–volume–mix bridge widely used in CFO analytics:

\[ \Delta R = \underbrace{(P_1 - P_0) Q_1}_{\text{price}} + \underbrace{P_0 (Q_1 - Q_0)}_{\text{volume}} + \underbrace{\text{mix effect}}_{\text{interaction}}. \]

12.2 Statistical / regression-based drivers

Fit a model to identify which inputs drive the KPI:

  • Linear regression with standardized coefficients (β weights).
  • Random Forest / Gradient Boosting feature importances (with care — correlated features distort).
  • SHAP values for both global and local driver attribution.
  • Partial Dependence Plots for shape of effect.

12.3 Causal vs correlational drivers

Driver dashboards often mislabel correlation as cause. Reduce risk by:

  • Holding everything else constant via segmentation.
  • Using lagged predictors to enforce temporal ordering.
  • Backing key drivers with experiments.

MODULE 13: Practical forecasting in a business setting

  • Always present a point forecast + interval — wider intervals signal uncertainty honestly.
  • Build at the most granular level you can forecast accurately, then aggregate; or build hierarchically with reconciliation (MinT, bottom-up, top-down).
  • Reserve a holdout window that mimics how the forecast will be used in production.
  • Track forecast bias (mean error) and accuracy separately. Persistent bias indicates a systemic miss; high variance indicates noise.
  • Document every assumption, especially levers vs exogenous drivers.

  2. Conformance — compare observed traces against a normative model; flag deviations (token replay, alignments).

MODULE 16: Lean and the Seven Wastes (TIMWOODS)

The eight wastes of Lean (the original seven plus the modern “Skills” / non-utilized talent):

  • Transport — moving things unnecessarily.
  • Inventory — work-in-progress or stock piling up.
  • Motion — people moving unnecessarily.
  • Waiting — idle time.

MODULE 18: Statistical Process Control (SPC)

Walter Shewhart (Bell Labs, 1924) introduced control charts. Donald Wheeler’s Understanding Statistical Process Control remains the practitioner bible.

Why control charts? Distinguish common-cause variation (inherent in the process) from special-cause (assignable, requiring action). Reacting to common-cause variation as if it were special — “tampering” — increases variation.

18.1 Control chart families

Chart Data type Use case
X-bar / R continuous; subgroups Manufacturing measurements
X-bar / s continuous; large subgroups as above
I-MR (Individuals, Moving Range) continuous; n=1 Service / process metrics
p-chart proportion defective call abandonment rate, error rate
np-chart count defective with fixed n as above when n fixed
c-chart count of defects per unit typos per page
u-chart defects per unit, variable area defects per kilometer of cable

18.2 Control limits

For an X-bar chart (subgroups of size \(n\)):

\[ \text{UCL} = \bar{\bar X} + A_2 \bar R, \quad \text{LCL} = \bar{\bar X} - A_2 \bar R, \]

where \(A_2\) depends on \(n\). More generally, the three-sigma rule:

\[ \text{UCL/LCL} = \mu \pm 3\sigma. \]

18.3 Western Electric / Nelson rules

Detect special-cause patterns:

  1. One point beyond 3σ.
  2. Nine points in a row on one side of the centerline.
  3. Six points trending up or down.
  4. Two of three consecutive points beyond 2σ on the same side.
  5. Four of five beyond 1σ on the same side.
  6. Fifteen points in a row within 1σ (over-control suspicion).
  7. Eight points in a row outside 1σ.

18.4 Process capability — Cp, Cpk

For a centered, normal process with spec limits LSL and USL:

\[ C_p = \frac{\text{USL} - \text{LSL}}{6\sigma}, \quad C_{pk} = \min\!\left( \frac{\text{USL} - \mu}{3\sigma}, \frac{\mu - \text{LSL}}{3\sigma} \right). \]

Rule of thumb: \(C_{pk} \ge 1.33\) is “capable”; \(\ge 1.67\) is excellent; \(\ge 2.0\) implies Six Sigma quality (3.4 DPMO).

Pp / Ppk use long-term sigma instead of within-subgroup sigma; usually \(C_{pk} \ge P_{pk}\).

MODULE 19: Theory of Constraints (ToC)

Eliyahu Goldratt, The Goal (1984). The throughput of a system is determined by its slowest step. The Five Focusing Steps:

  1. Identify the constraint.
  2. Exploit it (make sure it never starves or stalls).
  3. Subordinate everything else to the constraint.
  4. Elevate the constraint (invest to expand its capacity).
  5. Repeat — the constraint moves.

ToC introduces Throughput Accounting (Throughput = Sales − truly variable cost) and Drum-Buffer-Rope scheduling. In knowledge work, ToC manifests as WIP limits (Kanban) and bottleneck-aware sequencing.

MODULE 20: BPMN and process simulation

BPMN 2.0 (OMG standard) is the lingua franca of process modeling: pools, swim lanes, tasks, events, gateways, message flows.

Discrete-event simulation (SimPy in Python, AnyLogic, Arena) lets you model arrival distributions, service times, and resource constraints, then experiment with policy changes (more staff, smaller batch sizes) before committing to them.

MODULE 27: Validating Business Impact — Experimentation

27.1 A/B test fundamentals

The gold standard for causal claims. Randomly assign units to control and treatment, measure the difference in the target metric.

Sample size formula for two means with significance \(\alpha\), power \(1-\beta\), SD \(\sigma\), minimum detectable effect \(\delta\):

\[ n \approx \frac{(z_{1-\alpha/2} + z_{1-\beta})^2 \cdot 2 \sigma^2}{\delta^2}. \]

For two proportions \(p_1, p_2\) with mean \(\bar p\):

\[ n \approx \frac{(z_{1-\alpha/2} + z_{1-\beta})^2 \cdot [p_1(1-p_1) + p_2(1-p_2)]}{(p_1 - p_2)^2}. \]

Always plan with a minimum detectable effect (MDE) that is meaningful to the business — not just “statistically detectable”.

27.2 Common pitfalls

  • Peeking / sequential testing — stopping when significant inflates Type I rates. Use sequential testing methods (mSPRT, Bayesian alpha-spending, group sequential).
  • Sample Ratio Mismatch (SRM) — if your 50/50 split delivers 49/51, your randomization is broken; do not trust results. Use a chi-square SRM check on assignment counts.
  • Novelty / primacy effects — short tests overestimate new feature impact.
  • Network / spillover effects — users in marketplaces influence each other; consider cluster randomization.
  • Heterogeneous treatment effects — overall null may hide strong subgroup effects; pre-register cuts.
  • Multiple comparisons — testing many metrics or many cells inflates Type I.
  • p-hacking — choosing the test, segmentation, or metric post-hoc.
  • Underpowered tests — drawing strong null conclusions from a 70% power study is a Type II in waiting.
  • Drifting populations — pause during marketing pushes that change traffic mix.

27.3 Variance reduction — CUPED

Deng, Xu, Kohavi, Walker (2013). Use a covariate \(X\) (typically pre-experiment value of the metric) to reduce variance:

\[ Y^{\text{cuped}} = Y - \theta (X - \bar X), \quad \theta = \frac{\text{Cov}(Y, X)}{\text{Var}(X)}. \]

Variance reduction \(\approx \rho^2_{Y,X}\). Effective sample size can grow 30–60% without recruiting more users.

27.4 Beyond A/B — quasi-experiments

When randomization is impossible:

  • Difference-in-Differences (DiD) — compare changes in treated vs control over the same period.
  • Regression Discontinuity Design (RDD) — exploit a sharp cutoff (e.g., scholarship at GPA = 3.5).
  • Interrupted Time Series (ITS) — model a single series, look for a level/slope shift after intervention.
  • Synthetic Control (Abadie et al.) — weighted average of donors mimics treated unit’s pre-period.

27.5 Trustworthy experimentation maturity

Kohavi/Tang/Xu identify maturity levels: ad-hoc → standard process → platform → integrated decisions → cultural default. At maturity, every shipped change has a hypothesis, a guardrail set, a sample size, and a registered analysis plan.

MODULE 28: Monitoring Business Value

28.1 ROI and friends

  • ROI = \(\frac{\text{Gain} - \text{Cost}}{\text{Cost}}\). Use a defined time horizon.
  • Payback period = months until cumulative cash flow ≥ initial investment.
  • Discounted Payback — same, with discounted cash flows.
  • NPV = \(\sum_t \frac{CF_t}{(1+r)^t}\) — positive NPV → value-creating.
  • IRR — discount rate that makes NPV = 0. Beware of multiple IRRs with sign changes.
  • EVA (Economic Value Added) = NOPAT − WACC × Invested Capital.
  • Total Cost of Ownership (TCO) — includes capex, opex, training, change-management, opportunity cost.

28.2 Customer Lifetime Value (CLV / LTV)

Subscription model (simple):

\[ \text{LTV} = \frac{\text{ARPU} \times \text{Gross Margin}}{\text{Churn Rate}}. \]

Discounted version:

\[ \text{LTV} = \sum_{t=0}^{T} \frac{\text{ARPU}_t \cdot \text{Margin}_t \cdot \text{Retention}_t}{(1+r)^t}. \]

Customer Acquisition Cost (CAC) = total sales & marketing / new customers acquired in a period.

LTV/CAC rule of 3: <1 unsustainable, 1–3 weak, ≥3 healthy SaaS. Pair with CAC payback in months — under 12 is great, under 18 is normal.

28.3 Cohort retention curves

Track retention by acquisition cohort over tenure:

\[ \text{Retention}_{c, t} = \frac{\text{Active}_{c, t}}{\text{Cohort size}_c}. \]

Curves typically follow a steep early drop then flatten (the “retention smile” in healthy products). Look for:

  • The asymptote (plateau) — does the curve stabilize above 0? That fraction is your long-term loyal base.
  • Cohort-over-cohort improvement — newer cohorts should retain better as you iterate.
  • Seasonality — Q4 cohorts may behave differently.

MODULE 38: NPV / IRR / Payback in detail

38.1 NPV

\[ \text{NPV} = -C_0 + \sum_{t=1}^{T} \frac{CF_t}{(1+r)^t}. \]

  • \(C_0\) — initial investment (today).
  • \(CF_t\) — net cash flow in year \(t\).
  • \(r\) — discount rate (often WACC).

Positive NPV → invest; comparing mutually exclusive projects, pick the highest NPV.

38.2 IRR

IRR is the \(r\) that makes \(\text{NPV}(r) = 0\). Pitfalls:

  • Multiple IRRs when sign of cash flows changes more than once.
  • Reinvestment assumption (cash flows reinvested at IRR) is often unrealistic — prefer MIRR.

38.3 Payback and discounted payback

  • Payback = years until cumulative \(CF\) ≥ \(C_0\). Ignores time value and post-payback flows.
  • Discounted payback — same with discounted flows.
  • Useful for liquidity-constrained executives, but never the sole decision criterion.

38.4 Sensitivity and scenario analysis

  • One-way sensitivity (tornado) — vary each input ±X% while holding others fixed; rank by NPV impact.
  • Scenario analysis — combine inputs into best / base / worst cases.
  • Monte Carlo — sample inputs from distributions; report NPV distribution and probability of negative NPV.
import numpy as np
n = 100_000
growth = np.random.normal(0.06, 0.02, n)
margin = np.random.normal(0.30, 0.05, n)
spend  = 1_000_000
revenue = spend * (1 + growth)
npv = (revenue * margin - spend) / 1.10
print(np.mean(npv), np.percentile(npv, [5, 50, 95]))

  SELECT customer_id, MIN(DATE_TRUNC('month', order_dt)) AS cohort
  FROM fct_orders
  GROUP BY 1
),
activity AS (
  SELECT o.customer_id, DATE_TRUNC('month', o.order_dt) AS m, f.cohort
  FROM fct_orders o
  JOIN first_order f USING (customer_id)
)
SELECT
  cohort,
  DATEDIFF('month', cohort, m)                 AS age,
  COUNT(DISTINCT customer_id) * 1.0
    / FIRST_VALUE(COUNT(DISTINCT customer_id)) OVER (PARTITION BY cohort ORDER BY DATEDIFF('month', cohort, m))
                                               AS retention
FROM activity
GROUP BY 1, 2;

A.2 Python — A/B test analysis

import numpy as np
from scipy import stats

control = np.random.binomial(1, 0.10, size=20000)
treat   = np.random.binomial(1, 0.105, size=20000)

p1, p2 = control.mean(), treat.mean()
n1, n2 = len(control), len(treat)
p_pool = (control.sum() + treat.sum()) / (n1 + n2)
se = np.sqrt(p_pool * (1 - p_pool) * (1/n1 + 1/n2))
z  = (p2 - p1) / se
pval = 2 * (1 - stats.norm.cdf(abs(z)))
ci   = (p2 - p1) + np.array([-1, 1]) * 1.96 * np.sqrt(p1*(1-p1)/n1 + p2*(1-p2)/n2)

print(f"lift={p2-p1:.4f}, z={z:.2f}, p={pval:.4f}, 95% CI={ci}")

A.4 Python — CLV (subscription)

def ltv(arpu, gross_margin, monthly_churn, months=36, discount_monthly=0.01):
    cf = 0.0
    survive = 1.0
    for t in range(1, months + 1):
        survive *= (1 - monthly_churn)
        cf += (arpu * gross_margin * survive) / (1 + discount_monthly) ** t
    return cf

A.7 Python — Process capability

def capability(x, usl, lsl):
    mu, sigma = x.mean(), x.std(ddof=1)
    Cp = (usl - lsl) / (6 * sigma)
    Cpk = min((usl - mu) / (3 * sigma), (mu - lsl) / (3 * sigma))
    return Cp, Cpk

APPENDIX C: Pitfalls and Common Misconceptions

A compressed checklist of mistakes the assessment loves to surface.

  1. Treating ordinal scales as interval — averaging Likert scores without acknowledging the assumption.
  2. Reporting a mean for skewed distributions — use median + percentile range instead.
  3. Confusing correlation with causation — Anscombe, Datasaurus, spurious correlations site (Tyler Vigen).
  4. Simpson’s paradox — aggregate trend opposes within-group trend; always check segments.
  5. Survivorship bias — analyzing only the survivors (e.g., only customers who didn’t churn).
  6. Selection bias — non-random sample colors estimates.
  7. Look-ahead bias — using future information at training time.
  8. Goodhart / Campbell — metric gaming.
  9. p-hacking — choosing tests / cuts to find significance.
  10. Multiple comparisons without adjustment.
  11. Peeking at A/B tests without sequential corrections.
  12. Sample Ratio Mismatch — failure to validate randomization.
  13. Confusing correlation magnitude with effect size.
  14. Average treatment effect masking heterogeneous effects.
  15. Vanity metrics — cumulative counts, totals not normalized.
  16. Confusing leading and lagging indicators.
  17. Tampering with stable processes — reacting to common-cause variation as if special.
  18. Ignoring base rates — base rate fallacy in conditional probability.
  19. Overfitting forecasts to noise.
  20. Using \(R^2\) as the sole measure of model quality.
  21. Optimizing one KPI without guardrails.
  22. Confusing North Star Metric with vanity metric.
  23. Building dashboards before clarifying the decision.
  24. 3D charts, dual axes, pie charts with many slices, color rainbows for ordered data.
  25. Reporting precision beyond the data warrants (e.g., 0.7234521% conversion).
  26. Ignoring confidence intervals.
  27. Confusing CapEx and OpEx in business cases.
  28. Computing payback without considering opportunity cost.
  29. Comparing IRRs across projects of different sizes / durations.
  30. Using one discount rate for projects of different risk.

APPENDIX E: One-Page Decision Cheatsheet

Use this when shipping any analytics deliverable.

  1. Decision — what business decision will move because of this?
  2. Audience — who decides, who acts, who needs to know?
  3. Metric definition — numerator/denominator, exclusions, edge cases.
  4. Source of truth — lineage from raw → metric.
  5. Data quality — completeness, freshness, accuracy thresholds.
  6. Granularity — per what, per how often?
  7. Comparison — vs target, vs prior period, vs cohort, vs peer.
  8. Uncertainty — confidence interval, prediction interval, error bars.
  9. Action — if the number moves X, we do Y.
  10. Guardrails — what must NOT degrade for this to count as a win.
  11. Ownership — name on the dashboard; named escalation path.
  12. Sunset clause — when do we re-validate or retire this asset?
