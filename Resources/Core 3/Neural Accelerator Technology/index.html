Compression, Compilation, Search & Edge Deployment of Neural Networks

PART 1: QUANTIZATION — TURNING FP32 NETWORKS INTO INT8 (AND LOWER)

MODULE 1: NUMBER FORMATS, WHY WE QUANTIZE, AND THE ARITHMETIC OF LOW PRECISION

1.1 Why quantize at all?

Deep models are trained in 32-bit floating point (FP32). Each weight occupies 4 bytes, each activation 4 bytes, every multiply-accumulate (MAC) is an FP32 operation. For a 7-billion-parameter LLM that means 28 GB just for weights, before you even consider activations, KV-cache, optimizer state, gradient buffers, or batch dimension. On an edge SoC with 4 GB of LPDDR4 and a 50 mW power envelope this is hopeless.

Quantization replaces FP32 numbers by integers (typically INT8, INT4 or even INT2) or by lower-precision floating point (FP16, BF16, FP8, FP4). The objective function of quantization is roughly:

\[ \min_{Q} \; \mathbb{E}_{x \sim \mathcal{D}}\big[\, \mathcal{L}(f_{Q}(x), y) - \mathcal{L}(f_{FP32}(x), y) \,\big] \quad \text{subject to memory, latency, and energy budgets.} \]

Quantization touches four simultaneous benefits, and you should be able to enumerate all of them on demand:

  1. Memory footprint — INT8 is 4× smaller than FP32; INT4 is 8× smaller. This matters both for storage (on-disk model size) and for runtime DRAM/SRAM occupancy.
  2. Memory bandwidth — modern inference is memory-bound. If you can stream half as many bytes from HBM/DRAM to compute, you double effective throughput. Roofline analysis: more arithmetic intensity per byte loaded.
  3. Compute throughput — INT8 tensor cores on NVIDIA Ampere give 4× the TOPS of FP16; Hopper FP8 Tensor Cores give another 2×. Edge accelerators (Edge TPU, Hexagon HVX, Apple Neural Engine) are typically integer-only — FP simply does not execute.
  4. Energy per inference — energy is dominated by data movement. A 32-bit DRAM access costs roughly 640 pJ while a 32-bit FP MAC costs ~3.7 pJ and an 8-bit INT MAC ~0.2 pJ (Horowitz, ISSCC 2014). Quantization is the single biggest lever on battery life.

Reference: Jay Alammar’s “A Visual Guide to Quantization” and Lilian Weng’s “Large Transformer Model Inference Optimization” lay out these motivations beautifully — read them.

Important nuances:

  • BF16 vs FP16 — BF16 keeps the FP32 exponent (8 bits), so it can represent the same dynamic range; it just loses mantissa precision. This makes BF16 vastly easier for training (no loss scaling needed) at the cost of slightly noisier gradients. FP16 is more precise for representable magnitudes but underflows easily — that’s why mixed-precision training uses a loss scaler.
  • TF32 is a NVIDIA quirk: the input operands are rounded to 10 mantissa bits (like FP16) but keep the FP32 exponent; accumulation happens in FP32. It is bit-compatible with FP32 from the caller’s perspective: torch.matmul on Ampere uses TF32 by default.
  • FP8 comes in two flavors: E4M3 (4-bit exponent, 3-bit mantissa, biased to 7) for forward activations / weights — high precision, narrow range; E5M2 (5-bit exponent, 2-bit mantissa) for backward gradients — wider range needed for gradient tails. Hopper’s Transformer Engine dynamically switches between the two.
  • NF4 (NormalFloat 4) is an information-theoretically optimal 4-bit data type for weights that are zero-mean, unit-variance Gaussian (which neural-net weights approximately are). Each of the 16 codewords is the quantile of the standard normal distribution. Used in QLoRA.

1.3 The linear-affine quantization formula

For uniform affine (asymmetric) quantization of a real tensor \(r\) into an \(b\)-bit integer tensor \(q\):

\[ q = \text{clip}\big(\text{round}(r / s) + z, \; q_{\min}, q_{\max}\big) \]

\[ \hat{r} = s \cdot (q - z) \]

where: - \(s\) is the scale, a positive real number (FP32); - \(z\) is the zero-point, an integer in the same range as \(q\), so that the real value 0 maps exactly to integer \(z\) (important: zero-padding in conv layers must round-trip exactly, else the network shifts); - \(q_{\min}, q_{\max}\) are the integer bounds — for INT8 unsigned: 0..255; for INT8 signed: -128..127.

Given the floating-point range \([\alpha, \beta]\) that the tensor will inhabit (estimated by calibration), the scale and zero-point are derived as:

\[ s = \frac{\beta - \alpha}{q_{\max} - q_{\min}}, \quad z = \text{round}\Big(q_{\min} - \frac{\alpha}{s}\Big). \]

For symmetric quantization we set \(z = 0\) and use a signed integer range that mirrors zero:

\[ s = \frac{\max(|\alpha|, |\beta|)}{2^{b-1} - 1}, \quad \hat{r} = s \cdot q. \]

Symmetric quantization is faster (no zero-point subtraction inside the inner loop, no overflow handling) and is the default for weights which tend to be roughly zero-centered. Asymmetric is used for activations after ReLU because they live entirely in \([0, +\infty)\) and you don’t want to throw away half of the integer codes.

1.4 Per-tensor vs per-channel vs per-group

The granularity of \((s, z)\) controls how much information you keep:

Granularity Scope of one \((s,z)\) pair Storage overhead Accuracy When to use
Per-tensor Whole tensor Negligible Worst Activations on simple models
Per-row / per-token One scale per row (LLM activations) Small Better for LLMs SmoothQuant, LLM.int8() outlier rows
Per-channel (per-axis) One scale per output channel of a Conv/Linear One vector Much better for conv Default for weights in TFLite/TensorRT
Per-group One scale per contiguous group of \(g\) elements (e.g. 128) Bigger Best for INT4 GPTQ, AWQ default group=128
Per-element A scale for each value = original FP Pointless —

Why per-channel for conv weights? A convolution’s output channels are computed independently; if one channel’s weights are tiny and another’s are huge, a single per-tensor scale forces all weights to use the bigger range, wasting bits on the small-weight channel. Per-channel quantization restores precision proportional to each channel’s actual range.

Why per-group for INT4 LLMs? At 4 bits, even per-channel scales leak too much accuracy. A per-group scale (typically every 128 weights along the input dim) packs much higher fidelity with only ~0.5 bit/weight of overhead.

1.5 Symmetric INT8 matmul — the integer arithmetic walkthrough

This is the canonical question for the assessment. Given symmetric INT8 weights \(W_q\) with per-channel scale vector \(s_w\) and asymmetric INT8 input activations \(X_q\) with per-tensor scale \(s_x\) and zero-point \(z_x\), compute INT32 output \(Y\) then dequantize:

\[ Y_{ij} = \sum_k (X_{ik} - z_x)(W_{kj}) \cdot s_x \cdot s_{w,j} = s_x s_{w,j} \Big[\sum_k X_{ik} W_{kj} - z_x \sum_k W_{kj}\Big]. \]

Note the second term \(z_x \sum_k W_{kj}\) is a bias-like constant that can be folded into the layer’s bias at compile time. Then the inner loop is pure INT8 × INT8 → INT32 accumulation, no zero-point arithmetic at all. This is exactly how TensorRT / ONNX Runtime / TFLite implement quantized matmul, and exactly what NVIDIA’s IMMA / Intel’s VNNI / Arm’s SDOT instructions accelerate.

// Conceptual inner loop after offset-folding
int32_t acc = 0;
for (int k = 0; k < K; ++k)
    acc += static_cast<int32_t>(X_q[i*K+k]) * static_cast<int32_t>(W_q[k*N+j]);
// acc is INT32; rescale to INT8 output range:
float real = (acc - folded_bias) * s_x * s_w[j];
Y_q[i*N+j] = saturating_cast_int8(round(real / s_y) + z_y);

The requantize step (INT32 accumulator → INT8 output for the next layer) is usually implemented as multiply-by-fixed-point-scale + right-shift to avoid a runtime float divide.

MODULE 2: POST-TRAINING QUANTIZATION (PTQ)

PTQ converts an FP32 trained model directly into a quantized model without re-training. All you need is the FP32 weights and a tiny calibration set (50–500 unlabeled samples).

2.1 Dynamic vs Static PTQ

Aspect Dynamic PTQ Static PTQ
Weights Quantized offline (INT8) Quantized offline (INT8)
Activations Quantized on the fly at runtime Quantized offline; scales baked into the graph
Calibration set required No Yes (a few hundred unlabeled samples)
Speed Limited by per-call quantization overhead Maximum (pure integer pipeline)
Accuracy Usually higher (per-batch range) Slightly lower but acceptable
Typical use RNNs / Transformers where activations vary wildly across inputs CNN inference, edge deployment

PyTorch APIs: torch.ao.quantization.quantize_dynamic(model, {nn.Linear}, dtype=torch.qint8) for dynamic; torch.ao.quantization.prepare(model) + calibration + convert(model) for static. ONNX Runtime offers quantize_dynamic and quantize_static. TensorRT only supports static (it freezes scales into the engine).

2.2 Weight-only vs activation-weight quantization

For huge LLMs, the bottleneck is memory bandwidth from HBM to SM for the weight matrix. If you quantize weights only to INT4/INT8 but compute in FP16 (i.e. dequantize on the fly inside the kernel), you get the full bandwidth saving without touching activation distributions at all. This is what GPTQ and AWQ produce (W4A16). For edge or CNN models, you usually want W8A8 so the integer tensor cores fire on both operands.

2.3 Calibration techniques

Calibration is the act of estimating \([\alpha, \beta]\) for each activation tensor by feeding a small set of representative inputs through the FP32 model and collecting statistics.

Min-Max: \[ \alpha = \min_{x \in \text{calib}} x, \quad \beta = \max_{x \in \text{calib}} x. \] Simple, but a single outlier sample can ruin the range. Used as a baseline.

Percentile (e.g. 99.9%): \[ \alpha = \text{quantile}(x, 0.001), \quad \beta = \text{quantile}(x, 0.999). \] Saturates the 0.1% extreme tails — usually a big win.

MSE (Mean-Squared Error) — pick \((\alpha, \beta)\) to minimise: \[ (\alpha^*, \beta^*) = \arg\min_{\alpha, \beta} \mathbb{E}_x [(x - Q(x; \alpha, \beta))^2]. \] Solved by grid-search over symmetric thresholds; gives the best L2 reconstruction.

KL-divergence (a.k.a. entropy calibration, TensorRT default) — choose the threshold \(\beta\) so that the distribution of clipped/quantized FP values is closest in KL divergence to the original FP distribution. Algorithm (from Migacz, GTC 2017):

  1. Build a 2048-bin histogram of the absolute activations on calibration data.
  2. For each candidate threshold \(T\) (corresponding to bin index \(i\) from 128 to 2048):
    1. Truncate the histogram at bin \(i\), lumping everything beyond into the last bin → \(P\).
    2. Quantize the first \(i\) bins down to 128 bins → \(Q\).
    3. Expand \(Q\) back to \(i\) bins → \(Q'\).
    4. Compute \(KL(P \| Q')\).
  3. Pick the \(T\) with minimum KL divergence.

This is the gold-standard for activations and ships in TensorRT, OpenVINO and ONNX Runtime.

2.4 AdaRound — learning the rounding direction

Reference: Nagel et al. 2020, “Up or Down? Adaptive Rounding for Post-Training Quantization”.

Naive “round-to-nearest” (RTN) minimises per-weight error but not layer output error. AdaRound formulates rounding as a per-weight binary decision: round each weight up or down, with a learned soft mask \(h(V)\):

\[ \tilde{W} = s \cdot \big(\lfloor W/s \rfloor + h(V)\big), \quad h(V) = \text{clip}(\sigma(V)(\zeta - \gamma) + \gamma, 0, 1) \]

with \(\zeta = 1.1, \gamma = -0.1\) so \(h(V) \in \{0,1\}\) at convergence (this is the rectified sigmoid from Louizos et al.). Optimize \(V\) to minimise the layer-wise reconstruction loss:

\[ \mathcal{L}_{\text{AdaRound}} = \| W X - \tilde{W} X \|_F^2 + \lambda \sum_{i,j} (1 - |2 h(V_{ij}) - 1|^\beta). \]

The second term is an “annealing regulariser” that forces \(h(V)\) toward 0 or 1 as training progresses. AdaRound recovers many percentage points of accuracy on ResNet-50 at INT4 vs. RTN, with no labels and tiny compute.

2.5 BRECQ — block reconstruction

Reference: Li et al. 2021, “BRECQ: Pushing the Limit of Post-Training Quantization by Block Reconstruction”.

AdaRound reconstructs layer-by-layer, but errors accumulate across layers. BRECQ reconstructs a residual block at a time (the granularity of “block” is chosen between layer / stage / block by analyzing the Hessian’s diagonal block structure). The reconstruction loss is the Fisher-Information-weighted output error:

\[ \mathcal{L}_{\text{BRECQ}} = (Y - \tilde{Y})^\top \, \hat{F} \, (Y - \tilde{Y}) \]

where \(\hat{F}\) is the empirical Fisher (= per-sample squared gradients). BRECQ pushes INT4 inference to near-FP accuracy on classical CNNs.

2.6 OBQ → GPTQ — sequential weight quantization with Hessian information

Reference: Frantar & Alistarh 2022 “Optimal Brain Quantization”; Frantar et al. 2023 “GPTQ: Accurate Post-Training Quantization for Generative Pre-trained Transformers”.

OBQ is the modern reformulation of the classical Optimal Brain Surgeon (OBS) idea. Given a layer with input \(X\) and FP32 weight \(W\), quantizing weight \(w_i\) introduces error in \(XW\) which can be partially compensated by updating the remaining weights along the Hessian \(H = 2 X X^\top\).

The closed-form update when quantizing weight index \(q\) is:

\[ \delta_q = -\frac{w_q - \text{quant}(w_q)}{[H^{-1}]_{qq}}, \quad \Delta W = \delta_q \, [H^{-1}]_{:,q}. \]

OBQ greedily picks the next \(q\) to quantize as the one with minimum induced error. This is \(O(d_{col}^3)\) per row — feasible for small models, not for 175B-parameter GPT-3.

GPTQ scales OBQ to billion-parameter models by:

  1. Fixing the quantization order to be column-by-column left-to-right (no greedy search → no per-row \(O(d_{col}^3)\)).
  2. Using Cholesky decomposition of \(H\) for numerically stable updates.
  3. Quantizing in blocks of 128 columns at a time — within a block, do dense updates; across blocks, do lazy batched updates. This makes GPTQ embarrassingly cache-friendly.
  4. Quantizing a 175B model on a single A100 in a few hours, with near-FP16 perplexity at INT4.

GPTQ is the canonical W4A16 method for LLMs as of 2023–2024.

2.7 SmoothQuant — taming activation outliers in LLMs

Reference: Xiao et al. 2022, “SmoothQuant: Accurate and Efficient Post-Training Quantization for Large Language Models”.

In big transformers, activations are wildly outlier-prone (a handful of channels carry magnitudes 100× the median; see Dettmers’ “LLM.int8()” paper). Weights are well-behaved. SmoothQuant’s insight: migrate the difficulty from activations to weights via a per-channel diagonal scale \(s\):

\[ Y = (X \, \text{diag}(s)^{-1}) \, (\text{diag}(s) \, W). \]

Mathematically equivalent, but now \(\hat{X} = X s^{-1}\) has flatter per-channel statistics (easy to quantize) while \(\hat{W} = sW\) absorbs the outliers (still easy because weights are smooth). Choose \(s_j = \max_i |X_{ij}|^\alpha / \max_i |W_{ji}|^{1-\alpha}\) with \(\alpha = 0.5\) as a balance knob. Result: W8A8 transformers without accuracy loss.

2.8 AWQ — activation-aware weight quantization

Reference: Lin et al. 2023, “AWQ: Activation-aware Weight Quantization for LLM Compression and Acceleration”.

Observation: not all weights are equally important. About 1% of weights handle most of the activation magnitude (the “salient” weights). If we scale up those salient weights before quantization, the quantization error on them is proportionally smaller — and we can later scale the activation down by the same factor at runtime (or fuse into LayerNorm). The optimal scale per channel is chosen by a tiny grid search to minimise the output reconstruction error:

\[ s^* = \arg\min_s \big\| W X - Q(W \cdot s) \cdot (X / s) \big\|^2. \]

AWQ does not require backprop or gradient computation, and ships in llama.cpp, MLC-LLM, and vllm.

import torch
from torch.ao.quantization import (
    get_default_qconfig, QConfigMapping, prepare_fx, convert_fx

)

model_fp32 = MyResNet().eval()

example_inputs = (torch.randn(1, 3, 224, 224),)
prepared = prepare_fx(model_fp32, qmap, example_inputs)

# Calibration: feed ~256 unlabeled images
with torch.no_grad():
    for x in calib_loader:
        prepared(x)

quantized = convert_fx(prepared)
torch.jit.save(torch.jit.script(quantized), "resnet_int8.pt")

MODULE 3: QUANTIZATION-AWARE TRAINING (QAT)

When PTQ leaves more than ~1% accuracy on the table — typically at INT4 weights, INT8 activations on small or under-trained models, or on detection / segmentation tasks — you train with quantization in the loop.

3.1 Fake quantization and the Straight-Through Estimator (STE)

Reference: Jacob et al. 2018, “Quantization and Training of Neural Networks for Efficient Integer-Arithmetic-Only Inference”.

You insert a fake-quant node in the forward pass:

\[ \text{fq}(x; s, z) = s \cdot \big(\text{clip}(\text{round}(x/s) + z, q_{\min}, q_{\max}) - z\big). \]

The output is still FP32 (matching the original computation graph) but it has been rounded to one of \(2^b\) representable values. This lets the rest of the layers see the “quantized” tensor during forward, so the model learns weights that are robust to that rounding.

The problem: \(\text{round}(\cdot)\) has gradient zero almost everywhere. Solution: the Straight-Through Estimator (STE) — pretend the gradient through round is the identity, with clipping outside the range:

\[ \frac{\partial \text{fq}(x)}{\partial x} = \mathbf{1}_{q_{\min} \le \text{round}(x/s) + z \le q_{\max}}. \]

In PyTorch:

class FakeQuantSTE(torch.autograd.Function):
    @staticmethod
    def forward(ctx, x, s, qmin, qmax):
        ctx.save_for_backward((x/s).round().clamp(qmin, qmax))
        return (x/s).round().clamp(qmin, qmax).mul(s)

    @staticmethod
    def backward(ctx, g):
        (q,) = ctx.saved_tensors
        mask = (q > 0) & (q < 1)        # placeholder; real impl uses qmin/qmax
        return g * mask, None, None, None

3.2 Learnable scale: PACT and LSQ

PACT (Choi et al. 2018, “PArameterized Clipping acTivation”): make the clipping threshold \(\alpha\) for activations a learnable parameter, regularised toward small values:

\[ y = \text{PACT}(x; \alpha) = 0.5(|x| - |x - \alpha| + \alpha), \quad \mathcal{L} \leftarrow \mathcal{L} + \lambda \alpha^2. \]

The gradient \(\partial y / \partial \alpha\) is well-defined (it’s 1 when \(x > \alpha\), else 0), so \(\alpha\) trains alongside the weights. PACT was foundational for sub-4-bit activation QAT.

LSQ (Esser et al. 2020, “Learned Step Size Quantization”): make the scale \(s\) itself a learnable parameter, with a specially designed gradient that scales with \(s\):

\[ \frac{\partial \hat{x}}{\partial s} = \begin{cases} -x/s + \text{round}(x/s) & q_{\min} < x/s < q_{\max} \\ q_{\min} & x/s \le q_{\min} \\ q_{\max} & x/s \ge q_{\max} \end{cases} \]

with a learning-rate multiplier \(1/\sqrt{N q_{\max}}\) to balance the magnitudes of weight and scale updates. LSQ is the de-facto QAT primitive in most modern frameworks.

3.3 BatchNorm folding

For inference, you must fold BatchNorm into the preceding Conv:

\[ y = \gamma \cdot \frac{Wx + b - \mu}{\sqrt{\sigma^2 + \epsilon}} + \beta = W' x + b', \quad W' = \frac{\gamma W}{\sqrt{\sigma^2 + \epsilon}}, \quad b' = \beta - \frac{\gamma(\mu - b)}{\sqrt{\sigma^2 + \epsilon}}. \]

During QAT this is tricky because BN running stats are updated by gradient steps. The trick (Krishnamoorthi 2018, “Quantizing deep convolutional networks for efficient inference”) is to simulate folding during training: compute \(W'\) on the fly using current BN stats, fake-quant \(W'\), then forward through it. The model converges to weights that are robust to folded-and-quantized BN at inference time.

3.4 QAT recipe

  1. Start from a converged FP32 checkpoint.
  2. Insert fake-quant observers (per-channel symmetric weights, per-tensor asymmetric activations).
  3. Train for ~5–10% of the original training budget with a learning rate ~10×–100× smaller than the original.
  4. Optionally freeze BN running stats after a few epochs (so they don’t drift with quantization noise).
  5. Convert to a real integer model.
import torch
from torch.ao.quantization import (
    get_default_qat_qconfig, QConfigMapping, prepare_qat_fx, convert_fx,
)
model = MyResNet().train()
qmap = QConfigMapping().set_global(get_default_qat_qconfig("x86"))
model_prepared = prepare_qat_fx(model, qmap, (torch.randn(1,3,224,224),))
optimizer = torch.optim.SGD(model_prepared.parameters(), lr=1e-4, momentum=0.9)
for epoch in range(5):
    for x, y in train_loader:
        loss = criterion(model_prepared(x), y); loss.backward()
        optimizer.step(); optimizer.zero_grad()
model_prepared.eval()
model_int8 = convert_fx(model_prepared)

MODULE 4: MIXED-PRECISION QUANTIZATION (MPQ)

Not all layers should run at the same precision. The first/last layers of a ResNet, and the QKV projections of an LLM, are far more sensitive to quantization than the middle of the network.

4.1 HAQ — RL-based per-layer bit allocation

Reference: Wang et al. 2019, “HAQ: Hardware-Aware Automated Quantization with Mixed Precision”.

HAQ trains a DDPG reinforcement-learning agent to assign a precision (2/4/6/8 bits) to each layer, with a reward that combines:

\[ R = -\mathcal{L}_{\text{val}}(\text{model after PTQ}) \quad \text{subject to} \quad \text{latency}(\pi) \le L_{\text{target}}. \]

State features per layer: input/output dim, kernel size, stride, n_params, FLOPs. The agent learns that early conv layers want 8 bits while pointwise mid-network conv layers can tolerate 4. Latency is measured on the real target hardware (e.g. Edge TPU), not estimated.

4.2 HAWQ-v2 — Hessian-based sensitivity

Reference: Dong et al. 2019, 2020, “HAWQ” series.

The sensitivity of a layer to quantization is captured by the trace of its weight Hessian:

\[ \Omega_i = \text{Tr}(H_i) \cdot \| W_i - Q(W_i) \|^2 \approx \text{expected loss increase from quantizing layer } i. \]

The Hessian trace is estimated cheaply via Hutchinson’s stochastic estimator:

\[ \text{Tr}(H) \approx \frac{1}{m} \sum_{j=1}^m v_j^\top H v_j, \quad v_j \sim \mathcal{N}(0, I), \]

and Hessian-vector products use the standard double-backward trick: \(Hv = \nabla(\nabla L \cdot v)\). Layers with large \(\Omega_i\) are kept at higher precision; small-\(\Omega_i\) layers go to INT4 or INT2.

4.3 ZeroQ — data-free MPQ

Reference: Cai et al. 2020, “ZeroQ: A Novel Zero Shot Quantization Framework”.

When you have no calibration data (privacy or shipping constraints), synthesize a “distilled” dataset by inverting the BatchNorm statistics of the FP32 model: optimize random Gaussian images \(\hat{x}\) so that the per-layer mean/variance of \(\hat{x}\) match the stored running BN stats:

\[ \hat{x}^* = \arg\min_{\hat{x}} \sum_{l} \big( \|\mu_l(\hat{x}) - \mu_l^{\text{BN}}\|^2 + \|\sigma_l^2(\hat{x}) - (\sigma_l^{\text{BN}})^2\|^2 \big). \]

Use the synthetic data to run any calibration method and Hessian sensitivity analysis.

4.4 Mixed-precision in modern toolchains

  • TensorRT auto-selects FP32 / FP16 / INT8 per layer via its builder’s “precision constraints” — driven by per-layer profiling. You can also explicitly set layer.set_precision(trt.int8).
  • NVIDIA Transformer Engine in Hopper does dynamic FP8 scaling per tensor per step.
  • ONNX Runtime supports per-op fallback to higher precision when an op doesn’t have an INT8 kernel.

PART 2: PRUNING — MAKING NETWORKS SPARSE

MODULE 5: WHY AND WHAT TO PRUNE

The classic justification: most weights in a trained neural network contribute very little. Han et al. 2015 (“Learning both Weights and Connections for Efficient Neural Networks”) showed AlexNet retained accuracy after removing 90% of weights — if the remaining 10% were re-fine-tuned.

Pruning interacts with hardware: unstructured sparsity reduces parameter count but typically does not speed up dense matrix multiplication on GPUs/CPUs (the matrix shape doesn’t change). Structured sparsity (whole filters/heads/blocks) directly removes computation. Semi-structured N:M sparsity (NVIDIA Ampere 2:4) lives in between.

5.1 Granularity ladder

Granularity Removed unit Hardware speedup Accuracy cost
Element-wise (unstructured) Individual weight Only with sparse kernels (CSR/CSC) Lowest
2:4 / N:M 2 out of every 4 contiguous weights 2× on Ampere sparse tensor cores Low
Vector / block A row block of weights Yes (with block-sparse matmul) Medium
Filter / channel An entire output channel of a Conv Yes (just shrink dims) Higher
Attention head A QKV head in a transformer Yes (head_dim×3 fewer ops) Variable
Layer An entire layer Yes (just drop it) Highest

MODULE 6: UNSTRUCTURED MAGNITUDE PRUNING

6.1 Han et al.’s “Deep Compression” pipeline

Reference: Han, Mao & Dally 2016, “Deep Compression: Compressing Deep Neural Networks with Pruning, Trained Quantization and Huffman Coding”. Best Paper at ICLR 2016.

Three-stage recipe:

  1. Prune — set all weights below a magnitude threshold \(\tau\) to zero.
  2. Quantize — cluster the remaining weights with k-means into \(2^b\) centroids; replace each weight by its centroid index (weight sharing).
  3. Huffman-code the indices.

Result on AlexNet: 35× compression with no accuracy loss; VGG-16: 49×. This paper is the historical wellspring of all modern compression work.

6.2 One-shot vs iterative magnitude pruning (IMP)

One-shot: prune to target sparsity \(s\) in a single step, then fine-tune. Simple, fast, but fragile at high sparsity.

Iterative magnitude pruning (IMP): prune a small fraction (e.g. 20%), fine-tune for a few epochs, repeat. Crucial for sparsities above ~80%.

def iterative_magnitude_pruning(model, dataloader, sparsities=[0.5, 0.75, 0.9, 0.95]):
    for s in sparsities:
        prune_unstructured(model, sparsity=s)   # set the lowest |w| to zero
        fine_tune(model, dataloader, epochs=2)  # let the survivors compensate
    return model

6.3 Gradual magnitude pruning (GMP)

Reference: Zhu & Gupta 2018, “To Prune, or Not to Prune”.

Instead of pruning-then-retraining as a separate phase, gradually grow the sparsity schedule during training itself:

\[ s_t = s_f + (s_i - s_f)\left(1 - \frac{t - t_0}{n \Delta t}\right)^3, \quad t \in \{t_0, t_0 + \Delta t, \dots, t_0 + n \Delta t\}. \]

The cubic schedule lets the model adapt to early aggressive pruning while easing into the final sparsity. GMP is what most production pipelines (TensorFlow Model Optimization Toolkit) use.

6.4 The Lottery Ticket Hypothesis

Reference: Frankle & Carbin 2019, “The Lottery Ticket Hypothesis: Finding Sparse, Trainable Neural Networks”. ICLR 2019 Best Paper.

Claim: a randomly-initialized dense network contains winning tickets — sparse subnetworks that, when trained in isolation from the same original init, match the dense network’s accuracy. Algorithm:

  1. Initialize weights \(W_0\).
  2. Train the network to convergence.
  3. Prune the bottom-\(p\%\) magnitude weights → mask \(m\).
  4. Rewind the surviving weights to \(W_0\) (not random re-init).
  5. Re-train the sparse network \(m \odot W_0\).

Frankle’s “Linear Mode Connectivity and the Lottery Ticket Hypothesis” (2020) refined this to late-rewind: rewind to weights at step \(k\) (a few epochs into training) rather than step 0, which works for larger nets like ResNet-50 on ImageNet.

6.5 Movement pruning

Reference: Sanh, Wolf & Rush 2020, “Movement Pruning: Adaptive Sparsity by Fine-Tuning”.

For transfer-learning fine-tuning (e.g. pre-trained BERT for a downstream task), magnitude pruning is the wrong criterion because the pre-trained large weights may be important for general language but not for the task. Movement pruning scores each weight by how its magnitude changes during fine-tuning:

\[ S_{ij} = -\sum_t \big(\frac{\partial \mathcal{L}}{\partial W_{ij}}\big)_t \cdot W_{ij,t}. \]

A weight that moves toward zero gets a low score (pruned); a weight that moves away from zero (regardless of sign) gets a high score (kept). Outperforms magnitude pruning by ~10 points at 95% sparsity on BERT/SQuAD.

6.6 SparseGPT and Wanda — one-shot pruning for LLMs

SparseGPT (Frantar & Alistarh 2023): the OBQ/GPTQ machinery, but instead of quantizing a weight column you prune it (set to zero) and compensate the remaining weights. Achieves 50% unstructured sparsity on OPT-175B with negligible perplexity loss, in a single forward pass over a tiny calibration set.

Wanda (Sun et al. 2023, “A Simple and Effective Pruning Approach for Large Language Models”): an even simpler one-shot criterion — score each weight by magnitude × input-activation norm:

\[ S_{ij} = |W_{ij}| \cdot \| X_j \|_2. \]

Pick the lowest scores within each row → prune. No second-order information, no compensation, no fine-tuning. Works because LLM activations have wildly varying per-channel magnitudes; multiplying by \(\|X_j\|\) converts magnitude pruning into “effective output contribution” pruning. Matches SparseGPT at 50% / 2:4 sparsity on LLaMA.

MODULE 7: STRUCTURED PRUNING

7.1 Filter / channel pruning

Drop entire output channels of a Conv (and the corresponding input channels of the next layer). Real GPU speedup because the layers shrink.

Criterion options: - L1 norm of the filter (Li et al. 2016, “Pruning Filters for Efficient ConvNets”): \(S_f = \sum_{c,h,w} |W_{f,c,h,w}|\). Drop the lowest. - L2 norm, BN scale \(\gamma\) (Liu et al. 2017, “Network Slimming”): add \(L_1\) regularization on BN \(\gamma\), then prune channels whose \(\gamma\) went to zero. - Geometric Median (FPGM, He et al. 2019): prune the filter closest to the centroid of all filters, the redundant one.

7.2 Attention-head pruning

Reference: Michel et al. 2019 “Are Sixteen Heads Really Better Than One?”, Voita et al. 2019.

In multi-head attention, many heads are essentially duplicates. Score each head by \(|h_i^\top \partial \mathcal{L}/\partial h_i|\) (Taylor expansion of loss when masking the head) and prune the lowest. Typically half of the heads can be removed with little drop on machine translation and GLUE.

7.3 N:M (2:4) semi-structured sparsity

Reference: NVIDIA Ampere whitepaper; Mishra et al. 2021, “Accelerating Sparse Deep Neural Networks”.

NVIDIA Ampere/Hopper tensor cores support 2:4 structured sparsity: in every contiguous block of 4 weights along the input axis, exactly 2 must be zero. The hardware stores a 2-bit metadata index per pair (which two are non-zero) and skips the zero MACs:

\[ \text{2:4 weight tensor} \;\;\to\;\; \text{2 values + 2-bit mask per 4-element group}. \]

Effective compute saving: 2×. Crucially this can be applied on top of INT8 or FP16, giving a 2× speedup with negligible accuracy loss when combined with QAT-style fine-tuning. The recipe is “ASP” (Apex Sparsity Pipeline): pre-train → mask the lowest 2 of every 4 weights → fine-tune for a few epochs.

# NVIDIA ASP-style pseudocode
mask = build_2_4_mask(W, axis=-1)        # 1 means keep, 0 means prune
W *= mask                                # zero out the pruned half
for epoch in fine_tune_loop:
    train_step()
    W.data *= mask                       # re-impose mask after each optimizer step

7.4 Block-sparse and N:M for LLMs

Block-sparse matmul kernels (e.g. OpenAI’s blocksparse, NVIDIA cuSPARSELt) exploit dense BLAS by treating an entire \(k \times k\) block of zeros as a skipped multiply. Practical block sizes 32×32 or 16×16.

MODULE 8: WHAT IS AND ISN’T A SPEEDUP

A pruning method only matters if it translates to wall-clock or energy savings. The mapping:

Pruning Storage Memory bandwidth Compute
Unstructured dense format (CSR) ✅ smaller ✅ less data moved ❌ usually slower (irregular memory)
2:4 on Ampere ✅ smaller ✅ less ✅ 2× faster
Channel pruning ✅ smaller ✅ less ✅ proportional
Layer pruning ✅ smaller ✅ less ✅ proportional

This is why interview answers must always couple sparsity pattern with target hardware.

PART 3: KNOWLEDGE DISTILLATION (KD)

MODULE 9: THE HINTON DISTILLATION FRAMEWORK

Reference: Hinton, Vinyals & Dean 2015, “Distilling the Knowledge in a Neural Network”. The original (and still canonical) KD paper.

9.1 Response-based distillation

The “teacher” is a large, accurate (and ideally well-calibrated) model. The “student” is small. The student is trained to match the teacher’s softened output distribution plus the true labels:

\[ \mathcal{L}_{\text{KD}} = \alpha T^2 \cdot \text{KL}\big(p^{\text{teacher}}_T \, \big\| \, p^{\text{student}}_T\big) + (1-\alpha) \cdot \mathcal{L}_{\text{CE}}(y^{\text{true}}, p^{\text{student}}). \]

where the softened distribution is the softmax with temperature \(T\):

\[ p^{(i)}_T = \frac{\exp(z_i / T)}{\sum_j \exp(z_j / T)}. \]

Why temperature? A confident teacher (e.g. 0.99 on the right class, ~1e-6 on others) gives almost no information about which wrong classes are plausibly similar. Raising \(T\) flattens the distribution and exposes the dark knowledge — e.g. “this 7 looks a bit like a 1”. The student learns these similarities.

Why the \(T^2\) factor? Because the softmax gradient scales as \(1/T\), gradients from the KD term scale as \(1/T^2\); multiplying by \(T^2\) keeps the relative gradient magnitude of KD and CE losses balanced as \(T\) changes.

Typical hyperparameters: \(T \in [2, 10]\), \(\alpha \in [0.5, 0.9]\).

9.2 Why distillation works

Three complementary explanations:

  1. Soft labels are denser supervision. A one-hot label gives \(\log K\) bits per example; a softmax distribution gives up to \(H(p)\) bits, capturing inter-class similarities.
  2. Regularization. The student is forced to match a smooth function (the teacher), preventing overconfidence on training data.
  3. Privileged information. The teacher can be trained with more data, ensembles, or stronger augmentation; KD transfers that to the student at inference cost of a small model.

9.3 Offline, online, and self-distillation

Mode Teacher Student Notes
Offline Pre-trained, frozen Trained from scratch or fine-tuned Most common; e.g. DistilBERT
Online Trained jointly with the student Trained from scratch Both nets share/track each other; Deep Mutual Learning
Self-distillation Same architecture as the student, possibly an earlier checkpoint or deeper layer Itself “Be Your Own Teacher” — even with no extra params, you gain accuracy

MODULE 10: FEATURE-BASED AND RELATION-BASED DISTILLATION

10.1 FitNets — hint-based feature distillation

Reference: Romero et al. 2015, “FitNets: Hints for Thin Deep Nets”.

Match intermediate feature maps of the student to those of the teacher:

\[ \mathcal{L}_{\text{hint}} = \| r(f^{\text{student}}_l) - f^{\text{teacher}}_{l'} \|_F^2, \]

where \(r\) is a learnable 1×1 conv projecting the student’s channel count up to the teacher’s. Stage 1: pre-train the student with the hint loss. Stage 2: fine-tune with the Hinton soft-label loss. Particularly effective when the student is deeper than wide (a “thin” net).

10.2 Attention Transfer (AT)

Reference: Zagoruyko & Komodakis 2017.

Instead of matching raw feature maps, match the spatial attention maps — sum of squared activations over channels:

\[ A(f) = \sum_c |f_{c,h,w}|^p, \quad \mathcal{L}_{\text{AT}} = \|A(f^s)/\|A(f^s)\|_2 - A(f^t)/\|A(f^t)\|_2\|_p. \]

Cheap, no projection needed, and ImageNet wins.

10.3 Relation-based: RKD and CRD

RKD (Park et al. 2019, “Relational Knowledge Distillation”): match pairwise distances and triplet angles between student and teacher embeddings:

\[ \mathcal{L}_{\text{RKD-D}} = \sum_{i,j} \big| d^s_{ij} - d^t_{ij} \big|, \quad d_{ij} = \frac{\|f_i - f_j\|_2}{\mu}. \]

This is invariant to coordinate transformations of the teacher’s feature space — useful when student and teacher have completely different architectures.

CRD (Tian et al. 2020, “Contrastive Representation Distillation”): use a contrastive loss (InfoNCE) to pull positive (same-sample) teacher/student features together while pushing negatives apart. State of the art on CIFAR/ImageNet KD as of 2020.

10.4 DistilBERT, TinyBERT, MobileBERT, MiniLM

A capsule summary of NLP KD work — all heavily cited in the literature:

  • DistilBERT (Sanh et al. 2019): 6-layer student initialized from alternating layers of 12-layer BERT-base; loss = MLM + soft-label KL + cosine alignment on hidden states. 60% smaller, 60% faster, 97% of BERT’s GLUE score.
  • TinyBERT (Jiao et al. 2020): adds attention-matrix and hidden-state distillation at every transformer block, plus a two-stage procedure (general distillation on unlabeled corpora, then task-specific distillation on labeled data).
  • MobileBERT (Sun et al. 2020): bottleneck attention with inverted residual structure, distilled from a “teacher” of the same depth but wider hidden size. Designed for direct on-device CPU inference (<25 ms on Pixel 4).
  • MiniLM (Wang et al. 2020): match only the self-attention scaled-dot-product matrix and the value-relation matrix of the last layer. Architecture-agnostic — student and teacher can have completely different depths.

MODULE 11: DESIGNING TEACHER-STUDENT PAIRS FOR EDGE

A practical checklist when you design a student for an edge SoC:

  1. Inference-time SLA first. Profile the inference latency budget on the target hardware (e.g. 30 fps → 33 ms total). Subtract pre/post-processing. The student must fit.
  2. Pick an architecture family the hardware likes. Depthwise-separable convolutions on mobile DSPs/NPUs; group conv on Edge TPU; standard dense conv on Apple Neural Engine.
  3. Match teacher–student capacity, but not too closely. A 4× compression ratio is the usual sweet spot. Beyond 10× you saturate.
  4. Pre-train with self-supervision (e.g. SimCLR/MoCo or MLM) before KD. Particularly for small students; gives them a strong feature backbone.
  5. Use intermediate-feature distillation in addition to soft-label KD — the gradients from intermediate matches stabilize early training.
  6. Augment aggressively — the teacher is robust to augmentation; the student should be too. CutMix and Mixup are common.
  7. Distill on unlabeled domain data. KD does not need labels, so you can use any in-domain data you have.

Pseudocode for joint distillation:

loss = (
    alpha * (T**2) * F.kl_div(F.log_softmax(s_logits/T, dim=-1),
                              F.softmax(t_logits/T, dim=-1),
                              reduction='batchmean')
  + (1 - alpha) * F.cross_entropy(s_logits, y_true)
  + beta * sum(F.mse_loss(proj(s_feat[l]), t_feat[lp])
               for (l, lp, proj) in hint_pairs)
)

PART 4: NEURAL ARCHITECTURE SEARCH (NAS)

MODULE 12: SEARCH SPACES

A NAS algorithm is the product of three components: search space, search strategy, performance estimation strategy. Different combinations recover the entire NAS literature.

12.1 Chain-structured / macro spaces

A linear stack of operations chosen from a fixed catalogue ({3×3 conv, 5×5 conv, 3×3 depthwise, identity, max pool…}) with kernel size, stride and width as hyperparameters. Used in Zoph & Le’s original NASNet (2017) but the space is astronomically large.

12.2 Cell-based spaces (NASNet, AmoebaNet, DARTS)

Search for a small reusable cell (a few nodes, each node = a sum of two ops applied to two previous nodes), then stack copies of it. Drastically smaller search space — millions of cells, not \(10^{20}\) full nets.

A cell is a directed acyclic graph with \(N\) intermediate nodes; each takes inputs from the two previous cells’ outputs and applies an op. The “normal cell” preserves spatial resolution; the “reduction cell” halves it.

12.3 Hierarchical search spaces

Build a tree: motifs at the bottom (a single conv), composed into cells (Liu et al. 2018), composed into networks. Allows controllable diversity.

12.4 MBConv-based mobile spaces (MnasNet, EfficientNet, MobileNetV3)

Reference: Tan et al. 2019 “MnasNet”; Tan & Le 2019 “EfficientNet”; Howard et al. 2019 “MobileNetV3”.

A network is a sequence of stages; each stage is a stack of inverted residual + linear bottleneck blocks (MBConv) parameterised by:

  • kernel size \(k \in \{3, 5, 7\}\)
  • expansion ratio \(e \in \{3, 4, 6\}\)
  • depth (number of blocks per stage) \(d \in \{1, 2, 3, 4\}\)
  • width multiplier per stage \(w\)
  • presence of Squeeze-and-Excitation (SE)

This single space underlies essentially every modern mobile NAS work.

12.5 Hardware-aware constraints

For an edge processor, the search space must encode:

  • Maximum receptive field (memory) per layer.
  • Allowed input/output channel multiples (e.g. multiples of 8 for AVX-512 / VNNI; multiples of 16 for tensor cores).
  • Banned ops (e.g. no atomic operations on a particular DSP; no 5×5 depthwise on Hexagon HVX without slowing down).
  • DMA-friendly tensor sizes.
  • Aligned activation memory footprint \(\le\) on-chip SRAM (otherwise spills to external DRAM dominate latency).

A good hardware-aware NAS space is a tightly curated subset that, by construction, only produces models that map well to the accelerator’s instruction set.

MODULE 13: SEARCH STRATEGIES

13.1 Reinforcement learning (NASNet, MnasNet)

A controller (an LSTM) emits a sequence of architectural decisions (op, kernel, stride…). Each sample is a candidate network. Train the candidate briefly, record its validation accuracy, use that as reward, update the controller with REINFORCE / PPO.

Loss for the controller: \[ \nabla J(\theta) = \mathbb{E}_{a \sim \pi_\theta}\big[\, (R - b) \nabla \log \pi_\theta(a) \,\big] \]

with \(R = \text{Accuracy} - \lambda \cdot |\text{latency} - L^*|\) (the Pareto reward of MnasNet).

NASNet famously took ~2000 GPU-days on CIFAR-10. Modern variants amortize cost dramatically.

13.2 Evolutionary / regularised aging-evolution (AmoebaNet)

Reference: Real et al. 2019, “Regularized Evolution for Image Classifier Architecture Search”.

Maintain a population of \(P\) architectures, each with a fitness (validation accuracy). At each step: 1. Sample \(S\) candidates from the population (tournament of size \(S\)). 2. Pick the best as the parent. 3. Mutate the parent (change one decision: op type, kernel, connection). 4. Train the child briefly; add to the population. 5. Remove the oldest member (not the worst). This “aging” regularisation prevents the population from collapsing onto a single early winner.

Surprisingly strong: AmoebaNet beats NASNet under the same compute budget.

13.3 Gradient-based: DARTS, ENAS

Reference: Liu, Simonyan & Yang 2019, “DARTS: Differentiable Architecture Search”.

Relax the discrete op choice in each edge of the cell to a convex combination:

\[ \bar{o}^{(i,j)}(x) = \sum_{o \in \mathcal{O}} \frac{\exp(\alpha^{(i,j)}_o)}{\sum_{o'} \exp(\alpha^{(i,j)}_{o'})} \cdot o(x). \]

Now everything is differentiable. Optimise architecture parameters \(\alpha\) on a held-out validation loss while optimising weights \(w\) on the training loss — a bilevel problem:

\[ \min_\alpha \mathcal{L}_{\text{val}}(w^*(\alpha), \alpha), \quad \text{s.t. } w^*(\alpha) = \arg\min_w \mathcal{L}_{\text{train}}(w, \alpha). \]

DARTS approximates \(w^*(\alpha)\) by a single gradient step (“first-order DARTS”). The discrete final architecture is obtained by argmax_o α^(i,j)_o per edge. DARTS runs in a few GPU-days, not 2000.

ENAS (Pham et al. 2018) shares weights across all subnetworks via a single supernet, so each candidate’s evaluation is essentially free.

13.4 Supernet / once-for-all (OFA, ProxylessNAS, FBNet)

Reference: Cai, Gan & Han 2020, “Once-for-All: Train One Network and Specialize it for Efficient Deployment”.

Train a single elastic supernet in which you can dynamically toggle depth, width, kernel size, and resolution. After supernet training, you can extract any of \(10^{19}\) sub-networks without re-training, just by sampling the desired configuration. Then run a quick evolutionary search over sub-networks against the target hardware’s latency to find the Pareto-optimal one.

OFA is the modern industrial NAS workhorse: one \(10^4\)-GPU-hour pre-training, then any number of target devices for free.

ProxylessNAS (Cai et al. 2019) directly searches on the target task (no CIFAR proxy) and target hardware (latency in the loss).

FBNet (Wu et al. 2019) is a similar Gumbel-Softmax differentiable NAS with a latency lookup table.

13.5 Performance estimation — the bottleneck

Naively, each candidate must be fully trained to compute its accuracy. Tricks to avoid this:

  • Early stopping — train for 5 epochs, extrapolate.
  • Weight sharing (ENAS / OFA) — sub-networks inherit weights from a trained supernet.
  • Predictor models — train a small GBM or GNN to predict accuracy from architecture encoding.
  • Zero-shot / training-free proxies — score architectures with a single forward pass, no training at all.

MODULE 14: PROXY METRICS AND LATENCY PREDICTION

14.1 Latency lookup tables

For mobile inference, the most accurate latency model is empirical. Decompose any candidate into a sum of independent layers (works because mobile NPUs do little cross-layer parallelism):

\[ \text{lat}(\pi) \approx \sum_l \text{LUT}[\text{op}_l, c_{\text{in}}, c_{\text{out}}, k, s, H, W]. \]

Construct the LUT by benchmarking every layer config you’ll search over on the actual device — typically a few hundred ms per measurement, a few hours total. Used by MnasNet, ProxylessNAS, FBNet.

14.2 Learned latency predictors

For more complex hardware (DSPs with cross-layer fusion, GPUs with batch effects), train a small neural net (e.g. a GBM, MLP, or GCN) to predict latency from architecture features. Trained on ~5000 (architecture, measured latency) pairs.

14.3 Zero-shot / training-free proxies

A bombshell line of work: predict architecture accuracy with a single batch forward pass on a random init, no training.

  • SNIP (Lee et al. 2019): score = \(|g \odot w|\) at random init. The architectures whose pre-training gradient signals are larger end up training better.
  • GradSign (Zhang et al. 2022): use the sign correlation of gradients across mini-batches.
  • Zen-Score (Lin et al. 2021): use the network’s expressivity proxy — Gaussian complexity of random-init feature distributions.
  • NASWOT (Mellor et al. 2021): score = log-determinant of the binary activation pattern’s correlation matrix at init.

These proxies correlate ~0.6–0.8 with trained accuracy and can rank thousands of candidates in minutes.

14.4 Sensitivity analysis

After a model is trained, you want to know “how much does each layer matter for accuracy/latency?”. The standard tool is the Taylor expansion of the loss around the current weights:

\[ \Delta \mathcal{L}_l \approx \big| g_l^\top w_l + \tfrac{1}{2} w_l^\top H_l w_l \big|. \]

For each layer, compute (i) latency cost on the target device, (ii) accuracy sensitivity. Build a layer-wise Pareto frontier: layers with high latency cost and low sensitivity are first to compress (low bits / high prune ratio); layers with low latency cost or high sensitivity stay near FP16.

PART 5: ON-DEVICE AND FEDERATED TRAINING

MODULE 15: ON-DEVICE TRAINING FUNDAMENTALS

Most “edge AI” today is inference-only. Real on-device training is hard because:

  • Training memory is dominated by activations (you need them for the backward pass), not weights. A 100 MB model with batch=32 might need 4 GB of activation memory.
  • Float ops on mobile NPUs are typically much slower than INT ops.
  • Battery and thermal envelopes are tight.

Counter-measures: - Activation checkpointing / gradient recompute: throw away most activations during forward, recompute them on demand in backward. Roughly √depth memory at √depth extra FLOPs (Chen et al. 2016). - Mixed-precision training: weights/activations in FP16/BF16, master copy of weights in FP32. - Sparse updates: only update a subset of parameters (LoRA, BitFit, IA³, adapters). - TinyTL (Cai et al. 2020, “TinyTL: Reduce Memory, Not Parameters for Efficient On-Device Learning”): freeze weights, train only biases — drops memory by ~6×.

MODULE 16: FEDERATED LEARNING (FL)

Reference: McMahan et al. 2017, “Communication-Efficient Learning of Deep Networks from Decentralized Data”.

16.1 FedAvg — the foundational algorithm

\(K\) clients each hold local data \(D_k\). A central server orchestrates the rounds:

  1. Server sends global weights \(W_t\) to a random subset \(S_t\) of clients.
  2. Each client \(k\) runs a few epochs of SGD locally: \(W_t^k \leftarrow W_t^k - \eta \nabla \mathcal{L}_k(W_t^k)\).
  3. Each client sends its updated weights (or the delta \(\Delta_t^k = W_t^k - W_t\)) back.
  4. Server averages: \(W_{t+1} = \sum_k \frac{n_k}{n} W_t^k\) (weighted by local dataset size).

Key advantages: data never leaves the device, communication cost is the only network burden, scales to millions of clients (Google’s Gboard).

16.2 FedProx, FedOpt, Personalized FL

FedProx (Li et al. 2018): adds a proximal regulariser to the client objective to handle system heterogeneity (slow clients can do fewer local steps):

\[ \min_W \mathcal{L}_k(W) + \frac{\mu}{2} \|W - W^{\text{global}}\|^2. \]

FedOpt / FedAdam / FedYogi (Reddi et al. 2020): use adaptive server-side optimizers (Adam, AdaGrad) on the aggregated pseudo-gradient \(\bar{\Delta}_t\). Roughly 3–10× faster convergence than vanilla FedAvg on non-IID data.

Personalized FL (per-FedAvg, pFedMe, Ditto): a global model + a small per-client head, jointly optimised. Each client gets a model tailored to its data distribution while still benefiting from the global signal.

16.3 Gradient compression

Server-client communication of full gradients is the dominant cost in FL. Compression strategies:

Technique Compression Math
Top-K sparsification 100×–1000× Send only the K largest-magnitude gradient elements
signSGD 32× (1 bit/grad) Send only the sign; aggregate via majority vote
QSGD (Alistarh et al. 2017) 8–32× Stochastic uniform quantization with unbiased estimator
TernGrad ~16× Quantize to {-1, 0, +1}
PowerSGD (Vogels et al. 2019) 100× Low-rank approximation \(G \approx PQ^\top\) via power iteration
EF21 (Richtárik et al. 2021) depends Error feedback: accumulate compression residuals; eliminates bias

Error feedback is the key idea: after compression, store the difference \(e_t = g_t - C(g_t)\) on the client and add it to the next gradient \(g_{t+1} \leftarrow g_{t+1} + e_t\). This recovers convergence guarantees even for biased compressors.

16.4 Privacy in FL

The data may not leave the device, but the gradients can leak it (“Deep Leakage from Gradients”, Zhu et al. 2019). Standard privacy machinery:

  • Differential Privacy SGD (DP-SGD) (Abadi et al. 2016): clip per-sample gradients to \(\ell_2\) norm \(C\), add Gaussian noise \(\mathcal{N}(0, \sigma^2 C^2 I)\) to the aggregated batch gradient. Privacy budget \((\epsilon, \delta)\) tracked via the moments accountant or RDP accountant.
  • Secure aggregation (Bonawitz et al. 2017): use cryptographic masks (pairwise secret sharing) so the server only sees \(\sum_k \Delta_k\), never individual \(\Delta_k\).
  • Homomorphic encryption (BGV/CKKS): server computes encrypted aggregation. Currently too slow for large models.
  • Trusted execution environments (Intel SGX, Apple Secure Enclave): hardware-isolated computation. Fast but vulnerable to side-channel attacks.
  • Personalized DP — different privacy budgets per client / per data point.

MODULE 18: GRAPH-LEVEL OPTIMIZATIONS

18.1 Operator (kernel) fusion

The biggest single source of speedup in deep learning compilers. The goals: - Eliminate intermediate memory writes/reads. - Increase arithmetic intensity (FLOPs per byte loaded). - Reduce kernel launch overhead.

Vertical fusion — chain producer-consumer ops into a single kernel: - Conv → BatchNorm → ReLU → “ConvBnRelu” - MatMul → Add bias → GELU → “FusedFC” - LayerNorm → Linear → “FusedLN-Linear” (used in many transformer compilers)

Horizontal fusion — merge sibling ops that share the same input: - Multi-head attention’s three projections (Q, K, V) share the input → fuse into one matmul of width \(3 d_{\text{model}}\).

Conv–BN fold in detail — at inference time, BN’s affine parameters are absorbed into the conv weights (the math from §3.3): \[ W' = \gamma W / \sqrt{\sigma^2 + \epsilon}, \quad b' = \beta - \gamma(\mu - b)/\sqrt{\sigma^2 + \epsilon}. \] Now the BN op vanishes entirely.

TVM/Ansor and TensorRT do this automatically. PyTorch’s torch.fx graph mode and ONNX Runtime’s optimization_level=ORT_ENABLE_ALL apply hundreds of fusion rules.

18.2 Constant folding

Any subgraph whose inputs are all compile-time constants is evaluated at compile time:

Reshape(Const([1,2,3,4]), shape=[2,2])   →   Const([[1,2],[3,4]])
Add(Const(a), Const(b))                  →   Const(a+b)

In transformers this is huge: positional encodings, mask tensors, scale constants (\(1/\sqrt{d_k}\)) all collapse to constants.

18.3 Common subexpression elimination (CSE)

If two subgraphs compute the same value, replace with one. In transformer encoders this finds shared key/value projections across attention heads.

18.4 Dead code elimination (DCE)

Remove any op whose output isn’t reachable from a graph output. Often produced by aggressive fusion: a producer op gets folded into one consumer but had a second consumer that was also fused, leaving the original producer dead.

18.5 Algebraic simplification

x + 0 = x, x * 1 = x, x * 0 = 0, concat(split(x)) = x, transpose(transpose(x, p), p^{-1}) = x. Helpful after layout transforms inject reorder ops.

18.6 Layout selection

A “layout” is the in-memory ordering of tensor axes. Conv inputs can be:

  • NCHW — channels-major. Good for cuDNN convolutions on NVIDIA.
  • NHWC — channels-last. Friendlier for AVX-512 / AMX / mobile NPUs.
  • NCHWc / NCHW16c — channel-blocked. Optimal for AVX-512 because \(c\)=16 lines up with vector width.
  • NC/xHWx — Intel oneDNN’s blocked layout.

A compiler chooses the layout that lets the inner kernel saturate the vector unit. Inserting and fusing-away Transpose ops in between layers is a major optimization (TVM, ONNX Runtime’s LayoutTransformer).

18.7 Quantize/Dequantize (Q/DQ) propagation

In quantized graphs, the IR is full of Q → op → DQ → op → Q → … pairs. A compiler propagates them through arithmetic ops to merge adjacent Q/DQ where possible, eliminating redundant scale-and-cast round-trips. The result is a long INT8 pipeline with Q and DQ only at the boundary.

18.8 Auto-tuning kernels

For each fused operator on each target backend, you still need to choose: - tile sizes for the loop nest; - vectorization width; - thread block / warp configuration; - memory layout; - whether to use shared memory, registers, or device-specific units.

AutoTVM uses XGBoost-guided random search over hand-written templates (“schedules”). Ansor removes the need for templates by auto-generating the schedule search space from a TE (tensor-expression) description. MetaSchedule is TVM’s modern reinforcement-learning + cost-model autotuner. NVIDIA’s TensorRT ships its own internal autotuner that selects from a closed catalogue of pre-optimized cuDNN kernels at engine-build time.

MODULE 19: MEMORY MANAGEMENT

19.1 Memory layout transformations

A 4-D tensor [N, C, H, W] can be stored row-major or with channel blocking. The right layout transform turns strided memory accesses into contiguous ones, enabling SIMD and cache prefetcher to fire. Layout transforms are cheap if they get fused away, expensive if they linger as standalone copies.

Concretely, NCHW16c means: outer index is N, then \(C/16\), then H, W, then inner 16-element channel block. This lets a single AVX-512 vector instruction load 16 channels at one (N,H,W) position simultaneously.

19.2 Tensor allocation and memory reuse

A naive interpreter allocates fresh memory for every intermediate tensor; a compiler runs liveness analysis to reuse buffers. Two tensors with non-overlapping lifetimes can share storage. The compiler builds an interference graph (tensors = nodes, edges = overlapping lifetimes) and does graph coloring — exactly the same algorithm a CPU compiler uses for register allocation.

For dynamic shapes the problem becomes a 2-D bin-packing / Integer Linear Program:

\[ \min \max_t \sum_{T_j \in \text{Live}(t)} \text{size}(T_j) \quad \text{s.t. allocation overlap rules.} \]

Solved exactly with ILP for small graphs, with greedy first-fit / best-fit decreasing for large graphs.

19.3 In-place ops

When the producer’s tensor isn’t used elsewhere, we can write the consumer’s result into the producer’s buffer. ReLU is the canonical example: y = max(0, x) overwrites x if no one else reads it. PyTorch’s relu_(x), x += y, x.clamp_() all use this.

19.4 Activation checkpointing

Recompute activations in the backward pass instead of storing them. Strategies: - Block-wise — checkpoint every \(\sqrt{N}\) layers. Memory \(O(\sqrt{N})\), time \(\le 2\times\) baseline. - Selective — keep cheap-to-recompute activations on disk/SRAM, recompute expensive ones. Megatron’s “selective activation recomputation” stores only the attention dropout mask + softmax outputs. - Auto-grad — torch.utils.checkpoint.checkpoint(fn, *args) in PyTorch.

19.5 Memory pools / arena allocators

Allocate a single big chunk of memory at engine startup and sub-allocate from it. Avoids per-tensor malloc/free overhead and fragmentation. TensorRT, ONNX Runtime, and TFLite all do this.

19.6 Memory transfer pipelines on heterogeneous devices

Modern SoCs have multiple memories: CPU DRAM, GPU HBM, NPU SRAM, DSP TCM. Transfers between them are the dominant latency for small models.

Techniques: - Async DMA — overlap compute and transfer (CUDA streams; OpenCL command queues). - Double buffering / ping-pong — while NPU computes on buffer A, DMA fills buffer B. - Pinned host memory — non-pageable pages on the CPU, twice the H2D bandwidth. - GPUDirect Storage — NVMe → GPU memory directly, bypassing CPU.

19.7 The Roofline model

Reference: Williams et al. 2009. Indispensable for diagnosing performance:

\[ \text{Attainable performance (FLOP/s)} = \min(\pi, \beta \cdot AI), \]

where \(\pi\) is peak FLOP/s, \(\beta\) is peak memory bandwidth, and arithmetic intensity \(AI = \text{FLOPs} / \text{bytes moved}\). A workload sits at one of two regimes:

  • Compute-bound (\(\beta AI > \pi\)): you’ve saturated the FPUs. Quantization / fusion still help by reducing total work.
  • Memory-bound (\(\beta AI < \pi\)): the bandwidth wall. Fusion (raise AI), quantization (reduce bytes), prefetching are the leverage.

LLM inference at batch size 1 is deeply memory-bound — that’s why W4A16 (~4× bandwidth saving) is the standard.

MODULE 20: TAXONOMY OF PARALLELISM

20.1 Data parallelism

Each worker has a full copy of the model, processes a different shard of the batch, then all_reduce gradients. Bandwidth = \(2P (N-1)/N\) bytes per round where \(P\) is parameter count and \(N\) is number of workers. Implemented via NCCL ring-allreduce.

20.2 Tensor parallelism (Megatron-LM)

Reference: Shoeybi et al. 2019, “Megatron-LM: Training Multi-Billion Parameter Language Models Using Model Parallelism”.

For an FC layer \(Y = X W\) with \(W \in \mathbb{R}^{d_{in} \times d_{out}}\): - Row parallel: split \(W\) along the input dim, \(X\) is replicated, partial outputs are summed via all_reduce. - Column parallel: split \(W\) along the output dim, output is concatenated.

For a transformer MLP block (two linear layers), the standard pattern is column-parallel → activation → row-parallel, which gives only one all-reduce in forward (and one in backward), placed after the second matmul. For attention, split the heads across devices: each device computes a subset of heads with its own QKV projections, then a row-parallel output projection sums them.

20.3 Pipeline parallelism (GPipe, PipeDream)

Split a deep model into \(S\) stages, one per device. Forward activations of stage \(s\) are sent to stage \(s+1\); gradients flow back.

  • GPipe: split a minibatch into \(M\) microbatches, pipeline them through. Bubble (idle time) = \((S-1)/M\) of total time. Memory: must keep activations of all \(M\) microbatches.
  • PipeDream-1F1B: alternate forward and backward — once a stage finishes the forward of microbatch \(i\), do the backward of microbatch \(i - (S-1)\) immediately. Same bubble fraction but constant activation memory.
  • PipeDream-Flush / Interleaved 1F1B (Megatron-3): assign multiple non-contiguous chunks per device, halving the pipeline bubble.

20.4 ZeRO and FSDP

Reference: Rajbhandari et al. 2020, “ZeRO: Memory Optimizations Toward Training Trillion Parameter Models”.

The memory footprint of training has three components: parameters \(P\), gradients \(G\), optimizer states \(O\). For Adam with FP16 training, \(O = 12P\) bytes (FP32 master weights + momentum + variance), so \(P + G + O \approx 16P\) bytes.

  • ZeRO-1: shard \(O\) across data-parallel workers. Memory: \(P + G + O/N\).
  • ZeRO-2: also shard \(G\). Memory: \(P + (G+O)/N\).
  • ZeRO-3 / FSDP: also shard \(P\) itself. Memory: \((P+G+O)/N\). Each layer’s full weights are all-gathered on the fly when needed, then discarded.

ZeRO-3 / FSDP makes 70B-parameter training feasible on commodity 8×A100 nodes. The cost is one extra all-gather per layer.

20.5 OpenCL primitives — multithreading at the kernel level

OpenCL exposes a parallel programming model with three nested levels:

  • NDRange — global iteration space. A 3D index (get_global_id(0), get_global_id(1), get_global_id(2)).
  • Work-group — a tile of work-items that share local memory (a few KB of fast on-chip SRAM) and a barrier. Maps to GPU thread block / SIMD lane group.
  • Work-item — the smallest unit of execution. Maps to GPU thread / SIMD lane.
  • Wavefront / sub-group — a subgroup of work-items that execute in lockstep (AMD wavefront = 64; NVIDIA warp = 32). Specialized intrinsics like sub_group_reduce_add use this.

Data parallelism in OpenCL: launch one work-item per output element (or per output tile). Each work-item computes its slice of the output independently.

Task parallelism in OpenCL: enqueue multiple independent kernels on different command queues; use events to express dependencies. Used when kernels operate on different data and can overlap with each other and with host-device transfers.

// OpenCL data-parallel matmul (one work-item per output element)
__kernel void matmul(__global const float* A, __global const float* B,
                     __global float* C, int K) {
    int i = get_global_id(0);
    int j = get_global_id(1);
    float acc = 0.0f;
    for (int k = 0; k < K; ++k) acc += A[i*K + k] * B[k*get_global_size(1) + j];
    C[i*get_global_size(1) + j] = acc;
}

A tuned implementation will use __local shared-memory tiles, barrier(CLK_LOCAL_MEM_FENCE) sync, and 4-wide float4 vector loads.

MODULE 21: HOW TO PICK A PARALLELISM STRATEGY

A practical 2024-era decision tree for training a model with parameter count \(P\) on \(N\) GPUs of memory \(M\) each:

if P + 16P < M:          # tiny: < 1 GPU
    use single-GPU
elif 16P / N < M:        # fits with ZeRO-3 across N
    use FSDP / ZeRO-3
elif fits in one node (e.g. 8 GPUs via NVLink):
    use TP within a node + DP across nodes  (Megatron-LM style)
else:
    combine TP × PP × DP × ZeRO              (3D / 4D parallelism)

A canonical configuration for a 175B-parameter model on 1024 A100s: TP=8, PP=16, DP=8 — exactly Megatron’s GPT-3 reproduction.

PART 8: COMPRESSING VISION MODELS FOR EDGE

MODULE 22: ARCHITECTURAL FAMILIES FOR EDGE VISION

22.1 MobileNetV1 — depthwise separable conv

Reference: Howard et al. 2017.

A standard \(k \times k\) conv with \(C_{\text{in}}\) input channels and \(C_{\text{out}}\) output channels costs \(k^2 C_{\text{in}} C_{\text{out}} H W\) MACs. Depthwise separable conv splits it into:

  1. Depthwise conv: a \(k \times k\) filter per channel (no cross-channel mixing) → \(k^2 C_{\text{in}} H W\) MACs.
  2. Pointwise conv: a \(1 \times 1\) conv that mixes channels → \(C_{\text{in}} C_{\text{out}} H W\) MACs.

Total: \((k^2 C_{\text{in}} + C_{\text{in}} C_{\text{out}}) H W\) MACs. Reduction factor:

\[ \frac{k^2 C_{\text{in}} C_{\text{out}}}{k^2 C_{\text{in}} + C_{\text{in}} C_{\text{out}}} \approx \frac{1}{1/C_{\text{out}} + 1/k^2}. \]

For \(k=3\) and \(C_{\text{out}} = 64\): ~8× fewer MACs.

22.2 MobileNetV2 — inverted residual + linear bottleneck

Reference: Sandler et al. 2018.

A standard residual block is “wide–narrow–wide” with a non-linearity at the narrow end. MobileNetV2 reverses this: “narrow–wide–narrow”, with the non-linearity only in the expanded middle. The narrow ends are linear (no ReLU) because applying ReLU to a low-dimensional representation destroys information.

Block structure: 1. \(1 \times 1\) conv (expansion factor \(t = 6\), ReLU6). 2. \(3 \times 3\) depthwise conv (stride 1 or 2, ReLU6). 3. \(1 \times 1\) conv to project back to low dim, no activation. 4. Residual skip if shapes match.

22.3 MobileNetV3 — NAS + SE + h-swish

Reference: Howard et al. 2019.

MNV3 is the output of MnasNet-driven NAS + manual tweaks: - Squeeze-and-Excitation gates in select blocks. - h-swish activation: \(x \cdot \text{ReLU6}(x+3)/6\) — a piecewise-linear approximation of swish that’s friendly to integer quantization. - Reduced channels in the early layers, dropped 5 layers from the tail.

22.4 EfficientNet — compound scaling

Reference: Tan & Le 2019.

Given a baseline architecture (B0, itself from NAS), scale depth, width, and resolution jointly with a compound coefficient \(\phi\):

\[ \text{depth} = \alpha^\phi, \quad \text{width} = \beta^\phi, \quad \text{resolution} = \gamma^\phi \]

subject to \(\alpha \beta^2 \gamma^2 \approx 2\) (so FLOPs scale as \(2^\phi\)). The constraint is empirical: scaling FLOPs by a factor of 2 with this rule produces a model that is ~consistently better than independent scaling.

22.5 Detection / segmentation backbones

  • MobileNet-SSDLite: drop dense conv detection heads in favour of depthwise separable ones. Quantize to INT8 → real-time on Pixel CPUs.
  • EfficientDet: BiFPN feature fusion + compound-scaled EfficientNet backbone.
  • YOLOv5/v8 nano variants: deliberately shallow + wide channel multipliers for mobile.

MODULE 23: A PRACTICAL VISION-FOR-EDGE PIPELINE

A concrete recipe for shipping an object detector on a phone NPU:

  1. Pick an INT8-friendly architecture — MobileNetV2/V3 backbone, SSDLite head, ReLU6 activations.
  2. Pre-train on ImageNet, fine-tune on the target detection dataset (COCO/your data).
  3. Knowledge-distill from a larger teacher (EfficientDet-D4) — both response and feature-based.
  4. Apply structured pruning on the backbone: channel pruning to 0.7× width.
  5. Quantize to INT8 with QAT (PyTorch FX QAT, fold BN, per-channel weight scales).
  6. Export to ONNX, then TFLite (for Android NNAPI) or Core ML (for iOS Neural Engine).
  7. Run the toolchain’s optimizer (TFLite Converter with optimizations=[Optimize.DEFAULT]).

MODULE 24: OPERATOR FUSION FOR VISION

Vision graphs are dominated by Conv–BN–Activation motifs and post-processing (NMS).

  • Conv → BN → ReLU6 → fused ConvBnReluClip op (TFLite default after optimizations=[DEFAULT]).
  • Conv → BN → Add (skip) → ReLU (residual block) → fused into one kernel that holds the skip connection in registers.
  • For SSD post-processing: fuse softmax → top-K → NMS into a single kernel (TensorRT’s EfficientNMS).
  • For semantic segmentation: fuse bilinear upsample → conv → softmax (TVM does this; NVIDIA’s segmentation TRT plugins do this).

PART 9: COMPRESSING LANGUAGE MODELS FOR EDGE (AND LLM INFERENCE OPTIMIZATION)

MODULE 25: WHERE DOES LLM LATENCY COME FROM?

Reference: Lilian Weng, “Large Transformer Model Inference Optimization” (a must-read).

LLM inference has two distinct phases with very different characteristics:

Phase Workload Bottleneck Optimisation lever
Prefill (process the input prompt) One big matmul \(L_{\text{prompt}} \times d\) Compute-bound Tensor parallelism, FlashAttention
Decode (generate tokens one at a time) \(1 \times d\) matmul per layer per token Memory bandwidth (load weights) Quantization, KV cache, speculative decoding

At batch=1 decode, each generated token requires loading the entire weight matrix of every layer from HBM. For LLaMA-2-70B in FP16 that’s 140 GB read per token — at A100’s 2 TB/s HBM bandwidth, the lower bound is ~70 ms/token. INT4 weights (35 GB) → ~17.5 ms/token. This is why W4A16 is everywhere.

MODULE 26: KV CACHE AND ATTENTION OPTIMIZATIONS

26.1 The KV cache

In an autoregressive transformer, at step \(t\) the attention layer needs the keys and values of all previous tokens \(1..t-1\). Recomputing them at every step is \(O(t^2)\). The KV cache stores them:

\[ K_{1:t} \in \mathbb{R}^{t \times d_k}, \quad V_{1:t} \in \mathbb{R}^{t \times d_v} \quad \text{per layer per head}. \]

For LLaMA-2-70B at sequence length 4096, the KV cache is ~10 GB per batch. KV cache dominates LLM memory at long context.

26.2 Multi-Query and Grouped-Query Attention (MQA / GQA)

Reference: Shazeer 2019 “Fast Transformer Decoding”; Ainslie et al. 2023 “GQA”.

  • MHA: \(H\) heads, each with its own K and V → KV cache of \(H \cdot t \cdot d_k\) per layer.
  • MQA: \(H\) query heads share a single K, V → KV cache of \(t \cdot d_k\) per layer. ~\(H \times\) smaller.
  • GQA: \(H\) query heads share K, V in groups of \(g\) → KV cache of \((H/g) \cdot t \cdot d_k\). LLaMA-2-70B uses \(g = 8\).

GQA preserves most of MHA’s quality while making KV cache and decode bandwidth practical.

26.3 FlashAttention

Reference: Dao et al. 2022 (FA1), 2023 (FA2), 2024 (FA3).

Standard attention reads \(QK^\top\) (\(O(N^2)\) memory) into HBM, applies softmax, then multiplies by \(V\). FlashAttention does the softmax in tiled streaming form, never materializing \(N \times N\) scores in HBM:

  1. Tile \(Q\) and \(K, V\) along the sequence dim into blocks of size \(B_r, B_c\).
  2. For each query tile, iterate over key tiles; maintain a running softmax normalization \((m_i, \ell_i)\) in registers/SMEM.
  3. Update \(O_i \leftarrow \text{rescale}(O_i, m_i, m_i^{\text{new}}) + \tilde{p}_{ij} V_j\).

The online-softmax trick uses: \[ m_{ij} = \max(m_i, \max_j S_{ij}), \quad \ell_{ij} = e^{m_i - m_{ij}} \ell_i + \sum_j e^{S_{ij} - m_{ij}}. \]

Result: attention with \(O(N)\) memory (not \(O(N^2)\)) and 2–4× wall-clock speedup. FA2 doubles parallelism by parallelising over the sequence axis too; FA3 adds Hopper-specific FP8 + warp-specialisation.

26.4 PagedAttention and vLLM

Reference: Kwon et al. 2023, “Efficient Memory Management for Large Language Model Serving with PagedAttention”.

The KV cache is allocated as one giant contiguous tensor per request, causing fragmentation (different requests have different lengths). PagedAttention borrows from the OS virtual-memory playbook: - KV cache is split into fixed-size pages (typically 16 tokens). - A logical-to-physical page table per sequence. - Pages can be shared across sequences with the same prefix (prompt) — “copy-on-write” semantics for divergent generations.

Yields 2–4× higher throughput in batch inference; powers vLLM, the de-facto open-source LLM server.

26.5 Continuous batching

Reference: Yu et al. 2022, “Orca”.

Static batching wastes compute when sequences finish at different times. Continuous batching schedules at iteration granularity: as soon as one sequence emits its EOS, a new one slides into its slot. Combined with PagedAttention, this is how vLLM/TGI hit 90%+ GPU utilization on production serving.

26.6 Speculative decoding, Medusa, EAGLE

Reference: Leviathan et al. 2023; Cai et al. 2024 (Medusa); Li et al. 2024 (EAGLE).

Speculative decoding: a small fast draft model generates \(K\) tokens; the big target model verifies them in a single forward pass that returns logits for positions \(1..K+1\). Accepted tokens cost only the verification step; rejected tokens are replaced by the target’s distribution. Expected speedup ~2–3×.

Medusa: instead of a separate draft model, add \(K\) extra decoding heads to the target model. Head \(k\) predicts the \((k+1)\)-th future token. Verification is a tree-attention pass: all \(K^2\) candidate continuations checked in one forward.

EAGLE: predicts the feature (penultimate layer) of the next token rather than the token itself, then re-projects. Much higher acceptance rate (~3× speedup with tiny draft cost).

26.7 Prefix / prompt caching

If many requests share a long system prompt, cache its KV across requests. PagedAttention does this naturally (shared pages). On a chatbot with a 10K-token system prompt, ~95% of decode-time work for the first response token is saved.

26.8 RoPE optimizations

Rotary positional embedding applies a rotation by \(\theta_t = t \omega_i\) to each (Q, K) pair. Tricks: - Pre-compute \(\cos\theta, \sin\theta\) tables. - Apply RoPE in-place on the K cache only at write time, not at read time — saves bandwidth. - YaRN / NTK-aware RoPE to extend context length without retraining.

27.1 LLM.int8()

Reference: Dettmers et al. 2022.

Per-channel INT8 weights and vector-wise INT8 activations, except for the ~0.1% of channels with extreme activation magnitudes (“emergent outlier features”) — those rows stay in FP16. The matrix multiplication is split into two parts: INT8×INT8 fast path + small FP16×FP16 slow path; results are summed. First quantization recipe to handle the activation outlier problem in 175B-scale transformers.

27.2 QLoRA

Reference: Dettmers et al. 2023.

QLoRA enables fine-tuning a 65B model on a single 48GB A6000 by combining: 1. 4-bit NF4 quantization of all weights (frozen). 2. Double quantization — quantize the per-block scales themselves to 8 bits, saving ~0.4 bit/param. 3. Paged optimizer states — Nvidia unified memory spills CPU RAM when GPU is full. 4. LoRA trainable adapters in FP16.

LoRA adapter: \(W \to W + B A\) with \(A \in \mathbb{R}^{r \times k}\), \(B \in \mathbb{R}^{d \times r}\), \(r \ll d, k\). Only \(A, B\) are trained.

MODULE 28: DISTILLING LLMs FOR EDGE

Distilling generative LMs is subtly different from classifier KD:

  • DistilBERT / TinyBERT / MiniLM for encoders — already covered. Match hidden states + attention matrices.
  • Sequence-level KD for decoders / seq2seq — train the student to maximise likelihood of the teacher’s argmax decoding instead of the gold reference (Kim & Rush 2016 “Sequence-Level Knowledge Distillation”).
  • MiniLLM (Gu et al. 2024) — reverse-KL objective (\(\text{KL}(p_s \| p_t)\) instead of forward-KL) — discourages the student from spreading probability over modes the teacher doesn’t cover. Crucial for free-form generation.
  • On-policy distillation / GKD (Agarwal et al. 2023) — sample from the student during training, then compute teacher’s logits on those student samples — closes the train/test mismatch.
  • Task-specific distillation — for a tiny domain-specific bot, distill GPT-4’s responses on your domain into a 1B student. The DPO-style pipeline.

MODULE 29: TRANSFORMER ARCHITECTURE TWEAKS FOR MEMORY-CONSTRAINED ENVIRONMENTS

A grab-bag of tricks the assessment may quiz on:

  • Weight tying between input and output embeddings — halves the embedding parameters.
  • ALiBi positional bias instead of learned positional embeddings — no positional parameters at all.
  • SwiGLU activations are slightly more expensive than ReLU but yield more capacity per param; many small LMs use them.
  • RMSNorm instead of LayerNorm — drops the mean centering and bias; ~5–10% faster, same quality.
  • Parallel attention/FFN (PaLM-style) — compute attention and MLP in parallel from the same input, sum at the end. Better for pipeline parallelism.
  • Sliding-window attention (Mistral) — each token attends to only a fixed window of recent tokens; combined with cached sink tokens for long context.
  • MoE (Switch Transformer, Mixtral 8×7B) — only \(k\) of \(N\) experts active per token. 4× the parameters for the same FLOPs.
  • Hybrid SSM-Transformer (Mamba, Jamba, RWKV) — sequence mixing with linear-time recurrences instead of \(O(N^2)\) attention.

MODULE 30: CACHING REPEATED INFERENCE

In production, many prompts are near-duplicates (e.g. system prompt + user query templates). Three caching levels:

  1. Exact prompt cache — keyed on hash of the full prompt; return the cached completion. Cheapest, brittle.
  2. Prefix KV cache — cache the KV state at the end of the system prompt; new requests reuse it (PagedAttention page sharing).
  3. Semantic cache — embed the prompt, look up nearest-neighbour in a vector index; if cosine similarity > threshold, return the cached response. Used by LangChain, GPTCache.

Each level trades cache hit rate for fidelity.

MODULE 34: A SUMMARY MENTAL MODEL

To master this domain, hold the following five questions in your head whenever you look at a model:

  1. What number format does the target hardware actually execute? (Edge TPU = INT8 only; Hopper = FP8; CPU = FP32/FP16/INT8/BF16.)
  2. Is the workload compute-bound or memory-bound? (Conv on big batches = compute; LLM decode = memory; small CNN on phone = mixed.)
  3. What does the search space of mappable architectures look like? (Are you bound to depthwise separable conv? Do you need to use specific block shapes?)
  4. Where do the bytes actually move? (DRAM ↔︎ SRAM ↔︎ register; if you can’t draw this you can’t optimize this.)
  5. What is the latency, energy, and accuracy SLA? (One of these is always your binding constraint; identify which.)

APPENDIX A: A WORKED INT8 PTQ EXAMPLE BY HAND

Suppose a weight row \(w = [-1.2, 0.4, 2.7, -3.1, 0.0]\) to be quantized to symmetric INT8.

  1. \(\max |w| = 3.1\).
  2. Scale \(s = 3.1 / 127 \approx 0.02441\).
  3. \(q = \text{round}(w/s) = [-49, 16, 111, -127, 0]\).
  4. Dequant \(\hat{w} = s \cdot q = [-1.196, 0.391, 2.710, -3.100, 0.000]\).
  5. Quantization error vector: \([-0.004, 0.009, -0.010, 0.000, 0.000]\) — MSE = \(4.6\mathrm{e}{-5}\).

Now repeat with per-channel asymmetric uint8 on the activation row \(x = [0.0, 5.0, 0.2, 1.4, 0.8]\): 1. \(\alpha = 0, \beta = 5.0\). 2. \(s = 5.0 / 255 \approx 0.0196\), \(z = 0\). 3. \(q = [0, 255, 10, 71, 41]\). 4. Dequant \(\hat{x} = [0, 5.0, 0.196, 1.392, 0.804]\).

This is what every Q/DQ node in an ONNX graph is doing per element.

APPENDIX D: DERIVATION — WHY \(T^2\) IN HINTON DISTILLATION

Let \(z_i\) be the teacher logits and \(v_i\) the student logits. With temperature \(T\):

\[ p_i = \frac{\exp(z_i/T)}{\sum_j \exp(z_j/T)}, \quad q_i = \frac{\exp(v_i/T)}{\sum_j \exp(v_j/T)}. \]

The cross-entropy loss between \(p\) and \(q\) (a function of \(v\)):

\[ \mathcal{L}_{\text{soft}}(v) = -\sum_i p_i \log q_i. \]

Its gradient with respect to \(v_k\) is:

\[ \frac{\partial \mathcal{L}_{\text{soft}}}{\partial v_k} = \frac{1}{T}(q_k - p_k). \]

If \(z_i\) and \(v_i\) are both near zero (small logits, the high-\(T\) limit), expanding \(q_k - p_k\) to first order:

\[ q_k - p_k \approx \frac{1}{T}\Big( (v_k - \bar v) - (z_k - \bar z) \Big) \quad \implies \quad \frac{\partial \mathcal{L}_{\text{soft}}}{\partial v_k} \approx \frac{1}{T^2}(v_k - z_k - \text{mean shift}). \]

So the gradient magnitude scales as \(1/T^2\). To make the soft loss gradient comparable in scale to the hard cross-entropy gradient, multiply \(\mathcal{L}_{\text{soft}}\) by \(T^2\):

\[ \boxed{\mathcal{L}_{\text{KD}} = \alpha T^2 \, \text{KL}(p \| q) + (1-\alpha) \, \mathcal{L}_{\text{CE}}.} \]

This is the most-asked piece of distillation math.

APPENDIX E: DERIVATION — STE GRADIENT FOR FAKE QUANT

Let \(Q(x) = s \cdot \text{round}(x/s)\). Then \(Q\) is piecewise constant; its true derivative is zero almost everywhere and undefined at the breakpoints \(x = s(k+1/2)\). The Straight-Through Estimator simply pretends the derivative of round is the identity within the representable range:

\[ \frac{\partial Q(x)}{\partial x} \approx \mathbf{1}\big[ q_{\min} \le \text{round}(x/s) \le q_{\max} \big]. \]

Outside the clipping range, the gradient is zero (the rounded value is saturated). Inside, the gradient is 1. The bias this introduces is bounded:

\[ \big|\mathbb{E}[\text{STE grad}] - \nabla_x \mathcal{L}(\hat{x})\big| \le \mathcal{O}(s), \]

so as scale \(s \to 0\) (i.e. more bits), the STE estimator becomes unbiased. In practice, networks happily train with STE down to 2–4 bit precision.

APPENDIX F: DERIVATION — GPTQ COLUMN UPDATE

Setup: layer output is \(Y = X W\) with \(W \in \mathbb{R}^{d_{in} \times d_{out}}\), \(X \in \mathbb{R}^{N \times d_{in}}\). We process rows of \(W\) independently because columns of \(Y\) are independent. Within one row \(w \in \mathbb{R}^{d_{in}}\), the reconstruction loss is:

\[ \mathcal{L}(w) = \|X w - X w^* \|_2^2, \]

where \(w^*\) is the original FP32 row and \(w\) is the to-be-quantized row. The Hessian is \(H = 2 X^\top X\) which is shared across all rows. Quantizing element \(w_q\) to \(\hat{w}_q\) produces a residual \(\delta = w_q - \hat{w}_q\), which we want to compensate by perturbing the remaining elements:

\[ \Delta w_{-q} = - \delta \, \frac{[H^{-1}]_{-q,\,q}}{[H^{-1}]_{qq}}. \]

(Standard OBS update.) Choosing to quantize in left-to-right column order, GPTQ Cholesky-decomposes \(H = LL^\top\) once and uses the lower triangle to do the updates in \(O(d_{in}^2)\) instead of the OBQ greedy \(O(d_{in}^3)\). The block-128 trick:

for block in blocks_of_128_cols:
    for q in block:
        quantize w[q]; compute delta; update w[block beyond q] densely
    update w[future blocks] in batch using cached factorisation

That’s the whole algorithm.

CLOSING — HOW THIS ALL HANGS TOGETHER

The four pillars of neural acceleration — quantization, pruning, distillation, NAS — are not independent. They compose:

  • Quantize → distill (you can distill from an FP32 teacher into an INT8 student).
  • Prune → quantize (Han et al.’s Deep Compression).
  • NAS → quantize (Once-for-All searches over candidates and per-layer bit widths jointly).
  • Distill → prune (TinyBERT-style two-stage).

The compiler then takes whatever lossy small model you’ve produced and squeezes the hardware-software boundary one more time: fuses ops, picks the right layouts, packs memory, plans the parallelism — and emits a sequence of native instructions for the target accelerator.

Underlying all of this is one single principle, make every byte and every FLOP count, because the cost of computation is no longer transistors; it is data movement. The neural accelerator engineer’s job is to move the least number of bytes per useful prediction. Quantization shrinks each byte; pruning skips bytes; distillation builds a smaller model that needs fewer bytes; NAS finds the architecture that hits the byte budget; the compiler lays out the bytes optimally; the parallelism strategy distributes the bytes across devices. Master this one principle and the field’s bewildering inventory of techniques becomes a coherent toolbox.
