MLOps Fundamentals — From the ML Lifecycle to Resilient Production Systems

This is an exhaustive, expert-level study guide for MLOps Fundamentals: integrating the ML lifecycle with reproducibility, automating CI/CD, deploying scalable inference, and maintaining resilient operations in production and constrained environments. Sections are intentionally long-form: prose, tables, math, and runnable code. References inline are to Sculley et al. (“Hidden Technical Debt in ML Systems,” NeurIPS 2015), Zinkevich (“Rules of Machine Learning,” Google), Google’s “MLOps: Continuous delivery and automation pipelines in machine learning” whitepaper and “Practitioners guide to MLOps”, Chip Huyen’s Designing Machine Learning Systems, Burkov’s Machine Learning Engineering, Eugene Yan (eugeneyan.com), Made With ML (Goku Mohandas), Lilian Weng’s Lil’Log, Breck et al. (“ML Test Score”), Gama et al. (concept drift survey), and the official docs for MLflow, Kubeflow, TFX, KServe, BentoML, EvidentlyAI, Great Expectations, and DVC.

PART 1: THE ML LIFECYCLE AND THE CASE FOR MLOps

MODULE 1: THE ML LIFECYCLE

1.1 What MLOps Is (And Is Not)

MLOps is the discipline of applying DevOps principles to machine learning systems, extended to handle the unique artifacts of ML: data and models. The goal is to reliably and efficiently move ML from research to production, then keep it healthy. Google’s whitepaper defines MLOps as the union of ML system development (Dev) and ML system operations (Ops), with continuous integration (CI), continuous delivery (CD), and continuous training (CT) as core practices.

  • MLOps is not a single tool; it is a set of practices, patterns, and an organizational culture.
  • MLOps is not DevOps with a model file attached; ML systems have additional moving parts (data, features, labels, hardware, drift) that demand new pipelines and tests.
  • MLOps is not optional: without it, ML projects accumulate technical debt at compounding rates (Sculley et al., 2015).

Three foundational dimensions of MLOps (Burkov, Huyen): 1. Code lifecycle: the application, training scripts, feature transforms. 2. Data lifecycle: ingestion, validation, transformation, versioning, lineage. 3. Model lifecycle: experimentation, training, evaluation, registry, deployment, monitoring, retraining.

All three must be versioned, testable, observable, and automatable.

1.2 The Full ML Lifecycle

A canonical end-to-end pipeline (Huyen 2022; Made With ML):

  1. Problem framing — define business objective, success metrics (e.g., revenue lift, F1, AUC, latency budget), constraints (compliance, fairness, fallback behavior), feasibility study. Decide whether ML is needed at all (heuristics often win — see Zinkevich’s Rule #1: don’t be afraid to launch a product without ML).
  2. Data collection / ingestion — source identification, data contracts with upstream teams, schema definition, sampling strategy, label collection (in-product, crowdsourcing, weak supervision, semi-supervised).
  3. Data labeling — annotation guidelines, inter-annotator agreement (Cohen’s \(\kappa\), Krippendorff’s \(\alpha\)), active learning, label noise audits.
  4. EDA (Exploratory Data Analysis) — distributions, missingness patterns, leakage hunts, outliers, segment imbalance.
  5. Feature engineering — transformations, embeddings, aggregations, feature stores (point-in-time correctness).
  6. Model development — algorithm choice, baselines (always have a non-ML baseline), iteration.
  7. Training — hyperparameter search, distributed training, checkpoints, reproducibility.
  8. Evaluation — offline metrics, slice-based evaluation, behavioral tests, fairness audits, calibration.
  9. Validation / Pre-deployment — schema validation, ML Test Score (Breck et al.), invariance and directional expectation tests (Ribeiro et al., CheckList).
  10. Deployment — containerization, packaging, deployment strategy (shadow, canary, blue/green, A/B).
  11. Monitoring — performance, drift, throughput, latency, cost, fairness over time.
  12. Continuous training / retraining — triggered by drift, performance drop, schedule, or new data.
  13. Decommissioning — sunset old versions, archive, document residual risks.

The loop is never one-shot; production ML is a perpetual cycle. Huyen calls this the iterative process of ML systems.

1.3 CRISP-DM, TDSP, and MLOps

Framework Origin Core Phases Where MLOps fits
CRISP-DM (Cross-Industry Standard Process for Data Mining, 1999) SPSS / DaimlerChrysler Business Understanding → Data Understanding → Data Preparation → Modeling → Evaluation → Deployment MLOps automates the Deployment and Monitoring → back to Modeling arcs
TDSP (Team Data Science Process) Microsoft Business Understanding → Data Acquisition & Understanding → Modeling → Deployment → Customer Acceptance MLOps templatizes the artifacts and the handoffs between roles

CRISP-DM is process-centric; MLOps is system-centric. Modern teams typically wrap CRISP-DM phases with MLOps automation: every CRISP-DM transition (e.g., Modeling → Deployment) becomes a CI/CD pipeline.

MODULE 2: WHY MLOPS — TECHNICAL DEBT IN ML SYSTEMS

2.1 Sculley et al., “Hidden Technical Debt in ML Systems” (NeurIPS 2015)

The single most-cited paper in MLOps. Its central claim: only a tiny fraction of a production ML system is the ML code itself; the surrounding infrastructure (configuration, data collection, feature extraction, data verification, machine resource management, analysis tools, process management tools, serving infrastructure, monitoring) dwarfs it. This is often visualized as the “small box, huge box” diagram.

The paper enumerates the unique technical debts of ML:

1. Entanglement / CACE (Changing Anything Changes Everything) — ML systems mix signals; tweaking any input, hyperparameter, or sampling decision can shift model behavior in non-local ways. Mitigations: isolate models for sub-problems, serve ensembles, monitor end-to-end.

2. Correction cascades — building model B on top of model A’s outputs (and C on B…) creates fragile chains of dependencies. Mitigations: revisit the underlying model rather than band-aiding cascades.

3. Undeclared consumers — downstream systems silently consume predictions, creating hidden feedback loops. Mitigations: access controls, service mesh, contracts.

4. Data dependencies cost more than code dependencies. - Unstable data dependencies — input signals that change underneath you (a partner team rescales a feature, breaking your model). - Underutilized data dependencies — legacy features that contribute little but remain wired in. - Mitigations: data versioning, lineage, schema enforcement, periodic feature ablation audits.

5. Feedback loops — the model’s predictions affect future training data (e.g., a recommender’s choices shape what users see and click). - Direct feedback loops (model influences its own training data) and hidden feedback loops (two models in the same product interact through user behavior). - Mitigations: holdouts, exploration policies, off-policy evaluation, ghost users.

6. ML system anti-patterns: - Glue code — vast amounts of code transforming data into the shapes a particular framework wants. - Pipeline jungles — a sprawl of scrape/clean/join scripts that grew organically. - Dead experimental code paths — if experiment_X: ... branches that never get cleaned up. - Abstraction debt — ML frameworks lacking strong abstractions for relational data, stream/batch unification, feature stores. - Common smells: plain-old-data types everywhere, multi-language workflows (Python ↔︎ Scala ↔︎ C++) with brittle interfaces, prototype-in-prod.

7. Configuration debt — production ML systems have hundreds of configuration knobs (feature flags, schedule cron, sampling rates, hyperparams). Bad configs cause silent failures. Mitigations: config as code, schema validation, diffs in code review, defaults that fail loudly.

8. Dealing with the real world — fixed thresholds in dynamic systems, monitoring gaps, reproducibility (e.g., non-deterministic training), cultural debt (researchers vs engineers).

These debts compound. MLOps is the systematic repayment plan.

2.2 Zinkevich, “Rules of Machine Learning” (Google)

A practitioner manifesto. Key rules to internalize:

  • Rule #1: Don’t be afraid to launch a product without machine learning. Heuristics first.
  • Rule #2: Make metrics design and implementation a priority. Track everything before you decide what matters.
  • Rule #4: Keep the first model simple and get the infrastructure right. Infra is the hard part.
  • Rule #14: Starting with an interpretable model makes debugging easier.
  • Rule #16: Plan to launch and iterate. Three reasons to launch a new model: new features, retuning regularization, new objective.
  • Rule #29: The best way to make sure that you train like you serve is to save the set of features used at serving time, then pipe those features to a log to use them at training time — i.e., training/serving skew is the #1 silent killer; close the loop with feature logging.
  • Rule #31: Beware that if you join data from a table at training and serving time, the data in the table may change — point-in-time correctness.

These rules ground MLOps in lived experience.

2.3 Why MLOps: Reliability and Efficiency

The case for MLOps reduces to two business outcomes:

Driver What it Looks Like Without MLOps With MLOps
Reliability Models silently degrade; outages caused by data schema changes; no rollback path; no on-call playbook for ML; legal/PR risk from biased or stale predictions Drift monitoring; automated retraining; canary rollouts; versioned data, code, models; reproducible builds; SLO/SLI dashboards
Efficiency Weeks to deploy a model; experiments are not reproducible; one engineer is the only person who can ship; >50% of data scientists’ time on infra plumbing Hours to deploy; reproducible experiments; self-service platform; data scientists ship; on-call burden distributed
Velocity Each new model requires bespoke deployment; technical debt slows future work Templates, golden paths; parallel teams ship safely
Risk / Compliance No audit trail; no model lineage; manual reviews Model cards, data sheets, lineage graphs, signed artifacts
Cost Over-provisioned GPUs; orphaned endpoints; duplicate data pipelines Autoscaling, spot/preemptible, fractional GPUs, multi-model endpoints

Huyen’s framing: MLOps is the practice of treating ML as software, plus the unique challenges of data and models.

3.1 Modular Code (Notebooks Are Not Production)

Jupyter notebooks are excellent for exploration; they are not a production artifact. Common anti-pattern: deploying a .ipynb as the prediction service.

Refactor checklist: - Extract src/ package with sub-modules: data/, features/, models/, train/, predict/, evaluation/, utils/. - Each function has a single responsibility, type hints, and a docstring. - Notebooks call into src/, never duplicate code. - Use cookiecutter-data-science or cookiecutter-poetry to bootstrap structure.

3.2 Type Hints and Static Analysis

  • Type hints clarify contracts and enable static checking.
from __future__ import annotations
import numpy as np
import pandas as pd

def featurize(df: pd.DataFrame, *, cols: list[str]) -> np.ndarray:
    """Convert selected columns to a float32 matrix.

    Raises:
        KeyError: if any column is missing.
    """
    missing = [c for c in cols if c not in df.columns]
    if missing:
        raise KeyError(f"Missing columns: {missing}")
    return df[cols].to_numpy(dtype=np.float32)

pydantic for stricter validation (models for configs, environment variables, payloads):

from pydantic import BaseModel, Field, field_validator

class TrainConfig(BaseModel):
    lr: float = Field(gt=0, lt=1)
    batch_size: int = Field(ge=1)
    model_name: str

    @field_validator("model_name")
    @classmethod
    def must_be_known(cls, v: str) -> str:
        if v not in {"resnet50", "vit_b_16"}:
            raise ValueError("unknown model")
        return v

12-factor app principles apply: - Config in the environment, not in code. - One codebase, many deploys. - Treat backing services (DB, feature store, blob store) as attached resources. - Stateless processes; persist state in backing services. - Logs as event streams to stdout. - Disposability (fast startup, graceful shutdown). - Dev/prod parity (same Docker image across envs).

3.6 Testing for ML Systems

Five layers of tests (per Breck et al., “ML Test Score”):

Layer What you test Tools
Unit tests Pure functions (featurizers, transforms) pytest, hypothesis
Data tests Schema, types, ranges, uniqueness, freshness Great Expectations, Pandera, TFDV
Model tests Prediction shapes, calibration, invariance, directional pytest, CheckList, Deepchecks
Integration tests Pipeline end-to-end on a tiny dataset pytest, Airflow’s test mode
Infrastructure tests Deployment manifests, IaC, resource limits kubeval, conftest, terratest

Example unit test for a featurizer:

import numpy as np
from my_project.features import featurize

def test_featurize_shape_and_dtype():
    df = pd.DataFrame({"a": [1, 2, 3], "b": [4.0, 5.0, 6.0]})
    X = featurize(df, cols=["a", "b"])
    assert X.shape == (3, 2)
    assert X.dtype == np.float32

def test_featurize_missing_column_raises():
    with pytest.raises(KeyError, match="b"):
        featurize(pd.DataFrame({"a": [1]}), cols=["a", "b"])

Example model behavioral test (Ribeiro et al., CheckList):

def test_invariance_to_capitalization():

4.1 Git Practices

  • Trunk-based development with short-lived branches.
  • Conventional Commits: feat:, fix:, refactor:, chore:, docs:, test:.
  • PR templates with: motivation, summary, metrics impact, rollback plan, model card link.
  • CODEOWNERS for ML-critical paths (models, schemas).
  • Tagged releases with semver (e.g., model-v1.4.0); align with Model Registry tags.

4.2 Code Review for ML

A reviewer checklist that catches ML-specific bugs:

  • Is randomness seeded? Is non-determinism documented?
  • Are train/val/test splits time-aware where appropriate? Is there leakage?
  • Is the feature computation logic identical at train and serve time?
  • Are there fairness slices in the eval?
  • Are metrics reported with confidence intervals (bootstrap)?
  • Are model artifacts logged with a version and a config hash?
  • Are external dependencies (data tables, APIs) versioned/pinned?

PART 3: REPRODUCIBILITY, DATA/MODEL VERSIONING, EXPERIMENT TRACKING

MODULE 5: REPRODUCIBILITY

5.1 The Five Pillars of Reproducibility

A run is reproducible only if all five are pinned:

  1. Code — git SHA, plus dependency lock file.
  2. Data — content hash of the exact training dataset (DVC, lakeFS, Delta Lake versioning).
  3. Environment — Docker image digest (sha256:...), CUDA / cuDNN versions.
  4. Configuration — full Hydra/YAML/json with overrides recorded.
  5. Randomness — all seeds set deterministically; non-determinism explicitly disabled where possible.

Add a sixth in regulated contexts: hardware (GPU model, microcode), since some kernels are non-deterministic across architectures.

5.2 Seeding Everything

import os, random, numpy as np, torch

def seed_everything(seed: int = 42) -> None:
    os.environ["PYTHONHASHSEED"] = str(seed)
    random.seed(seed)
    np.random.seed(seed)
    torch.manual_seed(seed)
    torch.cuda.manual_seed_all(seed)
    torch.backends.cudnn.deterministic = True
    torch.backends.cudnn.benchmark = False
    # Strict determinism (PyTorch ≥ 1.8); requires CUBLAS_WORKSPACE_CONFIG
    os.environ.setdefault("CUBLAS_WORKSPACE_CONFIG", ":4096:8")
    torch.use_deterministic_algorithms(True, warn_only=True)

Caveats: - Atomic reductions on GPUs are inherently non-deterministic in some ops. - Multi-GPU + NCCL all-reduce order may differ across runs. - TF32 on Ampere/Hopper changes precision; pin it (torch.backends.cuda.matmul.allow_tf32).

5.3 Reproducible Data Splits

  • Use deterministic hash-based splits: hash(record_id) % 100 < 80 → train.
  • For time series: time-based (prefix train, suffix test); never random shuffle.
  • For grouped data (users, sessions): split by group to prevent leakage.
  • Record the exact split indices as an artifact (DVC params.yaml or MLflow artifact).

PART 4: CI FOR ML — DATA TESTS, MODEL TESTS, SCHEMA VALIDATION

MODULE 9: WHAT “CI” MEANS FOR ML

In classical software CI, the unit is a commit. In ML CI, the units are {commit, data version, config}. A green CI must validate all three.

9.1 The ML CI Pipeline

A typical pipeline triggered by a PR:

  1. Static checks — ruff, mypy, black, pre-commit.
  2. Unit tests — pytest on src/.
  3. Schema/data tests — Great Expectations / Pandera on a sample.
  4. Small-batch training smoke test — train for 1 epoch on 1% of data; verify metrics on a frozen tiny holdout reach a sanity floor.
  5. Model behavioral tests — invariance, directional, fairness slices.
  6. Performance regression test — compare candidate vs current production model on a fixed eval set; block if regression > epsilon.
  7. Container build — Docker image build & vulnerability scan (Trivy/Snyk).
  8. Artifact promotion — push to registry with PR-tagged label.
  9. Pull-request report — CML posts metrics, plots, drift report, diff vs main as a PR comment.

PART 5: CD AND CONTINUOUS TRAINING — GOOGLE’S MLOPS MATURITY LEVELS

MODULE 10: LEVELS 0, 1, 2

The Google MLOps whitepaper defines three levels:

Level 0 — Manual

  • Data and model preparation are manual.
  • Disconnect between data science and operations (researchers hand a model over).
  • Infrequent releases (months).
  • No CI, no CD, no active performance monitoring.
  • This is where most ML projects begin.

Level 1 — ML pipeline automation (CT)

  • The training pipeline is automated and reproducible.
  • Continuous Training (CT): retrain automatically based on triggers (new data, drift, schedule).
  • Components: feature store, metadata store, model registry, orchestrator.
  • The same pipeline runs in dev, staging, prod (with different inputs).
  • Still manual deployment of new pipelines themselves (no CI/CD on pipeline code).

Level 2 — CI/CD pipeline automation

  • The pipelines themselves are subject to CI/CD.
  • Source code → tests → build → deploy of pipeline components.
  • Automated end-to-end: a commit can trigger pipeline re-build, model re-train, evaluation, and deployment, all gated by quality checks.

11.5 Idempotency, Retries, Backfills

  • Idempotent task: running it twice produces the same result. Achieved with deterministic outputs and write-then-rename to a content-hashed path.
  • Retries with exponential backoff for transient failures (network, spot eviction).
  • Backfills: ability to re-run historical runs against a fixed pipeline definition. Important for replaying after a bug fix.
  • SLAs: per-DAG / per-task deadlines; missed SLAs page on-call.

MODULE 15: RELEASE STRATEGIES (SHADOW, CANARY, BLUE/GREEN, A/B, BANDIT)

15.1 Shadow Deployment (a.k.a. “Dark Launch”)

  • The new model receives production traffic in parallel with the current model.
  • Its predictions are logged but not returned to the user.
  • Purpose: validate latency, throughput, and prediction distribution under real load before any user impact.
  • Costs: 2× inference compute during shadow.
@app.post("/predict")
async def predict(req: Req):
    primary_task = asyncio.create_task(primary_model.predict(req))
    asyncio.create_task(log_shadow(shadow_model, req))  # fire and forget
    return {"score": await primary_task}

  • Route a small fraction (e.g., 1%) of traffic to the new version; gradually ramp (1% → 5% → 25% → 100%) while monitoring SLOs.
  • Automated rollback if error rate or KPI drops past a threshold.

15.3 Blue/Green

  • Two full environments (blue=current, green=candidate). Swap a single router pointer to move traffic 100% in one step.
  • Fast rollback (flip back). Cost: ~2× while both are live.

15.4 A/B Testing

  • Randomly split users into A and B groups; measure a primary KPI; compute statistical significance.
  • Power analysis before launch:

\[ n \;\approx\; \frac{2\,(z_{1-\alpha/2} + z_{1-\beta})^2 \,\sigma^2}{\Delta^2} \]

where \(\Delta\) is the minimum detectable effect, \(\sigma^2\) the variance, \(\alpha\) the significance level, \(1-\beta\) the power.

  • Stratified randomization for balance across segments.
  • Interleaving (especially for ranking): show items from A and B in the same list; less noisy than naive A/B.
  • Pitfalls: peeking, multiple comparisons, network effects (recommenders), Simpson’s paradox.

15.5 Multi-Armed Bandit (MAB)

  • Adaptive: route more traffic to better-performing variants in real time.
  • Algorithms: \(\varepsilon\)-greedy, UCB, Thompson sampling.
  • Tradeoff: faster regret minimization, but harder to interpret as a clean experiment; statistical inference is non-trivial.

15.6 Feature Flags

  • Decouple release from deployment.
  • LaunchDarkly, Unleash, Split, ConfigCat.
  • Use cases: dark launches, kill switches, targeted rollouts (premium users only), regional rollouts.

16.5 Feature Stores

A feature store is a centralized store for production-grade features with two flavors:

  • Offline store (warehouse: BigQuery, Snowflake, Redshift, Parquet on S3) for training.

PART 9: MONITORING — PERFORMANCE, DRIFT, OBSERVABILITY

MODULE 17: MODEL PERFORMANCE MONITORING

17.1 The Ground-Truth Lag Problem

You usually do not have labels at serve time. Examples: - Fraud: confirmation can take weeks (chargebacks). - Loan default: months. - Click prediction: minutes. - Medical diagnosis: never.

This is the ground-truth lag. It forces you to monitor proxies in the meantime:

  • Input drift: change in input distribution → model is operating out-of-distribution.
  • Prediction drift: shift in the model’s output distribution.
  • Action drift: change in downstream user behavior (CTR, conversion).
  • Calibration drift: mismatch between predicted probability and observed frequency.

When labels finally arrive, recompute true performance and reconcile.

17.2 Per-Slice Monitoring

A global metric can hide subgroup regressions (Simpson’s paradox). Monitor:

  • Per-country, per-language, per-device.
  • Per-customer-tier.
  • Per-protected-class (gender, age band, race) — required for fairness audits.
  • Per-time-of-day, per-day-of-week.

Set per-slice SLOs where appropriate.

17.3 Online Evaluation

When labels arrive quickly (clicks): - Real-time A/B reports: lift over control. - Counterfactual evaluation: off-policy estimators (IPS, doubly robust) for systems with bandit/RL learning. - Holdouts: 1% of traffic permanently shown the previous model; the honest baseline.

MODULE 18: DATA AND CONCEPT DRIFT

18.1 Definitions

Let \(X\) be inputs and \(Y\) the label. Decompose the joint \(P(X, Y) = P(Y \mid X) \, P(X)\).

Type Formal Practical
Covariate shift \(P(X)\) changes, \(P(Y\mid X)\) does not New geography sends more young users; the relationship age→churn is unchanged
Prior shift / Label shift \(P(Y)\) changes, \(P(X\mid Y)\) does not Fraud rate spikes seasonally
Concept drift \(P(Y\mid X)\) changes What “spam” looks like evolves; same features, new mapping
Train/serve skew Training distribution ≠ serving distribution due to a pipeline bug, not the world changing Feature computed differently at train vs serve time

Gama et al. (“A Survey on Concept Drift Adaptation,” 2014) classify drift as: - Sudden (abrupt change point) - Gradual (slow transition between regimes) - Incremental (continuous slow change) - Recurring (seasonality)

18.2 Detecting Drift: Statistical Tests

Population Stability Index (PSI) — for binned distributions \(p\) (reference) and \(q\) (current):

\[ \text{PSI}(p, q) \;=\; \sum_{i=1}^{B} \bigl(p_i - q_i\bigr) \,\ln\!\frac{p_i}{q_i} \]

Rule of thumb: \(\text{PSI} < 0.1\) → no significant shift; \(0.1 \le \text{PSI} < 0.25\) → moderate; \(\ge 0.25\) → significant.

Kolmogorov–Smirnov (KS) test — for continuous 1D distributions:

\[ D \;=\; \sup_{x} \,\bigl| F_p(x) - F_q(x) \bigr| \]

Two-sample p-value flags drift. Sensitive to large samples (everything is “significant”); pair with effect size.

Chi-square test — for categorical features:

\[ \chi^2 \;=\; \sum_i \frac{(O_i - E_i)^2}{E_i} \]

where \(O_i\) are observed counts and \(E_i\) expected counts under the reference.

KL divergence:

\[ D_{\mathrm{KL}}(p \,\|\, q) \;=\; \sum_i p_i \,\ln\!\frac{p_i}{q_i} \]

Asymmetric; undefined when \(q_i = 0\). Use Jensen–Shannon divergence (symmetric, bounded):

\[ \mathrm{JSD}(p, q) \;=\; \tfrac{1}{2} D_{\mathrm{KL}}(p \,\|\, m) + \tfrac{1}{2} D_{\mathrm{KL}}(q \,\|\, m),\quad m=\tfrac{1}{2}(p+q) \]

Wasserstein (Earth Mover’s) distance — for 1D:

\[ W_1(p, q) \;=\; \int_{-\infty}^{\infty} \bigl|F_p(x) - F_q(x)\bigr|\,dx \]

Captures distance even when supports do not overlap; robust to small bin counts.

Maximum Mean Discrepancy (MMD) — kernel-based test for multivariate drift:

\[ \mathrm{MMD}^2(p, q) \;=\; \mathbb{E}_{x,x'\sim p}\!\bigl[k(x,x')\bigr] + \mathbb{E}_{y,y'\sim q}\!\bigl[k(y,y')\bigr] - 2\,\mathbb{E}_{x\sim p, y\sim q}\!\bigl[k(x,y)\bigr] \]

Page–Hinkley test / CUSUM — sequential change-point detection; flags when cumulative deviation from a baseline crosses a threshold.

Embedding drift: for unstructured data (text, images), monitor the distribution of embeddings (e.g., centroid distance, MMD over embeddings, classifier-based two-sample test).

18.3 Concept Drift Detectors

  • DDM (Drift Detection Method) (Gama et al., 2004): monitors classifier error rate; flags when it exceeds a threshold of \(\mu + 3\sigma\).
  • EDDM (Early DDM): monitors distance between misclassifications.
  • ADWIN (Adaptive Windowing) (Bifet & Gavaldà, 2007): adaptive sliding window with Hoeffding-bound statistical guarantees.
  • HDDM, Page–Hinkley, KSWIN are also common.

20.4 Pitfalls of Automated Retraining

  • Feedback loops: training on data shaped by your model’s actions biases the next model. Use exploration / holdouts to break the loop.
  • Label noise amplification: bad upstream labeling pipeline contaminates each retrain.
  • Catastrophic forgetting in online learning; consider replay buffers.
  • Schema changes that break the retrain pipeline silently (the model still trains, but on a different feature definition).

MODULE 21: PRODUCTION DEBUGGING & INCIDENT RESPONSE

21.1 Train–Serve Skew

The #1 silent killer in ML production.

Causes: - Feature transform implemented twice (training in pandas, serving in Java) and they drift. - Time-of-day issues (training pulled latest snapshot; serving uses point-in-time). - Encoder/categorical maps differ. - Different default values for missing features.

Mitigations: - One feature transform library (importable from both training and serving). - Feature store as the canonical source. - Schemas with TFDV/Pandera/GE enforced both sides. - Skew detector: log a sample of serving features and compare distributions to training nightly.

21.2 Data Leakage

The model “knows” the future or target in training: - Target encoding leaking the label. - Future timestamps in train. - Identifiers correlated with label (e.g., record_id that increments with positives).

Detection: ablations, suspiciously high metrics, partial dependence plots showing implausible monotonicity.

21.3 Rollback Strategies

  • Immediate rollback: blue/green flip; feature flag off; service mesh route weight = 0.
  • Gradual rollback: reverse canary (100% → 25% → 0).
  • Model rollback: registry alias prod → previous version; redeploy.
  • Data rollback: when an upstream data bug corrupted training data, roll back the dataset version and rebuild from a known-good snapshot.
  • Schema rollback: if a schema change broke things, revert both schema and consumers.

Every model should have a runbook describing: - How to detect failure - How to roll back in <5 minutes - Who to page - How to communicate to stakeholders

21.4 Dark Launches and Ghost Evaluation

  • Dark launch: new feature/model receives traffic but its output is invisible to users; ghost rank/score.
  • Replay: stream historical traffic against the candidate to surface latency and prediction-distribution issues offline.

21.5 Incident Postmortems

Blameless postmortems for ML incidents — beyond the usual: - Root cause in {code, data, config, model, env, infra}. - Detection time (when did monitoring see it?), mitigation time, resolution time. - Affected slices and customers. - Followups as tickets (monitoring gaps, new tests, runbook entries). - Knowledge sharing: add learnings to the ML test suite.

21.6 Chaos Engineering for ML (“Gameday”)

  • Inject corrupted features (NaNs, swapped categories) — does the model fail closed or fail open?
  • Kill the feature store — does serving degrade gracefully?
  • Throttle the model endpoint — does the upstream fall back?
  • Replay an old data snapshot to test drift alerts.

24.5 ML-Specific Attacks and Defenses

Attack Defense
Data poisoning Anomaly detection on training data; robust aggregation in federated learning; provenance of training data
Adversarial examples Adversarial training; certified defenses; input validation
Model inversion / membership inference Differential privacy; reduce overfitting; limit query rates
Model stealing Rate limiting; output truncation; watermarking
Backdoor / trojan Activation clustering; spectral signatures; trusted training data only
Prompt injection (LLMs) Input sanitization; structured outputs; tool/agent permissioning; allow-listing

24.6 Disaster Recovery & Reliability

  • Backups: data lake snapshots, registry exports, config in git.
  • RPO/RTO targets: e.g., RPO 1 h, RTO 4 h.
  • Multi-region: active-active or active-passive replication.
  • Failover drills quarterly.
  • Chaos testing: kill pods, throttle networks, simulate region loss.

MODULE 25: COST OPTIMIZATION

25.1 Compute

  • Spot / preemptible for training and (carefully) for inference behind a queue.
  • Fractional GPUs / MIG for low-utilization workloads.
  • Right-sizing: profile actual GPU util (DCGM / nvidia-smi), CPU, memory — eliminate waste.
  • Autoscale to zero for low-traffic endpoints (KServe, Cloud Run).
  • Batch where possible: schedule heavy compute in off-peak.

25.2 Inference

  • Quantize (INT8/FP16) and distill for production.
  • Multi-model endpoints to consolidate idle capacity.
  • Dynamic batching to maximize GPU utilization.
  • Cache embeddings/features/responses for hot paths.

25.3 Storage

  • Tiered storage (hot → warm → cold/glacier) for data lakes.
  • TTL on logs, predictions, and experiment artifacts.
  • Deduplicate datasets (content-hash storage in DVC/lakeFS).
  • Compress (zstd) and columnize (Parquet) — orders of magnitude wins.

25.4 FinOps for ML

  • Tag every resource (team, model, env).
  • Per-model cost dashboards: GPU-hours, GBs, $/1k predictions.
  • Set budgets and alerts; chargeback / showback to product teams.
  • Make engineers cost-aware via PR comments (“this image is 12 GB; smaller base?”).

APPENDIX F: KEY EQUATIONS (REVIEW SHEET)

  • PSI: \(\text{PSI} = \sum_i (p_i - q_i)\,\ln(p_i/q_i)\)
  • KS statistic: \(D = \sup_x |F_p(x) - F_q(x)|\)
  • Chi-square: \(\chi^2 = \sum_i (O_i - E_i)^2 / E_i\)
  • KL: \(D_{\mathrm{KL}}(p\,\|\,q) = \sum_i p_i \ln(p_i/q_i)\)
  • JS: \(\mathrm{JSD}(p,q) = \tfrac{1}{2}D_{\mathrm{KL}}(p\,\|\,m) + \tfrac{1}{2}D_{\mathrm{KL}}(q\,\|\,m), \; m=\tfrac{1}{2}(p+q)\)
  • Wasserstein-1 (1D): \(W_1(p,q) = \int |F_p(x) - F_q(x)|\,dx\)
  • MMD\(^2\): \(\mathbb{E}_p[k(x,x')] + \mathbb{E}_q[k(y,y')] - 2\mathbb{E}_{p,q}[k(x,y)]\)
  • A/B sample size: \(n \approx 2(z_{1-\alpha/2}+z_{1-\beta})^2 \sigma^2 / \Delta^2\)
  • Expected Calibration Error: \(\mathrm{ECE} = \sum_{b=1}^{B} \frac{|S_b|}{N}\,\bigl|\mathrm{acc}(S_b) - \mathrm{conf}(S_b)\bigr|\)
  • Brier score: \(\mathrm{BS} = \tfrac{1}{N}\sum_{i=1}^{N}(p_i - y_i)^2\)
  • Drift score (NannyML CBPE estimate of accuracy without labels) — see NannyML docs.
  • Error budget: \(\text{EB} = 1 - \text{SLO}\); burn rate \(= (\text{observed error rate}) / (\text{error budget})\).
