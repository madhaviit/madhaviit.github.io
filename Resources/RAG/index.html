Retrieval-Augmented Generation (RAG)

PART 1: MASTERING GENERAL CONCEPTS & TERMINOLOGY OF RAG

MODULE 1: INTRODUCING RAG

1.1 Define Retrieval-Augmented Generation (RAG) and Explain Its Purpose

Definition. Retrieval-Augmented Generation (RAG) is an architecture that augments a generative language model (LLM) with an external knowledge source at inference time. Instead of relying only on the parameters learned during pretraining, the model first retrieves relevant passages from an external corpus (documents, web pages, internal wikis, databases) and then generates an answer conditioned on both the user’s query and the retrieved context.

The term was coined in the 2020 paper “Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks” (Lewis et al., Facebook AI). The original architecture combined a dense retriever (DPR) with a seq2seq generator (BART). Today, “RAG” is used as an umbrella term for any pipeline of the form:

user query  →  retrieve relevant context  →  prompt LLM with (query + context)  →  generated answer

Purpose of RAG. RAG exists to decouple knowledge from language ability:

  • The LLM provides the language-modeling capability (understanding the query, composing fluent answers, reasoning).
  • The retrieval system provides the knowledge, kept outside the model weights so it can be updated, audited, scoped, and attributed without retraining.

Concretely, a RAG system is designed to:

  1. Ground generated answers in verifiable source documents (reducing hallucination).
  2. Inject knowledge the LLM never saw (private documents, post-training events, domain-specific corpora).
  3. Provide citations so users can audit each claim.
  4. Keep responses fresh by updating the index instead of retraining the model.
  5. Reduce cost compared with continually fine-tuning the LLM on new knowledge.

Mental model. Think of the LLM as an extremely well-spoken intern with no internet access, and the retriever as the librarian who hands them the three most relevant pages before they’re asked to write an answer.

1.2 Why RAG Enhances LLMs by Retrieving Relevant External Knowledge

A standalone LLM stores everything it “knows” inside its weights (parametric memory). This is powerful but has structural limitations that RAG directly addresses.

Problem 1 — Knowledge cutoff. An LLM is frozen at the date its training data was collected. Anything that happened after that date is invisible to the model. Retrieving from a continuously updated index gives access to current information without retraining.

Problem 2 — Hallucination. When asked about topics on which the LLM has weak or no memory, it fills the gap with plausible-sounding but fabricated content. By placing actual passages in the prompt and instructing the model to answer only from those passages, RAG converts open-ended generation into a constrained reading-comprehension task, which dramatically reduces fabrication.

Problem 3 — Lack of attribution. A pure LLM cannot reliably say where a fact came from. RAG returns explicit document IDs/URLs alongside each retrieved chunk, so the answer can carry citations.

Problem 4 — Private/proprietary knowledge. Most enterprise knowledge (internal wikis, contracts, support tickets, codebases) was never in the pretraining corpus. Putting that data behind a retriever is the cheapest, safest, most auditable way to make an LLM “know” it. Sensitive data stays in your store; only the relevant snippets enter the prompt.

Problem 5 — Long-tail facts. LLMs memorize popular facts well and rare facts poorly. Retrieval shines on the long tail because a document only needs to exist in the corpus — it does not need to have been seen many times during pretraining.

Problem 6 — Cost and freshness of fine-tuning. Continually fine-tuning a multi-billion-parameter model to absorb new knowledge is expensive, slow, brittle (catastrophic forgetting), and not auditable. Updating a vector index is cheap, instant, and revertible.

Problem 7 — Source control and governance. You can add, remove, or redact a document from the index in seconds. You cannot easily “remove” a fact from a model’s weights.

Why retrieval works at all (the mechanism in one line): the user’s query and each candidate document are projected into the same vector space; documents whose vectors lie closest to the query vector are semantically most related and are surfaced as context.

MODULE 2: COMPARING RAG AND TRADITIONAL LLMs

2.1 Differentiating RAG-based and Non-Retrieval-Based Generative Models

A non-retrieval-based (vanilla) LLM answers using only its parametric memory. A RAG-based system augments the same LLM with a retrieval step that fetches grounding context from an external store.

Dimension Vanilla / Non-retrieval LLM RAG-based System
Knowledge source Frozen in model weights (parametric) External, indexed corpus (non-parametric) + parametric base
Knowledge freshness Stuck at training cutoff Updated by re-indexing documents
Hallucination risk High, especially for long-tail / out-of-domain facts Lower; model is constrained to provided context
Attribution None (cannot cite sources reliably) Each chunk has an ID/URL → answer can carry citations
Updating cost Fine-tuning or full retraining Re-embed and upsert documents
Private data Must be fine-tuned in (risky, leaks possible) Stays in your store; only relevant chunks enter prompt
Inference cost per query One LLM call One (or more) embedding / retrieval call(s) + one LLM call with longer prompt
Latency Lower (single call) Higher (retrieval adds ~tens to hundreds of ms; longer prompt → more tokens to process)
Determinism / auditability Opaque Retrieval logs show exactly which chunks were used
Failure mode Confident wrong answers “I don’t know” when context is missing or irrelevant
Best for Style, reasoning, common knowledge Knowledge-intensive Q&A, enterprise search, support bots

Architectural difference in one line. Vanilla LLM = decoder over a query. RAG = retriever + decoder over (query, retrieved context).

Key vocabulary distinction.

  • Parametric memory — knowledge baked into model weights during training.
  • Non-parametric memory — knowledge stored outside the model, retrieved on demand. RAG combines both.

2.2 Identifying Scenarios Where RAG Is Beneficial Over Standard LLMs

Choose RAG when one or more of these apply. These are the situational stems most likely to appear on the assessment.

Choose RAG when…

  1. The knowledge changes often — news, prices, stock levels, schedules, regulations. The vanilla model goes stale; the index doesn’t.
  2. The knowledge is private/proprietary — internal documentation, customer records, contracts, code. You don’t want to fine-tune it into weights you may need to share or revoke.
  3. Citations are required — legal, medical, scientific, journalistic, compliance use cases where every claim must be traceable.
  4. The domain is long-tail or specialized — rare diseases, niche statutes, obscure APIs. The base model never saw enough of it to memorize.
  5. Hallucination is costly — customer-facing support, healthcare, finance. Grounding in retrieved text is the cheapest hallucination defense.
  6. You need fine-grained access control — different users should see different subsets of knowledge. Filter the retriever by metadata; the LLM never sees what the user shouldn’t.
  7. You have a large knowledge base that doesn’t fit in the context window — millions of pages of documentation. Retrieval picks the few relevant chunks per query.
  8. You need fast iteration on knowledge — content changes daily; retraining every day is infeasible.

Prefer a standard (non-RAG) LLM when…

  • The task is purely stylistic or compositional (rewriting, summarization of the input itself, creative writing) — no external facts are needed.
  • The required knowledge is general-purpose and stable, well within the model’s training distribution (e.g., “explain how a binary search works”).
  • Latency budgets are extremely tight and the answer doesn’t depend on any external corpus.
  • The corpus is so small that you can simply paste it into the prompt every time.

Prefer fine-tuning instead of (or in addition to) RAG when…

  • You need to change the style, format, or behaviour of the model, not its knowledge. RAG cannot teach a model to follow a new output schema or speak in a brand voice; fine-tuning can.
  • The knowledge is so deeply intertwined with reasoning that retrieval cannot surface the right chunk reliably.

In practice, mature systems often combine both: fine-tune for behaviour, RAG for knowledge.

MODULE 3: DESCRIBING CORE RAG COMPONENTS

3.1 Roles of the Retriever, Retriever Store, and Generator

A RAG system has three core components. Know each by name and by responsibility — the assessment may test this with match-the-pairs or fill-in-the-blank items.

1. Retriever

  • Role: Given a user query, return the top-k most relevant pieces of context from the store.
  • Responsibilities:
    • Convert the query into a representation comparable with stored documents (e.g., dense embedding, sparse term vector, or both).
    • Execute a similarity search against the store.
    • Apply any filters (metadata, recency, access control).
    • Return ranked candidates, optionally with scores.
  • Implementations: dense retrievers (BERT-family bi-encoders, OpenAI/Cohere/BGE embedders), sparse retrievers (BM25, TF-IDF), or hybrid combinations.

2. Retriever store (a.k.a. document store, vector store, index)

  • Role: Persist the corpus in a form the retriever can search efficiently.
  • Responsibilities:
    • Store the original document chunks (text + metadata such as source URL, author, timestamp, ACLs).
    • Store the indexed representation: an inverted index for sparse retrieval (BM25), and/or a vector index for dense retrieval (FAISS, Chroma, etc.).
    • Support insert / update / delete (so the corpus can evolve) and metadata filtering.
  • Examples: FAISS, Chroma, Elasticsearch (BM25), and similar.

3. Generator

  • Role: Produce the final natural-language answer using the user’s query and the retrieved context.
  • Responsibilities:
    • Accept a prompt of the form “Use the following context to answer the question … … ”.
    • Synthesize an answer grounded in that context.
    • Optionally produce citations referencing the chunks it used.
    • Refuse / say “I don’t know” when the context is insufficient (when prompted to do so).
  • Implementations: any instruction-following LLM (GPT-4 family, Claude, Llama, Mistral, Gemini, etc.).

3.2 How These Components Interact to Improve Response Quality

The three components form a pipeline. There are two phases — indexing (offline) and querying (online).

Indexing phase (offline, run when documents are added or changed):

  1. Load raw documents from sources (PDFs, HTML, Markdown, databases).
  2. Split each document into chunks of manageable size.
  3. Embed each chunk using an embedding model (for dense retrieval) and/or tokenize it for the inverted index (for sparse retrieval).
  4. Write (chunk_text, embedding, metadata) into the retriever store.

Query phase (online, runs per user request):

  1. User query arrives at the system.
  2. Retriever embeds (or tokenizes) the query and searches the store for the top-k most similar chunks.
  3. Retriever store returns those chunks plus their metadata.
  4. The pipeline constructs a prompt that contains the system instruction, the retrieved context, and the original query.
  5. Generator (the LLM) reads the prompt and produces an answer, optionally with citations.
  6. The answer is returned to the user.

How each component improves response quality:

  • The retriever narrows the LLM’s attention to a small slice of highly relevant text, so the model spends its context window on signal rather than noise.
  • The retriever store, by being external and updatable, keeps the knowledge fresh, scoped, and auditable.
  • The generator turns retrieved snippets — which on their own are unstructured passages — into a coherent, contextual, query-specific answer.

Why all three are necessary. Drop the retriever and you’re back to a vanilla LLM that hallucinates on long-tail questions. Drop the store and the retriever has nothing to search. Drop the generator and you’ve built a search engine, not a conversational answer.

Interaction diagram (text form):

                      ┌──────────────┐
   user query  ───▶   │  Retriever   │  ──── embed query
                      └──────┬───────┘
                             │ similarity search
                             ▼
                      ┌──────────────┐
                      │  Retriever   │  returns top-k chunks
                      │    Store     │     + metadata
                      └──────┬───────┘
                             │
                             ▼
                      ┌──────────────┐
   prompt builder ─▶  │  Generator   │ ──▶ grounded answer
   (query + chunks)   │    (LLM)     │      (+ citations)
                      └──────────────┘

MODULE 4: USING VECTOR DATABASES & EMBEDDINGS

4.1 How Embeddings Work and Their Role in Similarity-Based Retrieval

Definition. An embedding is a fixed-length numerical vector that represents the meaning of a piece of text (or image, audio, etc.) in a continuous vector space. Texts with similar meaning have vectors that lie close together; unrelated texts lie far apart.

A good embedding model has the property that:

similarity(embed("how to reset my password"),
           embed("steps for changing the login passcode"))   →  high

similarity(embed("how to reset my password"),
           embed("recipe for chocolate cake"))                →  low

How they are produced. An embedding model is a neural network — typically a transformer encoder (e.g., BERT-family) — trained with a contrastive objective so that semantically related text pairs end up nearby in the output vector space and unrelated pairs end up far apart. The encoder reads the text and emits one vector (commonly 384, 768, 1024, 1536, or 3072 dimensions).

Similarity metrics used at retrieval time:

  • Cosine similarity — (a·b) / (‖a‖·‖b‖). Measures angle, not magnitude. Range [−1, 1]; higher is more similar. The default for most text embeddings (which are usually L2-normalized).
  • Dot product — a·b. Equivalent to cosine when vectors are normalized; otherwise also reflects magnitude. Common with FAISS for speed.
  • Euclidean (L2) distance — ‖a − b‖. Smaller is more similar. Less common for text embeddings.

Role in similarity-based retrieval (the full mechanism):

  1. Offline: embed every chunk in the corpus and store the vectors in a vector index.
  2. Online: embed the incoming query with the same model.
  3. Use the similarity metric to find the top-k chunk vectors closest to the query vector.
  4. Return those chunks as the retrieval result.

Because the comparison is over vectors, the retriever finds passages that are semantically related even when they share no surface words with the query — for example, “how do I cancel my subscription” matches a chunk about “terminating membership”.

Important practical rule. Always embed queries with the same embedding model that was used to embed the documents; otherwise the vectors are not comparable.

4.2 Differentiating Various Embedding Models

Embedding models differ along several axes that matter for choosing the right one (this is also exactly what skill 9.1 will ask you to weigh).

Differentiating axes:

Axis What it means Why it matters
Modality Text-only, multimodal (text + image), code, etc. Must match your data.
Architecture Bi-encoder (one vector per text, fast) vs. cross-encoder (joint scoring of query+doc, slow but accurate) Bi-encoders for retrieval; cross-encoders for re-ranking.
Dimensionality 384, 768, 1024, 1536, 3072 … Higher dim = often higher accuracy, but more storage and slower search.
Context length Max tokens the model accepts Caps your chunk size.
Language coverage English-only vs. multilingual Critical for non-English corpora.
Domain specialization General web text vs. code / biomedical / legal Domain-specific models often beat general ones in that domain.
Provider / hosting API (OpenAI, Cohere, Voyage, Google) vs. self-hosted open-weight (BGE, E5, GTE, Instructor, SBERT/MiniLM) Trade off cost, latency, privacy.
Licensing Commercial-use? On-prem allowed? Matters for enterprise.

Two important model families:

  • Bi-encoder (a.k.a. dual encoder). Encodes query and document independently into vectors; similarity is computed by a cheap dot product. Used as the actual retriever because the document vectors can be pre-computed and indexed. Examples: SBERT/MiniLM, BGE, E5, GTE, OpenAI text-embedding-3-*, Cohere embed-*.
  • Cross-encoder. Takes (query, document) as a single input and outputs a relevance score directly. Far more accurate but cannot pre-index — it must run a forward pass per candidate. Used for re-ranking the top-N from a bi-encoder. Examples: cross-encoder/ms-marco-MiniLM, BGE-reranker, Cohere Rerank.

General vs. specialized. A general embedding model trained on broad web text is a safe default. For very technical corpora (legal contracts, medical literature, source code), a domain-specialized embedder — or a general embedder fine-tuned on your domain — typically gives meaningfully better retrieval (covered in 9.2).

One-line picker. Start with a strong general bi-encoder, profile recall on your own evaluation set, and only switch to a heavier / domain-specific / fine-tuned model when the numbers demand it.

MODULE 5: APPLYING INDEXING TECHNIQUES

5.1 Comparing BM25, FAISS, and Chroma for Indexing and Retrieving Documents

These three are the canonical names the syllabus calls out. They are not all the same kind of thing — knowing the category each falls into is the most common assessment trap.

BM25 — sparse, lexical, keyword-based.

  • Category: A ranking function (not a database) built on top of an inverted index of terms. Implementations live in Lucene/Elasticsearch/OpenSearch, Whoosh, rank_bm25, etc.
  • How it works: For each document it stores which terms occur and how often. At query time, it scores each document by Σ IDF(term) · TF-normalization(term, doc), rewarding documents that contain rare query terms many times while penalising over-long documents.
  • Strengths: No training; instantaneous to update; very fast; transparent (you can see exactly which terms matched); excellent on queries that rely on rare keywords, names, IDs, error codes.
  • Weaknesses: Purely lexical. A query about “car” will not match a document that only talks about “automobile”. No semantic understanding, no synonym handling, no paraphrase robustness.
  • Use it when: keywords matter (codes, legal terms, product SKUs); when you have no embeddings; or as the sparse half of a hybrid retriever.

FAISS — dense vector similarity search library (Facebook AI Similarity Search).

  • Category: A library (not a managed database) for fast similarity search over large vector collections. You give it vectors; you get back nearest neighbors.
  • How it works: Builds an Approximate Nearest Neighbor (ANN) index over your embeddings. Supports many index types: flat (exact), IVF (Inverted File — clusters vectors and only searches a few clusters per query), HNSW (Hierarchical Navigable Small World graph — graph-based ANN, very fast), PQ (Product Quantization — compresses vectors to save memory).
  • Strengths: Extremely fast on millions–billions of vectors; many index types to trade speed for accuracy; runs on CPU or GPU; battle-tested.
  • Weaknesses: Pure search library — no persistence, no metadata filtering, no REST API, no built-in document storage. You have to maintain the original text and metadata yourself.
  • Use it when: you need maximum-performance ANN search and are willing to wire up your own storage / serving layer.

Chroma — dense vector database.

  • Category: A vector database — an end-to-end system that stores embeddings together with the original text and metadata, and exposes a simple API.
  • How it works: Wraps an ANN backend (e.g., HNSW) and adds persistence, collections, metadata filtering, and a Python-first API. Includes a default embedding function so you can hand it raw text.
  • Strengths: Easy to get started; persists data; metadata filters; embedding function bundled; designed specifically for LLM/RAG workflows.
  • Weaknesses: Less raw throughput than tuned FAISS at very large scale; relatively young; fewer ANN knobs.
  • Use it when: you want a one-stop vector DB for a RAG prototype or small-to-medium production workload, with minimal plumbing.

Side-by-side comparison:

Aspect BM25 FAISS Chroma
What it is Lexical ranking function over an inverted index ANN search library over dense vectors Vector database
Representation Sparse term vectors (TF-IDF-like) Dense embeddings Dense embeddings (+ stored text/metadata)
Matches on Exact / stemmed keyword overlap Semantic similarity Semantic similarity
Handles synonyms / paraphrase No Yes (depends on embeddings) Yes (depends on embeddings)
Stores documents/metadata Yes (e.g., via Lucene) No (vectors only) Yes
Metadata filtering Yes Manual / not built-in Built-in
Update cost Cheap Cheap (or rebuild for some index types) Cheap
Best at Rare keywords, IDs, names Massive-scale ANN Easy-to-use RAG store
Typical role in RAG Sparse retriever / hybrid component Underlying ANN engine Drop-in vector store

One-line rule. BM25 = keywords. FAISS = a fast ANN engine you wrap yourself. Chroma = a friendly vector DB built on top of an ANN engine.

5.2 How Indexing Strategies Affect Retrieval Speed and Accuracy

Every index choice is a point on the speed / accuracy / memory triangle. You cannot maximize all three at once.

Flat (exact) index. - How: Stores all vectors as-is; at query time computes similarity to every vector (brute force). - Accuracy: Perfect (top-k is exact). - Speed: Slow — O(N · d) per query, where N = number of vectors, d = dimensionality. - Memory: Highest — every full vector stored. - Use when: Corpus is small (≲10⁵ vectors) or you need a ground-truth baseline to evaluate ANN indexes against.

IVF (Inverted File) index. - How: Cluster the vectors into nlist partitions (centroids learned via k-means). At query time, look up the query’s nearest nprobe centroids and search only inside those partitions. - Accuracy: Tunable via nprobe. Higher nprobe = more clusters searched = higher recall, slower. - Speed: Much faster than flat (sub-linear in N). - Memory: Same as flat unless combined with PQ. - Use when: Mid-to-large corpora where you can afford a small recall loss for a big speedup.

HNSW (Hierarchical Navigable Small World) graph index. - How: Builds a multi-layer graph where each node is a vector connected to its nearest neighbors. Search greedily walks the graph from the top layer down. - Accuracy: Very high recall, often >95% of exact at sensible parameters. - Speed: Extremely fast at query time — logarithmic-ish. - Memory: Higher than IVF because the graph itself is stored. - Use when: You want the best speed-vs-accuracy default. Used under the hood by Chroma and many vector DBs. - Knobs: M (graph degree — higher = more accurate but more memory) and efSearch (search effort at query time — higher = higher recall but slower).

Product Quantization (PQ) / IVF-PQ. - How: Compress each vector into a short code by quantizing sub-vectors against learned codebooks. Drastically reduces memory and speeds up distance computations. - Accuracy: Lossy — recall drops with stronger compression. - Speed: Very fast, especially when memory-bound. - Memory: The lowest — vectors can be 8–32× smaller. - Use when: Corpus is huge (10⁸+ vectors) and memory is the bottleneck.

Inverted index (BM25). - Speed: Very fast (postings lists per term). - Accuracy: Exact for the lexical scoring function. But its “accuracy” relative to user intent is limited because it only matches surface terms. - Memory: Modest — proportional to vocabulary × postings, far less than dense vectors.

How each strategy trades off — summary table:

Index Recall vs. exact Query speed Memory Update friendliness
Flat (brute force) 1.0 Slow Highest Trivial
IVF (flat) tunable, ~0.9–0.99 Fast Same as flat Re-train centroids occasionally
IVF-PQ lower (lossy) Very fast Very low Same
HNSW very high (>0.95) Very fast High (graph) Incremental inserts supported
BM25 (inverted) exact for lexical, but only lexical Very fast Low–medium Trivial inserts

Practical guidance:

  • Small corpus (a few thousand chunks): a flat index is fine; you’ll never notice the cost.
  • Medium corpus (10⁵–10⁷ chunks): HNSW (used by Chroma and most vector DBs) is the default sweet spot.
  • Huge corpus (10⁸+ chunks): IVF-PQ or similar memory-aware index; accept a modest recall hit for tractable memory.
  • Heavy keyword / ID lookups: keep BM25 alongside (hybrid — see 11.1).

The core trade-off in one sentence: moving from exact to approximate search buys you orders of magnitude in speed and memory at the cost of a small, controllable drop in recall.

MODULE 6: IDENTIFYING RAG PIPELINE TYPES

6.1 Identifying Different Retrieval Strategies

“Retrieval strategy” = how the retriever decides which chunks to return. The main families, ordered from simplest to most elaborate:

1. Sparse / lexical retrieval (e.g., BM25, TF-IDF). Matches on terms in the query and document. Excellent on rare keywords (names, codes, identifiers); blind to synonyms and paraphrases.

2. Dense / semantic retrieval (embedding-based). Both query and documents are embedded with the same model; nearest-neighbor search returns semantically related chunks. Handles paraphrase and synonyms; can miss exact-keyword matches that don’t show up clearly in vector space.

3. Hybrid retrieval (sparse + dense). Run both retrievers in parallel and merge their results — commonly by score fusion (e.g., weighted sum after normalisation, or Reciprocal Rank Fusion). Captures the strengths of both: rare-term precision from BM25, semantic recall from embeddings. The default “strong” baseline in modern RAG.

4. Retrieve-then-rerank (two-stage retrieval). First stage: a fast bi-encoder (and/or BM25) returns a generous top-N (say 50–100). Second stage: a slow but accurate cross-encoder re-scores those N candidates and keeps the top-k (say 5–10) for the LLM. Big quality win for small latency cost because the cross-encoder only runs on N items, not the full corpus.

5. Filtered / metadata-aware retrieval. Apply structured filters (date range, document type, language, tenant, ACL) before or alongside the similarity search, so the LLM only sees chunks the user is allowed to see and that are relevant in time/source.

6. Query transformation strategies. Modify the query before retrieval to improve recall: - Query rewriting — the LLM rewrites a messy user question into a cleaner search query. - Multi-query — the LLM generates several paraphrased queries; results from all are merged. - Step-back / decomposition — for complex questions, break the query into sub-questions and retrieve for each. - HyDE (Hypothetical Document Embeddings) — ask the LLM to draft a plausible answer first, then embed and retrieve using that draft.

7. Multi-hop / iterative retrieval. The system retrieves, reads, decides what’s still missing, formulates a follow-up query, and retrieves again — repeating until enough evidence is assembled. Needed for questions that require chaining facts across documents.

6.2 Selecting the Appropriate Retrieval Method Based on the Use Case

A short decision guide; the assessment likes situational questions of the form “Given this use case, which retrieval strategy fits best?”

Use case / signal Recommended strategy Why
Lookups dominated by exact codes, names, error IDs BM25 / sparse (or hybrid leaning on BM25) Keywords are the strongest signal; embeddings can dilute them.
Free-form natural-language questions over prose Dense / semantic Handles paraphrase, synonyms, conceptual matches.
Mixed traffic (some keyword, some natural language) Hybrid (BM25 + dense) Best general-purpose default.
Top-k must be tight and highly precise (long, expensive LLM call) Retrieve-then-rerank with a cross-encoder Cross-encoder cleans up the top of the list.
Multi-tenant, permissioned, or time-sensitive corpus Filtered retrieval (dense or hybrid + metadata filters) Enforces access control and recency at the retriever.
Vague or messy user queries Add query rewriting / multi-query / HyDE in front of any base retriever Improves recall on poorly formed queries.
Questions whose answer requires combining facts from multiple documents Multi-hop / iterative retrieval Single-shot retrieval cannot assemble the chain.
Tiny corpus (≲ a few hundred chunks) Just put it all in the prompt; or flat dense index Retrieval gymnastics aren’t needed.
Huge corpus (10⁸+ chunks) Dense (IVF-PQ / HNSW) + hybrid + rerank Need ANN for scale, BM25 for precision, rerank for quality.

Selection heuristic in three questions:

  1. Do my users search with keywords or with natural language? → leans BM25 vs dense.
  2. Is the top of the result list good enough as-is, or do I need to clean it up? → decides whether to add a re-ranker.
  3. Does a single retrieval round contain enough evidence, or do I need to chain? → decides single-shot vs multi-hop.

MODULE 7: EVALUATING RETRIEVAL PERFORMANCE

7.1 Defining Precision@K, Recall@K, MRR, and F1-score

For each query, you have a ground-truth set of relevant documents and a ranked list of K returned documents.

Let: - K = number of results examined. - relevant@K = number of returned items in the top-K that are actually relevant. - total_relevant = total number of relevant documents that exist for this query in the corpus.

Precision@K — Of the K results I returned, what fraction are actually relevant?

Precision@K = relevant@K / K
  • Measures how clean the top-K list is.
  • Range: [0, 1]. Higher is better.
  • Insensitive to documents you missed outside the top-K.
  • Use when the cost of showing irrelevant results is high (e.g., you can only feed 5 chunks to the LLM and noise hurts the answer).

Recall@K — Of all the relevant documents that exist, what fraction did I surface in the top-K?

Recall@K = relevant@K / total_relevant
  • Measures how complete the retrieval is.
  • Range: [0, 1]. Higher is better.
  • Insensitive to how many irrelevant items are mixed in.
  • Use when missing a relevant document is costly (e.g., legal discovery, compliance search, or any RAG step where a re-ranker / LLM downstream can ignore noise but cannot recover a missing fact).

MRR — Mean Reciprocal Rank. On average, how high in the ranking does the first relevant result appear?

For a single query whose first relevant result is at rank rᵢ, the reciprocal rank is 1/rᵢ (and 0 if no relevant result appears). Averaged over a set of Q queries:

MRR = (1/Q) · Σᵢ (1 / rᵢ)
  • Range: [0, 1]. 1.0 means the first hit is always relevant; 0.5 means it’s typically at rank 2; etc.
  • Cares only about the rank of the first relevant result, not how many relevant results follow.
  • Use when there is essentially one correct answer per query (factoid QA, “find the right document”) and where the user only looks at / the LLM only uses the very top result.

F1-score (in retrieval). Harmonic mean of precision and recall — a single number that punishes imbalance between the two.

F1 = 2 · (Precision · Recall) / (Precision + Recall)
  • Range: [0, 1]. Higher is better.
  • Variant most relevant here: F1@K uses Precision@K and Recall@K.
  • Useful when you want a single metric that balances both, and don’t want a system that wins on recall by returning the entire corpus or wins on precision by returning one obvious result.

Quick numeric example. Suppose 6 relevant documents exist for a query, you return K=5, and 3 of the 5 are relevant, with the first relevant one at rank 2.

  • Precision@5 = 3/5 = 0.60
  • Recall@5 = 3/6 = 0.50
  • Reciprocal Rank for this query = 1/2 = 0.50
  • F1@5 = 2 · 0.60 · 0.50 / (0.60 + 0.50) ≈ 0.545

7.2 Choosing the Right Metric Based on Task Requirements

Pick the metric whose definition matches the cost structure of your task.

Task / requirement Best primary metric Reasoning
Factoid QA — one document holds the answer; user only needs the first hit MRR Captures “did the right thing come up first?”
Top-k feeds an LLM with a tight context window — every wasted slot hurts the answer Precision@K Penalises irrelevant chunks crowding out good ones.
Legal / compliance / discovery — missing a relevant doc is unacceptable Recall@K (often with large K) Penalises gaps in coverage.
Balanced quality with no clear asymmetry between false positives and false negatives F1@K Single score balancing precision and recall.
Cost of false positives ≠ cost of false negatives Use Precision and Recall separately, not F1 F1 weights them equally; report both so the trade-off is visible.
Comparing two retrievers / picking K Sweep K and plot Precision@K, Recall@K, MRR Lets you see the actual trade-off curve, not just one point.

Picking K itself:

  • Small K (3–5) when downstream LLM context is tight or each chunk is expensive.
  • Larger K (20–100) when followed by a re-ranker that will trim the list.
  • For Recall@K specifically, K must be large enough that meaningful coverage is achievable.

End-to-end vs. retriever-only evaluation. Precision@K / Recall@K / MRR / F1 evaluate retrieval in isolation. A high-recall retriever can still feed a bad final answer if the LLM ignores the context. In practice, measure both: retrieval metrics for the retriever, plus answer-level metrics (exact match, F1 over answer tokens, or human/LLM grading) for the full system.

PART 2: IMPLEMENTING RAG IN PRACTICE

MODULE 8: BUILDING A RAG PIPELINE

8.1 Constructing a Basic RAG Pipeline Using LangChain, LlamaIndex, or Custom Retrieval Logic

A basic RAG pipeline always has the same six logical stages, regardless of which framework you use:

  1. Load documents from a source.
  2. Split them into chunks.
  3. Embed each chunk and store it in a vector store.
  4. Retrieve top-k chunks for an incoming query.
  5. Prompt the LLM with the query + retrieved context.
  6. Generate the answer.

Skeleton in LangChain (Python):

from langchain_community.document_loaders import TextLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain_community.vectorstores import Chroma
from langchain.chains import RetrievalQA

docs = TextLoader("docs/handbook.txt").load()

splitter = RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=50)
chunks = splitter.split_documents(docs)

vectorstore = Chroma.from_documents(
    documents=chunks,
    embedding=OpenAIEmbeddings(model="text-embedding-3-small"),
    persist_directory="./chroma_db",
)

retriever = vectorstore.as_retriever(search_kwargs={"k": 4})

qa = RetrievalQA.from_chain_type(
    llm=ChatOpenAI(model="gpt-4o-mini", temperature=0),
    retriever=retriever,
    return_source_documents=True,
)

answer = qa.invoke({"query": "What is the parental-leave policy?"})
print(answer["result"])

Skeleton in LlamaIndex (Python):

from llama_index.core import SimpleDirectoryReader, VectorStoreIndex, Settings
from llama_index.embeddings.openai import OpenAIEmbedding
from llama_index.llms.openai import OpenAI

Settings.embed_model = OpenAIEmbedding(model="text-embedding-3-small")
Settings.llm         = OpenAI(model="gpt-4o-mini", temperature=0)

documents = SimpleDirectoryReader("docs/").load_data()
index     = VectorStoreIndex.from_documents(documents)

query_engine = index.as_query_engine(similarity_top_k=4)
print(query_engine.query("What is the parental-leave policy?"))

Skeleton with custom retrieval logic (no framework):

import numpy as np
from openai import OpenAI

client = OpenAI()

def embed(text: str) -> np.ndarray:
    v = client.embeddings.create(model="text-embedding-3-small", input=text).data[0].embedding
    return np.array(v, dtype=np.float32)

def chunk(text: str, size: int = 500, overlap: int = 50) -> list[str]:
    out, i = [], 0
    while i < len(text):
        out.append(text[i : i + size])
        i += size - overlap
    return out

with open("docs/handbook.txt") as f:
    chunks = chunk(f.read())
chunk_vecs = np.stack([embed(c) for c in chunks])

def retrieve(query: str, k: int = 4) -> list[str]:
    q = embed(query)
    sims = chunk_vecs @ q / (np.linalg.norm(chunk_vecs, axis=1) * np.linalg.norm(q) + 1e-12)
    top = np.argsort(-sims)[:k]
    return [chunks[i] for i in top]

def answer(query: str) -> str:
    context = "\n\n".join(retrieve(query))
    prompt  = (
        "Answer the question using ONLY the context below. "
        "If the answer is not in the context, say you don't know.\n\n"
        f"CONTEXT:\n{context}\n\nQUESTION: {query}"
    )
    resp = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": prompt}],
        temperature=0,
    )
    return resp.choices[0].message.content

All three implementations execute the same six logical stages — only the syntactic wrapping differs.

8.2 Implementing a Simple End-to-End Retrieval and Generation Loop

The minimum viable RAG loop, broken into its ordered steps. Memorize the order — assessment items often ask you to reorder these.

1. INGEST
   1a. Load raw documents from source.
   1b. Clean / normalize text (remove boilerplate, fix encoding).
   1c. Split into chunks (size and overlap chosen — see Module 10).
   1d. Embed each chunk with the embedding model.
   1e. Upsert (chunk_text, embedding, metadata) into the vector store.

2. QUERY
   2a. User submits a question.
   2b. (Optional) Rewrite or expand the query.
   2c. Embed the query with the SAME embedding model.
   2d. Search the vector store for top-k nearest chunks.
   2e. (Optional) Apply metadata filters / re-rank.

3. GENERATE
   3a. Build a prompt: system instructions + retrieved context + user question.
   3b. Call the LLM.
   3c. Parse and return the answer (optionally with citations to chunk IDs).

Minimal end-to-end loop (framework-agnostic pseudocode):

def rag_answer(question: str) -> str:
    q_emb   = embed(question)
    chunks  = vector_store.search(q_emb, k=4)
    context = "\n\n".join(c.text for c in chunks)
    prompt  = build_prompt(question, context)
    return llm.generate(prompt)

Sample prompt template (a strong default):

You are a helpful assistant. Use ONLY the context below to answer the question.
If the answer is not in the context, reply: "I don't know based on the provided context."
Cite the source IDs in square brackets when you use information from a chunk.

CONTEXT:
[1] {chunk_1_text}
[2] {chunk_2_text}
[3] {chunk_3_text}

QUESTION: {user_question}

ANSWER:

What each step contributes to quality.

  • Cleaning prevents boilerplate (nav bars, footers) from drowning out real content.
  • Chunking decides how much of the surrounding context travels with each retrieved hit.
  • Same-model embedding for query and docs is mandatory; otherwise the vectors live in different spaces.
  • The “answer ONLY from context” instruction is what actually reduces hallucination at generation time.

8.3 Identifying Different Packages to Construct a RAG Pipeline

Know the major options at a glance — the assessment may test “which of these is a RAG framework?” style items.

Package Type What it gives you When to pick it

LlamaIndex Data-framework for LLMs (Python, TS) Strong document ingestion, indexing, query engines, response synthesizers Document-heavy RAG where ingestion and indexing are the bulk of the work

MODULE 9: SELECTING EMBEDDING MODEL

9.1 Choosing the Right Embedding Model Based on Accuracy, Cost, and Latency Trade-offs

There is no universally “best” embedding model — only the best one for your corpus, your latency budget, and your cost ceiling. Score each candidate along three axes.

Accuracy.

  • Measure on your own data, not just public leaderboards. Build a small evaluation set of (query, relevant chunk) pairs from your corpus and compute Recall@K / MRR per candidate model.
  • Larger / higher-dimensional models generally retrieve better, with diminishing returns.
  • Domain match matters more than raw size: a 384-dim model trained on code may beat a 1536-dim general model on code.
  • Multilingual matters if any meaningful share of queries or documents is non-English.

Cost.

  • Hosted (API) embeddings: charged per token embedded. Costs accumulate from two places — bulk re-embedding of the corpus (one-time but big) and per-query embedding (small but continuous).
  • Self-hosted (open-weight) embeddings: no per-token fee, but you pay for GPU/CPU, engineering, and ops.
  • Higher dimensions cost more to store (vector size × number of chunks) and more to search.
  • Re-ranking with a cross-encoder costs more per query than bi-encoder retrieval — budget for it explicitly.

Latency.

  • Per-query embedding latency: typically tens of ms for hosted APIs; sub-10 ms for small self-hosted models on a GPU.
  • Larger embedding dimension → slightly slower vector search.
  • If you re-rank with a cross-encoder, that step usually dominates per-query latency.
  • Index choice (Module 5) affects search latency more than embedding choice does.

Decision matrix:

Constraint dominates Reasonable default
Accuracy above all (research, evaluation, recall-critical legal) A large hosted embedder + cross-encoder reranker
Cost above all (huge corpus, tight budget) Compact open-weight bi-encoder (e.g., MiniLM-class, 384-dim), self-hosted
Latency above all (interactive UX, real-time) Small embedder + HNSW index + no reranker (or a small one)
Balanced production system Mid-size hosted embedder (e.g., 768/1024-dim) with rerank on top of a hybrid retriever

Practical workflow:

  1. Pick 3–5 candidate embedding models spanning sizes/providers.
  2. Embed a sample of your corpus and your eval queries with each.
  3. Measure Recall@K and MRR on your eval set.
  4. Measure per-query latency and project monthly cost at your traffic.
  5. Pick the smallest / cheapest model that hits your accuracy bar.

Anti-patterns.

  • Picking the model that tops a public leaderboard without measuring on your own data.
  • Switching embedding models without re-embedding the entire corpus (vectors from different models are not comparable).
  • Maxing out dimensionality “to be safe” and then complaining about cost and latency.

9.2 Comparing the Effectiveness of Pretrained vs Fine-tuned Embedding Models

Pretrained embedding model. A model trained by someone else (OpenAI, Cohere, BAAI, Microsoft, Google, etc.) on a large general-purpose corpus. You use it out of the box.

  • Pros: Zero training effort, immediate use, strong general performance, well-supported.
  • Cons: Trained on general web text — may underperform on highly specialized vocabulary (legal, biomedical, internal product jargon, code identifiers); cannot capture domain conventions it never saw.

Fine-tuned embedding model. A pretrained model further trained on your data — typically with a contrastive objective over (query, relevant_passage) pairs from your domain, often with hard negatives.

  • Pros: Aligns the vector space to your notion of similarity; usually meaningfully better Recall@K and MRR on the domain it was fine-tuned for; can specialise on terminology, document structure, and user query style.
  • Cons: Requires labeled (or weakly labeled) training pairs; engineering effort; risk of overfitting to a small dataset; risk of catastrophic forgetting on general queries; must be re-fine-tuned as the domain evolves.

Effectiveness comparison.

Aspect Pretrained Fine-tuned
Out-of-domain quality Good baseline Often worse if fine-tuned narrowly
In-domain quality Decent → poor on niches Often substantially better
Cost to obtain Free (or API fee) Engineering + compute + labels
Time to deploy Minutes Days to weeks
Risk of regression Low Real (overfit, forget)
Maintenance Vendor handles upgrades You handle drift and retraining

When to fine-tune (rules of thumb):

  • Your domain vocabulary is meaningfully different from general web text (legal, medical, internal product names, code).
  • You can assemble at least a few hundred to a few thousand (query, relevant_passage) pairs — even weak supervision (e.g., from search logs or LLM-generated questions) often suffices.
  • A strong pretrained baseline has already been measured on your eval set and falls short of the bar.
  • The retrieval quality bottleneck has been isolated to the embedder — not to chunking, not to ranking, not to the LLM.

When to stick with pretrained:

  • You don’t have evaluation data yet — fine-tuning without an eval set is flying blind.
  • Your domain is mainstream and the pretrained model already meets your bar.
  • The expected lift over the pretrained baseline is small relative to the engineering cost.

A common third path: keep a pretrained bi-encoder for first-stage retrieval (cheap, broad, low-maintenance) and add a fine-tuned cross-encoder re-ranker on top (focused, expensive per item but only run on the top-N).

MODULE 10: OPTIMIZING CHUNKING STRATEGIES

10.1 Differentiating Between Fixed-Size and Semantic Chunking Approaches

Fixed-size chunking.

  • Definition: Cut the text into chunks of a predetermined size — measured in characters, tokens, words, or sentences — usually with a fixed overlap between consecutive chunks.
  • How it works: Walk through the document; emit a chunk every N units, starting N − overlap units after the previous chunk’s start.
  • Pros: Trivial to implement, deterministic, very fast, predictable token cost per chunk.
  • Cons: Boundary-blind — happily cuts in the middle of a sentence, a code block, a table row, or a logical idea. The retrieved chunk may start mid-thought and end mid-thought.
  • Mitigations: Add overlap (commonly 10–20% of chunk size) so adjacent chunks share some text; prefer a recursive splitter that tries to break on paragraph → sentence → word boundaries first and only falls back to a hard cut.

Semantic chunking.

  • Definition: Cut the text where the meaning changes, not where a fixed counter rolls over. Boundaries fall between distinct ideas, sections, or topics.
  • How it works (common approach): Embed each sentence (or small unit); walk through the document, and start a new chunk wherever the cosine similarity between consecutive sentences drops below a threshold — i.e., the topic just shifted. Variations: split by document structure (Markdown headings, HTML sections), by speaker turns, or by an LLM that proposes semantic boundaries.
  • Pros: Each chunk is internally coherent, which improves retrieval precision (the chunk is “about one thing”) and improves the LLM’s ability to use it (no awkward mid-sentence starts).
  • Cons: More expensive (per-sentence embedding or LLM calls during ingestion); chunk sizes vary widely; harder to bound context-window cost per retrieved chunk.

Side-by-side comparison:

Aspect Fixed-size Semantic
Boundary rule Counter-based (chars/tokens) Meaning shift / structure
Chunk size Uniform Variable
Implementation cost Trivial Moderate to high (extra embeddings or LLM calls)
Risk of cutting mid-idea High Low
Retrieval precision Lower (noisier chunks) Higher
Best when Homogeneous prose, prototypes, tight budgets Heterogeneous documents (long PDFs, mixed headings, transcripts)

Middle-ground option: recursive structure-aware splitting. Splits first on document structure (headings, paragraphs, sentences) and only resorts to a hard character/token cut as a fallback. Cheap like fixed-size, much cleaner boundaries. Often the strongest default before reaching for truly semantic chunking.

10.2 Optimizing Chunk Size for Better Retrieval Accuracy While Minimizing Context-Window Usage

Chunk size is the single highest-leverage knob in a RAG pipeline. It directly trades retrieval precision against context completeness and token cost.

Effects of chunk size:

  • Small chunks (≈100–300 tokens).
    • Pros: Each chunk is tightly focused → higher Precision@K; cheaper context window per chunk; the LLM’s attention isn’t spread across irrelevant text.
    • Cons: Important context may be split across chunks; relevant information may sit just outside the retrieved chunk; the same topic gets fragmented into many similar chunks (hurts diversity).
  • Large chunks (≈1000–2000+ tokens).
    • Pros: Each chunk carries more surrounding context, so the LLM has more to work with; fewer chunks total in the store.
    • Cons: Lower Precision@K (the chunk also contains a lot of irrelevant text); fewer distinct chunks fit in the context window; embeddings of very long chunks are “averaged” and lose specificity, so retrieval relevance drops.
  • Overlap.
    • 10–20% overlap between adjacent chunks is the typical default; protects against information that would otherwise straddle a chunk boundary.
    • Too much overlap = duplicated content in the index → wasted storage and the retriever returning two near-duplicate chunks for the same query.

Context-window arithmetic to keep in mind.

tokens_used_per_query ≈ system_prompt + (k × chunk_size) + query + answer_buffer

So picking k and chunk_size is a joint decision: if your model has an 8k context and you reserve 1k for system + question + answer, you have ~7k tokens left for retrieved chunks. With chunk_size = 500 you can comfortably feed k = 10–12; with chunk_size = 1500 you can only fit k ≈ 4. The first option gives more diversity, the second gives more depth per chunk.

Optimization workflow:

  1. Build a small evaluation set: queries paired with the chunks (or documents) that contain the correct answer.
  2. Sweep a grid of chunk sizes (e.g., 128, 256, 512, 1024, 2048 tokens) × overlaps (0, 10%, 20%).
  3. For each combination, rebuild the index and measure Recall@K, Precision@K (or MRR) and the average tokens consumed per query.
  4. Plot accuracy vs. tokens-used; pick the smallest chunk size whose accuracy is within a small margin of the best.
  5. Re-check on a held-out query set so you’re not overfitting the eval queries.

Practical defaults (good starting points, not laws):

  • General prose: 300–500 tokens, 10–15% overlap.
  • Long technical PDFs / specs: 500–800 tokens, 10–20% overlap, structure-aware splitter.
  • Conversational transcripts: split on speaker turns, then merge to a target size.
  • Source code: split on function / class boundaries, never mid-statement.

Why “smaller and more chunks” is not automatically better. As chunks get smaller, individual chunks lose context the LLM needs to interpret them; retrieval can return five tiny snippets that, separately, don’t actually contain the answer even though together they would. The sweet spot is “small enough to be on-topic, large enough to stand alone.”

MODULE 11: OPTIMIZING RETRIEVAL PERFORMANCE

11.1 Improving Retrieval Results Using Re-Ranking Methods or Hybrid Search

Two of the highest-impact, well-understood upgrades over plain dense retrieval.

A. Hybrid search (sparse + dense fusion).

Idea. Run a sparse retriever (BM25) and a dense retriever (embeddings) in parallel over the same corpus, then fuse their result lists. You get keyword precision from BM25 and semantic recall from embeddings.

Why it helps. BM25 nails queries that depend on exact tokens (product codes, names, error strings) that embeddings often dilute. Dense retrieval nails paraphrases and conceptual matches that BM25 misses entirely. Their failure modes are nearly disjoint, so fusion is robust.

How to fuse — two common methods:

  1. Score fusion (weighted sum). Normalize each retriever’s scores (e.g., min-max) and combine them: combined = α · dense + (1 − α) · sparse. Tune α on a held-out set (often around 0.5–0.7 in favor of dense for natural-language queries).
  2. Rank fusion (Reciprocal Rank Fusion, RRF). Ignore raw scores; combine by ranks:
RRF_score(doc) = Σ over retrievers   1 / (k + rank_in_that_retriever(doc))

k is a small constant (commonly 60). RRF is robust because it sidesteps cross-retriever score-scale issues.

B. Re-ranking with a cross-encoder.

Idea. First-stage retriever (bi-encoder and/or BM25) returns a generous top-N (say 50–100). A heavier cross-encoder then scores each (query, candidate) pair jointly and reorders them; keep the top-k (say 5–10) for the LLM.

Why it helps. A bi-encoder encodes query and document independently; it must commit to a single vector per document, which can miss subtle query-document interactions. A cross-encoder reads them together with full attention and produces a much more reliable relevance score. The cost — one forward pass per candidate — is manageable because it runs only on the top-N, not the whole corpus.

When to add a re-ranker: when Precision@k matters (small k feeding an expensive LLM call), when the first-stage retrieval is high-recall but noisy, or when downstream answer quality is bottlenecked by retrieval quality rather than coverage.

How re-ranking and hybrid combine in practice (a strong production recipe):

query
  ├── BM25         → top 100 by lexical score
  └── dense (bi-encoder) → top 100 by cosine

merge via RRF → top 100 unique candidates

cross-encoder rerank → top 5–10

feed to LLM

This pattern — hybrid retrieve → fuse → re-rank → generate — is the modern default for serious RAG systems.

11.2 Tuning Retriever Hyperparameters to Boost Relevance

A retriever exposes several knobs. Tune each against the same offline eval set (queries + relevance labels), measuring Recall@K, Precision@K, MRR, and end-to-end answer quality.

Hyperparameters worth tuning:

Knob What it controls How to think about tuning
k (top-k) How many chunks to return / send to the LLM Too small → misses relevant chunks (low recall). Too large → noise dilutes the LLM’s prompt (low precision, higher cost). Sweep k and watch both recall and answer quality.
Similarity score threshold Minimum similarity to accept a chunk Set so that clearly irrelevant chunks are dropped even if they fall in the top-k. Useful when “no good match” should mean “no chunk”, not “the best of a bad bunch”.
Hybrid fusion weight (α) / RRF constant Sparse vs. dense balance Tune α on a held-out set; for keyword-heavy queries lower α (give BM25 more weight), for natural-language queries higher α.
Number of first-stage candidates (N) How many candidates the re-ranker sees Larger N → better chance the true positive is in the candidate pool, but slower and pricier reranking. Pick the smallest N at which recall plateaus.
Re-ranker top_k_after_rerank Final size of context handed to the LLM Constrained by context window and answer-quality measurements.
Chunk size and overlap Granularity and redundancy of the index Re-tune whenever the embedder, document set, or query style changes (see 10.2).
Embedding model choice The vector space itself Treat as a hyperparameter; benchmark candidates on your data (see 9.1).
Index parameters (HNSW M, efSearch; IVF nlist, nprobe; PQ codes) Speed-vs-accuracy on ANN search Raise efSearch / nprobe until recall stops improving meaningfully; then back off for latency.
Metadata filters Pre-filter by date, source, language, tenant Use to enforce recency / access control; effectively raises precision on filtered slices.
Query transformations Rewrite, multi-query, HyDE Treat as on/off knobs; measure whether they actually help on your eval set (they don’t always).

Tuning workflow:

  1. Lock down an evaluation set of queries with known relevant chunks; without this, “tuning” is just vibes.
  2. Establish a baseline with default settings; record Recall@K, Precision@K, MRR, answer quality.
  3. One knob at a time. Sweep one hyperparameter through a small grid; pick the value that maximises the chosen metric.
  4. Re-evaluate end-to-end. A change that helps retrieval metrics can still hurt answer quality (e.g., bigger k that adds noisy chunks). Always check both.
  5. Iterate in the order of expected impact: chunk size → embedding model → k → hybrid fusion → re-ranker on/off → ANN index params.
  6. Lock and monitor. Once tuned, lock the configuration and monitor metrics in production; re-tune when the corpus, query distribution, or models change.

Common pitfalls:

  • Tuning everything at once — you can’t tell what helped.
  • Tuning on the same set you’ll later report numbers on — you’ll overfit.
  • Optimising retrieval metrics in isolation while ignoring final answer quality.
  • Cranking k arbitrarily high to “play it safe” — pollutes the LLM context with noise and increases cost.
  • Mixing embedding models between query and index without re-embedding — silently destroys retrieval quality.

KEY WORKFLOWS TO KNOW (Order-the-Steps Questions)

End-to-end RAG pipeline (build + serve):

  1. Load documents from sources.
  2. Clean and normalize text.
  3. Split into chunks (size + overlap).
  4. Embed chunks with the chosen embedding model.
  5. Upsert (chunk_text, embedding, metadata) into the vector store.
  6. At query time: embed the user query with the SAME model.
  7. Search the store for the top-k chunks.
  8. (Optional) Re-rank with a cross-encoder.
  9. Build a prompt with system instructions + retrieved context + question.
  10. Call the LLM; return the grounded answer (with citations).

Retrieval evaluation workflow:

  1. Assemble an evaluation set of queries with known relevant chunks.
  2. Run the retriever to get top-K for each query.
  3. Compute Precision@K, Recall@K, MRR (and F1@K) per query.
  4. Average across queries; report per-metric.
  5. Compare against a baseline / previous configuration.
  6. Identify the failure modes (low recall vs low precision) and pick the right intervention.

Chunk-size optimization workflow:

  1. Fix the embedder, retriever, and LLM; vary only chunking parameters.
  2. Define a grid: chunk sizes × overlaps.
  3. For each combination: rebuild the index.
  4. Measure retrieval metrics and average tokens-per-query on the eval set.
  5. Plot accuracy vs. tokens; pick the smallest chunk size whose accuracy is within a small margin of the best.
  6. Validate on a held-out query set; lock in.

Hybrid + re-rank workflow:

  1. Run BM25 → top-N candidates.
  2. Run dense retriever → top-N candidates.
  3. Merge with RRF (or weighted score fusion).
  4. Take the merged top-N (e.g., 50–100).
  5. Score each (query, candidate) pair with a cross-encoder.
  6. Keep the top-k (e.g., 5–10) for the LLM.
  7. Generate the answer.

SCENARIO CHEATSHEET (Situational Questions)

“Should I use RAG or a vanilla LLM?” (Skill 2.2)

  • Knowledge changes often → RAG.
  • Knowledge is private / proprietary → RAG.
  • Citations / auditability required → RAG.
  • Domain is long-tail or specialized → RAG.
  • Hallucinations are costly → RAG.
  • Per-user access control needed → RAG.
  • Task is pure style / rewriting / general reasoning with no external facts → Vanilla LLM.
  • Need to change behaviour or output format → Fine-tune (in addition to RAG, not instead).

“Which retrieval strategy fits this use case?” (Skill 6.2)

  • Queries are keyword-heavy (codes, names, IDs) → BM25 (or hybrid leaning on BM25).
  • Queries are conversational / paraphrased → Dense / semantic.
  • Mixed query traffic → Hybrid (BM25 + dense).
  • Small k feeding an expensive LLM call → add Cross-encoder re-ranker.
  • Per-tenant / time-sensitive / permissioned corpus → Metadata-filtered retrieval.
  • Vague or under-specified user queries → add Query rewriting / multi-query / HyDE.
  • Answer requires chaining facts across documents → Multi-hop / iterative retrieval.
  • Tiny corpus → skip retrieval; put everything in the prompt.
