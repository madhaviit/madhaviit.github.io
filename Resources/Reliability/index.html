Workera ELO / SAGE Assessment: Study Notes

Explainability, Interpretability, Fairness, Reliability, Privacy & Safety

PART 1: EXPLAINABILITY AND INTERPRETABILITY

MODULE 1: FOUNDATIONS OF EXPLAINABILITY

1.1 When Is Explainability Needed?

Explainability is not always required, but certain conditions strongly call for it. Understanding when to apply it is crucial for the assessment.

High-stakes decision contexts — situations where model outputs directly affect people’s lives, rights, or well-being: - Credit scoring and loan approval - Medical diagnosis and treatment recommendations - Criminal recidivism prediction (e.g., COMPAS) - Hiring and promotion decisions - Insurance underwriting - Child welfare risk scoring

Regulatory and legal requirements — jurisdictions and industries with explicit mandates: - GDPR Article 22 (EU): Right to an explanation for automated decisions - ECOA (US): Adverse action notices for credit denial must list reasons - Healthcare regulations requiring model transparency - FINRA requirements for algorithmic trading explanations

Debugging and model improvement — when practitioners need to: - Understand why a model is underperforming - Identify spurious correlations learned from data - Detect data leakage (model relying on proxy variables) - Verify the model is using features for the right reasons

Trust and adoption — when end users or stakeholders need to: - Trust an AI system before acting on its outputs - Override model predictions when domain knowledge conflicts - Build confidence in a new deployed system

Accountability and contestability — when affected individuals need to: - Challenge or appeal automated decisions - Understand what they can change to receive a different outcome (actionability) - Identify discriminatory patterns affecting protected groups

When explainability is less critical: - Low-stakes automated decisions (e.g., movie recommendations) - Contexts where performance is the only goal and no human is affected - Purely internal research experiments without deployment

1.2 Matching Explanation Purpose to Appropriate Audiences

Different stakeholders have fundamentally different needs from explanations. A mismatch leads to confusion, misuse, or rejection of the explanation.

Audience Primary Goal Preferred Explanation Type Key Concerns
Data Scientists / ML Engineers Debug, improve model Feature importances, SHAP values, partial dependence plots, surrogate models Technical fidelity, statistical validity
Domain Experts (Doctors, Lawyers) Validate model behavior Case-based reasoning, rule extraction, attention maps Alignment with domain knowledge
Business Stakeholders / Managers Justify deployment, understand risk Summary dashboards, high-level feature contributions Business impact, ROI, regulatory risk
Regulators / Auditors Ensure compliance Model cards, fairness metrics, global feature importance Reproducibility, legal compliance
End Users / Affected Individuals Understand and contest decisions Contrastive/counterfactual explanations, natural language Clarity, actionability, fairness
Legal / Compliance Teams Regulatory review Documentation, adverse action reasons, audit trails Liability, right to explanation

Key principle — contrastive explanations for end users: People naturally ask “Why was I rejected instead of approved?” rather than “What caused my outcome?” Counterfactual explanations (“If your income were $5,000 higher, you would have been approved”) directly address this need.

Key principle — global vs. local explanations by audience: - Engineers → both global (model behavior overall) and local (individual predictions) - End users → primarily local explanations for their specific case - Regulators → primarily global explanations for systemic behavior

1.3 Core Explainability Terms

Master these definitions precisely — matching-the-pairs questions frequently test these.

Interpretability — the degree to which a human can understand the cause of a decision; an intrinsic property of a model (e.g., a linear model is inherently interpretable).

Explainability — the ability to present model behavior in human-understandable terms, often via post-hoc methods applied to opaque models. Broader than interpretability.

Transparency — openness about the model’s architecture, training data, and decision process; a property of the overall system, not just the model.

Accountability — the ability to attribute responsibility for model outcomes to specific actors (developers, deployers, regulators).

Fidelity — how accurately an explanation reflects the true behavior of the model it is explaining.

Global Explanation — describes the overall behavior of a model across all inputs (e.g., which features matter most on average).

Local Explanation — describes the model’s behavior for a specific input instance (e.g., why this particular loan was denied).

Model-agnostic method — an explanation technique that can be applied to any model regardless of its architecture (e.g., LIME, SHAP).

Model-specific method — an explanation technique that works only for a particular model type (e.g., attention weights for transformers, feature importances for tree models).

Intrinsic interpretability — a model that is interpretable by design (linear regression, decision trees, rule-based systems).

Post-hoc explainability — explanations generated after training, applied to an already-trained model (e.g., SHAP applied to a neural network).

Surrogate model — a simpler, interpretable model trained to approximate the behavior of a complex model.

Contrastive explanation — explains a decision relative to an alternative outcome (“Why A and not B?”).

Counterfactual explanation — a what-if explanation that identifies the minimum change to an input that would change the prediction.

Saliency map — a visualization highlighting which parts of an input (e.g., pixels in an image) most influenced the prediction.

Partial Dependence Plot (PDP) — shows the marginal effect of one or two features on the predicted outcome, averaged over all other features.

Individual Conditional Expectation (ICE) plot — like a PDP but shows the effect for each individual instance, revealing heterogeneous effects.

Feature importance — a score indicating how much a feature contributes to model predictions.

SHAP value — a value that quantifies the contribution of each feature to a specific prediction, based on Shapley values from game theory.

1.4 Assessing Explanation Quality: Fidelity, Clarity, Stability, and Usefulness

The four pillars of explanation quality. You may be asked to evaluate or rank explanations using these criteria.

Fidelity (Faithfulness) - Does the explanation accurately reflect what the model actually computes? - High fidelity: the explanation would lead you to correctly predict what the model would do in new cases - Measured by: how well the surrogate/explanation model matches the original model’s predictions - Risk of low fidelity: users make decisions based on a misleading explanation - Example failure: a saliency map highlights “dog’s face” in an image, but the model is actually relying on the collar to predict breed

Clarity (Comprehensibility) - Is the explanation understandable to its intended audience? - Measured by: user studies, task completion rates, comprehension tests - Tradeoff: technically accurate explanations can be incomprehensible to lay users - Best practice: tailor the explanation format to the audience (SHAP plots for engineers; natural language for patients)

Stability (Robustness) - Does the explanation remain consistent when the input changes slightly? - High stability: similar inputs produce similar explanations - Measured by: perturbation tests — slightly vary input, check if explanation changes drastically - Unstable explanations undermine trust and suggest the explanation method is unreliable - LIME is known to have stability issues due to its random sampling; SHAP is generally more stable

Usefulness (Actionability) - Does the explanation enable the recipient to take meaningful action? - For end users: can they understand what to change to get a different outcome? - For engineers: does the explanation help them improve the model? - For regulators: does the explanation help them assess compliance? - Usefulness is audience-dependent — same explanation can be useful for one group and useless for another

Summary table:

Property Question It Answers Who Cares Most
Fidelity Does it reflect the model? Engineers, auditors
Clarity Is it understandable? End users, managers
Stability Is it consistent? Engineers, regulators
Usefulness Can you act on it? End users, domain experts

MODULE 2: APPLYING EXPLANATION METHODS

2.1 LIME — Local Interpretable Model-agnostic Explanations

What it does: LIME approximates the behavior of any black-box model locally around a specific prediction by training a simple interpretable model (usually linear regression) on perturbed samples near the input of interest.

How it works (step by step): 1. Take a specific input instance you want to explain (e.g., a loan application). 2. Generate perturbed versions of that instance (by randomly turning features on/off or adding noise). 3. Query the black-box model to get predictions for all perturbed samples. 4. Weight the perturbed samples by their proximity to the original instance. 5. Fit a simple interpretable model (e.g., linear regression, decision tree) on these weighted samples. 6. The coefficients of this simple model serve as the local explanation.

Key properties: - Model-agnostic: works with any classifier or regressor - Local: only explains one prediction at a time - Approximate: the explanation is an approximation, not the ground truth - Perturbation strategy varies by data type: - Tabular: randomly setting features to mean value or zero - Text: removing words/tokens - Images: “superpixel” segmentation — turning off coherent image regions

Strengths: - Easy to apply to any model - Produces human-readable explanations (feature weights) - Can handle text, images, and tabular data

Weaknesses: - Instability: random sampling means different runs produce different explanations - Kernel width sensitivity: the choice of neighborhood size heavily affects results - Sampling bias: perturbed samples may fall in unrealistic regions of input space - Explanation may not faithfully reflect global model behavior

LIME for text — key concept: Features are word presence/absence. A positive weight means the word pushes toward the predicted class; negative means it pushes against.

Code skeleton (tabular):

from lime.lime_tabular import LimeTabularExplainer

explainer = LimeTabularExplainer(
    X_train,
    feature_names=feature_names,
    class_names=class_names,
    mode='classification'
)

explanation = explainer.explain_instance(
    X_test[i],          # single instance to explain
    model.predict_proba, # black-box prediction function
    num_features=10      # top features to show
)

explanation.show_in_notebook()

2.2 SHAP — SHapley Additive exPlanations

Foundation — Shapley values from cooperative game theory: Shapley values fairly distribute a “payout” (the model’s prediction) among “players” (features) based on their marginal contributions across all possible feature coalitions.

The SHAP value for feature i is the average marginal contribution of feature i across all possible orderings of features being added to the model.

SHAP additive property: For any instance, the sum of all SHAP values equals the model’s prediction minus the base rate (expected model output):

f(x) = base_value + SHAP_1 + SHAP_2 + ... + SHAP_n

This makes SHAP values directly interpretable as signed contributions.

SHAP variants for different model types:

Variant Best For Speed
TreeSHAP Tree-based models (XGBoost, Random Forest) Very fast (exact)
DeepSHAP Neural networks Fast
KernelSHAP Any model (model-agnostic) Slow
LinearSHAP Linear models Instant

SHAP visualizations — know each one:

  • Force plot: Single prediction; shows features pushing prediction higher (red) or lower (blue) relative to base value.
  • Summary plot (beeswarm): All instances; shows distribution of SHAP values per feature, colored by feature value. Reveals global importance and direction of effect.
  • Dependence plot: SHAP value of one feature vs. its actual value; reveals non-linear effects and interactions (colored by another feature to reveal interactions).
  • Waterfall plot: Single prediction; shows step-by-step how each feature’s SHAP value builds up to the final prediction from the base value.
  • Bar plot: Global feature importance; average absolute SHAP values per feature.

Strengths: - Theoretically grounded (satisfies efficiency, symmetry, dummy, and additivity axioms) - Consistent: if a feature’s true contribution increases, its SHAP value never decreases - Provides both local and global explanations - More stable than LIME

Weaknesses: - Computationally expensive for KernelSHAP (exponential in features) - Assumes feature independence in some variants (can mislead with correlated features) - Can be hard to interpret for non-technical audiences - SHAP values can be manipulated by adversarial model design

Code skeleton:

import shap

explainer = shap.TreeExplainer(model)          # or shap.Explainer(model)
shap_values = explainer.shap_values(X_test)

shap.summary_plot(shap_values, X_test)         # global summary
shap.force_plot(explainer.expected_value,      # local explanation
                shap_values[0], X_test.iloc[0])
shap.dependence_plot("age", shap_values, X_test)  # feature interaction

2.3 Visualization Techniques for Interpreting Model Behavior

Partial Dependence Plots (PDP): - Show the marginal effect of a feature on the predicted outcome, averaged over all other features - Assumes feature independence (problematic with correlated features) - Best for: communicating overall direction and shape of feature effect to non-technical audiences - Plot shape interpretation: flat = feature has no effect; monotonic = consistent direction; non-monotonic = complex interaction

Individual Conditional Expectation (ICE) Plots: - Each line shows the effect of changing one feature for one specific instance - Reveals heterogeneity: some instances may respond oppositely to others - Centered ICE (c-ICE): all lines start at zero, making differences in slopes clearer - When to use over PDP: when you suspect interactions or subgroup-specific effects

Accumulated Local Effects (ALE) Plots: - Addresses PDP’s independence assumption by conditioning on neighborhoods - More reliable than PDP for correlated features - Less common but preferred in high-correlation settings

Saliency Maps (for images and text): - Gradient-based: compute gradient of output w.r.t. input pixels - GradCAM: uses gradients flowing into the last convolutional layer to produce coarse localization maps - Integrated Gradients: accumulates gradients along the path from a baseline to the input - Useful for: diagnosing whether a vision model is looking at the right region

Attention Visualization (for transformers): - Shows which input tokens a model attends to when producing an output - Caution: attention weights ≠ feature importance; models can attend to irrelevant tokens - Used for: initial inspection, not definitive explanation

Feature Importance Plots (tree models): - Mean Decrease in Impurity (MDI): fast but biased toward high-cardinality features - Permutation Importance: model-agnostic; measures drop in performance when feature is shuffled; can be computed on test set - SHAP-based importance: most reliable; addresses MDI’s bias

2.4 Surrogate Models

Definition: A surrogate model is a simpler, interpretable model trained to mimic the predictions of a complex black-box model.

Global surrogate: 1. Generate a large dataset of inputs (can be original training data or new samples). 2. Query the black-box model to get predictions for all inputs. 3. Train an interpretable model (linear regression, decision tree) on these input-prediction pairs. 4. Interpret the surrogate as an approximation of the black-box.

Fidelity assessment: Measure R² or agreement rate between the surrogate and the black-box on held-out data. Higher fidelity = more trustworthy surrogate.

Choice of surrogate model type: - Decision tree: Provides rule-based explanations; easy to visualize; risk of overfitting if too deep - Linear regression: Provides coefficient-based explanations; best when relationship is approximately linear - Rule lists (if-then rules): Highly interpretable; sparse explanations

Limitations: - A surrogate is an approximation, not the true model — always report fidelity - If the surrogate has low fidelity, its explanation is misleading - Global surrogates may miss local behavior; use local surrogates (LIME) for individual predictions

When to use surrogates: - When the original model is a proprietary black box with no internal access - When regulatory requirements demand an interpretable model for audit - When you want to communicate model logic to non-technical stakeholders

2.5 Post-Training Model Analysis

Concept: After training, systematic analysis of model behavior using interpretability techniques to understand what the model has learned.

Steps in post-training analysis: 1. Global feature importance: Which features drive predictions overall? (SHAP summary plot, permutation importance) 2. Feature effect analysis: How does each feature affect the prediction? (PDP, ALE plots) 3. Subgroup analysis: Does the model behave differently for demographic subgroups? 4. Error analysis: Which types of inputs does the model get wrong? Are errors clustered? 5. Concept probing: Does the model represent human-interpretable concepts internally? (for neural networks) 6. Layer-wise analysis: For deep networks, what is each layer learning? (activation maximization)

Activation maximization: Synthesizes the input that maximally activates a neuron — reveals what a unit “detects.” Used in computer vision to understand convolutional filters.

Probing classifiers: Train a linear classifier on internal representations to test if a concept (e.g., sentiment, POS tag) is encoded in a layer.

Checklist for post-training analysis: - [ ] Feature importance computed and cross-validated - [ ] Effects visualized for top features - [ ] Subgroup performance compared - [ ] Failure cases examined and categorized - [ ] Explanation quality assessed (fidelity, stability)

2.6 Counterfactual and Randomized Tests for Feature Sensitivity

Counterfactual explanations: - Answer: “What is the minimum change to this input that would change the prediction?” - Properties of a good counterfactual: - Validity: The counterfactual input actually receives the desired prediction - Proximity: The change from the original input is minimal - Sparsity: As few features changed as possible - Plausibility: The counterfactual is a realistic/realistic input (not out-of-distribution) - Actionability: Only features the user can actually change are modified (e.g., age is immutable)

Wachter counterfactuals: Minimize weighted distance between original and counterfactual subject to the prediction changing. Simple but may generate implausible instances.

DiCE (Diverse Counterfactual Explanations): Generates multiple diverse counterfactuals — useful for showing multiple paths to a different outcome.

Randomized feature sensitivity tests: - Feature permutation: Shuffle a feature across all instances; measure drop in performance. Larger drop = more important feature. - Feature ablation: Set a feature to zero or mean; measure performance change. - Input perturbation: Add small random noise to inputs; measure prediction stability (tests model robustness). - Interventional testing: Systematically vary one feature while holding others constant; verify the model responds as expected by domain knowledge.

Sensitivity analysis workflow: 1. Define the feature of interest. 2. Generate perturbed versions of test instances (vary only that feature). 3. Record model predictions for each perturbation level. 4. Plot the sensitivity curve (prediction vs. feature value). 5. Compare to domain expectations — does the model respond appropriately?

Why this matters for fairness: If the model is highly sensitive to race or gender (which should be irrelevant), this is detected through feature sensitivity tests.

PART 2: TECHNICAL FAIRNESS INTERVENTIONS

MODULE 3: DIAGNOSIS OF FAIRNESS ISSUES

3.1 Detecting Bias in Data

Types of bias in data — know each precisely:

Historical bias: The data reflects historical human prejudice. Even if the data collection process was unbiased, the underlying social patterns encode discrimination. Example: historical hiring data reflecting gender discrimination in STEM.

Representation bias (Underrepresentation): Some groups are sampled at lower rates than their true population frequency. The model has less data about these groups, leading to higher error rates. Example: clinical trial data overrepresenting white males; facial recognition datasets underrepresenting darker-skinned faces.

Measurement bias: Different groups are measured with different accuracy or instruments. Example: using arrest records as a proxy for crime — arrest rate reflects policing intensity, not actual crime, and policing is historically biased.

Aggregation bias: Combining data from different subgroups and fitting a single model ignores within-group variation. The model fails to capture group-specific patterns. Example: one diabetes prediction model for all ethnicities when the relationship between HbA1c and diabetes differs by ethnicity.

Label bias: The labels themselves are biased, even if the features are not. Example: loan default labels reflecting discriminatory lending practices (loans were denied to minorities so we have fewer default/repayment labels for them).

Temporal bias: Training data from a past time period does not reflect current population distributions or fairness standards.

Stereotype reinforcement: When training data encodes cultural stereotypes, the model learns and perpetuates them. Example: word embeddings placing “homemaker” closer to “woman” than “man.”

How to detect data bias: - Compute group-level statistics: mean, variance, missing data rate, label rates by demographic - Visualize distributions using histograms and density plots stratified by group - Check label rates: is the positive label rate similar across groups? - Check feature distributions: are there systematic differences? - Use demographic parity tests on the raw labels - Check for proxy variables: features highly correlated with protected attributes

3.2 Risks in Labels, Feature Selection, and Representation

Label risks:

  • Subjective labeling: Human annotators introduce bias when labeling subjective tasks (toxicity, sentiment, risk)
  • Feedback loops: If labels are derived from system decisions (e.g., hiring outcomes), and the system was biased, labels reflect that bias
  • Missing labels: Differential missingness across groups can bias training

Feature selection risks:

  • Proxy features: Features not directly protected but highly correlated with protected attributes (zip code ↔︎ race; credit score ↔︎ SES; name ↔︎ ethnicity/gender). Removing the protected attribute doesn’t help if proxies remain.
  • Disparate impact through feature selection: Some features may have unequal predictive power across groups, causing unequal treatment when used
  • Irrelevant features that encode group membership: Including name, photo, or other identifying information

Representation risks:

  • Class imbalance: If a protected group is a small minority, the model optimizes for majority-group performance
  • Intersection blindness: Looking only at gender OR race separately misses harm to intersectional groups (Black women may face unique discrimination not captured by either margin)
  • Distributional shift: Training data doesn’t represent the deployment population

3.3 Formulating Questions to Uncover Hidden Assumptions

A critical skill for the assessment. Before training, systematically question the model design.

Questions about the task definition: - What does the model’s output actually mean? What is being predicted, and does it capture what we care about? - Is the proxy outcome we’re predicting a fair proxy? (e.g., predicting “creditworthiness” using “repayment” — is repayment access affected by systemic factors?) - Who decided what the “correct” outcome is?

Questions about the data: - Who is represented in this dataset? Who is missing? - Over what time period was data collected? Does it reflect current reality? - How were labels assigned? By whom? With what incentives? - Are there features that should not be used? Why are they included?

Questions about the model: - What does the model optimize? Is accuracy the right metric here? - Does the model perform equally across demographic subgroups? - What are the consequences of false positives vs. false negatives for different groups?

Questions about deployment: - Who will use this model? In what contexts? - What happens to people who receive incorrect predictions? - Who can contest a decision? What recourse exists?

MODULE 4: FIXING FAIRNESS IN THE WORKFLOW

4.1 Pre-processing Methods for Bias Mitigation

Pre-processing interventions modify the training data before model training begins. They are model-agnostic — the corrected dataset can be used with any model.

Resampling: - Oversampling underrepresented groups: Duplicate instances from minority groups (e.g., SMOTE — Synthetic Minority Oversampling Technique). Ensures equal representation. - Undersampling majority groups: Remove instances from overrepresented groups. Risks losing information. - Stratified sampling: Ensure training data proportionally represents all subgroups.

Reweighting: - Assign higher training weights to underrepresented groups or incorrectly labeled instances - The model pays more attention to these examples during training - Reweighting factor: inverse of group probability — groups with lower representation get higher weights

Label flipping / Relabeling: - Identify and correct mislabeled instances, especially for groups where labeling was historically biased - Kamiran & Calders (2012): flip labels for borderline instances to achieve parity

Feature transformation: - Suppression: Remove protected attributes and their proxies - Fair representations: Learn a new feature space that removes group information while preserving predictive information (Zemel et al., Variational Fair Autoencoder) - Data augmentation: Generate synthetic data for underrepresented groups

Disparate impact remover: - Transforms feature values to remove correlation with the protected attribute while preserving rank ordering within groups - Adjusts features to achieve the “80% rule” (disparate impact threshold)

4.2 Causal and Adversarial Techniques

Causal approaches:

Causal graphs (DAGs): Model the causal relationships between variables, including protected attributes. Identify paths through which protected attributes affect predictions.

  • Direct discrimination: Protected attribute → prediction (direct causal path)
  • Indirect discrimination: Protected attribute → mediator → prediction (causal path through a feature)
  • Counterfactual fairness (Kusner et al.): A prediction is counterfactually fair if it would be the same for an individual in a counterfactual world where their protected attribute was different, with all causally downstream variables adjusted accordingly. More rigorous than standard fairness metrics.

Path-specific counterfactual fairness: Block only the illegitimate causal paths (direct discrimination) while allowing legitimate paths (e.g., education level can mediate gender effects if we consider it legitimate).

Adversarial debiasing:

Architecture: Two-component network 1. Predictor: Learns to predict the target label Y from features X 2. Adversary: Attempts to predict the protected attribute A from the predictor’s output

Training: The predictor is trained to minimize prediction loss while also maximizing the adversary’s prediction error (minimax game). This forces the predictor to make predictions from which the protected attribute cannot be recovered.

Predictor loss = prediction_loss - λ * adversary_accuracy
Adversary loss = adversary_prediction_loss

The hyperparameter λ controls the fairness-accuracy tradeoff.

Fairness VAE / Fair representation learning: - Encode data into a latent space where protected attribute information is removed - Decoder can reconstruct data without protected attribute information - Downstream models trained on fair representations inherit fairness properties

4.3 Fairness-Aware Training Methods

In-processing methods incorporate fairness constraints directly into the model training objective.

Fairness constraints as regularization: - Add a fairness penalty term to the loss function - Loss = prediction_loss + λ * fairness_violation - The model minimizes both prediction error and fairness violation simultaneously

Types of fairness constraints (know the definitions):

Fairness Criterion Definition Formula
Demographic Parity Prediction rate equal across groups P(Ŷ=1|A=0) = P(Ŷ=1|A=1)
Equalized Odds Equal TPR and FPR across groups P(Ŷ=1|Y=y, A=0) = P(Ŷ=1|Y=y, A=1) for y ∈ {0,1}
Equal Opportunity Equal TPR across groups (relaxed equalized odds) P(Ŷ=1|Y=1, A=0) = P(Ŷ=1|Y=1, A=1)
Predictive Parity Equal PPV (precision) across groups P(Y=1|Ŷ=1, A=0) = P(Y=1|Ŷ=1, A=1)
Individual Fairness Similar individuals receive similar predictions d(x,x’) small → d(f(x),f(x’)) small
Calibration Predicted probabilities match actual rates within each group P(Y=1|Ŷ=p, A=a) = p for all groups a

The impossibility theorem (Chouldechova 2017, Kleinberg et al. 2016): When base rates differ between groups, you cannot simultaneously satisfy calibration, equal FPR, and equal FPR. Fairness criteria are mutually exclusive under certain conditions. This means you must choose which criterion to prioritize based on the context and the costs of different types of errors.

Reduction approaches (Agarwal et al., Reductions framework): - Reduce fairness-constrained classification to a sequence of standard cost-sensitive classification problems - Iteratively re-weight training examples to satisfy fairness constraints - Implemented in Fairlearn (Microsoft)

Prejudice Remover (Kamishima et al.): - Adds a regularizer to logistic regression that penalizes mutual information between predictions and the protected attribute

4.4 Post-processing: Calibration on Group-Specific Thresholds

Post-processing methods modify model outputs after training to improve fairness. They don’t require access to model internals.

Threshold optimization: - Different decision thresholds for different demographic groups - Choose thresholds to equalize a target fairness metric (e.g., equalized odds) - Steps: 1. Get predicted probabilities from the trained model 2. For each group, find the threshold that achieves the fairness target 3. Apply group-specific thresholds at decision time

Reject option classification: - For predictions near the decision boundary, the model is uncertain - Give the favorable outcome to disadvantaged group members in the uncertain zone - Reduces discrimination in borderline cases

Calibration: - Platt scaling or isotonic regression to align predicted probabilities with actual frequencies - Apply separately per group to achieve within-group calibration - Relevant for COMPAS-style risk scores where calibration was achieved but other fairness criteria were violated

Roth-Morwitz algorithm (Equalized Odds Post-processing, Hardt et al. 2016): - Find a randomized threshold policy that equalizes TPR and FPR across groups - Operates only on predictions and protected attribute, not original features - Model-agnostic, easy to apply as a final step

4.5 Multi-Accuracy Auditing Across Subgroups

Multi-accuracy (Kim et al. 2019): - A predictor is “multi-accurate” if it is accurate simultaneously across all subgroups in a rich class of groups (not just demographic groups) - Extends standard accuracy auditing from top-level groups to intersectional subgroups

Multi-calibration: - The model’s predicted probabilities are calibrated for all subgroups, not just the overall population - Stronger guarantee than simple calibration: even for the intersection of being female, over 40, and from a specific zip code, the probability estimates are accurate

Auditing workflow: 1. Define the set of subgroups to audit (demographic, intersectional, behavioral) 2. Compute the target metric (accuracy, calibration, fairness metric) for each subgroup 3. Identify subgroups where the metric falls below the overall level (or a defined threshold) 4. Diagnose the cause: representation gap, label noise, or model error 5. Apply targeted interventions (resampling, reweighting, threshold adjustment) 6. Re-audit to verify improvement

Fairness tools and libraries:

Tool Developer Key Features
Fairlearn Microsoft Fairness metrics, reduction algorithms, dashboard
AI Fairness 360 (AIF360) IBM 70+ fairness metrics, 10+ bias mitigation algorithms
What-If Tool Google Interactive exploration of model behavior by subgroup
Aequitas University of Chicago Bias auditing toolkit for machine learning pipelines

MODULE 5: MONITORING FAIRNESS OVER TIME

5.1 Fairness Unit and Smoke Tests

Unit tests for fairness (individual components): - Test that a data preprocessing function does not introduce bias (e.g., imputation does not use group mean in a way that encodes group differences) - Test that feature encoding does not encode protected attribute information - Test that label assignment is consistent across groups

Smoke tests (basic sanity checks before deployment): - Does the model’s positive prediction rate differ by more than 20% across groups? (80% rule / four-fifths rule) - Does the model’s accuracy differ significantly across groups? - Does the model produce identical predictions for identical inputs regardless of protected attribute? - Is the model’s error rate stable across demographic subgroups?

Fairness testing frameworks: - Write fairness tests as part of CI/CD pipelines - Fail the pipeline if fairness metrics fall below thresholds - Use tools like Fairlearn’s MetricFrame to compute multiple metrics across multiple groups simultaneously

Counterfactual consistency test: - Generate pairs of inputs identical except for the protected attribute - Check that the model produces the same prediction for both - If not, the model uses the protected attribute (or a proxy) directly

5.2 Continuously Evaluating Models Against Fairness Metrics

Key fairness metrics to monitor continuously:

  • Demographic parity difference: |P(Ŷ=1|A=0) - P(Ŷ=1|A=1)|; target = 0
  • Equalized odds difference: Max difference in TPR and FPR across groups
  • Disparate impact ratio: P(Ŷ=1|A=minority) / P(Ŷ=1|A=majority); target ≥ 0.8 (four-fifths rule)
  • Calibration error by group: |P(Y=1|Ŷ=p, A=a) - p| for each group a
  • AUC-ROC by group: Measure discriminative power within each group

Monitoring pipeline: 1. Log predictions and (where available) outcomes along with group membership 2. Compute fairness metrics on a rolling window or fixed schedule 3. Alert when a metric crosses a predefined threshold 4. Investigate root causes (distribution shift, label drift, model degradation) 5. Retrain or recalibrate as needed

Concept/data drift and fairness: - Distributional shift affects groups unequally — a model that was fair at deployment may become unfair as the population changes - Monitoring for accuracy by group also detects fairness degradation - Covariate shift: input distribution changes; label shift: outcome rates change

5.3 Updating Test Sets as User Bases Evolve

Why test sets become stale: - User demographics evolve as the product grows - Societal norms change (what was acceptable language may become unacceptable) - Edge cases become common over time - New protected groups emerge (e.g., gender identity categories)

Best practices: - Rolling test sets: Continuously collect new labeled examples from production; retire old examples - Diverse annotation: Include annotators from underrepresented groups; use annotation guidelines that reflect current norms - Adversarial test sets: Collect examples specifically designed to challenge the model (Dynabench approach) - Red-teaming: Actively probe for failure cases in new subpopulations - Regular refresh schedule: Audit test set demographics quarterly or after significant user growth events - Version control for test sets: Track changes in test set composition alongside model versions

Feedback loop management: - If the model’s outputs influence future training data (e.g., a content recommendation model creates filter bubbles), the training distribution will drift toward model biases - Detect feedback loops through diversity monitoring and user behavior analysis - Break feedback loops with randomization, exploration, and diverse data collection

PART 3: RELIABILITY, PRIVACY, AND SAFETY

MODULE 6: BUILDING ROBUST MODELS

6.1 Designing Training Objectives That Reflect Real-World Goals

The alignment problem in ML: The objective function optimized during training is rarely identical to the true goal. This gap causes models to find unexpected solutions.

Common misalignments:

  • Optimizing accuracy vs. optimizing decision quality: Accuracy penalizes all errors equally, but false positives and false negatives often have very different costs.
  • Optimizing average performance vs. worst-case performance: A model with high average accuracy may fail catastrophically on rare but critical inputs.
  • Optimizing training distribution vs. deployment distribution: Training loss doesn’t measure deployment performance.

Solutions and design principles:

Cost-sensitive loss functions: - Assign different weights to different error types - loss = w_FP * FP_loss + w_FN * FN_loss - Set weights based on real-world costs of each error type

Task-specific metrics as objectives: - For medical screening: optimize recall (sensitivity) subject to a precision constraint - For fraud detection: optimize F-β score with β > 1 to weight recall more - For ranking: optimize NDCG or MAP rather than cross-entropy

Distributional robustness: - Optimize worst-case performance over a family of distributions (Distributionally Robust Optimization, DRO) - Ensures the model performs well even under distribution shift

Regularization for generalization: - L1/L2 regularization prevents overfitting to training distribution - Dropout, data augmentation, mixup improve robustness to input variation

Multi-objective optimization: - Define multiple objectives (accuracy, fairness, robustness) and find Pareto-optimal solutions - Use constrained optimization: maximize accuracy subject to fairness constraints

6.2 User-Driven Testing for Input Sensitivity

Behavioral testing (Ribeiro et al., CheckList): A framework for systematic model testing that goes beyond accuracy on a held-out set.

Three test types in CheckList:

  1. Minimum Functionality Tests (MFT): Test basic expected behaviors. Example: a sentiment model should classify “This is great!” as positive.
  2. Invariance Tests (INV): Perturbations that should NOT change the prediction. Example: changing a name from “John” to “Jamal” should not change a sentiment prediction.
  3. Directional Expectation Tests (DIR): Perturbations where prediction should change in a known direction. Example: adding “not” before a positive word should decrease positive sentiment score.

User-driven sensitivity testing: - Allow users to drag feature sliders and see how predictions change - Provide “what-if” interfaces (Google’s What-If Tool) - Show feature sensitivity curves for individual predictions - Let domain experts define test cases based on their knowledge

Stress testing: - Test on extreme values: minimum, maximum, boundary conditions - Test on adversarial examples (see Security section) - Test on out-of-distribution inputs (data from a different hospital, country, time period)

6.3 Identifying and Managing Edge Cases

Definition: Edge cases are inputs that occur rarely but for which the model’s failure would be high-cost or high-risk.

Sources of edge cases: - Rare demographic groups or rare feature combinations - Unusual events (weather anomalies, market crashes, pandemics) - Adversarial inputs - Data from new distributions not present during training

Identifying edge cases: - Uncertainty quantification: Flag inputs where model confidence is low (high entropy, high predictive variance in ensemble models) - Anomaly detection: Detect out-of-distribution inputs before the model makes a prediction - Error analysis: Cluster misclassified examples to find systematic failure modes - Human-in-the-loop review: Route low-confidence predictions to human reviewers

Managing edge cases at deployment: - Abstention/rejection option: Allow the model to refuse to make a prediction if uncertainty is too high - Confidence thresholding: Only act on predictions above a minimum confidence level - Escalation policies: Define procedures for routing edge cases to human experts - Monitoring dashboards: Track edge case occurrence rates and model performance on them

MODULE 7: PRIVACY AND LEGAL READINESS

7.1 Data Protection Principles During Model Development

Core principles (aligned with GDPR and general best practice):

Lawfulness, fairness, and transparency: Data must be collected and used lawfully; individuals must be informed.

Purpose limitation: Data collected for one purpose should not be used for an unrelated purpose without consent. Example: health data collected for treatment should not be used to train a commercial model without explicit consent.

Data minimization: Collect and retain only the data necessary for the specified purpose. Do not store more than needed.

Accuracy: Keep data up to date; have mechanisms to correct inaccurate data.

Storage limitation: Retain data only as long as necessary; have data deletion policies.

Integrity and confidentiality: Protect data against unauthorized access, loss, or destruction.

Accountability: Organizations must demonstrate compliance with these principles.

7.2 Regulatory Standards: GDPR and HIPAA

GDPR (General Data Protection Regulation — EU):

Key provisions relevant to ML: - Article 5: Data protection principles (above) - Article 9: Special categories of sensitive data: health, racial/ethnic origin, political opinions, religious beliefs, sexual orientation, biometric data → require explicit consent or other legal basis - Article 13/14: Transparency — inform individuals about automated processing - Article 15: Right of access — individuals can request their data - Article 17: Right to erasure (“right to be forgotten”) — complicates model training (retraining without deleted data) - Article 20: Right to data portability - Article 22: Right not to be subject to solely automated decisions with significant effects; right to human review and to contest decisions - Recital 71: Automated decisions should use appropriate mathematical and statistical procedures; should not concern special categories of data

HIPAA (Health Insurance Portability and Accountability Act — US):

Key provisions for ML: - Applies to Protected Health Information (PHI): individually identifiable health data - 18 de-identification identifiers must be removed for data to be considered de-identified (Safe Harbor method) - Expert determination: A statistician certifies that risk of re-identification is very small - Business Associate Agreement (BAA): Any third party (including cloud ML providers) handling PHI must sign a BAA - Minimum necessary rule: Only access the minimum PHI needed for the task - Security Rule: Technical safeguards for PHI (encryption, access controls, audit logs)

GDPR vs. HIPAA comparison:

7.3 Privacy-Preserving Practices

Data minimization: - Collect only features needed for the model - Aggregate where possible (use zip code instead of full address) - Use feature selection to remove unnecessary variables early

Access control: - Role-based access: only authorized personnel access sensitive data - Audit logs: track who accessed what data and when - Data compartmentalization: sensitive attributes stored separately

Anonymization and pseudonymization: - Anonymization: True removal of identifying information; irreversible - Pseudonymization: Replace identifiers with pseudonyms; re-identification possible with key - k-anonymity: Every record is indistinguishable from at least k-1 other records on quasi-identifying attributes - l-diversity: Each equivalence class has at least l well-represented sensitive values - t-closeness: Distribution of sensitive attribute in each equivalence class is close to overall distribution

Differential Privacy (DP): - Mathematical definition: a mechanism M is ε-differentially private if for any dataset D and D’ differing by one record, and any output S: P(M(D) ∈ S) ≤ e^ε * P(M(D’) ∈ S) - Adding carefully calibrated noise to model training or outputs - DP-SGD: Add Gaussian noise to gradients during training; clip gradients to bound sensitivity - ε (epsilon): Privacy budget — smaller ε = stronger privacy guarantee = more noise = lower accuracy - Used by: Apple, Google for analytics; PyTorch Opacus library for ML

Federated Learning: - Train models across multiple decentralized devices without centralizing raw data - Each device trains locally on its data; only model updates (gradients) are shared - Aggregated on a central server (FedAvg algorithm) - Privacy benefit: raw data never leaves the device - Residual risk: gradient sharing can still leak information (gradient inversion attacks)

Secure Multi-Party Computation (MPC): - Multiple parties jointly compute a function over their combined data without revealing their individual data to each other - Example: hospitals jointly train a model without sharing patient records

Homomorphic Encryption: - Perform computation on encrypted data; decrypt only the final result - Currently too slow for large-scale ML, but improving

MODULE 8: SECURITY AND ADVERSARIAL DEFENSE

8.1 Identifying Security Threats

Adversarial attacks:

Evasion attacks (inference time): Craft inputs that cause misclassification without being detected. - White-box attacks: Attacker has full knowledge of model architecture and parameters - FGSM (Fast Gradient Sign Method): add perturbation in the direction of gradient of loss w.r.t. input - PGD (Projected Gradient Descent): iterative FGSM with projection back onto ε-ball - C&W attack: optimization-based; more powerful but slower - Black-box attacks: Attacker only has access to model outputs - Transfer attacks: generate adversarial examples on a substitute model; transfer to target - Query-based attacks: use model predictions to estimate gradients

Poisoning attacks (training time): Inject malicious examples into training data to corrupt the model. - Clean-label poisoning: Inject examples with correct labels but adversarial features - Backdoor/Trojan attacks: Inject examples with a trigger pattern that causes misclassification at inference

Model extraction / Stealing: Repeatedly query a model to reconstruct its parameters or decision boundary.

Model inversion: Use model outputs to reconstruct sensitive training data. Example: reconstructing a face from a face recognition model’s confidence scores.

Membership inference: Determine whether a specific record was in the training data. Enabled by models overfitting to training data.

Prompt injection (for LLMs): Craft inputs that override the model’s system instructions or cause unintended behavior.

Data poisoning via crowdsourced labeling: Compromise annotators to label data incorrectly.

8.2 Defense Strategies

Input validation and preprocessing: - Detect out-of-distribution inputs before they reach the model - Feature squeeze: reduce input precision (e.g., color bit depth) to remove adversarial perturbations - Input transformation: JPEG compression, smoothing, resizing — destroy fine-grained adversarial patterns - Certifiable defenses: guarantee robustness within a defined ε-ball (randomized smoothing)

Adversarial training: - Include adversarial examples in training data - At each training step, generate adversarial examples for the current batch; train on both clean and adversarial - PGD adversarial training (Madry et al.) is the gold standard for robustness - Cost: significant training time increase; slight accuracy drop on clean data

Randomized smoothing: - Smooth the model’s predictions by averaging over Gaussian-noised copies of the input - Provides certified ℓ₂ robustness: guarantees that the prediction doesn’t change within a radius - Model-agnostic but requires many forward passes at inference time

Certified robustness: - Formally verify that no adversarial perturbation within a defined bound can change the prediction - Methods: interval bound propagation, Lipschitz-based bounds, linear programming relaxations - Currently practical only for small networks or tight ε bounds

Backdoor detection and mitigation: - Neural Cleanse: detect backdoors by finding small universal perturbations that flip predictions - STRIP (STRong Intentional Perturbation): detect backdoored inputs by superimposing random patterns - Fine-pruning: prune neurons that are unusually dormant on clean data but active on poisoned data

Differential privacy for defense: - DP training limits the influence of any single training point, making poisoning less effective - Also limits membership inference risk

Access controls and rate limiting: - Limit the number of queries a user can make (defends against model extraction and membership inference) - Add noise to predictions or return only top-k classes - Monitor for unusual query patterns

8.3 Evaluating Tradeoffs Between Security, Interpretability, and Performance

The security-accuracy tradeoff: - Adversarially trained models are more robust but less accurate on clean data (typically 1–10% drop) - Defense mechanisms (input smoothing, randomized smoothing) add latency - Stronger privacy guarantees (smaller ε in DP) reduce model utility

The interpretability-performance tradeoff: - Complex models (deep nets, ensembles) generally outperform interpretable models - Requiring interpretability may constrain model choice (e.g., using a decision tree instead of a neural network) - Post-hoc explanations partially resolve this by explaining complex models — but with reduced fidelity

The interpretability-security tradeoff: - Highly interpretable models are more vulnerable to adversarial attack (attacker can understand the model’s decision boundary) - Obfuscating model internals can be a weak security measure (security through obscurity is not sufficient) - SHAP explanations can be manipulated: a model can produce biased decisions while showing fair explanations on a scrutiny dataset

The privacy-utility tradeoff: - Stronger data protection (more noise in DP, more anonymization) → less accurate models - Federated learning preserves privacy but convergence is slower and accuracy may suffer - De-identification reduces dataset richness

Decision framework for high-risk contexts: 1. Identify the most critical risk: safety failure, privacy breach, discrimination, or adversarial attack 2. Prioritize the mitigation for the highest-risk failure mode 3. Quantify the tradeoff: how much accuracy/interpretability do you sacrifice? 4. Document the decision and the rationale 5. Define monitoring and escalation procedures

Example high-risk tradeoff (medical AI): - Use an interpretable model (logistic regression) even if a neural network would be more accurate, because clinicians need to verify the model’s reasoning before acting - Accept the accuracy cost in exchange for clinical trust and regulatory compliance - Apply differential privacy even if it slightly reduces model performance, because patient privacy is legally and ethically paramount

KEY WORKFLOWS TO KNOW (Order-the-Steps Questions)

LIME explanation workflow: 1. Select instance to explain 2. Generate perturbed samples around instance 3. Query black-box model on perturbed samples 4. Weight samples by proximity to original 5. Fit interpretable model on weighted samples 6. Interpret coefficients as explanation

SHAP analysis workflow: 1. Train model 2. Initialize SHAP explainer (match to model type) 3. Compute SHAP values for dataset 4. Generate summary plot for global importance 5. Generate force/waterfall plot for local explanation 6. Generate dependence plot for feature interactions

Bias audit workflow: 1. Define protected groups 2. Compute performance metrics per group 3. Compute fairness metrics (demographic parity, equalized odds) 4. Identify most problematic disparities 5. Diagnose root cause (data, features, labels, model) 6. Select and apply appropriate mitigation 7. Re-audit to verify improvement

Pre-processing bias mitigation workflow: 1. Identify underrepresented groups in training data 2. Choose mitigation strategy (resampling, reweighting, relabeling) 3. Apply mitigation to training set only (not test set) 4. Train model on mitigated data 5. Evaluate fairness and accuracy on original test set

Adversarial defense workflow: 1. Identify threat model (white-box vs. black-box, evasion vs. poisoning) 2. Evaluate model’s current robustness (accuracy on adversarial examples) 3. Select defense strategy (adversarial training, input preprocessing, certified defense) 4. Implement defense; retrain if necessary 5. Evaluate tradeoff: clean accuracy vs. robust accuracy 6. Deploy with monitoring for adversarial inputs
