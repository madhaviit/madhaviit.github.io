1 Vision-Language Model Architecture with PyTorch — The Complete Expert Guide

Every line in this document is written so that the reader walks away with the ability to draw any modern VLM on a whiteboard, justify every design choice with a paper, and code each component from a blank .py file. We move from first principles to the latest 2024–2025 architectures (LLaVA-NeXT, InternVL-2, Qwen2-VL, Florence-2, Idefics-3, Chameleon, Molmo).

1.1 What is a Vision-Language Model, really?

A Vision-Language Model (VLM) is a single differentiable function

\[ f_\theta : (\mathcal{I}, \mathcal{T}) \;\longrightarrow\; \mathcal{Y} \]

that consumes an image (or video) \(\mathcal{I}\) and a text \(\mathcal{T}\), and emits some target \(\mathcal{Y}\) — a class label, a similarity score, a caption, an answer, a bounding box, a segmentation mask, or another image. Internally every modern VLM decomposes into four building blocks:

  1. Visual encoder \(E_v\) — turns pixels into a sequence (or set) of visual tokens \(\mathbf{V} \in \mathbb{R}^{N_v \times d_v}\).
  2. Text encoder / language model \(E_t\) — turns sub-word tokens into textual representations \(\mathbf{T} \in \mathbb{R}^{N_t \times d_t}\).
  3. Fusion module \(F\) — aligns \(\mathbf{V}\) and \(\mathbf{T}\) in a shared space and lets them interact.
  4. Task head \(H\) — projects the fused representation to the output space.

Every architecture you will ever read about — CLIP, ALIGN, BLIP, BLIP-2, Flamingo, LLaVA, MiniGPT-4, Qwen-VL, Kosmos, PaLI, Florence-2, Chameleon — is a particular choice for these four blocks.

We will now master every block.

1.3 1. Comparing Model Architectures

Modern VLMs split cleanly into four design families. Memorise this taxonomy — examiners love it.

1.3.1 1.1 The Four Families

Family Canonical model Vision encoder Fusion Decoder Trained for
Dual-encoder (contrastive) CLIP, ALIGN, SigLIP ViT-L/14 none (dot-product) none Retrieval, zero-shot classification
Encoder-decoder (fusion-in-encoder) ViLBERT, LXMERT, ALBEF, BLIP ViT or Faster-RCNN cross-attention layers small text decoder VQA, captioning, ITM
Frozen-LLM (projection-style) Flamingo, BLIP-2, LLaVA, MiniGPT-4 frozen ViT (CLIP/EVA) Q-Former / MLP / Resampler frozen or LoRA LLM Open-ended dialog, instruction following
Early-fusion / unified tokenization Fuyu, Chameleon, Gemini-Nano, Unified-IO linear patch projector none (single transformer) single transformer Any-to-any modality

The progression is historical: dual-encoders proved alignment was learnable from scale; encoder-decoders showed cross-attention beats concatenation for VQA; frozen-LLM models exploited LLM emergent abilities while paying a tiny compute price; early-fusion models removed the modality boundary entirely.

1.3.2 1.2 Dual-Encoder: CLIP (Radford et al., 2021)

CLIP — Contrastive Language–Image Pre-training — is the bedrock. Two encoders, one objective:

\[ \mathcal{L}_{\text{CLIP}} = \tfrac{1}{2}\!\left[ \mathcal{L}_{i\to t} + \mathcal{L}_{t \to i} \right], \qquad \mathcal{L}_{i\to t} = -\frac{1}{B}\sum_{i=1}^B \log \frac{\exp(\langle \hat{\mathbf{v}}_i, \hat{\mathbf{t}}_i \rangle / \tau)}{\sum_{j=1}^B \exp(\langle \hat{\mathbf{v}}_i, \hat{\mathbf{t}}_j \rangle / \tau)} \]

where \(\hat{\mathbf{v}}_i\) and \(\hat{\mathbf{t}}_i\) are L2-normalised image and text embeddings, \(\tau\) is a learnable temperature, and \(B\) is the mini-batch size. Read Jay Alammar’s “Illustrated CLIP” and Lilian Weng’s “Contrastive Representation Learning” post for visual intuition.

# Minimal CLIP forward pass
import torch, torch.nn as nn, torch.nn.functional as F

class MiniCLIP(nn.Module):
    def __init__(self, vision_enc, text_enc, d=512):
        super().__init__()
        self.v = vision_enc
        self.t = text_enc
        self.v_proj = nn.Linear(vision_enc.out_dim, d, bias=False)
        self.t_proj = nn.Linear(text_enc.out_dim, d, bias=False)
        self.logit_scale = nn.Parameter(torch.tensor(2.6592))  # log(1/0.07)

    def forward(self, images, tokens):
        v = F.normalize(self.v_proj(self.v(images)), dim=-1)
        t = F.normalize(self.t_proj(self.t(tokens)), dim=-1)
        logits = self.logit_scale.exp() * v @ t.T
        return logits, logits.T   # image-to-text, text-to-image

Key properties:

  • Massive negative samples: with batch size \(B\), every image sees \(B-1\) negatives at no extra cost.
  • Modality gap: the two embedding clouds occupy distinct sub-regions of the unit sphere even after training (Liang et al., 2022). The cosine similarities are still discriminative but the absolute geometry is not aligned — relevant when you want to use CLIP embeddings in downstream classifiers.
  • Scaling laws: CLIP improves roughly as \(\text{acc} \propto \log(N_{\text{params}}) + \log(N_{\text{samples}})\).

1.3.3 1.3 SigLIP — better dual-encoder loss (Zhai et al., 2023)

SigLIP replaces the row-wise softmax with a pair-wise sigmoid:

\[ \mathcal{L}_{\text{SigLIP}} = -\frac{1}{B} \sum_{i=1}^B \sum_{j=1}^B \log \sigma\!\big(z_{ij}\cdot ( \tau \langle \hat{\mathbf{v}}_i, \hat{\mathbf{t}}_j\rangle + b )\big) \]

with \(z_{ij} = +1\) if \(i = j\) else \(-1\), and a learnable bias \(b\). Why is this important?

  • The softmax in CLIP forces global normalisation across the batch — every device must hold all logits during back-prop. SigLIP factorises and tolerates tiny batch sizes (down to 256) while matching CLIP at 32 k.
  • Robust to label-noise because each pair is independently labelled.
  • All modern VLMs (PaliGemma, Idefics-3, Molmo, InternVL-2) use SigLIP-So400m as the vision encoder.

1.3.4 1.4 Encoder-decoder VLMs — BLIP (Li et al., 2022)

BLIP introduces a Multi-modal Mixture-of-Encoder-Decoder (MED): one ViT plus a text transformer that can act as

  1. Unimodal text encoder (for ITC — image-text contrastive)
  2. Image-grounded text encoder (for ITM — image-text matching)
  3. Image-grounded text decoder (for LM — captioning)

All three modes share weights except for the self-attention and cross-attention layers, with prompt tokens distinguishing the role. Training combines three losses:

\[ \mathcal{L}_{\text{BLIP}} = \mathcal{L}_{\text{ITC}} + \mathcal{L}_{\text{ITM}} + \mathcal{L}_{\text{LM}}. \]

BLIP also introduces CapFilt — a captioner and filter trained on the same model that bootstraps clean captions from noisy web alt-text. Many “synthetic-caption” papers (LLaVA-1.5, ShareGPT-4V) descend from this idea.

1.3.5 1.5 Frozen-LLM: BLIP-2 and the Q-Former (Li et al., 2023)

BLIP-2 freezes a vision encoder (ViT-g, EVA-CLIP) and a language model (OPT, Flan-T5) and learns only a tiny Q-Former: 32 learnable query tokens that cross-attend to image features.

Pre-training has two stages:

  1. Representation learning — train Q-Former with ITC + ITM + ITG losses against the frozen vision encoder.
  2. Generative learning — connect Q-Former output to the frozen LLM via a linear projection and train on captioning.

Q-Former parameter count: ~188 M, versus 1.2 B for the LLM and 1 B for the ViT-g. This is the architectural pattern that lets the field iterate on “a tiny adapter, frozen giants on both sides.”

1.3.6 1.6 Frozen-LLM, simpler: LLaVA (Liu et al., 2023)

LLaVA proves that a single linear layer (LLaVA-1) or a 2-layer MLP (LLaVA-1.5) is sufficient to connect a frozen CLIP-ViT-L/14 to a frozen Vicuna/LLaMA-2. Training is two-stage:

  1. Feature alignment: train only the projector on 558 k LAION-CC-SBU caption pairs.
  2. Instruction tuning: unfreeze the LLM and projector, train on 150 k GPT-4-generated multimodal instructions.

LLaVA’s insight is that the LLM is already a multimodal reasoner if you can speak its language — and the projector’s job is to translate visual tokens into something the LLM treats as foreign-language tokens.

LLaVA-NeXT (1.6) and LLaVA-OneVision extend this with AnyRes dynamic tiling: an image is split into a 2×2 or 3×3 grid of 336² patches plus a global thumbnail, each tile is encoded separately, and tokens are concatenated. This unlocks 672² or 1008² resolution without retraining the ViT.

1.3.7 1.7 Cross-attention frozen-LLM: Flamingo / Idefics (Alayrac et al., 2022)

Flamingo interleaves gated cross-attention blocks inside a frozen LLM:

\[ \mathbf{h}^{(l+1)} = \mathbf{h}^{(l)} + \tanh(\alpha_l) \cdot \text{CrossAttn}(\mathbf{h}^{(l)}, \mathbf{V}) + \text{FFN}(\cdot) \]

The scalar gate \(\alpha_l\) is initialised to 0 so the LLM behaves identically to its uni-modal self at the start of training, then gradually opens. A Perceiver Resampler compresses arbitrary numbers of image features into 64 fixed tokens per image, allowing interleaved image-text sequences.

Flamingo handles few-shot multimodal prompting natively because the LLM still sees a single token stream.

1.3.8 1.8 Early-fusion: Fuyu / Chameleon / Unified-IO

Chameleon (Meta, 2024) and Fuyu-8B (Adept, 2023) eliminate the separate vision encoder. Pixels are turned into discrete tokens (Chameleon via VQ-VAE, Fuyu via a linear patch projector) and prepended to text tokens in a single decoder-only transformer.

Pros: no modality gap, generation across modalities, scaling laws are identical to text LLMs.

Cons: vision pre-training of a giant transformer is enormously expensive and discards the inductive biases of CNN/ViT — generally requires 10× more data.

1.3.9 1.9 How to compare two VLMs in one breath

A complete comparison must touch:

  1. Vision encoder choice and resolution (ViT-L/14 @ 224 vs SigLIP-So400m @ 384 vs AnyRes 672).
  2. LLM backbone and parameter count (Vicuna-7B vs LLaMA-2-13B vs Qwen-2.5-72B vs Phi-3).
  3. Fusion topology (concatenation / cross-attn / Q-Former / MLP).
  4. Training data scale and composition (image-text pairs, OCR, charts, video, agentic web traces).
  5. Pre-training objectives (ITC, ITM, LM, NTP, masked-anywhere).
  6. Token budget per image (1, 32, 64, 144, 256, 729, 2880…).
  7. Resolution strategy (fixed / dynamic tiling / native any-resolution / native-pack).
  8. Benchmark numbers (MMMU, MathVista, DocVQA, ChartQA, MMBench, VQAv2, AI2D, BLINK).

Memorise this list; it is the rubric for any architectural-comparison question.

1.4 2. Learning Representations

1.4.1 2.1 Why “representation learning” deserves its own section

A representation is a vector that carries information about the input in a form that is useful to downstream tasks while being invariant to nuisance variation. Bengio’s 2013 “Representation Learning: A Review and New Perspectives” defines the three desiderata: smoothness, multiple explanatory factors, and hierarchy. Lilian Weng’s “Self-Supervised Learning” post (Nov 2019, updated 2021) is the canonical free tutorial.

There are four ways a VLM learns representations.

1.4.2 2.2 Contrastive (InfoNCE) learning

The InfoNCE objective (van den Oord et al., 2018) is the workhorse:

\[ \mathcal{L}_{\text{InfoNCE}} = -\mathbb{E}\!\left[ \log \frac{\exp(s(q, k_+)/\tau)}{\exp(s(q,k_+)/\tau) + \sum_{k_-} \exp(s(q,k_-)/\tau)} \right] \]

where \(q\) is a query embedding, \(k_+\) is the positive (paired) key, and the sum runs over \(N-1\) negatives. As \(N \to \infty\), InfoNCE lower-bounds the mutual information \(I(q;k_+)\).

For VLMs the query/positive pair is an (image, text) sample; the negatives are the other items in the batch (CLIP) or items in a queue (MoCo / ALBEF). The key engineering questions are:

  • How many negatives? More is better up to ~32 k for CLIP; the marginal gain plateaus.
  • Hard-negative mining: ALBEF samples cross-modal negatives weighted by current similarity scores — this dramatically accelerates convergence.
  • Temperature \(\tau\): smaller \(\tau\) → sharper distribution → harder optimisation but better separation. Setting \(\tau\) learnable (CLIP) often beats fixed.
  • Momentum encoder: a slow EMA copy of the encoder produces consistent keys (He et al., MoCo). ALBEF uses a momentum distillation loss too.

1.4.3 2.3 Image-Text Matching (ITM)

A binary cross-entropy classifier sits on top of the fused [CLS] token and predicts “is this image-text pair a real match?” Half the batch is real pairs, half are hard negatives sampled with weights proportional to current similarity. ITM is cheap and effective but only useful when the model is encoder-style (it needs cross-attention).

1.4.4 2.4 Image-grounded language modelling (LM)

This is plain next-token prediction conditioned on the image:

\[ \mathcal{L}_{\text{LM}} = -\sum_{t=1}^T \log p_\theta(w_t \mid w_{<t}, \mathbf{V}). \]

It produces a generative model from the very start. Required for any model that must caption or chat. LLaVA and BLIP-2 use a masked variant where the loss is computed only on assistant tokens, not user-instruction tokens — this is the “selective masking” detail you should always mention.

1.4.5 2.5 Masked modelling (BEiT, FLAVA, MAE-style)

Mask 50–75 % of image patches and 15 % of text tokens; predict the missing ones from the remaining context. The image targets can be:

  • Raw pixels (MAE).
  • Discrete visual tokens from a pre-trained dVAE (BEiT, BEiT-3).
  • Features from a separate teacher (data2vec, EVA — predicts CLIP features).

Masked modelling rewards local representations, while contrastive learning rewards global invariances — combining them (FLAVA, BEiT-3) gives both.

1.4.6 2.6 Generative captioning and OCR self-supervision

Florence-2 (Microsoft, 2024) demonstrates that prompting a single sequence-to-sequence model with task-specific prompts (<OD> for detection, <CAPTION> for captioning, <OCR> for text reading) is sufficient — provided the pre-training data covers all tasks. The representation learned is the shared encoder hidden state.

1.4.7 2.7 The alignment / uniformity decomposition

Wang & Isola (2020) show that contrastive losses can be decomposed into

\[ \mathcal{L}_{\text{align}} = \mathbb{E}\big[\|f(x) - f(x^+)\|^2\big], \qquad \mathcal{L}_{\text{uniform}} = \log \mathbb{E}\big[\exp(-2 \|f(x) - f(y)\|^2)\big]. \]

A good encoder minimises both: positives collapse together (alignment), random samples spread on the unit sphere (uniformity). This decomposition is a great mental model for why contrastive learning works.

1.4.8 2.8 Probing learned representations

Three protocols to evaluate a frozen representation:

  1. Linear probe: freeze the encoder, train a linear classifier on a labelled dataset.
  2. k-NN probe: classify by the dominant label among the \(k\) nearest training embeddings.
  3. Zero-shot: in CLIP-style models, build a classifier by encoding text prompts and computing dot products.

A model that wins on zero-shot but loses on linear probe is producing a task-specific representation rather than a general one. Watch for this in papers.

1.4.9 2.9 Putting it all together — what to use when

  • Retrieval / zero-shot classification → InfoNCE only (CLIP, SigLIP).
  • VQA / classification head → InfoNCE + ITM + masked LM (ALBEF, BLIP).
  • Open-ended dialog / OCR / detection → generative LM on top of a frozen contrastive backbone (BLIP-2, LLaVA, Florence-2).
  • Any-to-any / image generation included → unified tokenisation + LM (Chameleon, Unified-IO 2).

1.5 3. Customizing Visual Encoders

1.5.1 3.1 The three encoder families

Family Examples Inductive bias Token count for 224² When to use
CNN ResNet-50, ConvNeXt-XL, RegNetY Locality, translation equivariance 49 (after 7×7 pool) Edge devices, dense prediction
Vision Transformer (ViT) ViT-L/14, DeiT, EVA-02 Almost none; global attention 256 (16² patches) or 196 (14²) Anything that scales
Hybrid / Hierarchical Swin, ConvNeXt-V2, BEiT-3 Hierarchical pooling + windowed attention varies Object detection, segmentation

A VLM almost always uses a ViT today because (a) it produces a clean sequence of tokens that drops straight into a transformer-based fusion module, and (b) it scales with data better than CNNs.

1.5.2 3.2 The ViT in one page

Given an image \(x \in \mathbb{R}^{H \times W \times 3}\) and patch size \(P\):

  1. Patchify: reshape into \(N = HW/P^2\) patches of size \(P \times P \times 3 = 3P^2\) floats, then linearly project to dimension \(d\). This is a convolution with kernel \(P\) and stride \(P\).

  2. Prepend [CLS] token: a learnable vector \(\mathbf{z}_{\text{cls}} \in \mathbb{R}^d\).

  3. Add positional embeddings \(\mathbf{p} \in \mathbb{R}^{(N+1)\times d}\) — learned (ViT) or sinusoidal (CLIP-RN) or RoPE-2D (Qwen2-VL).

  4. L transformer blocks: each is

    \[ \mathbf{z}^\prime = \mathbf{z} + \text{MHSA}(\text{LN}(\mathbf{z})), \quad \mathbf{z} = \mathbf{z}^\prime + \text{MLP}(\text{LN}(\mathbf{z}^\prime)). \]

  5. Final LN, output: either the [CLS] embedding or all \(N\) patch embeddings.

Variant cheat-sheet:

  • ViT-B/16: 12 layers, 12 heads, \(d=768\) — 86 M params.
  • ViT-L/14: 24 layers, 16 heads, \(d=1024\) — 304 M params.
  • ViT-H/14: 32 layers, 16 heads, \(d=1280\) — 632 M params.
  • ViT-g/14: 40 layers, 16 heads, \(d=1408\) — 1.0 B params.
  • SigLIP-So400m/14: 27 layers, 16 heads, \(d=1152\) — 400 M params, optimal “shape” by SoViT (Alabdulmohsin et al., 2023).
  • EVA-02-Large: ViT-L with RoPE-2D + xattn + sub-LN + masked image modelling pre-training; the strongest ~300 M parameter encoder publicly.

1.5.3 3.3 Loading a pre-trained ViT in PyTorch

import torch, torch.nn as nn
from transformers import CLIPVisionModel, SiglipVisionModel

# 1) CLIP-ViT-L/14 from OpenAI weights
clip = CLIPVisionModel.from_pretrained("openai/clip-vit-large-patch14-336")
# 2) SigLIP-So400m from Google (used by Gemini-Nano, PaliGemma, Idefics-3)
sig  = SiglipVisionModel.from_pretrained("google/siglip-so400m-patch14-384")

# Both return BaseModelOutputWithPooling: .last_hidden_state [B, 1+N, d], .pooler_output [B, d]
img = torch.randn(2, 3, 336, 336)

1.5.4 3.4 Customising the ViT — six surgical operations

You will be asked to modify an encoder. Six standard surgeries:

1.5.4.1 (a) Patch-size surgery

To support higher resolution without re-training, interpolate positional embeddings:

def interpolate_pos_embed(pos_embed, new_grid):
    # pos_embed: [1, 1+N_old, d]; new_grid: int H_new == W_new in patches
    cls, grid = pos_embed[:, :1], pos_embed[:, 1:]
    N_old = grid.shape[1]; gs_old = int(N_old**0.5)
    grid = grid.reshape(1, gs_old, gs_old, -1).permute(0, 3, 1, 2)
    grid = torch.nn.functional.interpolate(grid, size=(new_grid, new_grid),
                                           mode="bicubic", align_corners=False)
    grid = grid.permute(0, 2, 3, 1).reshape(1, new_grid * new_grid, -1)
    return torch.cat([cls, grid], dim=1)

CLIP-ViT-L/14 trained at 224 (16² patches) generalises to 336 (24²) with this trick alone.

1.5.4.2 (b) AnyRes / dynamic tiling

LLaVA-NeXT and Qwen2-VL split a high-res image into multiple ViT inputs.

def anyres_tiles(image, base=336, candidate_grids=((1,1),(1,2),(2,1),(2,2),(1,3),(3,1))):
    H, W = image.shape[-2:]
    best = max(candidate_grids,
               key=lambda g: min(W / (g[1]*base), H / (g[0]*base)))
    gh, gw = best
    image = torch.nn.functional.interpolate(
        image[None], size=(gh*base, gw*base), mode="bicubic", align_corners=False)[0]
    tiles = image.unfold(1, base, base).unfold(2, base, base)         # [3, gh, gw, base, base]
    tiles = tiles.permute(1, 2, 0, 3, 4).reshape(gh*gw, 3, base, base)
    thumb = torch.nn.functional.interpolate(
        image[None], size=(base, base), mode="bicubic", align_corners=False)[0]
    return torch.cat([thumb[None], tiles], dim=0)                     # [1+gh*gw, 3, b, b]

1.5.4.3 (c) RoPE-2D for any-resolution

Replace the 1-D positional embedding with rotary embeddings indexed by \((y, x)\):

\[ \text{RoPE}_{2D}(\mathbf{q}_{y,x}) = \begin{pmatrix} R_y & 0 \\ 0 & R_x \end{pmatrix} \mathbf{q}_{y,x} \]

where \(R_y\) and \(R_x\) are block-diagonal rotation matrices on half of the channels each. This is how Qwen2-VL and InternVL-2 support truly arbitrary resolutions.

1.5.4.4 (d) Register tokens (Darcet et al., 2024)

Add 4–8 extra learnable tokens to the input that are discarded at output. Empirically removes the “artifacts” in attention maps reported by Caron et al. and improves dense prediction.

1.5.4.5 (e) Token dropping / pruning

DynamicViT, ATS, ToMe — remove or merge tokens whose attention score is below a threshold to cut FLOPs by 30–50 %. ToMe (token-merging) is the most popular: at each layer, find the \(r\) most similar token pairs by cosine and average them.

1.5.4.6 (f) Multi-scale feature extraction

Concatenate features from layers \(\{l_1, l_2, l_3\}\) — e.g. layer 8 (textures), 16 (parts), 24 (objects). VAR, VeCLIP and LLaVA-1.5 with high-res use this trick.

1.5.5 3.5 Convolutional and hybrid encoders

When you need a low-token-count encoder for mobile or for video, ConvNeXt-V2 and EfficientFormer-V2 produce 49 or 196 tokens at much lower FLOPs. For dense prediction (detection, segmentation) the hierarchical encoders matter — Swin-Transformer uses shifted windows, InternImage uses deformable convolutions; the InternVL series uses these as the vision tower.

1.5.6 3.6 Practical PyTorch tips

  • torch.compile(encoder, mode="reduce-overhead") removes 20–30 % wall-clock on H100 for ViT-L.
  • Use torch.nn.functional.scaled_dot_product_attention (PyTorch ≥ 2.0) — it dispatches to Flash-Attention 2 automatically.
  • Freeze the encoder with for p in encoder.parameters(): p.requires_grad = False; encoder.eval() — the .eval() matters because of BatchNorm/Dropout (Dropout is no-op but LayerNorm running stats are not stored in CLIP-style ViTs, so .eval() is safe).
  • Cast pre-trained weights to bf16 with model.to(torch.bfloat16) when training fusion — the tiny accuracy loss is invisible and memory drops 50 %.

1.6 4. Extracting Visual Embeddings

1.6.1 4.1 Where to pull features from

A ViT block-by-block output has shape [B, 1+N, d]. Five canonical extraction strategies:

Strategy Tensor used Pros Cons
CLS token hidden_states[-1][:, 0] One vector per image, fast Loses spatial info
Mean pool of patches hidden_states[-1][:, 1:].mean(1) Robust, no CLS-collapse Loses CLS bias
All patch tokens hidden_states[-1][:, 1:] Spatial reasoning possible \(N\) tokens to fuse
Penultimate layer patches hidden_states[-2][:, 1:] Less “task-specialised” than the final One layer to specify
Multi-scale concat [h[l1], h[l2], h[l3]] Captures textures + objects 2-3× tokens

CLIP’s image_features uses CLS + a final projection layer. LLaVA-1.5 uses penultimate-layer patch tokens (24×24 = 576 tokens), justified empirically: the last layer is too task-specialised to the contrastive objective.

1.6.2 4.2 Standard PyTorch extraction

from transformers import CLIPVisionModel
import torch.nn as nn

class VisualFeatureExtractor(nn.Module):
    def __init__(self, model_name="openai/clip-vit-large-patch14-336", select_layer=-2):
        super().__init__()
        self.vit = CLIPVisionModel.from_pretrained(model_name, output_hidden_states=True)
        self.select_layer = select_layer  # -2 == penultimate

    @torch.no_grad()
    def forward(self, pixel_values):
        out = self.vit(pixel_values=pixel_values, output_hidden_states=True)
        feats = out.hidden_states[self.select_layer][:, 1:]   # drop CLS, [B, N, d]
        return feats

1.6.3 4.3 Projector designs — the bridge to language

The projector \(P : \mathbb{R}^{d_v} \to \mathbb{R}^{d_t}\) converts visual feature dimension to LLM embedding dimension.

1. Linear (LLaVA-1) — nn.Linear(1024, 4096). Two-step training: warm-up only this layer, then unfreeze everything.

2. MLP-2 (LLaVA-1.5) — Linear → GELU → Linear. +1.0 MMMU vs linear.

3. Q-Former (BLIP-2) — a 12-layer BERT initialised from BERT-base, with 32 learnable query tokens that cross-attend to image features and self-attend to each other. Output: 32 vectors regardless of input resolution.

class QFormer(nn.Module):
    def __init__(self, n_queries=32, d=768, n_layers=12, n_heads=12):
        super().__init__()
        self.queries = nn.Parameter(torch.randn(1, n_queries, d) * 0.02)
        self.layers = nn.ModuleList([QFormerLayer(d, n_heads) for _ in range(n_layers)])

    def forward(self, vision_tokens):
        q = self.queries.expand(vision_tokens.size(0), -1, -1)
        for layer in self.layers:
            q = layer(q, vision_tokens)
        return q                                      # [B, 32, d]

4. Perceiver Resampler (Flamingo) — like Q-Former but the queries also see learned positional embeddings indexed by time-of-frame (videos!), and the cross-attention is gated.

5. Pixel-Shuffle (InternVL-2) — concatenate \(2 \times 2\) neighbouring patch tokens then linearly project. Cuts token count 4× while keeping local information. Used in InternVL-2 to bring 24×24 down to 12×12 = 144 tokens.

6. C-Abstractor (Honeybee) — convolutions on the 2-D patch grid then flatten. Preserves locality, modest parameter count.

1.6.4 4.4 What about positional encoding for visual tokens inside the LLM?

There are three schools:

  • Use the ViT’s own pos-embed and add nothing: the LLM’s RoPE handles tokens by their position in the sequence but not their spatial position. This is what LLaVA does.
  • Add 2-D positional information via the projector (interleave spatial coordinates).
  • Use M-RoPE / RoPE-2D in the LLM itself (Qwen2-VL): the LLM is re-trained to interpret rotary frequencies along temporal, height and width axes.

1.6.5 4.5 Quick sanity check — do my visual embeddings preserve information?

After training, run a linear probe: freeze the encoder + projector and train a single linear classifier on ImageNet-1k. If accuracy drops more than 5 % vs the original ViT, your projector is destroying information.

1.7 5. Customizing Text Encoders

1.7.1 5.1 The decision tree

The first question is what role does text play in your VLM?

  • Symmetric retrieval (CLIP-style) → bidirectional encoder (BERT/RoBERTa) or a causal Transformer with attention-mask = bidirectional only for the [EOS] vector. CLIP uses a causal Transformer with EOS pooling.
  • Encoder-decoder VQA (BLIP-style) → BERT-base initialised, repurposed via prompt tokens.
  • Generative dialog (LLaVA-style) → a full LLM (Vicuna, LLaMA, Mistral, Phi, Qwen, Gemma).

1.7.2 5.2 Tokenisation primer

Three production algorithms:

Algorithm Used by Vocab Notes
BPE GPT-2, CLIP, LLaMA-1 32 k–50 k Greedy merging of frequent pairs
WordPiece BERT 30 k Likelihood-based merging
SentencePiece / Unigram T5, LLaMA-2, Mistral, Gemma 32 k–128 k Probabilistic, language-agnostic

For VLMs, the tokenizer must include special tokens for image placeholders, e.g. <image>, <video>, <roi>, <grounding>. LLaVA reserves the <image> token; Kosmos-2 reserves <box> and <phrase>; Florence-2 reserves <OD>, <OCR>, <CAPTION>.

1.7.3 5.3 BERT-family text encoder

A bidirectional encoder:

from transformers import AutoTokenizer, BertModel
tok = AutoTokenizer.from_pretrained("bert-base-uncased")
bert = BertModel.from_pretrained("bert-base-uncased")
out = bert(**tok(["a dog on grass"], return_tensors="pt", padding=True))
print(out.last_hidden_state.shape, out.pooler_output.shape)

The pooler_output is a Linear(d, d) → tanh applied to the [CLS] vector — only meaningful when fine-tuned with a classification head. For retrieval you usually use mean pooling of all token embeddings with attention mask.

1.7.4 5.4 CLIP-style causal text encoder

from transformers import CLIPTextModel, CLIPTokenizer
tok = CLIPTokenizer.from_pretrained("openai/clip-vit-large-patch14")
text = CLIPTextModel.from_pretrained("openai/clip-vit-large-patch14")
inputs = tok(["a photo of a cat"], padding=True, return_tensors="pt")
out = text(**inputs)
# CLIP pools at the EOS position
eos_idx = inputs.input_ids.argmax(dim=-1)
pooled = out.last_hidden_state[torch.arange(eos_idx.size(0)), eos_idx]

This pooling-at-EOS detail is famously subtle: because of greedy BPE, the largest token id in the sequence is the EOS token (49407 in CLIP’s tokenizer), so argmax selects it. If you fine-tune CLIP with a tokenizer that has a different id for EOS, you must change this.

1.7.6 5.6 Surgical modifications

Just like the visual side:

  • Add special tokens: tokenizer.add_special_tokens({"additional_special_tokens": ["<image>", "<im_patch>"]}) then model.resize_token_embeddings(len(tokenizer)). Critical: new embeddings are initialised randomly — initialise them to the mean of all existing token embeddings to stabilise training.
  • Extend context with RoPE re-scaling: replace \(\theta = 10000\) with \(\theta = 500000\) (NTK-aware scaling) or use YaRN (Peng et al., 2023). Done by Qwen2-VL to support 32 k visual+text tokens.
  • Replace the LLM with a Mixture-of-Experts version: same fusion module, swap the dense FFN for a routed FFN. Used by MoE-LLaVA, DeepSeek-VL2.
  • Add LoRA adapters to specific layers instead of full fine-tuning (see Optimization doc).
  • Freeze the LM entirely and rely on the projector to “speak” the LM’s language — the BLIP-2 / LLaVA-1 approach.

1.7.7 5.7 Causal masking with visual tokens

Inside the LLM, visual tokens occupy specific positions in the sequence. There are three masking conventions:

  • Pure causal: visual tokens are causally masked just like text. Easiest, default.
  • Visual-prefix bidirectional: visual tokens can attend to all other visual tokens but not text after them; text is causal. Used by some early models.
  • Bidirectional vision, causal text (Idefics-3): better preserves the ViT’s information.

The implementation detail is the attention mask, not a different module.

1.8 6. Extracting Textual Embeddings

1.8.1 6.1 Pooling choices

Pooling Formula Use case
CLS / [SOS] h[:, 0] BERT classifiers
EOS h[arange(B), eos_idx] CLIP retrieval
Mean (mask-aware) \(\frac{\sum_t m_t h_t}{\sum_t m_t}\) Sentence-BERT, MPNet
Max \(\max_t h_t\) rarely used
Last-token h[:, -1] LLM-based retrievers (E5-Mistral, GTE-Qwen2)

PyTorch mean-pool with mask:

def mean_pool(hidden, attention_mask):
    mask = attention_mask.unsqueeze(-1).float()
    return (hidden * mask).sum(1) / mask.sum(1).clamp(min=1e-6)

1.8.2 6.2 Sentence-Transformers and Matryoshka Representation Learning

When you want a single vector per sentence, the Sentence-BERT family (Reimers & Gurevych, 2019) fine-tunes BERT with a siamese contrastive loss. MRL (Kusupati et al., 2022) trains the same embedding to be useful when truncated to multiple dimensions \(\{64, 128, 256, 512, 768\}\) — relevant for VLMs because shorter embeddings mean cheaper retrieval at deployment time.

1.8.3 6.3 Text embeddings inside a generative VLM

When the LLM is the fusion module (LLaVA, Qwen-VL), there is no separate text embedding — the LLM’s own token embeddings are the textual representations. Your fusion module simply produces visual tokens with the same dimensionality and the LLM concatenates them. The “embedding extraction” question becomes: do I want the embedding before any attention, after the first layer, or at the final hidden state?

For downstream tasks where you need a fixed-size text vector from a decoder-only LM, use the last token’s hidden state (E5-Mistral approach), or apply an extra <|END|> token and pool there.

1.8.4 6.4 Practical PyTorch — multi-modal retrieval

import torch.nn.functional as F

def encode_text(text_model, tokenizer, sentences, device="cuda"):
    tok = tokenizer(sentences, padding=True, truncation=True,
                    max_length=77, return_tensors="pt").to(device)
    with torch.no_grad():
        out = text_model(**tok)
    eos_idx = tok.input_ids.argmax(dim=-1)
    h = out.last_hidden_state
    pooled = h[torch.arange(h.size(0), device=device), eos_idx]
    return F.normalize(pooled, dim=-1)

1.9.2 7.2 Cross-attention — first principles

Multi-head attention with separate sources for Q and K/V:

\[ \text{CrossAttn}(\mathbf{Q}, \mathbf{K}, \mathbf{V}) = \text{softmax}\!\left(\frac{\mathbf{Q}\mathbf{K}^\top}{\sqrt{d_k}}\right)\mathbf{V}, \qquad \mathbf{Q}=\mathbf{X}_t W_Q,\; \mathbf{K}=\mathbf{X}_v W_K,\; \mathbf{V}=\mathbf{X}_v W_V. \]

PyTorch implementation:

class CrossAttention(nn.Module):
    def __init__(self, d_text, d_vis, n_heads=8, d_head=64):
        super().__init__()
        d = n_heads * d_head
        self.n_heads = n_heads; self.d_head = d_head
        self.to_q = nn.Linear(d_text, d, bias=False)
        self.to_k = nn.Linear(d_vis,  d, bias=False)
        self.to_v = nn.Linear(d_vis,  d, bias=False)
        self.to_out = nn.Linear(d, d_text)

    def forward(self, x_text, x_vis, mask=None):
        B, Nt, _ = x_text.shape
        Nv = x_vis.shape[1]
        q = self.to_q(x_text).view(B, Nt, self.n_heads, self.d_head).transpose(1,2)
        k = self.to_k(x_vis ).view(B, Nv, self.n_heads, self.d_head).transpose(1,2)
        v = self.to_v(x_vis ).view(B, Nv, self.n_heads, self.d_head).transpose(1,2)
        attn = torch.nn.functional.scaled_dot_product_attention(q, k, v, attn_mask=mask)
        attn = attn.transpose(1,2).reshape(B, Nt, -1)
        return self.to_out(attn)

Note the use of scaled_dot_product_attention — PyTorch 2 dispatches to Flash-Attention 2 when shapes and dtypes align.

1.9.3 7.3 Gated cross-attention (Flamingo)

Add a learnable gate so the LLM starts as its uni-modal self:

class GatedXAttn(nn.Module):
    def __init__(self, d_text, d_vis, n_heads):
        super().__init__()
        self.xattn = CrossAttention(d_text, d_vis, n_heads)
        self.alpha_xattn = nn.Parameter(torch.zeros(1))
        self.ff   = nn.Sequential(nn.Linear(d_text, 4*d_text), nn.GELU(), nn.Linear(4*d_text, d_text))
        self.alpha_ff    = nn.Parameter(torch.zeros(1))

    def forward(self, x_text, x_vis):
        x_text = x_text + torch.tanh(self.alpha_xattn) * self.xattn(x_text, x_vis)
        x_text = x_text + torch.tanh(self.alpha_ff)    * self.ff(x_text)
        return x_text

Insert one such block every \(K\) LLM layers (Flamingo: \(K=4\) for the smaller variant). The initial zero-gate is critical: it makes the pre-trained LLM’s behaviour an exact starting point.

1.9.4 7.4 The Q-Former in detail

Q-Former has two pathways because it is trained with three losses (ITC, ITM, ITG) that need different attention masks. A single forward pass routes the queries through different self-attention masks depending on the task token:

  • For ITC: queries can only see each other; text can only see text. The two CLS representations contrast.
  • For ITM: queries see image features and full bidirectional text — used to compute a binary score.
  • For ITG: queries see image features; text is causally masked (so the model can be used as a captioner).
class QFormerLayer(nn.Module):
    def __init__(self, d=768, n_heads=12):
        super().__init__()
        self.self_attn  = nn.MultiheadAttention(d, n_heads, batch_first=True)
        self.cross_attn = nn.MultiheadAttention(d, n_heads, batch_first=True)
        self.ff = nn.Sequential(nn.Linear(d, 4*d), nn.GELU(), nn.Linear(4*d, d))
        self.ln1, self.ln2, self.ln3 = nn.LayerNorm(d), nn.LayerNorm(d), nn.LayerNorm(d)

    def forward(self, q, v):
        h = self.ln1(q)
        q = q + self.self_attn(h, h, h, need_weights=False)[0]
        h = self.ln2(q)
        q = q + self.cross_attn(h, v, v, need_weights=False)[0]
        q = q + self.ff(self.ln3(q))
        return q

1.9.5 7.5 MLP projector (LLaVA-1.5)

class MLPProjector(nn.Module):
    def __init__(self, d_vis, d_lm):
        super().__init__()
        self.proj = nn.Sequential(
            nn.Linear(d_vis, d_lm),
            nn.GELU(),
            nn.Linear(d_lm, d_lm),
        )
    def forward(self, x):                              # x: [B, N, d_vis]
        return self.proj(x)                            # -> [B, N, d_lm]

Used by 90 % of recent open-source VLMs. Initialization tip: zero-init the second linear’s bias and use Kaiming for weights to keep activations bounded.

1.9.6 7.6 Pixel-Shuffle / Patch-merging projector

InternVL-2’s trick to cut visual tokens 4×:

def pixel_shuffle_2d(x, scale=2):
    # x: [B, N, d] with N = h*w; halve spatial, multiply channels by scale**2
    B, N, d = x.shape
    h = w = int(N**0.5)
    x = x.view(B, h, w, d)
    x = x.view(B, h//scale, scale, w//scale, scale, d).permute(0, 1, 3, 2, 4, 5)
    x = x.reshape(B, (h//scale) * (w//scale), d * scale * scale)
    return x

class PixelShuffleProjector(nn.Module):
    def __init__(self, d_vis, d_lm, scale=2):
        super().__init__()
        self.scale = scale
        self.proj = nn.Sequential(
            nn.LayerNorm(d_vis * scale * scale),
            nn.Linear(d_vis * scale * scale, d_lm),
            nn.GELU(),
            nn.Linear(d_lm, d_lm),
        )
    def forward(self, x):
        return self.proj(pixel_shuffle_2d(x, self.scale))

1.9.7 7.7 Adapter / LoRA fusion

Insert a low-rank adapter into the LLM’s FFN or attention layers, with conditioning from visual features:

\[ \Delta \mathbf{h} = \alpha \cdot \text{cross-attn}(\mathbf{h}, \mathbf{V}) \cdot B^\top A^\top \]

where \(A \in \mathbb{R}^{r \times d}\), \(B \in \mathbb{R}^{d \times r}\), \(r \ll d\). LLaMA-Adapter-V2 does exactly this.

1.9.8 7.8 Choosing a fusion design — a decision table

Situation Recommended fusion
Train a CLIP-replacement on 1 B pairs None — dual encoder + InfoNCE/SigLIP
Train a chatbot on a frozen LLM with limited compute MLP projector + LoRA
Train a model that must answer about 100s of frames Q-Former + temporal pos-embed
Need few-shot in-context learning across images Gated cross-attention (Flamingo)

Need image generation too Unified VQ-VAE tokens + LM (Chameleon)

1.9.9 7.9 Common pitfalls

  • Forgetting LayerNorm on the projector input — visual features and LM embeddings live on different scales; un-normalised, the LM ignores them.
  • Wrong sequence length — never forget to pad and use attention_mask.
  • Position confusion: when you prepend visual tokens, the LLM’s RoPE indexes them as if they were text. If your LLM has RoPE θ=10000 trained on 4 k context, suddenly seeing 576 extra tokens can blow context length.
  • Mixed dtypes: keep projector in bfloat16 matching the LLM. A float32 projector silently up-casts the entire compute graph.

1.10 8. Assembling Multimodal Architecture

1.10.1 8.1 The seven-step recipe

Every modern projection-style VLM is assembled the same way. Memorise these seven steps.

  1. Pick the vision encoder. Default: SigLIP-So400m/14 @ 384. Faster + smaller: CLIP-ViT-L/14 @ 336. Best dense: DINOv2-Giant.
  2. Pick the LLM backbone. Default: Qwen2.5-7B-Instruct. Tiny: Phi-3-mini. Huge: LLaMA-3-70B.
  3. Pick the resolution strategy. Single-tile 384², or AnyRes 4-tile, or native any-resolution with M-RoPE.
  4. Pick the projector. Default: 2-layer MLP. Token-efficient: Pixel-shuffle ×2 + MLP. Few queries: Q-Former-32.
  5. Decide what is frozen. Stage 1 (alignment): freeze ViT and LLM, train projector. Stage 2 (instruction): unfreeze LLM (or LoRA), keep ViT frozen or unfreeze the last 4 layers.
  6. Choose the training data. Pre-train on captions (LAION-2B, COYO, DataComp). Instruction-tune on LLaVA-Mix-665k, ShareGPT-4V, LLaVA-OneVision-mid.
  7. Choose evaluation suite. MMMU, MathVista, DocVQA, ChartQA, AI2D, MMBench, BLINK, VQAv2, OK-VQA, TextVQA.

1.10.2 8.2 A complete reference implementation

import torch
import torch.nn as nn
from transformers import (
    AutoTokenizer, AutoModelForCausalLM,
    SiglipVisionModel, SiglipImageProcessor,
)


class VLM(nn.Module):
    """A LLaVA-style VLM: SigLIP vision tower + MLP projector + frozen-or-LoRA LLM."""

    def __init__(
        self,
        vision_name="google/siglip-so400m-patch14-384",
        llm_name="Qwen/Qwen2.5-7B-Instruct",
        image_token="<image>",
        select_layer=-2,
    ):
        super().__init__()
        self.tokenizer = AutoTokenizer.from_pretrained(llm_name)
        if image_token not in self.tokenizer.get_vocab():
            self.tokenizer.add_special_tokens({"additional_special_tokens": [image_token]})
        self.image_token_id = self.tokenizer.convert_tokens_to_ids(image_token)

        self.vision = SiglipVisionModel.from_pretrained(vision_name)
        self.processor = SiglipImageProcessor.from_pretrained(vision_name)
        self.llm = AutoModelForCausalLM.from_pretrained(
            llm_name, torch_dtype=torch.bfloat16, attn_implementation="flash_attention_2"
        )
        self.llm.resize_token_embeddings(len(self.tokenizer))

        d_vis = self.vision.config.hidden_size
        d_lm = self.llm.config.hidden_size
        self.projector = nn.Sequential(
            nn.LayerNorm(d_vis),
            nn.Linear(d_vis, d_lm),
            nn.GELU(),
            nn.Linear(d_lm, d_lm),
        ).to(torch.bfloat16)

        self.select_layer = select_layer

    def freeze_for_alignment(self):
        for p in self.vision.parameters():
            p.requires_grad = False
        for p in self.llm.parameters():
            p.requires_grad = False

    def encode_vision(self, pixel_values):
        out = self.vision(pixel_values=pixel_values, output_hidden_states=True)
        feats = out.hidden_states[self.select_layer]                    # [B, N, d_vis]
        return self.projector(feats)                                     # [B, N, d_lm]

    def _embed_inputs(self, input_ids, pixel_values):
        """Replace every <image> token's embedding with vision tokens."""
        inputs_embeds = self.llm.get_input_embeddings()(input_ids)       # [B, T, d_lm]
        vision_embeds = self.encode_vision(pixel_values)                 # [B, N, d_lm]
        merged, new_attn_mask = [], []
        for b in range(input_ids.size(0)):
            img_pos = (input_ids[b] == self.image_token_id).nonzero(as_tuple=True)[0]
            if len(img_pos) == 0:
                merged.append(inputs_embeds[b]); continue
            assert len(img_pos) == 1, "Use one <image> per sample in this minimal demo"
            p = img_pos.item()
            merged.append(torch.cat([inputs_embeds[b, :p],
                                     vision_embeds[b],
                                     inputs_embeds[b, p+1:]], dim=0))
        merged = nn.utils.rnn.pad_sequence(merged, batch_first=True)
        return merged

    def forward(self, input_ids, pixel_values, labels=None):
        inputs_embeds = self._embed_inputs(input_ids, pixel_values)
        outputs = self.llm(inputs_embeds=inputs_embeds, labels=labels)
        return outputs

This is essentially the architecture of LLaVA-1.5, Bunny, Imp, and most open-source 7B-class VLMs published in 2024. Read it line by line until you can rewrite it from memory.

from PIL import Image
model = VLM().eval().to("cuda")
model.freeze_for_alignment()

image = Image.open("dog.jpg").convert("RGB")
pixel_values = model.processor(images=image, return_tensors="pt")["pixel_values"].to("cuda", dtype=torch.bfloat16)

prompt = "<image>\nQuestion: What animal is in the picture?\nAnswer:"
tok = model.tokenizer(prompt, return_tensors="pt").to("cuda")

with torch.no_grad():
    out = model(input_ids=tok.input_ids, pixel_values=pixel_values)
print(out.logits.shape)

1.10.4 8.4 Native-resolution / dynamic-tile assembly (LLaVA-NeXT pattern)

The differences from LLaVA-1.5:

  1. processor returns multiple tiles plus a thumbnail.
  2. encode_vision runs the ViT on each tile (with possible gradient checkpointing), then concatenates patch tokens.
  3. A newline token is inserted between rows of tiles so the LLM can reason about layout.
def encode_anyres(self, tiles):
    # tiles: [B, T, 3, H, W] where T = num_tiles_per_image (variable)
    B, T = tiles.shape[:2]
    tiles_flat = tiles.flatten(0, 1)
    feats = self.encode_vision(tiles_flat)                  # [B*T, N, d_lm]
    feats = feats.view(B, T, -1, feats.size(-1))            # [B, T, N, d_lm]
    return feats.reshape(B, T*feats.size(2), -1)            # concat tiles

1.10.5 8.5 Beyond LLaVA — Qwen2-VL’s “naive dynamic resolution”

Qwen2-VL drops the tiling entirely. The ViT is retrained to operate on arbitrary \(H \times W\) via M-RoPE (multi-dimensional rotary embeddings). The vision encoder outputs a variable number of patch tokens; each is given a (t, y, x) triple consumed by the LLM’s M-RoPE. The trick that makes the LM happy: split RoPE channels into thirds — one for temporal, one for height, one for width.

1.10.6 8.6 Multi-image and video

For multiple images, each gets its own <image> placeholder and its own visual-token block:

<image>{img1}<image>{img2} What is different between these two pictures?

For video, sample \(T\) frames, run each through the ViT, concat with temporal embeddings, optionally compress via Q-Former with shared queries across frames (Video-LLaMA, Video-ChatGPT).

1.10.7 8.7 Sanity checks before training

Before launching a multi-day run, verify:

  1. One sample forward — loss.backward() does not throw.
  2. Loss decreases on a tiny set — overfit 16 samples and watch loss → ~0.
  3. Visual token count matches — print inputs_embeds.shape.
  4. <image> token is masked in labels — the LLM should not be supervised to predict the image token.
  5. No mixed-dtype warnings — everything bf16 except the vision encoder LayerNorms which can stay fp32.

1.10.8 8.8 Open research questions (good interview answers)

  • Token efficiency vs accuracy — is 144 tokens ever enough for OCR / charts? Most evidence says no; pixel-shuffle helps but DocVQA still wants ≥ 729 tokens.
  • Generative vs discriminative pre-training — does masked image modelling (BEiT-3, EVA) beat contrastive (SigLIP) when used as a VLM tower? Currently a tie; depends on downstream task.
  • Encoder-free VLMs — Fuyu / Chameleon throw away the ViT. The 2024 evidence is that they need 5–10× the data; not yet a free lunch.
  • Modality gap — does it matter for downstream? Liang et al. 2022 say yes, but recent work (Schrodi et al. 2024) argues the gap is partly a measurement artefact.
  • Visual tokenisation for generation — VQ-VAEs vs continuous tokens vs Magvit-v2 finite-scalar quantisation.

## What is a Vision-Language Model, really?

A **Vision-Language Model (VLM)** is a single differentiable function

$$
f_\theta : (\mathcal{I}, \mathcal{T}) \;\longrightarrow\; \mathcal{Y}
$$

that consumes an image (or video) $\mathcal{I}$ and a text $\mathcal{T}$, and emits some target $\mathcal{Y}$ — a class label, a similarity score, a caption, an answer, a bounding box, a segmentation mask, or another image. Internally every modern VLM decomposes into four building blocks:

1. **Visual encoder** $E_v$ — turns pixels into a sequence (or set) of visual tokens $\mathbf{V} \in \mathbb{R}^{N_v \times d_v}$.
2. **Text encoder / language model** $E_t$ — turns sub-word tokens into textual representations $\mathbf{T} \in \mathbb{R}^{N_t \times d_t}$.
3. **Fusion module** $F$ — aligns $\mathbf{V}$ and $\mathbf{T}$ in a shared space and lets them interact.
4. **Task head** $H$ — projects the fused representation to the output space.

Every architecture you will ever read about — CLIP, ALIGN, BLIP, BLIP-2, Flamingo, LLaVA, MiniGPT-4, Qwen-VL, Kosmos, PaLI, Florence-2, Chameleon — is a particular choice for these four blocks.

We will now master every block.

## 1. Comparing Model Architectures

Modern VLMs split cleanly into four design families. Memorise this taxonomy — examiners love it.

### 1.1 The Four Families

| Family | Canonical model | Vision encoder | Fusion | Decoder | Trained for |
|---|---|---|---|---|---|
| **Dual-encoder (contrastive)** | CLIP, ALIGN, SigLIP | ViT-L/14 | none (dot-product) | none | Retrieval, zero-shot classification |
| **Encoder-decoder (fusion-in-encoder)** | ViLBERT, LXMERT, ALBEF, BLIP | ViT or Faster-RCNN | cross-attention layers | small text decoder | VQA, captioning, ITM |
| **Frozen-LLM (projection-style)** | Flamingo, BLIP-2, LLaVA, MiniGPT-4 | frozen ViT (CLIP/EVA) | Q-Former / MLP / Resampler | frozen or LoRA LLM | Open-ended dialog, instruction following |
| **Early-fusion / unified tokenization** | Fuyu, Chameleon, Gemini-Nano, Unified-IO | linear patch projector | none (single transformer) | single transformer | Any-to-any modality |

The progression is historical: dual-encoders proved alignment was learnable from scale; encoder-decoders showed cross-attention beats concatenation for VQA; frozen-LLM models exploited LLM emergent abilities while paying a tiny compute price; early-fusion models removed the modality boundary entirely.

### 1.2 Dual-Encoder: CLIP (Radford et al., 2021)

CLIP — **C**ontrastive **L**anguage–**I**mage **P**re-training — is the bedrock. Two encoders, one objective:

$$
\mathcal{L}_{\text{CLIP}} = \tfrac{1}{2}\!\left[ \mathcal{L}_{i\to t} + \mathcal{L}_{t \to i} \right], \qquad
\mathcal{L}_{i\to t} = -\frac{1}{B}\sum_{i=1}^B \log \frac{\exp(\langle \hat{\mathbf{v}}_i, \hat{\mathbf{t}}_i \rangle / \tau)}{\sum_{j=1}^B \exp(\langle \hat{\mathbf{v}}_i, \hat{\mathbf{t}}_j \rangle / \tau)}
$$

where $\hat{\mathbf{v}}_i$ and $\hat{\mathbf{t}}_i$ are L2-normalised image and text embeddings, $\tau$ is a learnable temperature, and $B$ is the mini-batch size. Read Jay Alammar's *"Illustrated CLIP"* and Lilian Weng's *"Contrastive Representation Learning"* post for visual intuition.

```python
# Minimal CLIP forward pass
import torch, torch.nn as nn, torch.nn.functional as F

class MiniCLIP(nn.Module):
    def __init__(self, vision_enc, text_enc, d=512):
        super().__init__()
        self.v = vision_enc
        self.t = text_enc
        self.v_proj = nn.Linear(vision_enc.out_dim, d, bias=False)
        self.t_proj = nn.Linear(text_enc.out_dim, d, bias=False)
        self.logit_scale = nn.Parameter(torch.tensor(2.6592))  # log(1/0.07)

    def forward(self, images, tokens):
        v = F.normalize(self.v_proj(self.v(images)), dim=-1)
        t = F.normalize(self.t_proj(self.t(tokens)), dim=-1)
        logits = self.logit_scale.exp() * v @ t.T
        return logits, logits.T   # image-to-text, text-to-image
```

Key properties:

- **Massive negative samples**: with batch size $B$, every image sees $B-1$ negatives at no extra cost.
- **Modality gap**: the two embedding clouds occupy distinct sub-regions of the unit sphere even after training (Liang et al., 2022). The cosine similarities are still discriminative but the absolute geometry is not aligned — relevant when you want to use CLIP embeddings in downstream classifiers.
- **Scaling laws**: CLIP improves roughly as $\text{acc} \propto \log(N_{\text{params}}) + \log(N_{\text{samples}})$.

### 1.3 SigLIP — better dual-encoder loss (Zhai et al., 2023)

SigLIP replaces the row-wise softmax with a **pair-wise sigmoid**:

$$
\mathcal{L}_{\text{SigLIP}} = -\frac{1}{B} \sum_{i=1}^B \sum_{j=1}^B \log \sigma\!\big(z_{ij}\cdot ( \tau \langle \hat{\mathbf{v}}_i, \hat{\mathbf{t}}_j\rangle + b )\big)
$$

with $z_{ij} = +1$ if $i = j$ else $-1$, and a learnable bias $b$. Why is this important?

- The softmax in CLIP forces global normalisation across the batch — every device must hold all logits during back-prop. SigLIP factorises and tolerates **tiny batch sizes** (down to 256) while matching CLIP at 32 k.
- Robust to label-noise because each pair is independently labelled.
- All modern VLMs (PaliGemma, Idefics-3, Molmo, InternVL-2) use SigLIP-So400m as the vision encoder.

### 1.4 Encoder-decoder VLMs — BLIP (Li et al., 2022)

BLIP introduces a **Multi-modal Mixture-of-Encoder-Decoder (MED)**: one ViT plus a text transformer that can act as

1. Unimodal text encoder (for ITC — image-text contrastive)
2. Image-grounded text encoder (for ITM — image-text matching)
3. Image-grounded text decoder (for LM — captioning)

All three modes **share weights** except for the self-attention and cross-attention layers, with prompt tokens distinguishing the role. Training combines three losses:

$$
\mathcal{L}_{\text{BLIP}} = \mathcal{L}_{\text{ITC}} + \mathcal{L}_{\text{ITM}} + \mathcal{L}_{\text{LM}}.
$$

BLIP also introduces **CapFilt** — a captioner and filter trained on the same model that bootstraps clean captions from noisy web alt-text. Many "synthetic-caption" papers (LLaVA-1.5, ShareGPT-4V) descend from this idea.

### 1.5 Frozen-LLM: BLIP-2 and the Q-Former (Li et al., 2023)

BLIP-2 freezes a vision encoder (ViT-g, EVA-CLIP) **and** a language model (OPT, Flan-T5) and learns only a tiny **Q-Former**: 32 learnable query tokens that cross-attend to image features.

Pre-training has two stages:

1. **Representation learning** — train Q-Former with ITC + ITM + ITG losses against the frozen vision encoder.
2. **Generative learning** — connect Q-Former output to the frozen LLM via a linear projection and train on captioning.

Q-Former parameter count: ~188 M, versus 1.2 B for the LLM and 1 B for the ViT-g. This is the architectural pattern that lets the field iterate on "a tiny adapter, frozen giants on both sides."

### 1.6 Frozen-LLM, simpler: LLaVA (Liu et al., 2023)

LLaVA proves that a **single linear layer (LLaVA-1)** or a **2-layer MLP (LLaVA-1.5)** is sufficient to connect a frozen CLIP-ViT-L/14 to a frozen Vicuna/LLaMA-2. Training is two-stage:

1. **Feature alignment**: train only the projector on 558 k LAION-CC-SBU caption pairs.
2. **Instruction tuning**: unfreeze the LLM and projector, train on 150 k GPT-4-generated multimodal instructions.

LLaVA's insight is that **the LLM is already a multimodal reasoner if you can speak its language** — and the projector's job is to translate visual tokens into something the LLM treats as foreign-language tokens.

LLaVA-NeXT (1.6) and LLaVA-OneVision extend this with **AnyRes** dynamic tiling: an image is split into a 2×2 or 3×3 grid of 336² patches plus a global thumbnail, each tile is encoded separately, and tokens are concatenated. This unlocks 672² or 1008² resolution without retraining the ViT.

### 1.7 Cross-attention frozen-LLM: Flamingo / Idefics (Alayrac et al., 2022)

Flamingo *interleaves* gated cross-attention blocks **inside** a frozen LLM:

$$
\mathbf{h}^{(l+1)} = \mathbf{h}^{(l)} + \tanh(\alpha_l) \cdot \text{CrossAttn}(\mathbf{h}^{(l)}, \mathbf{V}) + \text{FFN}(\cdot)
$$

The scalar gate $\alpha_l$ is initialised to 0 so the LLM behaves identically to its uni-modal self at the start of training, then gradually opens. A **Perceiver Resampler** compresses arbitrary numbers of image features into 64 fixed tokens per image, allowing interleaved image-text sequences.

Flamingo handles few-shot multimodal prompting natively because the LLM still sees a single token stream.

### 1.8 Early-fusion: Fuyu / Chameleon / Unified-IO

Chameleon (Meta, 2024) and Fuyu-8B (Adept, 2023) eliminate the separate vision encoder. Pixels are turned into discrete tokens (Chameleon via VQ-VAE, Fuyu via a linear patch projector) and **prepended to text tokens** in a single decoder-only transformer.

Pros: no modality gap, generation across modalities, scaling laws are identical to text LLMs.

Cons: vision pre-training of a giant transformer is enormously expensive and discards the inductive biases of CNN/ViT — generally requires 10× more data.

### 1.9 How to compare two VLMs in one breath

A complete comparison must touch:

1. **Vision encoder choice and resolution** (ViT-L/14 @ 224 vs SigLIP-So400m @ 384 vs AnyRes 672).
2. **LLM backbone and parameter count** (Vicuna-7B vs LLaMA-2-13B vs Qwen-2.5-72B vs Phi-3).
3. **Fusion topology** (concatenation / cross-attn / Q-Former / MLP).
4. **Training data scale and composition** (image-text pairs, OCR, charts, video, agentic web traces).
5. **Pre-training objectives** (ITC, ITM, LM, NTP, masked-anywhere).
6. **Token budget per image** (1, 32, 64, 144, 256, 729, 2880…).
7. **Resolution strategy** (fixed / dynamic tiling / native any-resolution / native-pack).
8. **Benchmark numbers** (MMMU, MathVista, DocVQA, ChartQA, MMBench, VQAv2, AI2D, BLINK).

Memorise this list; it is the rubric for any architectural-comparison question.

## 2. Learning Representations

### 2.1 Why "representation learning" deserves its own section

A representation is a vector that **carries information about the input in a form that is useful to downstream tasks** while being invariant to nuisance variation. Bengio's 2013 *"Representation Learning: A Review and New Perspectives"* defines the three desiderata: **smoothness**, **multiple explanatory factors**, and **hierarchy**. Lilian Weng's *"Self-Supervised Learning"* post (Nov 2019, updated 2021) is the canonical free tutorial.

There are four ways a VLM learns representations.

### 2.2 Contrastive (InfoNCE) learning

The InfoNCE objective (van den Oord et al., 2018) is the workhorse:

$$
\mathcal{L}_{\text{InfoNCE}} = -\mathbb{E}\!\left[ \log \frac{\exp(s(q, k_+)/\tau)}{\exp(s(q,k_+)/\tau) + \sum_{k_-} \exp(s(q,k_-)/\tau)} \right]
$$

where $q$ is a query embedding, $k_+$ is the positive (paired) key, and the sum runs over $N-1$ negatives. As $N \to \infty$, InfoNCE lower-bounds the mutual information $I(q;k_+)$.

For VLMs the query/positive pair is an (image, text) sample; the negatives are the other items in the batch (CLIP) or items in a queue (MoCo / ALBEF). The key engineering questions are:

- **How many negatives?** More is better up to ~32 k for CLIP; the marginal gain plateaus.
- **Hard-negative mining**: ALBEF samples cross-modal negatives weighted by current similarity scores — this dramatically accelerates convergence.
- **Temperature $\tau$**: smaller $\tau$ → sharper distribution → harder optimisation but better separation. Setting $\tau$ learnable (CLIP) often beats fixed.
- **Momentum encoder**: a slow EMA copy of the encoder produces consistent keys (He et al., MoCo). ALBEF uses a momentum **distillation** loss too.

### 2.3 Image-Text Matching (ITM)

A binary cross-entropy classifier sits on top of the fused [CLS] token and predicts "is this image-text pair a real match?" Half the batch is real pairs, half are hard negatives sampled with weights proportional to current similarity. ITM is **cheap and effective** but only useful when the model is encoder-style (it needs cross-attention).

### 2.4 Image-grounded language modelling (LM)

This is plain next-token prediction conditioned on the image:

$$
\mathcal{L}_{\text{LM}} = -\sum_{t=1}^T \log p_\theta(w_t \mid w_{<t}, \mathbf{V}).
$$

It produces a *generative* model from the very start. Required for any model that must caption or chat. LLaVA and BLIP-2 use a masked variant where the loss is computed only on assistant tokens, not user-instruction tokens — this is the **"selective masking"** detail you should always mention.

### 2.5 Masked modelling (BEiT, FLAVA, MAE-style)

Mask 50–75 % of image patches and 15 % of text tokens; predict the missing ones from the remaining context. The image targets can be:

- Raw pixels (MAE).
- Discrete visual tokens from a pre-trained dVAE (BEiT, BEiT-3).
- Features from a separate teacher (data2vec, EVA — predicts CLIP features).

Masked modelling rewards **local** representations, while contrastive learning rewards **global** invariances — combining them (FLAVA, BEiT-3) gives both.

### 2.6 Generative captioning and OCR self-supervision

Florence-2 (Microsoft, 2024) demonstrates that prompting a single sequence-to-sequence model with task-specific prompts (`<OD>` for detection, `<CAPTION>` for captioning, `<OCR>` for text reading) is sufficient — provided the pre-training data covers all tasks. The representation learned is the **shared encoder hidden state**.

### 2.7 The alignment / uniformity decomposition

Wang & Isola (2020) show that contrastive losses can be decomposed into

$$
\mathcal{L}_{\text{align}} = \mathbb{E}\big[\|f(x) - f(x^+)\|^2\big], \qquad
\mathcal{L}_{\text{uniform}} = \log \mathbb{E}\big[\exp(-2 \|f(x) - f(y)\|^2)\big].
$$

A good encoder minimises both: positives collapse together (alignment), random samples spread on the unit sphere (uniformity). This decomposition is a great mental model for *why* contrastive learning works.

### 2.8 Probing learned representations

Three protocols to evaluate a frozen representation:

1. **Linear probe**: freeze the encoder, train a linear classifier on a labelled dataset.
2. **k-NN probe**: classify by the dominant label among the $k$ nearest training embeddings.
3. **Zero-shot**: in CLIP-style models, build a classifier by encoding text prompts and computing dot products.

A model that wins on zero-shot but loses on linear probe is producing a *task-specific* representation rather than a *general* one. Watch for this in papers.

### 2.9 Putting it all together — what to use when

- **Retrieval / zero-shot classification** → InfoNCE only (CLIP, SigLIP).
- **VQA / classification head** → InfoNCE + ITM + masked LM (ALBEF, BLIP).
- **Open-ended dialog / OCR / detection** → generative LM on top of a frozen contrastive backbone (BLIP-2, LLaVA, Florence-2).
- **Any-to-any / image generation included** → unified tokenisation + LM (Chameleon, Unified-IO 2).

## 3. Customizing Visual Encoders

### 3.1 The three encoder families

| Family | Examples | Inductive bias | Token count for 224² | When to use |
|---|---|---|---|---|
| **CNN** | ResNet-50, ConvNeXt-XL, RegNetY | Locality, translation equivariance | 49 (after 7×7 pool) | Edge devices, dense prediction |
| **Vision Transformer (ViT)** | ViT-L/14, DeiT, EVA-02 | Almost none; global attention | 256 (16² patches) or 196 (14²) | Anything that scales |
| **Hybrid / Hierarchical** | Swin, ConvNeXt-V2, BEiT-3 | Hierarchical pooling + windowed attention | varies | Object detection, segmentation |

A VLM almost always uses a ViT today because (a) it produces a clean sequence of tokens that drops straight into a transformer-based fusion module, and (b) it scales with data better than CNNs.

### 3.2 The ViT in one page

Given an image $x \in \mathbb{R}^{H \times W \times 3}$ and patch size $P$:

1. **Patchify**: reshape into $N = HW/P^2$ patches of size $P \times P \times 3 = 3P^2$ floats, then linearly project to dimension $d$. This is a convolution with kernel $P$ and stride $P$.
2. **Prepend [CLS] token**: a learnable vector $\mathbf{z}_{\text{cls}} \in \mathbb{R}^d$.
3. **Add positional embeddings** $\mathbf{p} \in \mathbb{R}^{(N+1)\times d}$ — learned (ViT) or sinusoidal (CLIP-RN) or RoPE-2D (Qwen2-VL).
4. **L transformer blocks**: each is

   $$
   \mathbf{z}^\prime = \mathbf{z} + \text{MHSA}(\text{LN}(\mathbf{z})), \quad \mathbf{z} = \mathbf{z}^\prime + \text{MLP}(\text{LN}(\mathbf{z}^\prime)).
   $$

5. **Final LN, output**: either the [CLS] embedding or all $N$ patch embeddings.

Variant cheat-sheet:

- **ViT-B/16**: 12 layers, 12 heads, $d=768$ — 86 M params.
- **ViT-L/14**: 24 layers, 16 heads, $d=1024$ — 304 M params.
- **ViT-H/14**: 32 layers, 16 heads, $d=1280$ — 632 M params.
- **ViT-g/14**: 40 layers, 16 heads, $d=1408$ — 1.0 B params.
- **SigLIP-So400m/14**: 27 layers, 16 heads, $d=1152$ — 400 M params, optimal "shape" by SoViT (Alabdulmohsin et al., 2023).
- **EVA-02-Large**: ViT-L with RoPE-2D + xattn + sub-LN + masked image modelling pre-training; the strongest ~300 M parameter encoder publicly.

### 3.3 Loading a pre-trained ViT in PyTorch

```python
import torch, torch.nn as nn
from transformers import CLIPVisionModel, SiglipVisionModel

# 1) CLIP-ViT-L/14 from OpenAI weights
clip = CLIPVisionModel.from_pretrained("openai/clip-vit-large-patch14-336")
# 2) SigLIP-So400m from Google (used by Gemini-Nano, PaliGemma, Idefics-3)
sig  = SiglipVisionModel.from_pretrained("google/siglip-so400m-patch14-384")

# Both return BaseModelOutputWithPooling: .last_hidden_state [B, 1+N, d], .pooler_output [B, d]
img = torch.randn(2, 3, 336, 336)
out = clip(pixel_values=img)
print(out.last_hidden_state.shape, out.pooler_output.shape)
```

### 3.4 Customising the ViT — six surgical operations

You will be asked to *modify* an encoder. Six standard surgeries:

#### (a) Patch-size surgery

To support higher resolution without re-training, **interpolate positional embeddings**:

```python
def interpolate_pos_embed(pos_embed, new_grid):
    # pos_embed: [1, 1+N_old, d]; new_grid: int H_new == W_new in patches
    cls, grid = pos_embed[:, :1], pos_embed[:, 1:]
    N_old = grid.shape[1]; gs_old = int(N_old**0.5)
    grid = grid.reshape(1, gs_old, gs_old, -1).permute(0, 3, 1, 2)
    grid = torch.nn.functional.interpolate(grid, size=(new_grid, new_grid),
                                           mode="bicubic", align_corners=False)
    grid = grid.permute(0, 2, 3, 1).reshape(1, new_grid * new_grid, -1)
    return torch.cat([cls, grid], dim=1)
```

CLIP-ViT-L/14 trained at 224 (16² patches) generalises to 336 (24²) with this trick alone.

#### (b) AnyRes / dynamic tiling

LLaVA-NeXT and Qwen2-VL split a high-res image into multiple ViT inputs.

```python
def anyres_tiles(image, base=336, candidate_grids=((1,1),(1,2),(2,1),(2,2),(1,3),(3,1))):
    H, W = image.shape[-2:]
    best = max(candidate_grids,
               key=lambda g: min(W / (g[1]*base), H / (g[0]*base)))
    gh, gw = best
    image = torch.nn.functional.interpolate(
        image[None], size=(gh*base, gw*base), mode="bicubic", align_corners=False)[0]
    tiles = image.unfold(1, base, base).unfold(2, base, base)         # [3, gh, gw, base, base]
    tiles = tiles.permute(1, 2, 0, 3, 4).reshape(gh*gw, 3, base, base)
    thumb = torch.nn.functional.interpolate(
        image[None], size=(base, base), mode="bicubic", align_corners=False)[0]
    return torch.cat([thumb[None], tiles], dim=0)                     # [1+gh*gw, 3, b, b]
```

#### (c) RoPE-2D for any-resolution

Replace the 1-D positional embedding with rotary embeddings indexed by $(y, x)$:

$$
\text{RoPE}_{2D}(\mathbf{q}_{y,x}) = \begin{pmatrix} R_y & 0 \\ 0 & R_x \end{pmatrix} \mathbf{q}_{y,x}
$$

where $R_y$ and $R_x$ are block-diagonal rotation matrices on half of the channels each. This is how Qwen2-VL and InternVL-2 support truly arbitrary resolutions.

#### (d) Register tokens (Darcet et al., 2024)

Add 4–8 extra learnable tokens to the input that are **discarded at output**. Empirically removes the "artifacts" in attention maps reported by Caron et al. and improves dense prediction.

#### (e) Token dropping / pruning

DynamicViT, ATS, ToMe — remove or merge tokens whose attention score is below a threshold to cut FLOPs by 30–50 %. ToMe (token-merging) is the most popular: at each layer, find the $r$ most similar token pairs by cosine and average them.

#### (f) Multi-scale feature extraction

Concatenate features from layers $\{l_1, l_2, l_3\}$ — e.g. layer 8 (textures), 16 (parts), 24 (objects). VAR, VeCLIP and LLaVA-1.5 with high-res use this trick.

### 3.5 Convolutional and hybrid encoders

When you need a **low-token-count** encoder for mobile or for video, ConvNeXt-V2 and EfficientFormer-V2 produce 49 or 196 tokens at much lower FLOPs. For dense prediction (detection, segmentation) the hierarchical encoders matter — **Swin-Transformer** uses shifted windows, **InternImage** uses deformable convolutions; the InternVL series uses these as the vision tower.

### 3.6 Practical PyTorch tips

- `torch.compile(encoder, mode="reduce-overhead")` removes 20–30 % wall-clock on H100 for ViT-L.
- Use `torch.nn.functional.scaled_dot_product_attention` (PyTorch ≥ 2.0) — it dispatches to Flash-Attention 2 automatically.
- Freeze the encoder with `for p in encoder.parameters(): p.requires_grad = False; encoder.eval()` — the `.eval()` matters because of BatchNorm/Dropout (`Dropout` is no-op but `LayerNorm` running stats are not stored in CLIP-style ViTs, so `.eval()` is safe).
- Cast pre-trained weights to bf16 with `model.to(torch.bfloat16)` when training fusion — the tiny accuracy loss is invisible and memory drops 50 %.

## 4. Extracting Visual Embeddings

### 4.1 Where to pull features from

A ViT block-by-block output has shape `[B, 1+N, d]`. Five canonical extraction strategies:

| Strategy | Tensor used | Pros | Cons |
|---|---|---|---|
| **CLS token** | `hidden_states[-1][:, 0]` | One vector per image, fast | Loses spatial info |
| **Mean pool of patches** | `hidden_states[-1][:, 1:].mean(1)` | Robust, no CLS-collapse | Loses CLS bias |
| **All patch tokens** | `hidden_states[-1][:, 1:]` | Spatial reasoning possible | $N$ tokens to fuse |
| **Penultimate layer patches** | `hidden_states[-2][:, 1:]` | Less "task-specialised" than the final | One layer to specify |
| **Multi-scale concat** | `[h[l1], h[l2], h[l3]]` | Captures textures + objects | 2-3× tokens |

CLIP's *image_features* uses CLS + a final projection layer. LLaVA-1.5 uses **penultimate-layer patch tokens** (24×24 = 576 tokens), justified empirically: the last layer is too task-specialised to the contrastive objective.

### 4.2 Standard PyTorch extraction

```python
from transformers import CLIPVisionModel
import torch.nn as nn

class VisualFeatureExtractor(nn.Module):
    def __init__(self, model_name="openai/clip-vit-large-patch14-336", select_layer=-2):
        super().__init__()
        self.vit = CLIPVisionModel.from_pretrained(model_name, output_hidden_states=True)
        self.select_layer = select_layer  # -2 == penultimate

    @torch.no_grad()
    def forward(self, pixel_values):
        out = self.vit(pixel_values=pixel_values, output_hidden_states=True)
        feats = out.hidden_states[self.select_layer][:, 1:]   # drop CLS, [B, N, d]
        return feats
```

### 4.3 Projector designs — the bridge to language

The projector $P : \mathbb{R}^{d_v} \to \mathbb{R}^{d_t}$ converts visual feature dimension to LLM embedding dimension.

**1. Linear (LLaVA-1)** — `nn.Linear(1024, 4096)`. Two-step training: warm-up only this layer, then unfreeze everything.

**2. MLP-2 (LLaVA-1.5)** — `Linear → GELU → Linear`. +1.0 MMMU vs linear.

**3. Q-Former (BLIP-2)** — a 12-layer BERT initialised from BERT-base, with 32 learnable query tokens that cross-attend to image features and self-attend to each other. Output: 32 vectors regardless of input resolution.

```python
class QFormer(nn.Module):
    def __init__(self, n_queries=32, d=768, n_layers=12, n_heads=12):
        super().__init__()
        self.queries = nn.Parameter(torch.randn(1, n_queries, d) * 0.02)
        self.layers = nn.ModuleList([QFormerLayer(d, n_heads) for _ in range(n_layers)])

    def forward(self, vision_tokens):
        q = self.queries.expand(vision_tokens.size(0), -1, -1)
        for layer in self.layers:
            q = layer(q, vision_tokens)
        return q                                      # [B, 32, d]
```

**4. Perceiver Resampler (Flamingo)** — like Q-Former but the queries also see learned positional embeddings indexed by time-of-frame (videos!), and the cross-attention is gated.

**5. Pixel-Shuffle (InternVL-2)** — concatenate $2 \times 2$ neighbouring patch tokens then linearly project. Cuts token count 4× while keeping local information. Used in InternVL-2 to bring 24×24 down to 12×12 = 144 tokens.

**6. C-Abstractor (Honeybee)** — convolutions on the 2-D patch grid then flatten. Preserves locality, modest parameter count.

### 4.4 What about positional encoding for visual tokens *inside the LLM*?

There are three schools:

- **Use the ViT's own pos-embed and add nothing**: the LLM's RoPE handles tokens by their *position in the sequence* but not their spatial position. This is what LLaVA does.
- **Add 2-D positional information via the projector** (interleave spatial coordinates).
- **Use M-RoPE / RoPE-2D in the LLM itself** (Qwen2-VL): the LLM is re-trained to interpret rotary frequencies along temporal, height and width axes.

### 4.5 Quick sanity check — do my visual embeddings preserve information?

After training, run a **linear probe**: freeze the encoder + projector and train a single linear classifier on ImageNet-1k. If accuracy drops more than 5 % vs the original ViT, your projector is destroying information.

## 5. Customizing Text Encoders

### 5.1 The decision tree

The first question is *what role* does text play in your VLM?

- **Symmetric retrieval (CLIP-style)** → bidirectional encoder (BERT/RoBERTa) or a causal Transformer with attention-mask = bidirectional only for the [EOS] vector. CLIP uses a causal Transformer with EOS pooling.
- **Encoder-decoder VQA (BLIP-style)** → BERT-base initialised, repurposed via prompt tokens.
- **Generative dialog (LLaVA-style)** → a full LLM (Vicuna, LLaMA, Mistral, Phi, Qwen, Gemma).

### 5.2 Tokenisation primer

Three production algorithms:

| Algorithm | Used by | Vocab | Notes |
|---|---|---|---|
| **BPE** | GPT-2, CLIP, LLaMA-1 | 32 k–50 k | Greedy merging of frequent pairs |
| **WordPiece** | BERT | 30 k | Likelihood-based merging |
| **SentencePiece / Unigram** | T5, LLaMA-2, Mistral, Gemma | 32 k–128 k | Probabilistic, language-agnostic |

For VLMs, the tokenizer **must include special tokens** for image placeholders, e.g. `<image>`, `<video>`, `<roi>`, `<grounding>`. LLaVA reserves the `<image>` token; Kosmos-2 reserves `<box>` and `<phrase>`; Florence-2 reserves `<OD>`, `<OCR>`, `<CAPTION>`.

### 5.3 BERT-family text encoder

A bidirectional encoder:

```python
from transformers import AutoTokenizer, BertModel
tok = AutoTokenizer.from_pretrained("bert-base-uncased")
bert = BertModel.from_pretrained("bert-base-uncased")
out = bert(**tok(["a dog on grass"], return_tensors="pt", padding=True))
print(out.last_hidden_state.shape, out.pooler_output.shape)
```

The `pooler_output` is a `Linear(d, d) → tanh` applied to the [CLS] vector — only meaningful when fine-tuned with a classification head. For retrieval you usually use **mean pooling of all token embeddings with attention mask**.

### 5.4 CLIP-style causal text encoder

```python
from transformers import CLIPTextModel, CLIPTokenizer
tok = CLIPTokenizer.from_pretrained("openai/clip-vit-large-patch14")
text = CLIPTextModel.from_pretrained("openai/clip-vit-large-patch14")
inputs = tok(["a photo of a cat"], padding=True, return_tensors="pt")
out = text(**inputs)
# CLIP pools at the EOS position
eos_idx = inputs.input_ids.argmax(dim=-1)
pooled = out.last_hidden_state[torch.arange(eos_idx.size(0)), eos_idx]
```

This pooling-at-EOS detail is famously subtle: because of greedy BPE, the **largest token id** in the sequence is the EOS token (49407 in CLIP's tokenizer), so `argmax` selects it. If you fine-tune CLIP with a tokenizer that has a *different* id for EOS, you must change this.

### 5.5 LLM backbones — the modern default

Decoder-only causal LMs dominate. You should know the parameter count, context length, tokenizer, and the position-embedding scheme for each:

| Model | Params | Context | Tokenizer | Pos-Embed |
|---|---|---|---|---|
| LLaMA-2 | 7 / 13 / 70 B | 4 k | SentencePiece-BPE, 32 k | RoPE θ=10000 |
| LLaMA-3 / 3.1 / 3.2 | 8 / 70 / 405 B | 8 k / 128 k | tiktoken, 128 k | RoPE θ=500000 |
| Mistral-7B / Mixtral-8x7B | 7 / 56 B | 32 k | BPE, 32 k | RoPE + sliding window |
| Qwen2 / Qwen2.5 | 0.5 / 1.5 / 7 / 14 / 32 / 72 B | 128 k | BPE, 152 k | RoPE w/ YARN |
| Gemma / Gemma-2 | 2 / 9 / 27 B | 8 k | SentencePiece, 256 k | RoPE |
| Phi-3 | 3.8 / 7 / 14 B | 4 k–128 k | tiktoken, 32 k | RoPE w/ LongRoPE |

### 5.6 Surgical modifications

Just like the visual side:

- **Add special tokens**: `tokenizer.add_special_tokens({"additional_special_tokens": ["<image>", "<im_patch>"]})` then `model.resize_token_embeddings(len(tokenizer))`. **Critical**: new embeddings are initialised randomly — initialise them to the *mean of all existing token embeddings* to stabilise training.
- **Extend context with RoPE re-scaling**: replace $\theta = 10000$ with $\theta = 500000$ (NTK-aware scaling) or use **YaRN** (Peng et al., 2023). Done by Qwen2-VL to support 32 k visual+text tokens.
- **Replace the LLM with a Mixture-of-Experts** version: same fusion module, swap the dense FFN for a routed FFN. Used by MoE-LLaVA, DeepSeek-VL2.
- **Add LoRA adapters** to specific layers instead of full fine-tuning (see Optimization doc).
- **Freeze the LM entirely** and rely on the projector to "speak" the LM's language — the BLIP-2 / LLaVA-1 approach.

### 5.7 Causal masking with visual tokens

Inside the LLM, visual tokens occupy specific positions in the sequence. There are three masking conventions:

- **Pure causal**: visual tokens are causally masked just like text. Easiest, default.
- **Visual-prefix bidirectional**: visual tokens can attend to all other visual tokens but not text after them; text is causal. Used by some early models.
- **Bidirectional vision, causal text** (Idefics-3): better preserves the ViT's information.

The implementation detail is the **attention mask**, not a different module.

## 6. Extracting Textual Embeddings

### 6.1 Pooling choices

| Pooling | Formula | Use case |
|---|---|---|
| **CLS / [SOS]** | `h[:, 0]` | BERT classifiers |
| **EOS** | `h[arange(B), eos_idx]` | CLIP retrieval |
| **Mean (mask-aware)** | $\frac{\sum_t m_t h_t}{\sum_t m_t}$ | Sentence-BERT, MPNet |
| **Max** | $\max_t h_t$ | rarely used |
| **Last-token** | `h[:, -1]` | LLM-based retrievers (E5-Mistral, GTE-Qwen2) |
| **Weighted (attention)** | $\sum_t \alpha_t h_t$ | learned, slightly better |

PyTorch mean-pool with mask:

```python
def mean_pool(hidden, attention_mask):
    mask = attention_mask.unsqueeze(-1).float()
    return (hidden * mask).sum(1) / mask.sum(1).clamp(min=1e-6)
```

### 6.2 Sentence-Transformers and Matryoshka Representation Learning

When you want a **single** vector per sentence, the Sentence-BERT family (Reimers & Gurevych, 2019) fine-tunes BERT with a siamese contrastive loss. **MRL** (Kusupati et al., 2022) trains the same embedding to be useful when truncated to multiple dimensions $\{64, 128, 256, 512, 768\}$ — relevant for VLMs because shorter embeddings mean cheaper retrieval at deployment time.

### 6.3 Text embeddings inside a generative VLM

When the LLM is *the* fusion module (LLaVA, Qwen-VL), there is no separate text embedding — the LLM's own token embeddings are the textual representations. Your fusion module simply produces visual tokens with the same dimensionality and the LLM concatenates them. The "embedding extraction" question becomes: *do I want the embedding before any attention, after the first layer, or at the final hidden state?*

For downstream tasks where you need a fixed-size text vector from a decoder-only LM, use the **last token's hidden state** (E5-Mistral approach), or apply an extra `<|END|>` token and pool there.

### 6.4 Practical PyTorch — multi-modal retrieval

```python
import torch.nn.functional as F

def encode_text(text_model, tokenizer, sentences, device="cuda"):
    tok = tokenizer(sentences, padding=True, truncation=True,
                    max_length=77, return_tensors="pt").to(device)
    with torch.no_grad():
        out = text_model(**tok)
    eos_idx = tok.input_ids.argmax(dim=-1)
    h = out.last_hidden_state
    pooled = h[torch.arange(h.size(0), device=device), eos_idx]
    return F.normalize(pooled, dim=-1)
```

## 7. Implementing Custom Fusion Modules

### 7.1 The fusion taxonomy

| Type | Where fusion happens | Example |
|---|---|---|
| **No fusion (late)** | Dot-product after both encoders | CLIP, SigLIP |
| **Cross-attention fusion** | Inside one transformer | Flamingo, ALBEF, BLIP |
| **Concatenative fusion** | Tokens prepended to LLM | LLaVA, MiniGPT-4, Idefics |
| **Resampler fusion** | Few learned queries summarise vision then concat | BLIP-2, Flamingo |
| **Adapter fusion** | Small modules inserted into LLM | LLaMA-Adapter, MMICL |
| **Unified fusion** | Single transformer over (vision_tokens, text_tokens) | Fuyu, Chameleon |

### 7.2 Cross-attention — first principles

Multi-head attention with separate sources for Q and K/V:

$$
\text{CrossAttn}(\mathbf{Q}, \mathbf{K}, \mathbf{V}) = \text{softmax}\!\left(\frac{\mathbf{Q}\mathbf{K}^\top}{\sqrt{d_k}}\right)\mathbf{V}, \qquad \mathbf{Q}=\mathbf{X}_t W_Q,\; \mathbf{K}=\mathbf{X}_v W_K,\; \mathbf{V}=\mathbf{X}_v W_V.
$$

PyTorch implementation:

```python
class CrossAttention(nn.Module):
    def __init__(self, d_text, d_vis, n_heads=8, d_head=64):
        super().__init__()
        d = n_heads * d_head
        self.n_heads = n_heads; self.d_head = d_head
        self.to_q = nn.Linear(d_text, d, bias=False)
        self.to_k = nn.Linear(d_vis,  d, bias=False)
        self.to_v = nn.Linear(d_vis,  d, bias=False)
        self.to_out = nn.Linear(d, d_text)

    def forward(self, x_text, x_vis, mask=None):
        B, Nt, _ = x_text.shape
        Nv = x_vis.shape[1]
        q = self.to_q(x_text).view(B, Nt, self.n_heads, self.d_head).transpose(1,2)
        k = self.to_k(x_vis ).view(B, Nv, self.n_heads, self.d_head).transpose(1,2)
        v = self.to_v(x_vis ).view(B, Nv, self.n_heads, self.d_head).transpose(1,2)
        attn = torch.nn.functional.scaled_dot_product_attention(q, k, v, attn_mask=mask)
        attn = attn.transpose(1,2).reshape(B, Nt, -1)
        return self.to_out(attn)
```

Note the use of `scaled_dot_product_attention` — PyTorch 2 dispatches to Flash-Attention 2 when shapes and dtypes align.

### 7.3 Gated cross-attention (Flamingo)

Add a learnable gate so the LLM starts as its uni-modal self:

```python
class GatedXAttn(nn.Module):
    def __init__(self, d_text, d_vis, n_heads):
        super().__init__()
        self.xattn = CrossAttention(d_text, d_vis, n_heads)
        self.alpha_xattn = nn.Parameter(torch.zeros(1))
        self.ff   = nn.Sequential(nn.Linear(d_text, 4*d_text), nn.GELU(), nn.Linear(4*d_text, d_text))
        self.alpha_ff    = nn.Parameter(torch.zeros(1))

    def forward(self, x_text, x_vis):
        x_text = x_text + torch.tanh(self.alpha_xattn) * self.xattn(x_text, x_vis)
        x_text = x_text + torch.tanh(self.alpha_ff)    * self.ff(x_text)
        return x_text
```

Insert one such block every $K$ LLM layers (Flamingo: $K=4$ for the smaller variant). The initial zero-gate is **critical**: it makes the pre-trained LLM's behaviour an exact starting point.

### 7.4 The Q-Former in detail

Q-Former has **two pathways** because it is trained with three losses (ITC, ITM, ITG) that need different attention masks. A single forward pass routes the queries through different self-attention masks depending on the task token:

- For ITC: queries can only see each other; text can only see text. The two CLS representations contrast.
- For ITM: queries see image features and full bidirectional text — used to compute a binary score.
- For ITG: queries see image features; text is causally masked (so the model can be used as a captioner).

```python
class QFormerLayer(nn.Module):
    def __init__(self, d=768, n_heads=12):
        super().__init__()
        self.self_attn  = nn.MultiheadAttention(d, n_heads, batch_first=True)
        self.cross_attn = nn.MultiheadAttention(d, n_heads, batch_first=True)
        self.ff = nn.Sequential(nn.Linear(d, 4*d), nn.GELU(), nn.Linear(4*d, d))
        self.ln1, self.ln2, self.ln3 = nn.LayerNorm(d), nn.LayerNorm(d), nn.LayerNorm(d)

    def forward(self, q, v):
        h = self.ln1(q)
        q = q + self.self_attn(h, h, h, need_weights=False)[0]
        h = self.ln2(q)
        q = q + self.cross_attn(h, v, v, need_weights=False)[0]
        q = q + self.ff(self.ln3(q))
        return q
```

### 7.5 MLP projector (LLaVA-1.5)

```python
class MLPProjector(nn.Module):
    def __init__(self, d_vis, d_lm):
        super().__init__()
        self.proj = nn.Sequential(
            nn.Linear(d_vis, d_lm),
            nn.GELU(),
            nn.Linear(d_lm, d_lm),
        )
    def forward(self, x):                              # x: [B, N, d_vis]
        return self.proj(x)                            # -> [B, N, d_lm]
```

Used by 90 % of recent open-source VLMs. **Initialization tip**: zero-init the second linear's bias and use Kaiming for weights to keep activations bounded.

### 7.6 Pixel-Shuffle / Patch-merging projector

InternVL-2's trick to cut visual tokens 4×:

```python
def pixel_shuffle_2d(x, scale=2):
    # x: [B, N, d] with N = h*w; halve spatial, multiply channels by scale**2
    B, N, d = x.shape
    h = w = int(N**0.5)
    x = x.view(B, h, w, d)
    x = x.view(B, h//scale, scale, w//scale, scale, d).permute(0, 1, 3, 2, 4, 5)
    x = x.reshape(B, (h//scale) * (w//scale), d * scale * scale)
    return x

class PixelShuffleProjector(nn.Module):
    def __init__(self, d_vis, d_lm, scale=2):
        super().__init__()
        self.scale = scale
        self.proj = nn.Sequential(
            nn.LayerNorm(d_vis * scale * scale),
            nn.Linear(d_vis * scale * scale, d_lm),
            nn.GELU(),
            nn.Linear(d_lm, d_lm),
        )
    def forward(self, x):
        return self.proj(pixel_shuffle_2d(x, self.scale))
```

### 7.7 Adapter / LoRA fusion

Insert a low-rank adapter into the LLM's FFN or attention layers, with conditioning from visual features:

$$
\Delta \mathbf{h} = \alpha \cdot \text{cross-attn}(\mathbf{h}, \mathbf{V}) \cdot B^\top A^\top
$$

where $A \in \mathbb{R}^{r \times d}$, $B \in \mathbb{R}^{d \times r}$, $r \ll d$. LLaMA-Adapter-V2 does exactly this.

### 7.8 Choosing a fusion design — a decision table

| Situation | Recommended fusion |
|---|---|
| Train a CLIP-replacement on 1 B pairs | None — dual encoder + InfoNCE/SigLIP |
| Train a chatbot on a frozen LLM with limited compute | MLP projector + LoRA |
| Train a model that must answer about 100s of frames | Q-Former + temporal pos-embed |
| Need few-shot in-context learning across images | Gated cross-attention (Flamingo) |
| Want highest token efficiency (long doc + image) | Pixel-shuffle + AnyRes |
| Need image generation too | Unified VQ-VAE tokens + LM (Chameleon) |

### 7.9 Common pitfalls

- **Forgetting LayerNorm** on the projector input — visual features and LM embeddings live on different scales; un-normalised, the LM ignores them.
- **Wrong sequence length** — never forget to pad and use `attention_mask`.
- **Position confusion**: when you prepend visual tokens, the LLM's RoPE indexes them as if they were text. If your LLM has RoPE θ=10000 trained on 4 k context, suddenly seeing 576 extra tokens can blow context length.
- **Mixed dtypes**: keep projector in `bfloat16` matching the LLM. A `float32` projector silently up-casts the entire compute graph.

## 8. Assembling Multimodal Architecture

### 8.1 The seven-step recipe

Every modern projection-style VLM is assembled the same way. Memorise these seven steps.

1. **Pick the vision encoder.** Default: SigLIP-So400m/14 @ 384. Faster + smaller: CLIP-ViT-L/14 @ 336. Best dense: DINOv2-Giant.
2. **Pick the LLM backbone.** Default: Qwen2.5-7B-Instruct. Tiny: Phi-3-mini. Huge: LLaMA-3-70B.
3. **Pick the resolution strategy.** Single-tile 384², or AnyRes 4-tile, or native any-resolution with M-RoPE.
4. **Pick the projector.** Default: 2-layer MLP. Token-efficient: Pixel-shuffle ×2 + MLP. Few queries: Q-Former-32.
5. **Decide what is frozen.** Stage 1 (alignment): freeze ViT and LLM, train projector. Stage 2 (instruction): unfreeze LLM (or LoRA), keep ViT frozen or unfreeze the last 4 layers.
6. **Choose the training data.** Pre-train on captions (LAION-2B, COYO, DataComp). Instruction-tune on LLaVA-Mix-665k, ShareGPT-4V, LLaVA-OneVision-mid.
7. **Choose evaluation suite.** MMMU, MathVista, DocVQA, ChartQA, AI2D, MMBench, BLINK, VQAv2, OK-VQA, TextVQA.

### 8.2 A complete reference implementation

```python
import torch
import torch.nn as nn
from transformers import (
    AutoTokenizer, AutoModelForCausalLM,
    SiglipVisionModel, SiglipImageProcessor,
)


class VLM(nn.Module):
    """A LLaVA-style VLM: SigLIP vision tower + MLP projector + frozen-or-LoRA LLM."""

    def __init__(
        self,
        vision_name="google/siglip-so400m-patch14-384",
        llm_name="Qwen/Qwen2.5-7B-Instruct",
        image_token="<image>",
        select_layer=-2,
    ):
        super().__init__()
        self.tokenizer = AutoTokenizer.from_pretrained(llm_name)
        if image_token not in self.tokenizer.get_vocab():
            self.tokenizer.add_special_tokens({"additional_special_tokens": [image_token]})
        self.image_token_id = self.tokenizer.convert_tokens_to_ids(image_token)

        self.vision = SiglipVisionModel.from_pretrained(vision_name)
        self.processor = SiglipImageProcessor.from_pretrained(vision_name)
        self.llm = AutoModelForCausalLM.from_pretrained(
            llm_name, torch_dtype=torch.bfloat16, attn_implementation="flash_attention_2"
        )
        self.llm.resize_token_embeddings(len(self.tokenizer))

        d_vis = self.vision.config.hidden_size
        d_lm = self.llm.config.hidden_size
        self.projector = nn.Sequential(
            nn.LayerNorm(d_vis),
            nn.Linear(d_vis, d_lm),
            nn.GELU(),
            nn.Linear(d_lm, d_lm),
        ).to(torch.bfloat16)

        self.select_layer = select_layer

    def freeze_for_alignment(self):
        for p in self.vision.parameters():
            p.requires_grad = False
        for p in self.llm.parameters():
            p.requires_grad = False

    def encode_vision(self, pixel_values):
        out = self.vision(pixel_values=pixel_values, output_hidden_states=True)
        feats = out.hidden_states[self.select_layer]                    # [B, N, d_vis]
        return self.projector(feats)                                     # [B, N, d_lm]

    def _embed_inputs(self, input_ids, pixel_values):
        """Replace every <image> token's embedding with vision tokens."""
        inputs_embeds = self.llm.get_input_embeddings()(input_ids)       # [B, T, d_lm]
        vision_embeds = self.encode_vision(pixel_values)                 # [B, N, d_lm]
        merged, new_attn_mask = [], []
        for b in range(input_ids.size(0)):
            img_pos = (input_ids[b] == self.image_token_id).nonzero(as_tuple=True)[0]
            if len(img_pos) == 0:
                merged.append(inputs_embeds[b]); continue
            assert len(img_pos) == 1, "Use one <image> per sample in this minimal demo"
            p = img_pos.item()
            merged.append(torch.cat([inputs_embeds[b, :p],
                                     vision_embeds[b],
                                     inputs_embeds[b, p+1:]], dim=0))
        merged = nn.utils.rnn.pad_sequence(merged, batch_first=True)
        return merged

    def forward(self, input_ids, pixel_values, labels=None):
        inputs_embeds = self._embed_inputs(input_ids, pixel_values)
        outputs = self.llm(inputs_embeds=inputs_embeds, labels=labels)
        return outputs
```

This is essentially the architecture of LLaVA-1.5, Bunny, Imp, and most open-source 7B-class VLMs published in 2024. Read it line by line until you can rewrite it from memory.

### 8.4 Native-resolution / dynamic-tile assembly (LLaVA-NeXT pattern)

The differences from LLaVA-1.5:

1. `processor` returns multiple tiles plus a thumbnail.
2. `encode_vision` runs the ViT on each tile (with possible gradient checkpointing), then concatenates patch tokens.
3. A **newline token** is inserted between rows of tiles so the LLM can reason about layout.

```python
def encode_anyres(self, tiles):
    # tiles: [B, T, 3, H, W] where T = num_tiles_per_image (variable)
    B, T = tiles.shape[:2]
    tiles_flat = tiles.flatten(0, 1)
    feats = self.encode_vision(tiles_flat)                  # [B*T, N, d_lm]
    feats = feats.view(B, T, -1, feats.size(-1))            # [B, T, N, d_lm]
    return feats.reshape(B, T*feats.size(2), -1)            # concat tiles
```

### 8.5 Beyond LLaVA — Qwen2-VL's "naive dynamic resolution"

Qwen2-VL drops the tiling entirely. The ViT is **retrained** to operate on arbitrary $H \times W$ via M-RoPE (multi-dimensional rotary embeddings). The vision encoder outputs a variable number of patch tokens; each is given a `(t, y, x)` triple consumed by the LLM's M-RoPE. The trick that makes the LM happy: split RoPE channels into thirds — one for temporal, one for height, one for width.

### 8.6 Multi-image and video

For multiple images, each gets its own `<image>` placeholder and its own visual-token block:

```
<image>{img1}<image>{img2} What is different between these two pictures?
```

For video, sample $T$ frames, run each through the ViT, concat with temporal embeddings, optionally compress via Q-Former with shared queries across frames (Video-LLaMA, Video-ChatGPT).

### 8.7 Sanity checks before training

Before launching a multi-day run, verify:

1. **One sample forward** — `loss.backward()` does not throw.
2. **Loss decreases on a tiny set** — overfit 16 samples and watch loss → ~0.
3. **Visual token count matches** — print `inputs_embeds.shape`.
4. **`<image>` token is masked in labels** — the LLM should not be supervised to predict the image token.
5. **No mixed-dtype warnings** — everything bf16 except the vision encoder LayerNorms which can stay fp32.

### 8.8 Open research questions (good interview answers)

- **Token efficiency vs accuracy** — is 144 tokens ever enough for OCR / charts? Most evidence says no; pixel-shuffle helps but DocVQA still wants ≥ 729 tokens.
- **Generative vs discriminative pre-training** — does masked image modelling (BEiT-3, EVA) beat contrastive (SigLIP) when used as a VLM tower? Currently a tie; depends on downstream task.
- **Encoder-free VLMs** — Fuyu / Chameleon throw away the ViT. The 2024 evidence is that they need 5–10× the data; not yet a free lunch.
- **Modality gap** — does it matter for downstream? Liang et al. 2022 say yes, but recent work (Schrodi et al. 2024) argues the gap is partly a measurement artefact.
- **Visual tokenisation for generation** — VQ-VAEs vs continuous tokens vs Magvit-v2 finite-scalar quantisation.
