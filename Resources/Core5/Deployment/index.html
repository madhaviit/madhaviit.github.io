1 Vision-Language Model Deployment with PyTorch — The Complete Expert Guide

Once a VLM is trained, the hard part begins: training it at scale across many GPUs and many nodes, then squeezing it into a phone, a smart-TV, or a kiosk while keeping latency low and accuracy intact. This document covers the full lifecycle: distributed training, communication tuning, edge-export formats, Tizen / mobile deployment, monitoring, and forensic troubleshooting.

1.2 1. Configuring Multi-GPU Training

1.2.1 1.1 The parallelism zoo

Before any code, you must know what you are sharding.

Parallelism What is split across devices? Best for
Data Parallel (DP / DDP) Mini-batch Model fits on one GPU
ZeRO-1 Optimizer states DP + memory savings
ZeRO-2 Optimizer states + grads More memory savings
ZeRO-3 / FSDP Optimizer + grads + params Models too big for one GPU
Tensor Parallel (TP) Single layer’s matrices Layers too big for one GPU
Pipeline Parallel (PP) Whole stages of layers Very deep models
Sequence / Context Parallel (SP/CP) The sequence dimension Very long contexts
Expert Parallel (EP) MoE experts MoE models

Modern VLMs use a combination — typically FSDP + TP, with SP for long-video models.

1.2.2 1.2 PyTorch DistributedDataParallel (DDP) — the baseline

DDP replicates the full model on every GPU and averages gradients via NCCL all-reduce after each backward.

# launch with:  torchrun --nproc_per_node=8 train.py
import os, torch, torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP
from torch.utils.data import DataLoader, DistributedSampler

def setup():
    dist.init_process_group(backend="nccl")
    local_rank = int(os.environ["LOCAL_RANK"])
    torch.cuda.set_device(local_rank)
    return local_rank

local_rank = setup()
model = build_vlm().to(local_rank)
model = DDP(model, device_ids=[local_rank], find_unused_parameters=False,
            gradient_as_bucket_view=True, broadcast_buffers=False)

sampler = DistributedSampler(dataset, shuffle=True, drop_last=True)
loader  = DataLoader(dataset, batch_size=8, sampler=sampler,
                     num_workers=4, pin_memory=True, persistent_workers=True)

opt    = torch.optim.AdamW(model.parameters(), lr=1e-4)
scaler = torch.amp.GradScaler("cuda")

for epoch in range(EPOCHS):
    sampler.set_epoch(epoch)                     # different shuffle every epoch
    for batch in loader:
        with torch.autocast("cuda", dtype=torch.bfloat16):
            loss = model(**batch).loss
        loss.backward()                          # all-reduce happens here
        opt.step(); opt.zero_grad(set_to_none=True)

Five DDP details that always come up:

  1. device_ids must contain a single device — the local rank.
  2. find_unused_parameters=True is required if any parameter is conditionally skipped in forward (e.g. a frozen LoRA branch). It costs CPU time; flip it off whenever possible.
  3. gradient_as_bucket_view=True saves memory by giving you a view into the all-reduce bucket buffer instead of allocating new gradient tensors.
  4. broadcast_buffers=False turns off the all-reduce of BatchNorm running stats — useless for ViTs/LLMs which use LayerNorm only.
  5. set_to_none=True in zero_grad() keeps the gradient pointer null instead of a zero tensor, slightly faster.

1.2.3 1.3 Gradient accumulation done right

To simulate a larger batch when memory is tight:

accum = 4
opt.zero_grad(set_to_none=True)
for step, batch in enumerate(loader):
    is_last = (step + 1) % accum == 0
    ctx = model.no_sync() if not is_last else nullcontext()
    with ctx:
        with torch.autocast("cuda", dtype=torch.bfloat16):
            loss = model(**batch).loss / accum
        loss.backward()
    if is_last:
        opt.step(); opt.zero_grad(set_to_none=True)

model.no_sync() skips the all-reduce during the inner accumulation steps — communication cost goes down by accum. Forget this and you will all-reduce on every micro-batch, paying the full network cost while gaining no math.

1.2.4 1.4 Fully Sharded Data Parallel (FSDP / FSDP2)

When the model does not fit on a single GPU, shard everything: parameters, gradients, optimizer states.

import functools
from torch.distributed.fsdp import (
    FullyShardedDataParallel as FSDP,
    MixedPrecision, ShardingStrategy, BackwardPrefetch,
)
from torch.distributed.fsdp.wrap import transformer_auto_wrap_policy
from transformers.models.qwen2.modeling_qwen2 import Qwen2DecoderLayer

mp = MixedPrecision(
    param_dtype=torch.bfloat16,
    reduce_dtype=torch.bfloat16,
    buffer_dtype=torch.bfloat16,
)

wrap_policy = functools.partial(
    transformer_auto_wrap_policy,
    transformer_layer_cls={Qwen2DecoderLayer},   # one shard per LLM block
)

model = FSDP(
    model,
    sharding_strategy=ShardingStrategy.FULL_SHARD,         # ZeRO-3
    auto_wrap_policy=wrap_policy,
    mixed_precision=mp,
    backward_prefetch=BackwardPrefetch.BACKWARD_PRE,       # overlap comm
    limit_all_gathers=True,
    device_id=torch.cuda.current_device(),
    use_orig_params=True,                                  # required for ParamGroups
)

Sharding strategies:

  • FULL_SHARD — ZeRO-3 (params + grads + opt states sharded).
  • SHARD_GRAD_OP — ZeRO-2 (grads + opt states only).
  • HYBRID_SHARD — full-shard inside a node, replicate across nodes (best for 2-node setups with fat intra-node NVLink and thin inter-node Ethernet).
  • NO_SHARD — equivalent to DDP.

FSDP2 (PyTorch ≥ 2.4) introduces fully_shard() as the preferred API — per-parameter sharding, cleaner ergonomics, full DTensor-native. Same conceptual model, nicer code.

1.2.5 1.5 Mixed precision: fp16 vs bf16 vs fp8

Format Range Precision Where to use
fp16 ±65 504 ~3 decimal Inference, V100
bf16 ±\(3.4\times 10^{38}\) ~2 decimal Default training on A100/H100
fp8 (e4m3 / e5m2) ±448 / ±57 344 ~1 decimal H100 only, large speedup

bf16 is the de facto training dtype because it has the same exponent range as fp32 — no GradScaler required, no overflow surprises. fp16 only stays alive on V100s that have no bf16 hardware. fp8 needs the Transformer Engine library (pip install transformer_engine) and explicit scaling factors.

1.2.6 1.6 Activation checkpointing

Recompute activations during backward instead of storing them — trades FLOPs for memory.

from torch.distributed.algorithms._checkpoint.checkpoint_wrapper import (
    apply_activation_checkpointing,
    CheckpointWrapper,
    CheckpointImpl,
)

def check_fn(submodule):
    return isinstance(submodule, Qwen2DecoderLayer)

apply_activation_checkpointing(
    model,
    checkpoint_wrapper_fn=functools.partial(
        CheckpointWrapper,
        checkpoint_impl=CheckpointImpl.NO_REENTRANT,
    ),
    check_fn=check_fn,
)

NO_REENTRANT is the modern variant and integrates cleanly with FSDP. Saves ~\(O(L)\) memory at the cost of one extra forward.

1.2.7 1.7 Tensor parallel (TP) and sequence parallel (SP)

When a single transformer block does not fit on a GPU (e.g. 70 B+), split the matrices themselves:

  • Column-wise split on W_q, W_k, W_v and W_up, W_gate of MLP.
  • Row-wise split on W_o and W_down.
  • All-gather along the head dimension before the row-parallel layer.

In PyTorch (≥ 2.3):

from torch.distributed.tensor.parallel import (
    parallelize_module, ColwiseParallel, RowwiseParallel, SequenceParallel,
)
from torch.distributed.device_mesh import init_device_mesh

mesh = init_device_mesh("cuda", (DP, TP), mesh_dim_names=("dp","tp"))
tp_mesh = mesh["tp"]

plan = {
    "self_attn.q_proj": ColwiseParallel(),
    "self_attn.k_proj": ColwiseParallel(),
    "self_attn.v_proj": ColwiseParallel(),
    "self_attn.o_proj": RowwiseParallel(),
    "mlp.gate_proj":  ColwiseParallel(),
    "mlp.up_proj":    ColwiseParallel(),
    "mlp.down_proj":  RowwiseParallel(),
}
for blk in model.model.layers:
    parallelize_module(blk, tp_mesh, plan)

Sequence parallel additionally splits LayerNorm activations along the sequence axis between TP collective calls — relevant when sequences are very long (video, long-context VLMs).

1.2.8 1.8 Pipeline parallel — when and how

Pipeline parallel (PP) splits the model into stages and pipelines micro-batches through them. PyTorch’s torch.distributed.pipelining (PiPPy) does this cleanly. PP shines when:

  • Model is deep (≥ 80 layers).
  • Inter-node bandwidth is the bottleneck (PP communicates only activations at stage boundaries, not gradients across all params).

For 7-13 B VLMs, PP is overkill; FSDP + TP is enough.

1.2.9 1.9 Recipes by model size

Size Single node 8×H100 80 GB Multi-node
1–3 B DDP + bf16 seldom needed
7–13 B FSDP FULL_SHARD + activation ckpt HYBRID_SHARD across 2 nodes
30–70 B FSDP + TP=2 + activation ckpt FSDP + TP=4 across N nodes
100 B+ FSDP + TP=8 + PP=2/4 scale TP first, then PP

1.3 2. Managing Multi-Node Training

1.3.1 2.1 What changes when you cross a node boundary

Inside a node, GPUs talk over NVLink/NVSwitch — 600–900 GB/s on A100/H100 nodes.

Across nodes, they go over Ethernet (25–100 Gbit/s) or InfiniBand (200–400 Gbit/s, ~5× faster). The bandwidth difference is 6–30×; the latency difference is even larger. Every architectural decision in multi-node training is a workaround for this fact.

1.3.2 2.2 The launcher: torchrun and srun

torchrun (PyTorch’s own launcher, supersedes torch.distributed.launch):

# Node 0 (master):
torchrun \
  --nnodes=4 --nproc_per_node=8 \
  --node_rank=0 --rdzv_id=vlm_run01 --rdzv_backend=c10d \
  --rdzv_endpoint=node0.cluster:29500 train.py

# Node 1..3: same command with --node_rank=1, 2, 3

torchrun is elastic by default — workers can join and leave thanks to the c10d rendezvous backend.

SLURM (srun) — the dominant academic / national-lab scheduler:

#!/bin/bash
#SBATCH --job-name=vlm
#SBATCH --nodes=4
#SBATCH --ntasks-per-node=8
#SBATCH --gpus-per-task=1
#SBATCH --cpus-per-task=16
#SBATCH --time=24:00:00
#SBATCH --partition=h100

export MASTER_ADDR=$(scontrol show hostnames "$SLURM_JOB_NODELIST" | head -n1)
export MASTER_PORT=29500
export OMP_NUM_THREADS=8
export NCCL_DEBUG=WARN

srun python -u train.py

Inside train.py you read RANK, LOCAL_RANK, WORLD_SIZE from environment variables that SLURM/torchrun set automatically.

1.3.3 2.3 The init_process_group dance

import os, torch, torch.distributed as dist

def setup():
    if "RANK" in os.environ and "WORLD_SIZE" in os.environ:
        dist.init_process_group(backend="nccl")
    else:
        # SLURM path
        rank   = int(os.environ["SLURM_PROCID"])
        world  = int(os.environ["SLURM_NTASKS"])
        local  = int(os.environ["SLURM_LOCALID"])
        dist.init_process_group(backend="nccl", rank=rank, world_size=world,
                                init_method=f"tcp://{os.environ['MASTER_ADDR']}:{os.environ['MASTER_PORT']}")
        os.environ["LOCAL_RANK"] = str(local)
    torch.cuda.set_device(int(os.environ["LOCAL_RANK"]))

1.3.4 2.4 NCCL — the actual collective library

NCCL is NVIDIA’s collective communication library; PyTorch’s nccl backend is a thin wrapper. The collectives you need to know:

  • all_reduce — every rank ends up with the sum (or other op) of every input. Used after loss.backward() to average gradients.
  • all_gather — every rank collects all inputs concatenated. Used by FSDP to materialise full parameters before forward.
  • reduce_scatter — sum, then scatter chunks back. Used by FSDP for gradients.
  • broadcast — one rank’s tensor goes to all others. Used to seed initial weights and to share the schedule.
  • barrier — synchronisation point.

NCCL chooses among several algorithms:

Algorithm Bandwidth-optimal? Latency Notes
Ring Yes for large messages \(O(P)\) steps classic
Tree Better latency \(O(\log P)\) newer default for small msgs
CollNet (SHARP) Hardware-accelerated on Mellanox switches very low InfiniBand only
NVLS (NVLink SHARP) Intra-node H100 NVSwitch very low DGX H100 only

You can profile NCCL with NCCL_DEBUG=INFO; it prints the algorithm chosen for each collective.

1.3.5 2.5 Topology and fabric

A simple mental picture: an 8×H100 DGX node looks like

    GPU0—NVLink—GPU1—NVLink—GPU2—…
       \        /
        NVSwitch (intra-node, 900 GB/s pairwise)
       /        \
   GPU4    …   GPU7
   |                  |
   NIC0  NIC1  NIC2  NIC3  (InfiniBand HCAs, 1:1 with GPUs)
       \    |    |    /
        switch fabric (200–400 Gbit/s per port)

For best inter-node performance you must:

  1. Use GPUDirect RDMA (NICs read GPU memory directly).
  2. Pin each GPU process to the NUMA node owning its HCA via numactl.
  3. Enable NCCL_IB_HCA to use the right HCA.
  4. Set NCCL_IB_GID_INDEX=3 for RoCE v2, NCCL_IB_GID_INDEX=0 for IB.

1.3.6 2.6 The 12 NCCL env vars every deployment engineer knows

export NCCL_DEBUG=WARN                  # INFO for first run, WARN afterwards

1.3.7 2.7 Fault tolerance

A 64-GPU job runs for a week and hits a memory error. What do you do?

  1. Checkpoint frequently (every 250–1000 steps) but asynchronously (next chapter).
  2. torchrun elastic restarts crashed workers automatically.
  3. torch.distributed.elastic monitors and signals workers.
  4. TorchSnapshot / Distributed Checkpoint (DCP) writes sharded checkpoints in parallel — orders of magnitude faster than gathering everything to rank 0.
import torch.distributed.checkpoint as dcp

state = {"model": model.state_dict(), "opt": opt.state_dict(), "step": step}
dcp.save(state, checkpoint_id=f"/ckpt/step_{step}")

Each rank writes its own shard; loading reverses the process. DCP is the default in PyTorch 2.4+.

1.3.8 2.8 Data loading at scale

A 64-GPU job demands ~64× the data throughput of a single GPU. Best practices:

  • Sharded data: WebDataset or MosaicML Streaming format — many tar shards on object store, each rank streams a disjoint set.
  • num_workers ≈ 4 × CPU/GPU with pin_memory=True.
  • persistent_workers=True to avoid re-spawning every epoch.
  • prefetch_factor=2 to overlap CPU and GPU.
  • Image decoding on GPU with NVIDIA DALI for VLMs — JPEG → tensor is often the bottleneck.

1.4 3. Optimizing Communication Overhead

1.4.1 3.1 The model: communication vs compute

For a parameter \(P\), an all-reduce takes time

\[ T_{\text{ar}}(P) \approx \alpha + \beta \cdot 2 \frac{N-1}{N} \cdot P \]

where \(\alpha\) is latency, \(\beta = 1/\text{bandwidth}\), and the factor \(\frac{2(N-1)}{N}\) comes from the ring-all-reduce algorithm. For \(N\) large, this becomes \(2P/B\) — twice the parameter bytes over the network.

A single bf16 7 B model has \(2 \cdot 7\text{B} = 14\) GB. On a 100 Gbit/s NIC = 12.5 GB/s, one all-reduce takes 1.1 seconds. Compare to ~0.4 seconds of compute for the backward pass → communication dominates unless we hide it.

1.4.2 3.2 Overlap with computation

PyTorch DDP automatically buckets gradients and starts all-reducing the bucket as soon as it is filled — overlap with the rest of the backward pass. Two knobs:

  • gradient_as_bucket_view=True — saves memory.
  • bucket_cap_mb (default 25 MB) — larger buckets = fewer collectives but less overlap. For 7-13 B models, 50-100 MB is often optimal.

For FSDP, set backward_prefetch=BackwardPrefetch.BACKWARD_PRE so the next shard’s parameters are all-gathered while you compute the current backward. The matching forward_prefetch=True does the analogous thing in forward.

1.4.3 3.3 Gradient compression

Trade communication for accuracy:

  • fp16 all-reduce with MixedPrecision(reduce_dtype=torch.float16) — halves bytes.
  • PowerSGD (Vogels et al., 2019) — rank-r low-rank approximation. Built into PyTorch DDP.
import torch.distributed.algorithms.ddp_comm_hooks.powerSGD_hook as ps
state = ps.PowerSGDState(process_group=None, matrix_approximation_rank=4,
                         start_powerSGD_iter=1_000)
model.register_comm_hook(state, ps.powerSGD_hook)

PowerSGD is essentially free for many models after a warm-up period.

1.4.4 3.4 Bucket sizes and parameter ordering

DDP reduces buckets in the order parameters are registered. To maximise overlap, register parameters in forward order (default if you wrap a nn.Module directly).

For FSDP, set the auto-wrap policy at the right granularity — each FSDP unit is a single all-gather. Too many small units = collective overhead; too few large units = no overlap.

1.4.5 3.5 ZeRO trade-offs

ZeRO-3 / FSDP FULL_SHARD has the lowest memory but the most communication: every parameter is gathered and re-discarded once per forward + once per backward. ZeRO-2 (SHARD_GRAD_OP) gathers parameters once (they are kept) but still scatters gradients. Choose:

  • Memory-bound? → ZeRO-3.
  • Communication-bound? → ZeRO-2 or HYBRID_SHARD.

HYBRID_SHARD is the sweet spot for 2–4 node setups: full-shard inside the fast NVLink island, replicate across the slow Ethernet/IB.

1.4.6 3.6 Compute–comm overlap for activations

When you use TP and SP, you must overlap the all-gather of activations before each row-parallel matmul with the previous compute. PyTorch’s parallelize_module does this automatically if you pass use_local_output=False.

1.4.7 3.7 NCCL tuning examples

A few real-world tunes:

  • For a 2-node A100-40 GB setup with 200 Gbit/s IB, NCCL_ALGO=Ring NCCL_PROTO=LL128 gives best throughput for 1 GB messages.
  • For an H100 NVL node, NCCL_NVLS_ENABLE=1 cuts intra-node all-reduce by ~30 %.
  • For noisy fabrics, NCCL_IB_TIMEOUT=22 NCCL_IB_RETRY_CNT=7 avoids spurious crashes; pair with elastic restart.

1.4.8 3.8 Profile communication

Three tools, all free:

  • torch.profiler — annotate ranges, dump Chrome trace, inspect collective calls.
  • NVIDIA Nsight Systems (nsys) — system-wide GPU + NCCL traces.
  • NCCL_DEBUG=INFO logs + the nccl-tests benchmark — measure raw all-reduce bandwidth.
from torch.profiler import profile, ProfilerActivity, schedule

with profile(
    activities=[ProfilerActivity.CPU, ProfilerActivity.CUDA],
    schedule=schedule(wait=1, warmup=1, active=3, repeat=1),
    on_trace_ready=lambda p: p.export_chrome_trace(f"trace_{dist.get_rank()}.json"),
    record_shapes=True, with_stack=True,
) as p:
    for step in range(5):
        ...
        p.step()

Open the trace in chrome://tracing; look for nccl:all_reduce rows that don’t overlap the compute rows above them. Anything not overlapped is wasted time.

1.5.2 4.2 torch.export — the modern foundation

PyTorch ≥ 2.2 introduced torch.export, which captures the model as an ExportedProgram — an FX graph with a sound, stable representation. This is the source format for both ExecuTorch and the modern ONNX exporter.

import torch

example_inputs = (torch.randn(1, 3, 384, 384),)
ep = torch.export.export(vision_model, example_inputs)
print(ep)                                  # ExportedProgram with input/output specs
torch.export.save(ep, "vision.pt2")

Dynamic shapes (variable resolution!) are declared explicitly:

from torch.export import Dim
H = Dim("H", min=224, max=1024)
W = Dim("W", min=224, max=1024)
ep = torch.export.export(model, (torch.randn(1,3,384,384),),
                         dynamic_shapes={"pixel_values": {2: H, 3: W}})

1.5.3 4.3 Export to ONNX

import torch
torch.onnx.export(
    model, example_inputs, "vlm.onnx",
    input_names=["pixel_values","input_ids"],
    output_names=["logits"],
    dynamic_axes={"pixel_values": {0: "B", 2: "H", 3: "W"},
                  "input_ids":   {0: "B", 1: "T"},
                  "logits":      {0: "B", 1: "T"}},
    opset_version=18,
    dynamo=True,                       # the new exporter, FX-based
)

The dynamo=True exporter is the future — it relies on torch.export under the hood and supports control flow and dynamic shapes far better than the legacy TorchScript-based exporter.

Validate the ONNX model:

import onnxruntime as ort, numpy as np
sess = ort.InferenceSession("vlm.onnx", providers=["CUDAExecutionProvider","CPUExecutionProvider"])
outs = sess.run(None, {"pixel_values": np.random.randn(1,3,384,384).astype(np.float32),
                       "input_ids":   np.array([[1,2,3]], dtype=np.int64)})

Two things to verify every time:

  1. Output deltas between torch_model(x) and onnx_runtime(x) should be < 1e-3 in fp32 (< 1e-2 in fp16). Anything larger is a buggy export.
  2. All operators supported by your target runtime. Run onnx.checker.check_model("vlm.onnx") and polygraphy inspect model vlm.onnx --mode=onnx to enumerate ops.

1.5.4 4.4 ExecuTorch — PyTorch’s mobile/edge runtime

ExecuTorch is the successor to PyTorch Mobile. The pipeline:

torch.export → quantization → to_edge → backend partitioning → to_executorch → .pte
from executorch.exir import to_edge
from executorch.backends.xnnpack.partition.xnnpack_partitioner import XnnpackPartitioner

ep   = torch.export.export(model, example_inputs)
edge = to_edge(ep)
edge = edge.to_backend(XnnpackPartitioner())          # delegate ops to XNNPACK
prog = edge.to_executorch()
with open("vlm.pte", "wb") as f:
    f.write(prog.buffer)

Backends ship out-of-the-box: XNNPACK (CPU, ARM/x86), Vulkan (mobile GPU), CoreML, Qualcomm QNN (Hexagon NPU), MediaTek, Apple MPS.

The .pte binary is loaded by ExecuTorch’s C++/Java/Swift runtime; you ship one file per platform.

1.5.5 4.5 TensorRT — fastest path on NVIDIA edge

Path of least resistance: ONNX → TensorRT via trtexec.

        --fp16 --workspace=4096 \
        --minShapes=pixel_values:1x3x224x224,input_ids:1x16 \
        --optShapes=pixel_values:1x3x384x384,input_ids:1x64 \

For Jetson Orin / AGX-Orin you compile the engine on the device (engines are device-specific). For data-center INT8/FP8, supply a calibration cache (see Optimization doc).

1.5.9 4.9 The export checklist

Before shipping any exported model, verify:

  1. Numerical parity vs PyTorch reference within tolerance.
  2. Latency measured per device (CPU, GPU, NPU) at target batch / shape.
  3. Memory peak under sustained load (some op kernels temporarily allocate huge buffers).
  4. Cold-start time — engine load + first inference; users care about app launch.
  5. Thermal behaviour — phones throttle after 30 s of sustained compute.
  6. Privacy / on-device requirements — no network calls leaking inputs.

1.6 5. Optimizing for Tizen Platform

1.6.1 5.1 Tizen in two paragraphs

Tizen is Samsung’s Linux-based OS for TVs, smartwatches, washing machines, fridges, and Galaxy IoT devices. It has been the dominant Smart-TV OS since 2015 and ships in tens of millions of devices a year. Apps come in two flavours: native (C/C++ using Tizen ML APIs) and Web/Tizen .NET (JavaScript / C#). Tizen Studio is the IDE.

For VLM deployment the runtime you target is NNStreamer + TensorFlow Lite (with optional GPU/NPU delegates), surfaced via the Tizen ML API (formerly Inference Tizen API).

1.6.2 5.2 The Tizen ML stack

   Application (C/C++, .NET, Web)

NNStreamer is Samsung’s GStreamer-based ML pipeline framework. It treats an ML model as a filter (tensor_filter) in a streaming pipeline, with tensor data flowing as GstBuffers. This is perfect for VLM use-cases like TV captioning of live video — you connect a camera/video source to a vision-encoder filter, fan out to a Q-Former filter, fan out to an LLM filter, and write text to a UI sink.

1.6.3 5.3 The recommended VLM-on-Tizen pipeline

  1. Export the vision encoder (e.g. SigLIP-So400m) to a small TFLite model, quantised INT8.
  2. Export the projector as a separate TFLite model (a few MB).
  3. Export the LLM to a quantised GGUF or TFLite-LLM (currently 4-bit AWQ/GPTQ is best).
  4. Wire them with NNStreamer, with the LLM decoder running in a tensor_filter that supports KV-cache.

1.6.7 5.7 Memory budget on Tizen TVs

A 2024 4K Tizen TV has ~3–4 GB total RAM, of which ~1 GB is available to apps. Practical VLM budget:

Projector 5–10 MB
LLM 1–3 B (INT4) 600–800 MB

1.6.8 5.8 Cold-start optimisation

Tizen apps are killed aggressively. Strategies:

  • Pre-load models on app launch in a background thread.
  • Pin tensors in RAM by holding the mv_inference_h handle.
  • Lazy-load the LLM only when the user actually asks a question (vision encoder always on for ambient features).

1.6.9 5.9 Privacy and DRM

Tizen Smart-TVs frequently process camera frames and microphone audio — Samsung’s Smart Hub privacy policy mandates that all VLM inference happens on-device unless the user explicitly opts in to cloud. Your deployment must run fully offline by default.

1.7.4 6.4 KV-cache and stateful LLM inference

The LLM in a VLM is autoregressive; without a KV cache, generating 100 tokens is 100× more expensive than a single forward. ExecuTorch and CoreML both support stateful models as of 2024:

  • CoreML state: declare states with ct.target.iOS18, ct.StateType. The state lives across predictions() calls.
  • ExecuTorch: explicit (input, kv_in) → (output, kv_out) signature; the runtime exposes MutableTensor to update in place.

A common mistake is to materialise the full KV cache at max-length. Use a ring buffer of kv_len = current_pos actual length.

1.7.5 6.5 Streaming visual encoders

For real-time camera apps, you cannot afford a full ViT pass per frame at 30 fps. Two tricks:

  1. Temporal sub-sampling: run vision encoder at 5 fps, interpolate features for in-between frames.
  2. Patch-residual encoders: re-run only patches whose mean intensity has changed (delta encoding).
  3. Frame-difference gating: a tiny CNN decides whether the scene changed enough to require re-encoding.

1.7.6 6.6 Tokenizer on device

Most LLM tokenizers are SentencePiece or BPE — port them with sentencepiece-android, tiktoken-rs, or compile the C library yourself. Never ship a Python tokenizer to a phone.

For VLMs, you must also reserve and map the <image> token correctly — the same special-token IDs you trained with.

1.7.7 6.7 Image preprocessing on device

Don’t allocate a float32 384×384×3 = 1.7 MB buffer on the JVM heap. Use Android’s ByteBuffer.allocateDirect (zero-copy to native), or MediaPipe Image, or iOS vImage for resize and normalisation. Saving 10 ms per frame here adds up over a long session.

1.7.8 6.8 Battery and thermal

A phone CPU at 100 % drops to half its clock after ~3 minutes. To stay below the thermal limit:

  • Pin VLM inference to efficiency cores when possible (SCHED_BATCH on Android).
  • Use NPU > GPU > CPU. NPUs are 10× more energy-efficient than CPUs.
  • Drop the frame rate when the user is idle.
  • Read BatteryManager.BATTERY_PROPERTY_CURRENT_NOW and back off when the device is hot.

1.8.8 7.8 Accuracy monitoring (the silent killer)

The model that was 75 % MMMU yesterday is now 65 % — why? Possible causes:

  1. Quantisation calibration drift (some weights got re-quantised on the latest release).
  2. Tokenizer mismatch (wrong special tokens).
  3. Image preprocessor mismatch (mean/std flipped, BGR vs RGB).
  4. Resolution mismatch (training was 384², inference is 336²).
  5. Sampling temperature changed.

Mitigation: a continuous golden-set evaluation — 200 hand-picked prompts run nightly through every deployed checkpoint; metrics piped into Prometheus. Alert on > 2 σ drops.

1.8.9 7.9 Cost monitoring

Per-token cost is the deployment KPI for cloud-served VLMs:

\[ \text{cost / 1k tokens} = \frac{\text{GPU hourly price} \cdot 3600}{\text{tokens / sec / replica}} / 1000 \]

If your H100 costs $4/hour and the deployment yields 200 tok/s, then \(4 \times 3600 / 200 / 1000 = \$0.072 / k\) tokens. Track this in Grafana right next to latency.

1.9 8. Troubleshooting Optimized Models

1.9.1 8.1 The five failure modes after optimisation

Symptom Likely cause
Accuracy collapse Quantisation calibration / dtype mismatch
Latency regression Operator fallback to CPU; missing kernel
OOM at inference KV cache overflow; static-shape graph too large
Non-deterministic output Reduction order in fp16/bf16; sampling temperature
Hang or crash on device Unsupported op; quantisation domain mismatch

1.9.2 8.2 Forensic methodology

A systematic recipe:

  1. Reproduce locally: capture the exact input that fails (image + tokens) and re-run through PyTorch reference.

  2. Compare layer outputs: instrument both reference and exported model with hooks, dump activations to disk, np.testing.assert_allclose layer-by-layer.

    activations = {}
def hook(name): return lambda m, i, o: activations.setdefault(name, o.detach().cpu().float().numpy())
for n, m in model.named_modules():
    m.register_forward_hook(hook(n))

  3. Find the first divergent layer. If it is a quantised conv, you have a calibration / scale problem. If it is a softmax, you have an fp16 overflow.

  4. Inspect operators: polygraphy run vlm.onnx --trt --validate --rtol 1e-3 --atol 1e-3 for TRT, executorch.exir.print_program for ExecuTorch.

  5. Bisect by partition: comment out half the model, re-export, see if the bug persists. Classic binary search.

1.9.3 8.3 Common accuracy bugs

1.9.3.1 bf16 → fp16 conversion

Some on-device runtimes (CoreML before iOS 17) silently downcast to fp16. The exponent range is smaller; intermediate exp() overflows. Fix: insert softmax in fp32 explicitly, or scale logits before softmax.

1.9.3.2 Wrong image normalisation

CLIP uses mean (0.48145466, 0.4578275, 0.40821073) and std (0.26862954, 0.26130258, 0.27577711). SigLIP uses mean (0.5, 0.5, 0.5) and std (0.5, 0.5, 0.5). Mix them up and your model will almost work — accuracy drops 5–10 % silently.

1.9.3.3 BGR vs RGB

OpenCV reads BGR; PIL reads RGB. Models almost always trained on RGB. Swap the channels and accuracy plummets.

1.9.3.4 Padding side mismatch

Decoder-only LLMs require left padding at inference time (so the rightmost token is the most recent). Encoder/BERT models want right padding. Mix them up and the model will generate gibberish for the longest sample of a batch.

1.9.3.5 Tokenizer mismatch

Adding <image> after training without re-training the embedding produces random initial values. The model “works” but is 10 % worse on benchmarks.

1.9.3.6 KV-cache rotary mismatch

After quantisation, some runtimes recompute rotary embeddings with slightly different precision. Result: the cache from step \(t\) doesn’t match the embedding at step \(t+1\), so generation drifts. Fix: compute rotary positions in fp32 even when weights are int4.

1.9.4 8.4 Common latency bugs

1.9.4.1 Operator fallback

onnxruntime will gladly run an unsupported op on CPU while the rest runs on GPU — silently. Detect with sess.get_providers() and per-op profiling. Common offenders: ScatterND, dynamic Reshape, Range with int64.

1.9.4.2 Wrong workspace size in TensorRT

--workspace=128 is too small for VLMs. Use --workspace=4096 or --memPoolSize=workspace:4096.

1.9.4.3 Missing Flash-Attention

If scaled_dot_product_attention falls back to the math kernel, you lose 2-3×. Check with torch.backends.cuda.sdp_kernel(enable_flash=True).

1.9.4.4 Lock contention in multi-threaded inference servers

A single Python interpreter is throttled by the GIL. Use multiple worker processes (gunicorn, uvicorn), or pure C++/Rust front-ends, or vLLM/TGI/TensorRT-LLM which release the GIL during compute.

1.9.5 8.5 Common OOM bugs

1.9.5.1 KV-cache overflow

A 7 B LLM with 32-layer × 32-head × 128-dim KV in bf16 costs \(32 \times 2 \times 32 \times 128 \times 2 = 524 288\) bytes per token. At 8 k context that’s 4 GB; at 32 k it’s 16 GB. Either:

  • Quantise the KV cache (fp8 / int8) — torch.compile does this with --quantize-kv.
  • Truncate the context window.
  • Use paged KV cache (vLLM PagedAttention).
  • Use grouped-query attention (already standard in LLaMA-3, Qwen-2).

1.9.5.2 Static-shape graph too large

If you compiled an ONNX graph with max_seq_len=8192, the runtime allocates buffers for 8192 every call. Use dynamic axes or split the engine into multiple seq-length buckets.

1.9.6 8.6 Non-determinism

bf16 reductions on GPU are non-associative; the order of summation depends on which SMs scheduled which blocks. To get bit-exact reproducibility:

torch.use_deterministic_algorithms(True, warn_only=False)
torch.backends.cudnn.deterministic = True
torch.backends.cudnn.benchmark = False
import os; os.environ["CUBLAS_WORKSPACE_CONFIG"] = ":4096:8"

Set sampling to greedy (do_sample=False) and seed everything (torch, numpy, Python’s random).

1.9.7 8.7 Hangs and crashes

On distributed jobs, the most common cause is a single rank exiting while others wait at an all-reduce. Symptoms: training stalls; nvidia-smi shows 0 % util on some GPUs. Cure:

  • NCCL_ASYNC_ERROR_HANDLING=1 — propagate errors across the world.
  • TORCH_NCCL_BLOCKING_WAIT=1 — block on collectives with timeout.
  • TORCH_NCCL_ASYNC_ERROR_HANDLING=1 (>= 2.4) — supersedes the above.
  • Wrap each dist.collective call inside a try/except, log rank-local errors, broadcast termination signal.

For on-device crashes, attach symbol files (addr2line on Android, dSYM on iOS) to logs so you can resolve native stack frames.

1.9.8 8.8 A diagnostic checklist you can recite

When something breaks after optimisation:

  1. Compare end-to-end outputs in fp32 reference vs target runtime.
  2. Re-confirm image preprocessing (mean, std, channel order, resize mode).
  3. Re-confirm tokenizer special tokens and pad direction.
  4. Inspect operator coverage and provider mapping on the target.
  5. Dump activations layer-by-layer; binary-search to first divergence.
  6. Re-export with debug names and shape inference enabled.
  7. Reproduce in a tiny repro; isolate to one block.
  8. Open an upstream issue with that repro.

1.9.9 8.9 The mind of a deployment engineer

You will not understand every line of a 3-million-line runtime stack. ## 1. Configuring Multi-GPU Training

### 1.1 The parallelism zoo

Before any code, you must know what you are sharding.

| Parallelism | What is split across devices? | Best for |
|---|---|---|
| **Data Parallel (DP / DDP)** | Mini-batch | Model fits on one GPU |
| **ZeRO-1** | Optimizer states | DP + memory savings |
| **ZeRO-2** | Optimizer states + grads | More memory savings |
| **ZeRO-3 / FSDP** | Optimizer + grads + params | Models too big for one GPU |
| **Tensor Parallel (TP)** | Single layer's matrices | Layers too big for one GPU |
| **Pipeline Parallel (PP)** | Whole stages of layers | Very deep models |
| **Sequence / Context Parallel (SP/CP)** | The sequence dimension | Very long contexts |
| **Expert Parallel (EP)** | MoE experts | MoE models |

Modern VLMs use a *combination* — typically FSDP + TP, with SP for long-video models.

### 1.2 PyTorch DistributedDataParallel (DDP) — the baseline

DDP replicates the full model on every GPU and averages gradients via **NCCL all-reduce** after each backward.

```python
# launch with:  torchrun --nproc_per_node=8 train.py
import os, torch, torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP
from torch.utils.data import DataLoader, DistributedSampler

def setup():
    dist.init_process_group(backend="nccl")
    local_rank = int(os.environ["LOCAL_RANK"])
    torch.cuda.set_device(local_rank)
    return local_rank

local_rank = setup()
model = build_vlm().to(local_rank)
model = DDP(model, device_ids=[local_rank], find_unused_parameters=False,
            gradient_as_bucket_view=True, broadcast_buffers=False)

sampler = DistributedSampler(dataset, shuffle=True, drop_last=True)
loader  = DataLoader(dataset, batch_size=8, sampler=sampler,
                     num_workers=4, pin_memory=True, persistent_workers=True)

opt    = torch.optim.AdamW(model.parameters(), lr=1e-4)
scaler = torch.amp.GradScaler("cuda")

for epoch in range(EPOCHS):
    sampler.set_epoch(epoch)                     # different shuffle every epoch
    for batch in loader:
        with torch.autocast("cuda", dtype=torch.bfloat16):
            loss = model(**batch).loss
        loss.backward()                          # all-reduce happens here
        opt.step(); opt.zero_grad(set_to_none=True)
```

Five DDP details that always come up:

1. **`device_ids`** *must* contain a single device — the local rank.
2. **`find_unused_parameters=True`** is required if any parameter is **conditionally skipped** in forward (e.g. a frozen LoRA branch). It costs CPU time; flip it off whenever possible.
3. **`gradient_as_bucket_view=True`** saves memory by giving you a view into the all-reduce bucket buffer instead of allocating new gradient tensors.
4. **`broadcast_buffers=False`** turns off the all-reduce of `BatchNorm` running stats — useless for ViTs/LLMs which use LayerNorm only.
5. **`set_to_none=True`** in `zero_grad()` keeps the gradient pointer null instead of a zero tensor, slightly faster.

### 1.3 Gradient accumulation done right

To simulate a larger batch when memory is tight:

```python
accum = 4
opt.zero_grad(set_to_none=True)
for step, batch in enumerate(loader):
    is_last = (step + 1) % accum == 0
    ctx = model.no_sync() if not is_last else nullcontext()
    with ctx:
        with torch.autocast("cuda", dtype=torch.bfloat16):
            loss = model(**batch).loss / accum
        loss.backward()
    if is_last:
        opt.step(); opt.zero_grad(set_to_none=True)
```

`model.no_sync()` **skips the all-reduce** during the inner accumulation steps — communication cost goes down by `accum`. Forget this and you will all-reduce on every micro-batch, paying the full network cost while gaining no math.

### 1.4 Fully Sharded Data Parallel (FSDP / FSDP2)

When the model does not fit on a single GPU, **shard everything**: parameters, gradients, optimizer states.

```python
import functools
from torch.distributed.fsdp import (
    FullyShardedDataParallel as FSDP,
    MixedPrecision, ShardingStrategy, BackwardPrefetch,
)
from torch.distributed.fsdp.wrap import transformer_auto_wrap_policy
from transformers.models.qwen2.modeling_qwen2 import Qwen2DecoderLayer

mp = MixedPrecision(
    param_dtype=torch.bfloat16,
    reduce_dtype=torch.bfloat16,
    buffer_dtype=torch.bfloat16,
)

wrap_policy = functools.partial(
    transformer_auto_wrap_policy,
    transformer_layer_cls={Qwen2DecoderLayer},   # one shard per LLM block
)

model = FSDP(
    model,
    sharding_strategy=ShardingStrategy.FULL_SHARD,         # ZeRO-3
    auto_wrap_policy=wrap_policy,
    mixed_precision=mp,
    backward_prefetch=BackwardPrefetch.BACKWARD_PRE,       # overlap comm
    limit_all_gathers=True,
    device_id=torch.cuda.current_device(),
    use_orig_params=True,                                  # required for ParamGroups
)
```

Sharding strategies:

- `FULL_SHARD` — ZeRO-3 (params + grads + opt states sharded).
- `SHARD_GRAD_OP` — ZeRO-2 (grads + opt states only).
- `HYBRID_SHARD` — full-shard inside a node, replicate across nodes (best for 2-node setups with fat intra-node NVLink and thin inter-node Ethernet).
- `NO_SHARD` — equivalent to DDP.

FSDP2 (PyTorch ≥ 2.4) introduces `fully_shard()` as the **preferred API** — per-parameter sharding, cleaner ergonomics, full DTensor-native. Same conceptual model, nicer code.

### 1.5 Mixed precision: fp16 vs bf16 vs fp8

| Format | Range | Precision | Where to use |
|---|---|---|---|
| **fp16** | ±65 504 | ~3 decimal | Inference, V100 |
| **bf16** | ±$3.4\times 10^{38}$ | ~2 decimal | Default training on A100/H100 |
| **fp8 (e4m3 / e5m2)** | ±448 / ±57 344 | ~1 decimal | H100 only, large speedup |

bf16 is the *de facto* training dtype because it has the same exponent range as fp32 — no `GradScaler` required, no overflow surprises. fp16 only stays alive on V100s that have no bf16 hardware. fp8 needs the Transformer Engine library (`pip install transformer_engine`) and explicit scaling factors.

### 1.6 Activation checkpointing

Recompute activations during backward instead of storing them — trades FLOPs for memory.

```python
from torch.distributed.algorithms._checkpoint.checkpoint_wrapper import (
    apply_activation_checkpointing,
    CheckpointWrapper,
    CheckpointImpl,
)

def check_fn(submodule):
    return isinstance(submodule, Qwen2DecoderLayer)

apply_activation_checkpointing(
    model,
    checkpoint_wrapper_fn=functools.partial(
        CheckpointWrapper,
        checkpoint_impl=CheckpointImpl.NO_REENTRANT,
    ),
    check_fn=check_fn,
)
```

`NO_REENTRANT` is the modern variant and integrates cleanly with FSDP. Saves ~$O(L)$ memory at the cost of one extra forward.

### 1.7 Tensor parallel (TP) and sequence parallel (SP)

When a single transformer block does not fit on a GPU (e.g. 70 B+), split the matrices themselves:

- **Column-wise split** on `W_q, W_k, W_v` and `W_up, W_gate` of MLP.
- **Row-wise split** on `W_o` and `W_down`.
- All-gather along the head dimension before the row-parallel layer.

In PyTorch (≥ 2.3):

```python
from torch.distributed.tensor.parallel import (
    parallelize_module, ColwiseParallel, RowwiseParallel, SequenceParallel,
)
from torch.distributed.device_mesh import init_device_mesh

mesh = init_device_mesh("cuda", (DP, TP), mesh_dim_names=("dp","tp"))
tp_mesh = mesh["tp"]

plan = {
    "self_attn.q_proj": ColwiseParallel(),
    "self_attn.k_proj": ColwiseParallel(),
    "self_attn.v_proj": ColwiseParallel(),
    "self_attn.o_proj": RowwiseParallel(),
    "mlp.gate_proj":  ColwiseParallel(),
    "mlp.up_proj":    ColwiseParallel(),
    "mlp.down_proj":  RowwiseParallel(),
}
for blk in model.model.layers:
    parallelize_module(blk, tp_mesh, plan)
```

Sequence parallel additionally splits LayerNorm activations along the sequence axis between TP collective calls — relevant when sequences are very long (video, long-context VLMs).

### 1.8 Pipeline parallel — when and how

Pipeline parallel (PP) splits the model into stages and pipelines micro-batches through them. PyTorch's `torch.distributed.pipelining` (PiPPy) does this cleanly. PP shines when:

- Model is *deep* (≥ 80 layers).
- Inter-node bandwidth is the bottleneck (PP communicates only activations at stage boundaries, not gradients across all params).

For 7-13 B VLMs, PP is overkill; FSDP + TP is enough.

### 1.9 Recipes by model size

| Size | Single node 8×H100 80 GB | Multi-node |
|---|---|---|
| 1–3 B | DDP + bf16 | seldom needed |
| 7–13 B | FSDP `FULL_SHARD` + activation ckpt | HYBRID_SHARD across 2 nodes |
| 30–70 B | FSDP + TP=2 + activation ckpt | FSDP + TP=4 across N nodes |
| 100 B+ | FSDP + TP=8 + PP=2/4 | scale TP first, then PP |

## 2. Managing Multi-Node Training

### 2.1 What changes when you cross a node boundary

Inside a node, GPUs talk over NVLink/NVSwitch — 600–900 GB/s on A100/H100 nodes.

Across nodes, they go over Ethernet (25–100 Gbit/s) or InfiniBand (200–400 Gbit/s, ~5× faster). The bandwidth difference is **6–30×**; the latency difference is even larger. *Every architectural decision in multi-node training is a workaround for this fact.*

### 2.2 The launcher: `torchrun` and `srun`

**torchrun** (PyTorch's own launcher, supersedes `torch.distributed.launch`):

```bash
# Node 0 (master):
torchrun \
  --nnodes=4 --nproc_per_node=8 \
  --node_rank=0 --rdzv_id=vlm_run01 --rdzv_backend=c10d \
  --rdzv_endpoint=node0.cluster:29500 train.py

# Node 1..3: same command with --node_rank=1, 2, 3
```

`torchrun` is **elastic** by default — workers can join and leave thanks to the c10d rendezvous backend.

**SLURM (`srun`)** — the dominant academic / national-lab scheduler:

```bash
#!/bin/bash
#SBATCH --job-name=vlm
#SBATCH --nodes=4
#SBATCH --ntasks-per-node=8
#SBATCH --gpus-per-task=1
#SBATCH --cpus-per-task=16
#SBATCH --time=24:00:00
#SBATCH --partition=h100

export MASTER_ADDR=$(scontrol show hostnames "$SLURM_JOB_NODELIST" | head -n1)
export MASTER_PORT=29500
export OMP_NUM_THREADS=8
export NCCL_DEBUG=WARN

srun python -u train.py
```

Inside `train.py` you read **`RANK`**, **`LOCAL_RANK`**, **`WORLD_SIZE`** from environment variables that SLURM/torchrun set automatically.

### 2.3 The init_process_group dance

```python
import os, torch, torch.distributed as dist

def setup():
    if "RANK" in os.environ and "WORLD_SIZE" in os.environ:
        dist.init_process_group(backend="nccl")
    else:
        # SLURM path
        rank   = int(os.environ["SLURM_PROCID"])
        world  = int(os.environ["SLURM_NTASKS"])
        local  = int(os.environ["SLURM_LOCALID"])
        dist.init_process_group(backend="nccl", rank=rank, world_size=world,
                                init_method=f"tcp://{os.environ['MASTER_ADDR']}:{os.environ['MASTER_PORT']}")
        os.environ["LOCAL_RANK"] = str(local)
    torch.cuda.set_device(int(os.environ["LOCAL_RANK"]))
```

### 2.4 NCCL — the actual collective library

NCCL is NVIDIA's collective communication library; PyTorch's `nccl` backend is a thin wrapper. The collectives you need to know:

- **`all_reduce`** — every rank ends up with the sum (or other op) of every input. Used after `loss.backward()` to average gradients.
- **`all_gather`** — every rank collects all inputs concatenated. Used by FSDP to materialise full parameters before forward.
- **`reduce_scatter`** — sum, then scatter chunks back. Used by FSDP for gradients.
- **`broadcast`** — one rank's tensor goes to all others. Used to seed initial weights and to share the schedule.
- **`barrier`** — synchronisation point.

NCCL chooses among several algorithms:

| Algorithm | Bandwidth-optimal? | Latency | Notes |
|---|---|---|---|
| **Ring** | Yes for large messages | $O(P)$ steps | classic |
| **Tree** | Better latency | $O(\log P)$ | newer default for small msgs |
| **CollNet (SHARP)** | Hardware-accelerated on Mellanox switches | very low | InfiniBand only |
| **NVLS** (NVLink SHARP) | Intra-node H100 NVSwitch | very low | DGX H100 only |

You can profile NCCL with `NCCL_DEBUG=INFO`; it prints the algorithm chosen for each collective.

### 2.5 Topology and fabric

A simple mental picture: an 8×H100 DGX node looks like

```
    GPU0—NVLink—GPU1—NVLink—GPU2—…
       \        /
        NVSwitch (intra-node, 900 GB/s pairwise)
       /        \
   GPU4    …   GPU7
   |                  |
   NIC0  NIC1  NIC2  NIC3  (InfiniBand HCAs, 1:1 with GPUs)
       \    |    |    /
        switch fabric (200–400 Gbit/s per port)
```

For best inter-node performance you must:

1. Use **GPUDirect RDMA** (NICs read GPU memory directly).
2. Pin each GPU process to the **NUMA node** owning its HCA via `numactl`.
3. Enable **NCCL_IB_HCA** to use the right HCA.
4. Set `NCCL_IB_GID_INDEX=3` for RoCE v2, `NCCL_IB_GID_INDEX=0` for IB.

### 2.6 The 12 NCCL env vars every deployment engineer knows

```bash
export NCCL_DEBUG=WARN                  # INFO for first run, WARN afterwards
export NCCL_ASYNC_ERROR_HANDLING=1      # crash on collective error
export NCCL_SOCKET_IFNAME=^lo,docker0   # which NIC for bootstrap
export NCCL_IB_HCA=mlx5_0,mlx5_2        # which HCAs to use
export NCCL_IB_TIMEOUT=22               # higher for flaky fabrics
export NCCL_IB_RETRY_CNT=7
export NCCL_NET_GDR_LEVEL=5             # max GPUDirect RDMA
export NCCL_P2P_LEVEL=NVL               # force NVLink intra-node
export NCCL_NVLS_ENABLE=1               # H100 NVSwitch SHARP
export NCCL_ALGO=Tree,Ring              # let NCCL choose
export NCCL_PROTO=Simple,LL,LL128       # protocols, lowest latency to highest BW
export OMP_NUM_THREADS=8                # avoid CPU oversubscription
```

### 2.7 Fault tolerance

A 64-GPU job runs for a week and hits a memory error. What do you do?

1. **Checkpoint frequently** (every 250–1000 steps) but **asynchronously** (next chapter).
2. **`torchrun` elastic** restarts crashed workers automatically.
3. **`torch.distributed.elastic`** monitors and signals workers.
4. **TorchSnapshot / Distributed Checkpoint (DCP)** writes sharded checkpoints in parallel — orders of magnitude faster than gathering everything to rank 0.

```python
import torch.distributed.checkpoint as dcp

state = {"model": model.state_dict(), "opt": opt.state_dict(), "step": step}
dcp.save(state, checkpoint_id=f"/ckpt/step_{step}")
```

Each rank writes its own shard; loading reverses the process. DCP is the default in PyTorch 2.4+.

### 2.8 Data loading at scale

A 64-GPU job demands ~64× the data throughput of a single GPU. Best practices:

- **Sharded data**: WebDataset or MosaicML Streaming format — many tar shards on object store, each rank streams a disjoint set.
- **`num_workers ≈ 4 × CPU/GPU`** with `pin_memory=True`.
- **`persistent_workers=True`** to avoid re-spawning every epoch.
- **`prefetch_factor=2`** to overlap CPU and GPU.
- **Image decoding on GPU** with **NVIDIA DALI** for VLMs — JPEG → tensor is often the bottleneck.

## 3. Optimizing Communication Overhead

### 3.1 The model: communication vs compute

For a parameter $P$, an all-reduce takes time

$$
T_{\text{ar}}(P) \approx \alpha + \beta \cdot 2 \frac{N-1}{N} \cdot P
$$

where $\alpha$ is latency, $\beta = 1/\text{bandwidth}$, and the factor $\frac{2(N-1)}{N}$ comes from the ring-all-reduce algorithm. For $N$ large, this becomes $2P/B$ — *twice the parameter bytes over the network*.

A single bf16 7 B model has $2 \cdot 7\text{B} = 14$ GB. On a 100 Gbit/s NIC = 12.5 GB/s, one all-reduce takes **1.1 seconds**. Compare to ~0.4 seconds of compute for the backward pass → communication dominates unless we hide it.

### 3.2 Overlap with computation

PyTorch DDP automatically buckets gradients and starts all-reducing the bucket as soon as it is filled — overlap with the rest of the backward pass. Two knobs:

- `gradient_as_bucket_view=True` — saves memory.
- `bucket_cap_mb` (default 25 MB) — larger buckets = fewer collectives but less overlap. For 7-13 B models, 50-100 MB is often optimal.

For FSDP, set `backward_prefetch=BackwardPrefetch.BACKWARD_PRE` so the next shard's parameters are *all-gathered* while you compute the current backward. The matching `forward_prefetch=True` does the analogous thing in forward.

### 3.3 Gradient compression

Trade communication for accuracy:

- **fp16 all-reduce** with `MixedPrecision(reduce_dtype=torch.float16)` — halves bytes.
- **PowerSGD** (Vogels et al., 2019) — rank-r low-rank approximation. Built into PyTorch DDP.

```python
import torch.distributed.algorithms.ddp_comm_hooks.powerSGD_hook as ps
state = ps.PowerSGDState(process_group=None, matrix_approximation_rank=4,
                         start_powerSGD_iter=1_000)
model.register_comm_hook(state, ps.powerSGD_hook)
```

PowerSGD is essentially free for many models after a warm-up period.

### 3.4 Bucket sizes and parameter ordering

DDP reduces buckets in the order parameters are registered. To maximise overlap, register parameters in *forward order* (default if you wrap a `nn.Module` directly).

For FSDP, set the auto-wrap policy at the right granularity — each FSDP unit is a single all-gather. Too many small units = collective overhead; too few large units = no overlap.

### 3.5 ZeRO trade-offs

ZeRO-3 / FSDP `FULL_SHARD` has the lowest memory but the most communication: every parameter is gathered and re-discarded once per forward + once per backward. ZeRO-2 (`SHARD_GRAD_OP`) gathers parameters once (they are kept) but still scatters gradients. Choose:

- Memory-bound? → ZeRO-3.
- Communication-bound? → ZeRO-2 or `HYBRID_SHARD`.

`HYBRID_SHARD` is the sweet spot for 2–4 node setups: full-shard inside the fast NVLink island, **replicate** across the slow Ethernet/IB.

### 3.6 Compute–comm overlap for activations

When you use TP and SP, you must overlap the **all-gather of activations** before each row-parallel matmul with the previous compute. PyTorch's `parallelize_module` does this automatically if you pass `use_local_output=False`.

### 3.7 NCCL tuning examples

A few real-world tunes:

- For a **2-node** A100-40 GB setup with 200 Gbit/s IB, `NCCL_ALGO=Ring NCCL_PROTO=LL128` gives best throughput for 1 GB messages.
- For an **H100 NVL** node, `NCCL_NVLS_ENABLE=1` cuts intra-node all-reduce by ~30 %.
- For **noisy fabrics**, `NCCL_IB_TIMEOUT=22 NCCL_IB_RETRY_CNT=7` avoids spurious crashes; pair with elastic restart.

### 3.8 Profile communication

Three tools, all free:

- **`torch.profiler`** — annotate ranges, dump Chrome trace, inspect collective calls.
- **NVIDIA Nsight Systems (`nsys`)** — system-wide GPU + NCCL traces.
- **NCCL_DEBUG=INFO logs** + the `nccl-tests` benchmark — measure raw all-reduce bandwidth.

```python
from torch.profiler import profile, ProfilerActivity, schedule

with profile(
    activities=[ProfilerActivity.CPU, ProfilerActivity.CUDA],
    schedule=schedule(wait=1, warmup=1, active=3, repeat=1),
    on_trace_ready=lambda p: p.export_chrome_trace(f"trace_{dist.get_rank()}.json"),
    record_shapes=True, with_stack=True,
) as p:
    for step in range(5):
        ...
        p.step()
```

Open the trace in `chrome://tracing`; look for **`nccl:all_reduce`** rows that *don't* overlap the compute rows above them. Anything not overlapped is wasted time.

### 4.1 The export landscape

| Format | Target runtime | Strengths | Notes |
|---|---|---|---|
| **TorchScript** | LibTorch C++ | Drop-in from PyTorch | Legacy in 2026; mostly replaced by ExecuTorch |
| **ExecuTorch (`.pte`)** | ExecuTorch runtime (mobile, embedded) | Native PyTorch path, partial-backend execution | Default for new PyTorch mobile in 2024+ |
| **ONNX (`.onnx`)** | onnxruntime, TensorRT, OpenVINO | Cross-vendor IR | Operator coverage gaps for new ops |
| **TensorRT (`.engine`)** | NVIDIA Jetson, Orin | Best NVIDIA performance | Vendor-locked |
| **TFLite (`.tflite`)** | Android, Tizen, microcontrollers | Mature mobile ecosystem | Needs conversion via ONNX or AI-Edge-Torch |
| **CoreML (`.mlpackage`)** | iOS / macOS | Neural Engine acceleration | Apple-only |
| **OpenVINO IR (`.xml`+`.bin`)** | Intel CPUs, NPUs | Best Intel performance | x86 / NPU only |
| **GGUF** | llama.cpp | Quantised LLM-only | Used for LLM backbone of VLMs |

### 4.2 `torch.export` — the modern foundation

PyTorch ≥ 2.2 introduced `torch.export`, which captures the model as an **ExportedProgram** — an FX graph with a sound, stable representation. This is the source format for both ExecuTorch and the modern ONNX exporter.

```python
import torch

example_inputs = (torch.randn(1, 3, 384, 384),)
ep = torch.export.export(vision_model, example_inputs)
print(ep)                                  # ExportedProgram with input/output specs
torch.export.save(ep, "vision.pt2")
```

Dynamic shapes (variable resolution!) are declared explicitly:

```python
from torch.export import Dim
H = Dim("H", min=224, max=1024)
W = Dim("W", min=224, max=1024)
ep = torch.export.export(model, (torch.randn(1,3,384,384),),
                         dynamic_shapes={"pixel_values": {2: H, 3: W}})
```

### 4.3 Export to ONNX

```python
import torch
torch.onnx.export(
    model, example_inputs, "vlm.onnx",
    input_names=["pixel_values","input_ids"],
    output_names=["logits"],
    dynamic_axes={"pixel_values": {0: "B", 2: "H", 3: "W"},
                  "input_ids":   {0: "B", 1: "T"},
                  "logits":      {0: "B", 1: "T"}},
    opset_version=18,
    dynamo=True,                       # the new exporter, FX-based
)
```

The `dynamo=True` exporter is the future — it relies on `torch.export` under the hood and supports control flow and dynamic shapes far better than the legacy TorchScript-based exporter.

Validate the ONNX model:

```python
import onnxruntime as ort, numpy as np
sess = ort.InferenceSession("vlm.onnx", providers=["CUDAExecutionProvider","CPUExecutionProvider"])
outs = sess.run(None, {"pixel_values": np.random.randn(1,3,384,384).astype(np.float32),
                       "input_ids":   np.array([[1,2,3]], dtype=np.int64)})
```

Two things to verify *every time*:

1. **Output deltas** between `torch_model(x)` and `onnx_runtime(x)` should be < 1e-3 in fp32 (< 1e-2 in fp16). Anything larger is a buggy export.
2. **All operators supported by your target runtime**. Run `onnx.checker.check_model("vlm.onnx")` and `polygraphy inspect model vlm.onnx --mode=onnx` to enumerate ops.

### 4.4 ExecuTorch — PyTorch's mobile/edge runtime

ExecuTorch is the successor to PyTorch Mobile. The pipeline:

```
torch.export → quantization → to_edge → backend partitioning → to_executorch → .pte
```

```python
from executorch.exir import to_edge
from executorch.backends.xnnpack.partition.xnnpack_partitioner import XnnpackPartitioner

ep   = torch.export.export(model, example_inputs)
edge = to_edge(ep)
edge = edge.to_backend(XnnpackPartitioner())          # delegate ops to XNNPACK
prog = edge.to_executorch()
with open("vlm.pte", "wb") as f:
    f.write(prog.buffer)
```

Backends ship out-of-the-box: **XNNPACK** (CPU, ARM/x86), **Vulkan** (mobile GPU), **CoreML**, **Qualcomm QNN** (Hexagon NPU), **MediaTek**, **Apple MPS**.

The `.pte` binary is loaded by ExecuTorch's C++/Java/Swift runtime; you ship one file per platform.

### 4.5 TensorRT — fastest path on NVIDIA edge

Path of least resistance: ONNX → TensorRT via `trtexec`.

```bash
trtexec --onnx=vlm.onnx --saveEngine=vlm.engine \
        --fp16 --workspace=4096 \
        --minShapes=pixel_values:1x3x224x224,input_ids:1x16 \
        --optShapes=pixel_values:1x3x384x384,input_ids:1x64 \
        --maxShapes=pixel_values:1x3x768x768,input_ids:1x256
```

For Jetson Orin / AGX-Orin you compile the engine on the device (engines are device-specific). For data-center INT8/FP8, supply a calibration cache (see Optimization doc).

### 4.6 CoreML (iOS / macOS)

```python
import coremltools as ct
mlmodel = ct.convert(
    "vlm.onnx",
    convert_to="mlprogram",
    compute_units=ct.ComputeUnit.ALL,     # CPU + GPU + Neural Engine
    minimum_deployment_target=ct.target.iOS17,

mlmodel.save("VLM.mlpackage")
```

iOS 18 supports **8-bit weight quantisation natively** and INT4 weight-only quantisation via `ct.optimize.coreml.linear_quantize_weights`.

### 4.7 TFLite via ai-edge-torch

Google's `ai-edge-torch` is the modern PyTorch → TFLite bridge:

```python
import ai_edge_torch
edge = ai_edge_torch.convert(model.eval(), sample_args)
edge.export("vlm.tflite")
```

It uses `torch.export` under the hood and handles dynamic shapes properly. The resulting `.tflite` runs in any TFLite runtime — Android, Tizen, embedded Linux.

### 4.8 OpenVINO

For Intel CPUs / iGPUs / NPUs (Meteor Lake, Lunar Lake) you go via ONNX:

```bash
ovc vlm.onnx --output_dir ovir/         # OpenVINO Model Converter
```

Inference:

```python
from openvino.runtime import Core
core = Core()
model = core.read_model("ovir/vlm.xml")
compiled = core.compile_model(model, device_name="NPU")     # or "GPU", "CPU"
```

OpenVINO is particularly strong for VLMs on AI PCs because the NPU has dedicated INT8 matmul hardware.

### 4.9 The export checklist

Before shipping any exported model, verify:

1. **Numerical parity** vs PyTorch reference within tolerance.
2. **Latency** measured **per device** (CPU, GPU, NPU) at target batch / shape.
3. **Memory peak** under sustained load (some op kernels temporarily allocate huge buffers).
4. **Cold-start time** — engine load + first inference; users care about app launch.
5. **Thermal behaviour** — phones throttle after 30 s of sustained compute.
6. **Privacy / on-device** requirements — no network calls leaking inputs.

## 5. Optimizing for Tizen Platform

### 5.1 Tizen in two paragraphs

Tizen is Samsung's Linux-based OS for **TVs, smartwatches, washing machines, fridges, and Galaxy IoT devices**. It has been the dominant Smart-TV OS since 2015 and ships in tens of millions of devices a year. Apps come in two flavours: **native** (C/C++ using Tizen ML APIs) and **Web/Tizen .NET** (JavaScript / C#). Tizen Studio is the IDE.

For VLM deployment the runtime you target is **NNStreamer** + **TensorFlow Lite** (with optional GPU/NPU delegates), surfaced via the **Tizen ML API** (formerly *Inference Tizen API*).

### 5.2 The Tizen ML stack

```
   Application (C/C++, .NET, Web)
         │
   Tizen ML API  (high-level)
         │
   NNStreamer  (GStreamer plugin, pipeline-style ML)
         │
   ML inference backends
   ├── TensorFlow Lite     (CPU, GPU, Hexagon, EdgeTPU)
   ├── ONNX Runtime        (CPU, GPU, NPU)
   ├── Snapdragon NPE
   ├── Samsung ENN         (Exynos NPU)
   └── OpenVINO            (Intel-based Tizen devices)
```

NNStreamer is Samsung's GStreamer-based ML pipeline framework. It treats an ML model as a *filter* (`tensor_filter`) in a streaming pipeline, with tensor data flowing as GstBuffers. This is perfect for VLM use-cases like **TV captioning of live video** — you connect a camera/video source to a vision-encoder filter, fan out to a Q-Former filter, fan out to an LLM filter, and write text to a UI sink.

### 5.3 The recommended VLM-on-Tizen pipeline

1. Export the **vision encoder** (e.g. SigLIP-So400m) to a small TFLite model, quantised INT8.
2. Export the **projector** as a separate TFLite model (a few MB).
3. Export the **LLM** to a quantised GGUF or TFLite-LLM (currently 4-bit AWQ/GPTQ is best).
4. Wire them with NNStreamer, with the LLM decoder running in a `tensor_filter` that supports KV-cache.

### 5.4 Calling a TFLite model from a Tizen native app

A minimal C example using Tizen ML API:

```c
#include <mv_inference.h>
#include <mv_inference_type.h>

mv_engine_config_h cfg;
mv_create_engine_config(&cfg);
mv_engine_config_set_string_attribute(cfg, MV_INFERENCE_MODEL_WEIGHT_FILE_PATH,
                                      "vision_int8.tflite");
mv_engine_config_set_string_attribute(cfg, MV_INFERENCE_BACKEND_TYPE,
                                      "tflite");
mv_engine_config_set_string_attribute(cfg, MV_INFERENCE_TARGET_DEVICE_TYPE,
                                      "npu");           // or "gpu" / "cpu"

mv_inference_h infer;
mv_inference_create(&infer);
mv_inference_configure(infer, cfg);
mv_inference_prepare(infer);

mv_source_h src;
mv_create_source(&src);
mv_source_fill_by_buffer(src, frame_buffer, frame_size, width, height, MEDIA_VISION_COLORSPACE_RGB888);

mv_inference_image_classify(src, infer, NULL, callback, NULL);
```

For VLMs you typically need the lower-level **inference-engine-tflite** API to get the raw tensor outputs (because you fuse them with text), then feed them into the next stage.

### 5.5 NNStreamer pipeline (GStreamer description)

```bash
gst-launch-1.0 \
  v4l2src ! videoconvert ! videoscale ! \
  video/x-raw,format=RGB,width=384,height=384 ! \
  tensor_converter ! \
  tensor_filter framework=tensorflow-lite model=vision_int8.tflite custom=Delegate:NPU ! \
  tensor_filter framework=tensorflow-lite model=projector_int8.tflite ! \

  tensor_decoder mode=detokenize ! \
  appsink
```

This pipeline does **end-to-end video → caption** entirely on the TV's NPU.

### 5.6 Samsung-specific NPU: ENN (Exynos Neural Network)

For Galaxy phones and select Tizen devices, the **ENN backend** offers ~3–10× better performance than TFLite-GPU. You convert via Samsung's **ENN converter**:

```
enn_converter --input vision.onnx --output vision.nnc --quant int8 \
              --target exynos2400
```

ENN supports per-channel INT8 weight + per-tensor INT8 activation, and FP16 fallback for unsupported ops. The runtime is loaded by NNStreamer with `framework=snpe-enn`.

### 5.7 Memory budget on Tizen TVs

A 2024 4K Tizen TV has ~3–4 GB total RAM, of which ~1 GB is available to apps. Practical VLM budget:

| Component | Budget |
|---|---|
| Vision encoder (INT8) | 80–150 MB |
| Projector | 5–10 MB |
| LLM 1–3 B (INT4) | 600–800 MB |
| KV cache (for 1k tokens) | 100–200 MB |
| Misc app | 100 MB |

This means for TVs you typically pick a *small* LLM (Phi-3-mini-3.8B INT4 ≈ 1.9 GB → too big; Qwen2.5-1.5B-Instruct INT4 ≈ 800 MB → fits).

### 5.8 Cold-start optimisation

Tizen apps are killed aggressively. Strategies:

- **Pre-load models on app launch** in a background thread.
- **Pin tensors in RAM** by holding the `mv_inference_h` handle.
- **Lazy-load the LLM** only when the user actually asks a question (vision encoder always on for ambient features).

### 5.9 Privacy and DRM

Tizen Smart-TVs frequently process camera frames and microphone audio — Samsung's **Smart Hub privacy policy** mandates that all VLM inference happens on-device unless the user explicitly opts in to cloud. Your deployment **must run fully offline** by default.

## 6. Implementing Mobile Inference

### 6.1 Hardware landscape

| Platform | Compute units | Frameworks | Notes |
|---|---|---|---|
| **Android (high-end)** | CPU (8 cores), GPU (Adreno/Mali), NPU (Hexagon, NPU0) | ExecuTorch, TFLite, ONNX-RT, MediaPipe, QNN | Hexagon HTP is the strongest mobile NPU |
| **Android (mid-tier)** | CPU + GPU only | TFLite GPU/XNNPACK | NPU may be absent |
| **iOS** | CPU (4-8 cores), GPU, Apple Neural Engine (ANE) | CoreML, ExecuTorch+CoreML, MLX, MPSGraph | ANE accepts mlprogram only |
| **Apple Vision Pro** | M2 + R1 | CoreML | Real-time spatial VLMs |

For VLMs in 2026, the realistic on-device targets are **3–8 B LLM, 0.4 B vision encoder, INT4 weights**, running 5–25 tokens/s.

### 6.2 ExecuTorch on Android

The standard mobile path. Steps:

1. Export the model (see §4.4).
2. Add the ExecuTorch AAR to your Android app:

   ```gradle
   implementation 'org.pytorch:executorch-android:0.4.0'
   ```

3. Load and run from Kotlin:

   ```kotlin
   import org.pytorch.executorch.Module
   import org.pytorch.executorch.EValue
   import org.pytorch.executorch.Tensor

   val module = Module.load(assetFilePath(this, "vlm.pte"))
   val pix = Tensor.fromBlob(pixelFloats, longArrayOf(1, 3, 384, 384))
   val ids = Tensor.fromBlob(tokenIds, longArrayOf(1, tokenIds.size.toLong()))
   val out = module.forward(EValue.from(pix), EValue.from(ids))
   val logits = out[0].toTensor()
   ```

4. To use the Qualcomm NPU, swap `XnnpackPartitioner` for `QnnPartitioner(QnnExecuTorchHtpBackendOptions(...))` during export.

### 6.3 ExecuTorch on iOS

```python
from executorch.backends.apple.coreml.partition.coreml_partitioner import CoreMLPartitioner
edge = edge.to_backend(CoreMLPartitioner(skip_ops_for_coreml_delegation=[]))
prog = edge.to_executorch()
```

Then in Swift:

```swift
import ExecuTorch
let module = try Module(filePath: pteURL.path)
let output = try module.forward([pixelTensor, idTensor])
```

If you prefer pure CoreML (no ExecuTorch), use `coremltools` (§4.6) and `MLModel`.

### 6.4 KV-cache and stateful LLM inference

The LLM in a VLM is autoregressive; without a KV cache, generating 100 tokens is 100× more expensive than a single forward. ExecuTorch and CoreML both support **stateful models** as of 2024:

- **CoreML state**: declare states with `ct.target.iOS18`, `ct.StateType`. The state lives across `predictions()` calls.
- **ExecuTorch**: explicit `(input, kv_in) → (output, kv_out)` signature; the runtime exposes `MutableTensor` to update in place.

A common mistake is to materialise the full KV cache at max-length. Use a *ring buffer* of `kv_len = current_pos` actual length.

### 6.5 Streaming visual encoders

For real-time camera apps, you cannot afford a full ViT pass per frame at 30 fps. Two tricks:

1. **Temporal sub-sampling**: run vision encoder at 5 fps, interpolate features for in-between frames.
2. **Patch-residual encoders**: re-run only patches whose mean intensity has changed (delta encoding).
3. **Frame-difference gating**: a tiny CNN decides whether the scene changed enough to require re-encoding.

### 6.6 Tokenizer on device

Most LLM tokenizers are SentencePiece or BPE — port them with **`sentencepiece-android`**, **`tiktoken-rs`**, or compile the C library yourself. **Never** ship a Python tokenizer to a phone.

For VLMs, you must also reserve and map the `<image>` token correctly — the same special-token IDs you trained with.

### 6.7 Image preprocessing on device

Don't allocate a `float32` 384×384×3 = 1.7 MB buffer on the JVM heap. Use Android's `ByteBuffer.allocateDirect` (zero-copy to native), or `MediaPipe Image`, or iOS `vImage` for resize and normalisation. Saving 10 ms per frame here adds up over a long session.

### 6.8 Battery and thermal

A phone CPU at 100 % drops to half its clock after ~3 minutes. To stay below the thermal limit:

- Pin VLM inference to **efficiency cores** when possible (`SCHED_BATCH` on Android).
- Use NPU > GPU > CPU. NPUs are 10× more energy-efficient than CPUs.
- Drop the frame rate when the user is idle.
- Read `BatteryManager.BATTERY_PROPERTY_CURRENT_NOW` and back off when the device is hot.

## 7. Monitoring Deployment Performance

### 7.1 What to measure

| Metric | Definition | Target |
|---|---|---|
| **Latency p50 / p95 / p99** | Time per request at the 50/95/99 percentile | Depends; for chat, p95 ≤ 1.5 s TTFT |
| **TTFT (Time To First Token)** | Latency to first generated token | Critical for UX |
| **TPOT (Time Per Output Token)** | Average inter-token latency | Steady-state generation speed |
| **Throughput** | Requests / second or tokens / second | Server capacity |
| **GPU SM utilisation** | % of streaming multiprocessors active | > 60 % is healthy |
| **GPU memory** | Bytes allocated | < 90 % to avoid OOM |
| **Energy per request** | Joules | Mobile: minimise |
| **Accuracy drift** | KPI vs validation set | < 1 % drop after quantisation |

### 7.2 Server-side: Prometheus + Grafana

Instrument the inference server with the **prometheus-client** library:

```python
from prometheus_client import Counter, Histogram, Gauge

inflight  = Gauge("vlm_inflight_requests", "Currently processing")
latency   = Histogram("vlm_request_latency_seconds", "End-to-end latency",
                      buckets=(.05,.1,.25,.5,1,2,5,10))
tokens    = Counter("vlm_tokens_generated_total", "Generated tokens")
gpu_util  = Gauge("vlm_gpu_util_percent", "SM utilization")

@app.post("/generate")
async def generate(req):
    inflight.inc()
    t0 = time.time()
    out = model.generate(**req.payload)
    latency.observe(time.time() - t0)
    tokens.inc(out.shape[1])
    inflight.dec()
    return out
```

Scrape with Prometheus, render with Grafana. The dashboard you want:

- p50/p95/p99 latency line.
- TTFT histogram.
- Tokens/sec rate.
- GPU memory bar.
- Error rate panel.
- Saturation: queue depth vs in-flight.

### 7.3 GPU telemetry: `dcgm-exporter`

NVIDIA's **DCGM** exporter exposes per-GPU SM utilisation, memory, temperature, power, NVLink throughput, ECC errors, PCIe replays. Run it as a sidecar; scrape with the same Prometheus.

### 7.4 PyTorch native profiling

For deep dives:

```python
import torch
from torch.profiler import profile, ProfilerActivity

with profile(
    activities=[ProfilerActivity.CPU, ProfilerActivity.CUDA],
    record_shapes=True, profile_memory=True, with_stack=True,

print(prof.key_averages().table(sort_by="cuda_time_total", row_limit=20))
```

Open the trace in `chrome://tracing`; look for:

- **Long blue gaps** between CUDA kernels — CPU-side bottleneck.
- **`aten::copy_`** with large tensor sizes — unwanted CPU↔GPU copies.
- **Many tiny kernels** — fusion opportunity (try `torch.compile`).

### 7.5 vLLM / TensorRT-LLM metrics

Both serving stacks expose detailed metrics out of the box:

- **vLLM**: `--enable-metrics` flag enables `/metrics` Prometheus endpoint with per-batch info.
- **TensorRT-LLM**: integrates with **Triton Inference Server**; Triton has its own Prometheus exporter.

The two queue-management metrics you must track:

- **Pending requests in the scheduler** — if > 0 for long periods, you're under-provisioned.
- **Pre-emption count** — vLLM's continuous batching may evict requests when memory is tight.

### 7.6 Distributed tracing

For multi-stage pipelines (preprocess → vision → projector → LLM → postprocess), use **OpenTelemetry** to get end-to-end span traces. Jaeger or Honeycomb work great.

### 7.7 Edge / mobile telemetry

Phones don't run Prometheus. Strategies:

- **Firebase Performance** / **iOS MetricKit** for latency histograms.
- **Custom thin telemetry**: aggregate p50/p95 locally, batch-upload daily (privacy-friendly).
- **Crash reporting**: Firebase Crashlytics for native crashes from the inference C++ code.

### 7.8 Accuracy monitoring (the silent killer)

The model that was 75 % MMMU yesterday is now 65 % — why? Possible causes:

1. Quantisation calibration drift (some weights got re-quantised on the latest release).
2. Tokenizer mismatch (wrong special tokens).
3. Image preprocessor mismatch (mean/std flipped, BGR vs RGB).
4. Resolution mismatch (training was 384², inference is 336²).
5. Sampling temperature changed.

Mitigation: a **continuous golden-set evaluation** — 200 hand-picked prompts run nightly through every deployed checkpoint; metrics piped into Prometheus. Alert on > 2 σ drops.

### 7.9 Cost monitoring

Per-token cost is the deployment KPI for cloud-served VLMs:

$$
\text{cost / 1k tokens} = \frac{\text{GPU hourly price} \cdot 3600}{\text{tokens / sec / replica}} / 1000
$$

If your H100 costs $4/hour and the deployment yields 200 tok/s, then $4 \times 3600 / 200 / 1000 = \$0.072 / k$ tokens. Track this in Grafana right next to latency.

## 8. Troubleshooting Optimized Models

### 8.1 The five failure modes after optimisation

| Symptom | Likely cause |
|---|---|
| **Accuracy collapse** | Quantisation calibration / dtype mismatch |
| **Latency regression** | Operator fallback to CPU; missing kernel |
| **OOM at inference** | KV cache overflow; static-shape graph too large |
| **Non-deterministic output** | Reduction order in fp16/bf16; sampling temperature |
| **Hang or crash on device** | Unsupported op; quantisation domain mismatch |

### 8.2 Forensic methodology

A systematic recipe:

1. **Reproduce locally**: capture the *exact* input that fails (image + tokens) and re-run through PyTorch reference.
2. **Compare layer outputs**: instrument both reference and exported model with hooks, dump activations to disk, `np.testing.assert_allclose` layer-by-layer.

   ```python
   activations = {}
   def hook(name): return lambda m, i, o: activations.setdefault(name, o.detach().cpu().float().numpy())
   for n, m in model.named_modules():
       m.register_forward_hook(hook(n))
   ```

3. **Find the first divergent layer**. If it is a quantised conv, you have a calibration / scale problem. If it is a softmax, you have an fp16 overflow.
4. **Inspect operators**: `polygraphy run vlm.onnx --trt --validate --rtol 1e-3 --atol 1e-3` for TRT, `executorch.exir.print_program` for ExecuTorch.
5. **Bisect by partition**: comment out half the model, re-export, see if the bug persists. Classic binary search.

### 8.3 Common accuracy bugs

#### bf16 → fp16 conversion

Some on-device runtimes (CoreML before iOS 17) silently downcast to fp16. The exponent range is smaller; intermediate `exp()` overflows. Fix: insert `softmax` in fp32 explicitly, or scale logits before softmax.

#### Wrong image normalisation

CLIP uses mean `(0.48145466, 0.4578275, 0.40821073)` and std `(0.26862954, 0.26130258, 0.27577711)`. SigLIP uses mean `(0.5, 0.5, 0.5)` and std `(0.5, 0.5, 0.5)`. Mix them up and your model will *almost* work — accuracy drops 5–10 % silently.

#### BGR vs RGB

OpenCV reads BGR; PIL reads RGB. Models almost always trained on RGB. Swap the channels and accuracy plummets.

#### Padding side mismatch

Decoder-only LLMs require **left padding** at inference time (so the rightmost token is the most recent). Encoder/BERT models want **right padding**. Mix them up and the model will generate gibberish for the longest sample of a batch.

#### Tokenizer mismatch

Adding `<image>` after training without re-training the embedding produces random initial values. The model "works" but is 10 % worse on benchmarks.

#### KV-cache rotary mismatch

After quantisation, some runtimes recompute rotary embeddings with slightly different precision. Result: the cache from step $t$ doesn't match the embedding at step $t+1$, so generation drifts. Fix: compute rotary positions in fp32 even when weights are int4.

### 8.4 Common latency bugs

#### Operator fallback

`onnxruntime` will gladly run an unsupported op on CPU while the rest runs on GPU — silently. Detect with `sess.get_providers()` and per-op profiling. Common offenders: `ScatterND`, dynamic `Reshape`, `Range` with int64.

#### Wrong workspace size in TensorRT

`--workspace=128` is too small for VLMs. Use `--workspace=4096` or `--memPoolSize=workspace:4096`.

#### Missing Flash-Attention

If `scaled_dot_product_attention` falls back to the math kernel, you lose 2-3×. Check with `torch.backends.cuda.sdp_kernel(enable_flash=True)`.

#### Lock contention in multi-threaded inference servers

A single Python interpreter is throttled by the GIL. Use multiple worker processes (gunicorn, uvicorn), or pure C++/Rust front-ends, or vLLM/TGI/TensorRT-LLM which release the GIL during compute.

### 8.5 Common OOM bugs

#### KV-cache overflow

A 7 B LLM with 32-layer × 32-head × 128-dim KV in bf16 costs $32 \times 2 \times 32 \times 128 \times 2 = 524 288$ bytes per token. At 8 k context that's 4 GB; at 32 k it's 16 GB. Either:

- Quantise the KV cache (fp8 / int8) — `torch.compile` does this with `--quantize-kv`.
- Truncate the context window.
- Use **paged KV cache** (vLLM PagedAttention).
- Use **grouped-query attention** (already standard in LLaMA-3, Qwen-2).

#### Static-shape graph too large

If you compiled an ONNX graph with `max_seq_len=8192`, the runtime allocates buffers for 8192 every call. Use dynamic axes or split the engine into multiple seq-length buckets.

### 8.6 Non-determinism

bf16 reductions on GPU are non-associative; the order of summation depends on which SMs scheduled which blocks. To get bit-exact reproducibility:

```python
torch.use_deterministic_algorithms(True, warn_only=False)
torch.backends.cudnn.deterministic = True
torch.backends.cudnn.benchmark = False
import os; os.environ["CUBLAS_WORKSPACE_CONFIG"] = ":4096:8"
```

Set sampling to greedy (`do_sample=False`) and seed everything (`torch`, `numpy`, Python's `random`).

### 8.7 Hangs and crashes

On distributed jobs, the most common cause is a **single rank exiting** while others wait at an all-reduce. Symptoms: training stalls; `nvidia-smi` shows 0 % util on some GPUs. Cure:

- `NCCL_ASYNC_ERROR_HANDLING=1` — propagate errors across the world.
- `TORCH_NCCL_BLOCKING_WAIT=1` — block on collectives with timeout.
- `TORCH_NCCL_ASYNC_ERROR_HANDLING=1` (>= 2.4) — supersedes the above.
- Wrap each `dist.collective` call inside a try/except, log rank-local errors, broadcast termination signal.

For on-device crashes, attach symbol files (`addr2line` on Android, dSYM on iOS) to logs so you can resolve native stack frames.

### 8.8 A diagnostic checklist you can recite

When something breaks after optimisation:

1. Compare end-to-end outputs in fp32 reference vs target runtime.
2. Re-confirm image preprocessing (mean, std, channel order, resize mode).
3. Re-confirm tokenizer special tokens and pad direction.
4. Inspect operator coverage and provider mapping on the target.
5. Dump activations layer-by-layer; binary-search to first divergence.
6. Re-export with debug names and shape inference enabled.
7. Reproduce in a tiny repro; isolate to one block.
8. Open an upstream issue with that repro.

### 8.9 The mind of a deployment engineer

You will not understand every line of a 3-million-line runtime stack. What you must build instead is **fast iteration** — small experiments, layer hooks, numpy assertions, a CI that exports + numerically tests every commit. A deployment engineer's competitive advantage is not memory; it is the *speed at which they can localise a bug*.
