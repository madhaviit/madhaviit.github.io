1 Vision-Language Model Optimization with PyTorch — The Complete Expert Guide

Optimization is where models go from “works on the demo notebook” to “wins on the leaderboard, runs on a phone, and recovers gracefully from a midnight crash”. This document covers the eight competencies you need to call yourself a VLM optimization expert: checkpointing, learning-rate scheduling, pre-training failure recovery, adaptation strategies, parameter-efficient fine-tuning, loss design, quantization, and knowledge distillation. Every section has both first-principles theory and runnable PyTorch.

1.2 1. Managing Checkpoint Strategies

1.2.1 1.1 Why checkpointing is harder than torch.save

A naïve torch.save(model.state_dict(), "ckpt.pt") works for a 100 M model on one GPU. For a 70 B model on 64 GPUs it (a) takes 8 minutes, (b) requires rank-0 to hold the whole model in CPU RAM, (c) corrupts on crash, and (d) leaves your optimizer states orphaned across half the cluster. A real checkpoint strategy answers six questions:

  1. What do we save? (weights, optimizer, LR scheduler, RNG state, dataloader cursor, scaler, EMA copy)
  2. When do we save? (every N steps, every K minutes, best validation metric, at exit)
  3. Where do we save? (local SSD, network FS, object store, sharded vs single file)
  4. How do we save? (sync vs async, sharded vs gathered, atomic vs partial)
  5. How do we resume? (bit-exact vs warm-start; same world size vs reshape)
  6. How do we retire? (rolling window, by metric, manual pinning)

Get any of these wrong and a week of training disappears in seconds.

1.2.2 1.2 What is in a complete checkpoint

state = {
    "model":          model.state_dict(),       # weights
    "optimizer":      optimizer.state_dict(),   # Adam m, v, step
    "scheduler":      scheduler.state_dict(),   # current step
    "scaler":         scaler.state_dict(),      # if using GradScaler
    "ema":            ema.state_dict(),         # exponential moving average
    "step":           global_step,
    "epoch":          epoch,
    "rng": {
        "python":      random.getstate(),
        "numpy":       np.random.get_state(),
        "torch":       torch.get_rng_state(),
        "cuda":        torch.cuda.get_rng_state_all(),
    },
    "sampler":        sampler.state_dict() if hasattr(sampler, "state_dict") else None,
    "config":         dataclasses.asdict(config),
    "git_sha":        get_git_sha(),
    "version":        "ckpt_v3",
}

Forgetting any of these creates a subtle resume bug. Forgetting the RNG state means a different shuffle order; forgetting the sampler state means re-doing seen samples; forgetting scaler means a different fp16 loss-scale.

1.2.3 1.3 Distributed Checkpoint (DCP) — the modern API

PyTorch 2.4+ ships torch.distributed.checkpoint (DCP). Every rank writes its own shard in parallel; resume gathers them back. Sharded checkpoints survive world-size changes (resharding-on-load) — invaluable for cluster failures.

import torch.distributed.checkpoint as dcp
from torch.distributed.checkpoint.state_dict import get_state_dict, set_state_dict

state_dict, optim_state = get_state_dict(model, optimizer)
ckpt = {"model": state_dict, "optim": optim_state, "step": step}
dcp.save(ckpt, checkpoint_id=f"s3://bucket/ckpts/step_{step}")

Resume:

state_dict, optim_state = get_state_dict(model, optimizer)
ckpt = {"model": state_dict, "optim": optim_state, "step": 0}
dcp.load(ckpt, checkpoint_id="s3://bucket/ckpts/step_5000")
set_state_dict(model, optimizer, model_state_dict=ckpt["model"], optim_state_dict=ckpt["optim"])

DCP integrates with FSDP, TP, PP. Don’t roll your own.

1.2.4 1.4 Async checkpointing

A 30-second dcp.save once per 1000 steps wastes 3 % of training time. Asynchronously snapshot the state in pinned host memory, then offload the write to a background thread:

from torch.distributed.checkpoint.state_dict_saver import async_save

future = async_save(ckpt, checkpoint_id=f"...{step}")   # returns immediately
# training continues
future.result()                                          # wait only at exit / next save

This makes checkpointing essentially free for any save period > 200 steps.

1.2.5 1.5 Atomicity

The checkpoint must be readable after a crash. Two strategies:

  • Write to a temp directory, then rename: dcp.save(..., "ckpt_TMP_{step}") then os.rename to ckpt_{step}. POSIX rename is atomic.
  • Two-phase commit: write all shards then a DONE marker; resume only loads if DONE exists.

1.2.6 1.6 Retention policy

Disk fills fast. A sensible default for a 7-day run:

  • Keep the last \(K=3\) recent checkpoints (rolling).
  • Keep the best by validation metric (pinned).
  • Keep one per epoch (sparse history).
def retain_recent(folder, k=3):
    ckpts = sorted(p for p in os.listdir(folder) if p.startswith("step_"),
                   key=lambda p: int(p.split("_")[1]))
    for old in ckpts[:-k]:
        shutil.rmtree(os.path.join(folder, old))

1.2.7 1.7 Exponential Moving Average (EMA)

EMA produces a parameter average that is usually 1–3 points better than the raw model on validation. The update rule, applied every step:

\[ \theta^{\text{EMA}}_t = \beta \cdot \theta^{\text{EMA}}_{t-1} + (1-\beta) \cdot \theta_t, \qquad \beta \in [0.999, 0.9999]. \]

A common variant warms up \(\beta\) from 0 over the first few thousand steps. Save both the live and EMA states in your checkpoint; serve from EMA.

class EMA:
    def __init__(self, model, decay=0.9999):
        self.decay = decay
        self.shadow = {n: p.detach().clone() for n, p in model.named_parameters() if p.requires_grad}

    @torch.no_grad()
    def update(self, model):
        for n, p in model.named_parameters():
            if p.requires_grad:
                self.shadow[n].mul_(self.decay).add_(p.detach(), alpha=1 - self.decay)

    def state_dict(self):  return self.shadow
    def load_state_dict(self, sd): self.shadow = {k: v.clone() for k, v in sd.items()}

1.2.8 1.8 Best-metric checkpointing

best_metric = float("-inf")
def maybe_save_best(val_metric, step):
    nonlocal best_metric
    if val_metric > best_metric:
        best_metric = val_metric
        dcp.save(ckpt, checkpoint_id=f"{folder}/best")

1.2.9 1.9 Object-store backends

For very large runs, save directly to S3 / GCS / Azure Blob. dcp.save accepts a custom FsspecWriter:

from torch.distributed.checkpoint.fsspec import FsspecWriter
dcp.save(ckpt, storage_writer=FsspecWriter("s3://bucket/ckpts/step_5000"))

For Samsung / on-prem clusters, Lustre or WekaIO is typical; DCP works against any POSIX FS.

1.3 2. Tuning Learning Rate Schedules

1.3.1 2.1 Why the schedule matters more than the optimizer

For transformers, learning-rate (LR) schedule and warmup are single most-tuned hyperparameters. The right schedule recovers 1–3 points; the wrong one diverges. The four pillars:

  1. Initial LR — base learning rate at peak.
  2. Warmup — linear ramp from 0 to the peak.
  3. Decay — cosine, linear, inverse-square-root, or constant.
  4. Final LR — value at the end of training (often 10 % of peak).

1.3.2 2.2 Warmup — why it exists

In the first few thousand steps, gradients are noisy and weights are far from a good solution. A large LR amplifies noise; a small LR makes effective progress impossible. Warmup linearly increases LR from 0 to peak over the first \(W\) steps:

\[ \eta_t = \eta_{\max} \cdot \min(1, t/W). \]

Typical \(W = 500\) to 5000 steps; longer for tiny batch sizes; can be in number of samples rather than steps for stability across hardware.

1.3.3 2.3 The big four schedules

1.3.3.1 (a) Cosine

\[ \eta_t = \eta_{\min} + \tfrac{1}{2}(\eta_{\max} - \eta_{\min})\left(1 + \cos\!\left(\pi \cdot \frac{t-W}{T-W}\right)\right). \]

The default for LLMs, ViTs, CLIP, BLIP-2, LLaVA. Smooth and forgiving.

1.3.3.2 (b) Linear

\[ \eta_t = \eta_{\max} \cdot \left(1 - \frac{t-W}{T-W}\right). \]

Faster decay early on; used by RoBERTa, T5.

1.3.3.3 (c) Inverse square root

\[ \eta_t = \eta_{\max} \cdot \sqrt{W/t}\quad (t > W). \]

The original Transformer schedule (Vaswani et al., 2017). Allows training to continue indefinitely; useful for never-ending pre-training.

1.3.3.4 (d) Constant with cooldown (D-Adaptation / Schedule-Free)

Keep LR constant during the bulk of training, then cool down at the end. Defazio & Mishchenko’s Schedule-Free (2024) eliminates the schedule entirely with a momentum-style rule:

\[ \theta_t \leftarrow \theta_{t-1} - \eta \nabla L(\bar{\theta}_t), \quad \bar{\theta}_t = (1-c_t)\bar{\theta}_{t-1} + c_t \theta_t. \]

Empirically matches cosine without needing to specify \(T\).

1.3.4 2.4 PyTorch implementations

from torch.optim.lr_scheduler import LambdaLR
import math

def cosine_with_warmup(optimizer, num_warmup, num_total, min_ratio=0.1):
    def lr_lambda(step):
        if step < num_warmup:
            return step / max(1, num_warmup)
        progress = (step - num_warmup) / max(1, num_total - num_warmup)
        return max(min_ratio, 0.5 * (1 + math.cos(math.pi * progress)))
    return LambdaLR(optimizer, lr_lambda)

HuggingFace transformers.get_cosine_schedule_with_warmup does the same; optimum exposes more exotic schedules.

1.3.5 2.5 Per-layer / discriminative LR

VLMs often fine-tune a frozen ViT at \(10^{-6}\), a fresh projector at \(10^{-3}\), and an LLM with LoRA at \(10^{-4}\). Build parameter groups:

optimizer = torch.optim.AdamW(
    [
        {"params": vision.parameters(),  "lr": 1e-6},
        {"params": projector.parameters(), "lr": 1e-3},
        {"params": [p for n,p in llm.named_parameters() if "lora_" in n], "lr": 1e-4},
    ],
    betas=(0.9, 0.95), weight_decay=0.05, eps=1e-8,
)

ULMFit (Howard & Ruder, 2018) introduced discriminative fine-tuning — lower LR for earlier layers, higher for later. For ViTs/LLMs this is implemented as a layer-wise LR decay factor \(\gamma\) (commonly 0.65–0.85):

\[ \eta_{\ell} = \eta_{\max} \cdot \gamma^{L - \ell} \]

where \(\ell\) is the layer index and \(L\) is the total layers.

1.3.6 2.6 Peak LR rules of thumb

Setting Suggested peak LR
ViT pre-training from scratch \(1\times 10^{-3}\) (AdamW, β2=0.95)
CLIP-style contrastive \(5\times 10^{-4}\) (linear)
LLM pre-training (decoder, 7 B) \(3\times 10^{-4}\)
LLM continued pre-training \(1\times 10^{-4}\)
VLM Stage-1 (projector only) \(1\times 10^{-3}\)
VLM Stage-2 (full instruction tune) \(2\times 10^{-5}\)
LoRA on a 7 B LLM \(1\times 10^{-4}\) to \(5\times 10^{-4}\)
QLoRA \(2\times 10^{-4}\)

These are starting points. A 30-minute LR-finder sweep (Smith, 2017) almost always pays for itself.

1.3.7 2.7 Batch size and LR — the linear / sqrt rules

For SGD-style optimisers, the linear-scaling rule (Goyal et al., 2017): when batch size goes \(k \times\), multiply LR by \(k\) and use linear warmup. For AdamW, the rule is sub-linear (often \(\sqrt{k}\) or even less). For contrastive losses (CLIP, SigLIP) where larger batch directly improves the objective, the optimal LR rises slower than \(\sqrt{k}\).

1.3.8 2.8 Schedule-Free and Lion

Two newer optimisers worth knowing:

  • Lion (Chen et al., 2023) — uses only the sign of the momentum; needs LR ≈ 1/10 of AdamW; memory-efficient.
  • Schedule-Free AdamW — eliminates the LR schedule; benchmark-competitive.

Both have first-class PyTorch implementations (from lion_pytorch import Lion, from schedulefree import AdamWScheduleFree).

1.3.9 2.9 Gradient clipping

Always clip gradients during VLM training:

torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)

The 1.0 norm is standard for LLMs; tighter (0.5) for ViTs pre-training; looser (5.0) for SGD-from-scratch. Clipping prevents the rare spike from blowing up Adam’s moments.

1.3.10 2.10 Diagnosing a misbehaving schedule

  • Loss plateaus immediately: warmup too short or LR too low.
  • Loss spikes after warmup: peak LR too high; reduce or extend warmup.
  • Loss diverges late: weight decay too low; learning rate too high.
  • Best validation arrives early: training too long; reduce \(T\).

1.4 3. Handling Pre-training Failures

1.4.1 3.1 The taxonomy of failures

Symptom Likely cause
Loss = NaN suddenly fp16 overflow; exploding gradient; bad sample
Loss diverges monotonically LR too high; weight init bad; missing LN
Loss plateaus at high value LR too low; loss bug; label mismatch
Loss spikes then recovers bad batch; well-behaved transformer thing
Grad norm explodes dead ReLU / very deep block; precision issue
GPU OOM partway dataloader memory leak; KV cache overflow
Throughput collapses filesystem hot-spot; sync stragglers
Validation diverges from train overfitting; eval data drift

1.4.2 3.2 The first 48 hours of pre-training — a checklist

Most failures appear in the first day. Hit each item:

  1. Verify the first batch: shapes, dtypes, image normalisation, tokens make sense; loss for one sample is reasonable.
  2. Overfit a tiny subset (e.g., 64 samples) — loss should hit ~0 in a few hundred steps. If it doesn’t, your model is broken before scaling matters.
  3. Initial loss == log(vocab) for an untrained LM (e.g., ≈ 10.5 for 32k vocab).
  4. Warmup curve is smooth (no sudden drops or spikes).
  5. Grad-norm is in [0.1, 10] range; if it is \(10^4\), you have a NaN-cascade in waiting.
  6. GPU utilisation > 60 % on every rank.
  7. Per-rank loss differs by < 5 % — bigger gaps mean data sharding bugs.
  8. EMA loss drifts only slowly — large EMA-loss vs raw-loss gap means too-fast LR.

1.4.3 3.3 NaN — the killer bug

The most common cause is fp16 overflow. Possible loci:

  • Softmax denominator — exponentiated logits > 65k overflow. Cast softmax to fp32 explicitly.
  • Loss scale too high in GradScaler — the scaler should auto-adjust; watch its history.
  • Adam \(\epsilon\) too small — bump to \(1\times 10^{-6}\) for fp16, \(1\times 10^{-8}\) for bf16.
  • Layer-norm divide-by-zero — bf16 helps; ensure \(\epsilon \geq 10^{-5}\).

Switch to bf16 in any 2025+ training run. It has the same exponent range as fp32 and is supported on every GPU since A100. fp16 is a legacy choice you should only use on V100.

If you must keep fp16, instrument:

def hook(module, input, output):
    if torch.is_tensor(output) and not torch.isfinite(output).all():
        print(f"non-finite at {module}; min/max={output.min().item()}/{output.max().item()}")
for name, m in model.named_modules(): m.register_forward_hook(hook)

1.4.4 3.4 Restart strategies

When a run NaNs at step 87 421:

  1. Roll back to the most recent checkpoint (say step 86 000).
  2. Skip the bad batch: advance the dataloader past the offending sample(s). If you don’t track samples per rank, you cannot do this — another reason to log sampler state.
  3. Lower LR by 2× temporarily and resume; usually fine after 1k steps.
  4. Clip more aggressively: e.g., reduce max_norm from 1.0 to 0.5.
  5. If it NaNs again at the same step: investigate the data. Often a corrupted JPEG, a single sample with insane CLIP score, or a near-empty text.

1.4.5 3.5 Loss spikes — sometimes you just wait

Transformers, especially LLM backbones at >1 B params, exhibit recoverable loss spikes during pre-training (Chowdhery et al., PaLM, 2022). The principled response is:

  • Don’t restart unless the spike is > 4 × the recent EMA loss and doesn’t recover within 1k steps.
  • Log gradient skew (kurtosis) — large skew predicts spikes.
  • Add embed-norm regularisation (Dehghani et al., ViT-22B) for huge models.

1.4.6 3.6 Throughput regressions

Train throughput halving overnight is usually:

  1. Storage filling up — checkpoint retention failed; SSD at 99 %.
  2. Stragglers — one rank slow because of a hot CPU or NIC.
  3. Power throttling — GPU drops from 700 W to 400 W when DC AC fails.
  4. NCCL congestion — a noisy neighbour on the IB switch.

Always log time_per_step per rank and alert on a 30 % deviation.

1.4.7 3.7 Data pipeline disasters

A subtle but common bug: a single corrupt shard that gets sampled by every rank eventually. Symptoms:

  • All ranks NaN around the same global step.
  • Replaying from checkpoint reproduces the NaN at the same step.

Fix: hash every sample, fingerprint shards, and skip on read errors:

import io, hashlib
def safe_decode(jpeg_bytes):
    try:
        img = Image.open(io.BytesIO(jpeg_bytes)).convert("RGB")
        img.load()
        return img
    except Exception:
        return None

1.4.8 3.8 Reproducibility for debugging

When investigating a failure, set torch.use_deterministic_algorithms(True), fixed seeds, and run on a single GPU. The aim isn’t reproducible production — it’s a reliable repro of the bug.

1.4.9 3.9 Logging is your time machine

You cannot diagnose what you didn’t log. Mandatory logs per step:

  • step, epoch, sample count, wall time, throughput.
  • raw loss, EMA loss, validation loss every \(V\) steps.
  • gradient norm (pre-clip), parameter norm.
  • LR per group, optimizer step count.
  • GPU memory, util, power.
  • on-rank zero only: image-grids of a fixed sample, a generated caption, attention map.

Use Weights & Biases, MLflow, or plain TensorBoard.

1.4.10 3.10 The “post-mortem” template

Whenever a run crashes, write a one-page post-mortem with:

  1. Symptom — what failed, when, on which rank.
  2. Trigger — root cause if known; hypothesis if not.
  3. Diagnosis steps — what you ran, what you saw.
  4. Resolution — code, config, or hardware change.
  5. Prevention — alert, test, or runbook entry.

A bug fixed and undocumented will recur.

1.5 4. Adapting Pretrained Models

1.5.1 4.1 The five adaptation strategies

Strategy What is updated? Compute Use case
Linear probe A single linear classifier on frozen features tiny Diagnostic, retrieval
Full fine-tune All parameters huge Domain shift, max accuracy
Last-N-layers The top N layers only medium When data is scarce
PEFT (LoRA / Adapters / Prompts) < 1 % of parameters small Personalisation, multi-tenant
Continued pre-training All params, same objective huge New language / domain corpus

For VLMs, the typical adaptation pipeline is:

  • Stage 1: train projector only (alignment).
  • Stage 2: instruction-tune projector + LLM (often LoRA on LLM, projector full).
  • Stage 3: domain-tune on small specialised data (LoRA on a copy of LLM; switchable at inference).

1.5.2 4.2 Catastrophic forgetting

When fine-tuning, the model’s original abilities decay. Mitigation:

  • Lower LR for layers carrying broad knowledge (early ViT layers, base LLM blocks).
  • Mix the original pre-training data into the fine-tune mix.
  • EWC (Kirkpatrick et al., 2017) — penalise changes weighted by Fisher information.
  • LoRA / Adapters — base weights frozen, so original abilities are mathematically preserved.

1.5.3 4.3 Selecting layers to unfreeze

For a ViT encoder, unfreezing the last 4 blocks typically wins; for an LLM, the top 8 blocks or all attention layers. A defensible heuristic:

  • Probe by layer index — train a linear classifier on each layer’s features. The “knee” of the curve identifies where task-specific information starts to dominate.
  • Unfreeze from the knee upward.

1.5.4 4.4 Domain adaptation — medical, legal, satellite, robotics

The recipe:

  1. Continued pre-training on a few B unlabelled images + captions (LM loss).
  2. Instruction tuning on tens of thousands of domain instructions.
  3. Evaluation on domain benchmarks (PathVQA, ChartQA, GeoChat, RoboVQA).

For medical VLMs, look at LLaVA-Med, RadFM, PathChat. For documents, LLaVA-NeXT-Doc, mPLUG-DocOwl, InternVL2-Doc.

def stage1_alignment(vlm, loader, steps=5000):
    vlm.freeze_for_alignment()                  # vision + LLM frozen
    opt = torch.optim.AdamW(vlm.projector.parameters(), lr=1e-3)
    schd = cosine_with_warmup(opt, 500, steps)
    train(vlm, loader, opt, schd, steps)

def stage2_instruction(vlm, loader, steps=20000):

    for p in vlm.vision.parameters(): p.requires_grad = False
    params = list(vlm.projector.parameters()) + [p for p in vlm.llm.parameters() if p.requires_grad]
    opt = torch.optim.AdamW(params, lr=2e-5)
    schd = cosine_with_warmup(opt, 1000, steps)
    train(vlm, loader, opt, schd, steps)

def stage3_domain(vlm, loader, steps=5000):
    # Train a new LoRA adapter named "medical" while keeping stage2 LoRA frozen
    add_lora(vlm.llm, r=8, alpha=16, name="medical")
    opt = torch.optim.AdamW(get_lora_params(vlm.llm, name="medical"), lr=1e-4)
    schd = cosine_with_warmup(opt, 200, steps)

1.5.6 4.6 Evaluating an adapted model

Always run three suites before declaring success:

  • Target benchmark — domain-specific.
  • General VLM benchmarks — MMMU, MMBench, MathVista. Drop > 2 pts = regression.
  • Smoke tests — your own held-out prompts. Sanity-check qualitatively.

A common mistake is to celebrate a domain win that masks a 10-point general drop.

1.6.2 5.2 LoRA — the canonical method (Hu et al., 2021)

For a frozen linear layer \(W \in \mathbb{R}^{d \times k}\), learn a low-rank update:

\[ W' = W + \Delta W, \qquad \Delta W = \tfrac{\alpha}{r} \cdot B A, \quad A \in \mathbb{R}^{r \times k}, \; B \in \mathbb{R}^{d \times r}. \]

Typical \(r \in \{4, 8, 16, 32, 64\}\), \(\alpha = 2r\). At inference, merge: \(W \leftarrow W + \tfrac{\alpha}{r} BA\) — zero overhead.

Pure-PyTorch implementation:

class LoRALinear(nn.Module):
    def __init__(self, base: nn.Linear, r=8, alpha=16, dropout=0.0):
        super().__init__()
        self.base = base
        for p in self.base.parameters(): p.requires_grad = False
        d_in, d_out = base.in_features, base.out_features
        self.A = nn.Parameter(torch.empty(r, d_in))
        self.B = nn.Parameter(torch.zeros(d_out, r))
        nn.init.kaiming_uniform_(self.A, a=math.sqrt(5))   # Kaiming for A
        nn.init.zeros_(self.B)                              # zero for B -> ΔW=0 at init
        self.scale = alpha / r
        self.drop = nn.Dropout(dropout) if dropout > 0 else nn.Identity()

    def forward(self, x):
        return self.base(x) + self.drop(x) @ self.A.T @ self.B.T * self.scale

The standard target modules in a transformer block are the four attention projections (q_proj, k_proj, v_proj, o_proj); for LLaMA-style models add the MLP projections (gate_proj, up_proj, down_proj).

1.6.3 5.3 QLoRA — Dettmers et al., 2023

The breakthrough: quantise the base model to 4-bit NormalFloat (NF4), keep LoRA in bf16, double-quantise the quantisation constants. Memory drops 4× while accuracy is preserved within 0.1 pts.

Why NF4? The empirical distribution of pre-trained LLM weights is roughly Gaussian; NF4 is the quantile-based 4-bit code-book that is optimal for a Gaussian under squared error. Standard INT4 / FP4 codebooks distribute levels uniformly, which is sub-optimal.

QLoRA also introduces paged optimisers — Adam states overflow into CPU memory in pages, eliminating the OOM that used to plague long-context fine-tuning.

1.6.4 5.4 DoRA (Weight-Decomposed Low-Rank Adaptation, Liu et al., 2024)

DoRA decomposes the weight matrix into magnitude and direction:

\[ W = m \cdot \frac{V}{\|V\|_c}, \qquad m \in \mathbb{R}^{1 \times k}, \; V \in \mathbb{R}^{d \times k}. \]

LoRA updates direction; DoRA additionally learns the per-column magnitude \(m\). Typically +0.5 to +1.5 points over LoRA at similar parameter count. Supported in HuggingFace PEFT: LoraConfig(use_dora=True).

1.6.5 5.5 Adapters (Houlsby et al., 2019 — Pfeiffer et al., 2020)

A bottleneck MLP inserted after each attention or FFN block:

\[ h \leftarrow h + W_{\text{up}} \cdot \sigma(W_{\text{down}} \cdot h), \quad W_{\text{down}} \in \mathbb{R}^{r \times d}, W_{\text{up}} \in \mathbb{R}^{d \times r}. \]

The Pfeiffer variant keeps only one adapter per block (after FFN); Houlsby has two (after attention and FFN). Adapters are slightly slower than LoRA at inference because they’re not mergeable; but they handle tasks where LoRA’s low-rank assumption is too restrictive (multi-task switching).

1.6.6 5.6 (IA)\(^3\) — Liu et al., 2022

Three learnable vectors per block: \(l_k, l_v, l_{ff}\). They re-scale keys, values, and FFN activations:

\[ \text{Attn}(Q, K, V) = \text{softmax}\!\left(\frac{Q (l_k \odot K)^\top}{\sqrt d}\right)(l_v \odot V), \quad \text{FFN}(x) = W_2(l_{ff} \odot \text{GELU}(W_1 x)). \]

Often 100× fewer parameters than LoRA. Competitive on many benchmarks.

1.6.7 5.7 Prefix and prompt tuning

Prompt tuning (Lester et al., 2021): prepend \(K\) learnable token embeddings to the input. Only those embeddings train.

Prefix tuning (Li & Liang, 2021): per layer, prepend \(K\) learnable key/value vectors to the attention. More expressive than prompt tuning.

Both work surprisingly well at very large model scales (≥ 11 B); at smaller scales they underperform LoRA.

1.6.8 5.8 PEFT for the visual branch

You can LoRA the ViT too. Target the attention projections plus the patch-embedding conv:

target = ["q_proj","k_proj","v_proj","out_proj","fc1","fc2"]   # CLIP ViT

For best accuracy, also unfreeze the last LayerNorm and the CLS token / positional embeddings — those are tiny and frequently bottleneck adaptation.

1.6.9 5.9 Multi-adapter serving

A production VLM may host dozens of LoRA adapters (one per customer, per language, per domain). S-LoRA and vLLM Multi-LoRA schedule a single base model and swap adapter weights per request. Memory cost is proportional to total adapter size, not to base × adapters.

1.6.10 5.10 PEFT vs full fine-tune — when does it lose?

PEFT generally matches full fine-tune within 0.5–1 pt on most benchmarks. It loses when:

  • The base model lacks the capability being fine-tuned (LoRA can’t add what isn’t there).
  • The domain shift is huge (medical scans vs natural images) — consider continued pre-training first.
  • The rank is too low for the task complexity — raise \(r\) from 8 to 64 or 128.

1.7 6. Developing Loss Functions

1.7.1 6.1 The loss is the soul of the model

Two models with identical architecture but different losses behave entirely differently. A loss expresses what counts as success. For VLMs, success means a combination of contrastive alignment, image-text matching, language generation, masking-prediction, and (sometimes) explicit reward signals. We will walk every loss with formula + PyTorch.

1.7.2 6.2 Cross-entropy (the LM loss)

Given a sequence \(w_1, \ldots, w_T\) and conditional images \(\mathbf{V}\):

\[ \mathcal{L}_{\text{LM}} = -\sum_{t=1}^T \log p_\theta(w_t \mid w_{<t}, \mathbf{V}). \]

Implementation detail: the LM loss is only computed over assistant tokens, never over system/user instruction tokens. Set labels[mask_user_tokens] = -100 so PyTorch’s CrossEntropyLoss(ignore_index=-100) skips them.

def shift_and_mask(input_ids, labels):
    # next-token prediction
    return input_ids[:, :-1], labels[:, 1:]

logits = llm(input_ids).logits
shift_logits, shift_labels = shift_and_mask(logits, labels)
loss = F.cross_entropy(shift_logits.reshape(-1, shift_logits.size(-1)),
                       shift_labels.reshape(-1), ignore_index=-100)

1.7.3 6.3 Contrastive InfoNCE / CLIP loss

Already introduced in the Architecture doc. The PyTorch one-liner:

def clip_loss(image_features, text_features, logit_scale):
    image_features = F.normalize(image_features, dim=-1)
    text_features  = F.normalize(text_features,  dim=-1)
    logits_per_image = logit_scale.exp() * image_features @ text_features.T
    labels = torch.arange(image_features.size(0), device=image_features.device)
    return 0.5 * (F.cross_entropy(logits_per_image, labels) +
                  F.cross_entropy(logits_per_image.T, labels))

The temperature \(\tau = 1/\text{logit\_scale}\). Clip it to [0.01, 100] to avoid blow-ups:

with torch.no_grad():
    logit_scale.clamp_(0, math.log(100))

1.7.4 6.4 SigLIP — pairwise sigmoid

\[ \mathcal{L}_{\text{SigLIP}} = -\frac{1}{B}\sum_{i,j} \log \sigma\!\big( z_{ij}(\tau s_{ij} + b) \big). \]

PyTorch:

def siglip_loss(img, txt, logit_scale, logit_bias):
    img = F.normalize(img, dim=-1); txt = F.normalize(txt, dim=-1)
    logits = logit_scale.exp() * img @ txt.T + logit_bias
    labels = 2 * torch.eye(img.size(0), device=img.device) - 1   # +1 on diag, -1 off-diag
    return -F.logsigmoid(labels * logits).mean()

1.7.5 6.5 Image-Text Matching (ITM)

A binary classifier on the fused [CLS] token:

def itm_loss(model, img, txt, hard_negatives_idx):
    pos = model.fuse(img, txt).cls       # positive pair
    neg = model.fuse(img, txt[hard_negatives_idx]).cls
    logits = model.itm_head(torch.cat([pos, neg], dim=0))   # [2B, 2]
    targets = torch.cat([torch.ones(pos.size(0)),
                         torch.zeros(neg.size(0))]).long().to(logits.device)
    return F.cross_entropy(logits, targets)

Hard negatives are sampled with weights proportional to current cosine similarity — see ALBEF eq. (10).

1.7.6 6.6 Masked image modelling losses

MAE (He et al., 2022) reconstructs masked patches in pixel space:

\[ \mathcal{L}_{\text{MAE}} = \frac{1}{|\mathcal{M}|} \sum_{i \in \mathcal{M}} \| p_\theta(x_i) - x_i \|_2^2 \]

with per-patch L2 on the normalised target patches (mean-std per patch).

BEiT reconstructs discrete visual tokens from a frozen dVAE:

\[ \mathcal{L}_{\text{BEiT}} = \mathbb{E}_{x, \mathcal{M}}\sum_{i\in\mathcal{M}} -\log p_\theta(z_i \mid x_{\mathcal{M}^c}) \]

where \(z_i\) are the dVAE’s tokens.

EVA (Fang et al., 2023) predicts CLIP features at masked positions — feature distillation + masked modelling.

1.7.7 6.7 Captioning + LM mixture

Multi-task training combines several losses:

\[ \mathcal{L} = \lambda_1 \mathcal{L}_{\text{ITC}} + \lambda_2 \mathcal{L}_{\text{ITM}} + \lambda_3 \mathcal{L}_{\text{LM}} + \lambda_4 \mathcal{L}_{\text{MIM}}. \]

Default weights: \(1, 1, 1, 1\) for BLIP-2 representation stage; \(0, 0, 1, 0\) for BLIP-2 generative stage; varied for FLAVA / BEiT-3.

1.7.8 6.8 DPO / KTO (preference losses)

For RLHF-style alignment of VLMs (LLaVA-RLHF, RLAIF-V):

\[ \mathcal{L}_{\text{DPO}} = -\mathbb{E}_{(x, y_w, y_l)} \log \sigma\!\Big(\beta \log \frac{\pi_\theta(y_w|x)}{\pi_{\text{ref}}(y_w|x)} - \beta \log \frac{\pi_\theta(y_l|x)}{\pi_{\text{ref}}(y_l|x)} \Big). \]

where \((y_w, y_l)\) are chosen/rejected responses, \(\pi_{\text{ref}}\) is the frozen reference model, \(\beta \approx 0.1\). KTO (Ethayarajh et al., 2024) variant works with unpaired labels.

def dpo_loss(pi_logp_chosen, pi_logp_rejected, ref_logp_chosen, ref_logp_rejected, beta=0.1):
    pi_diff  = pi_logp_chosen  - pi_logp_rejected
    ref_diff = ref_logp_chosen - ref_logp_rejected
    return -F.logsigmoid(beta * (pi_diff - ref_diff)).mean()

1.7.9 6.9 Auxiliary losses for stability

  • Z-loss (Chowdhery et al., PaLM): penalise the log-sum-exp of the LM head logits — \(\lambda_z \cdot \log^2 Z\) where \(Z = \sum \exp(\text{logits})\). Keeps logits well-scaled, prevents NaN.
  • Router-z loss for MoE (DeepSeek-MoE): balances expert load.
  • Sparsity / contrastive auxiliary for image-text retrieval to encourage uniformity.

1.7.10 6.10 Designing a loss — a checklist

  1. Is the loss bounded below? If not, expect divergence.
  2. Are gradients well-scaled when inputs are large? Add log-sum-exp tricks.
  3. Does the loss reduce to a known good loss when constants degenerate? Sanity.
  4. Is the loss invariant to your symmetry? (translation, scale, etc.) If yes, good — if not, your data needs more augmentation.
  5. Does it scale linearly with batch size, or quadratically? Contrastive scales \(O(B^2)\); ITM scales \(O(B)\).
  6. What does ChatGPT-as-a-judge think of the model trained on this loss? A surprising sanity check.

1.7.11 6.11 Implementing a custom loss in PyTorch

class VLMLoss(nn.Module):
    def __init__(self, lambda_itc=1.0, lambda_itm=1.0, lambda_lm=1.0):
        super().__init__()
        self.lambda_itc = lambda_itc
        self.lambda_itm = lambda_itm
        self.lambda_lm  = lambda_lm
        self.itm_head = nn.Linear(768, 2)

    def forward(self, vision_emb, text_emb, fused_cls, lm_logits, labels, hard_neg_idx, logit_scale):
        l_itc = clip_loss(vision_emb, text_emb, logit_scale)
        l_itm = F.cross_entropy(self.itm_head(fused_cls), labels)
        l_lm  = F.cross_entropy(lm_logits.reshape(-1, lm_logits.size(-1)),
                                labels.reshape(-1), ignore_index=-100)
        return self.lambda_itc * l_itc + self.lambda_itm * l_itm + self.lambda_lm * l_lm, {
            "itc": l_itc.detach(), "itm": l_itm.detach(), "lm": l_lm.detach(),
        }

1.8.2 7.2 The basic math

Symmetric uniform quantisation of a tensor \(x\) to \(b\)-bit signed integers:

\[ s = \frac{\max(|x|)}{2^{b-1} - 1}, \quad q = \mathrm{clip}\!\left(\mathrm{round}(x/s),\; -2^{b-1},\; 2^{b-1}-1\right), \quad \hat{x} = s \cdot q. \]

Asymmetric:

\[ s = \frac{x_{\max} - x_{\min}}{2^b - 1}, \quad z = \mathrm{round}(-x_{\min}/s), \quad q = \mathrm{clip}(\mathrm{round}(x/s) + z, 0, 2^b - 1). \]

Quantisation error \(\| x - \hat x \|\) is essentially uniform with variance \(s^2/12\) → smaller \(s\) means smaller error → group quantisation (one \(s\) per small group) reduces error dramatically over per-tensor.

1.8.3 7.3 Outliers — why naive INT8 fails

Dettmers (LLM.int8(), 2022) discovered: in transformers > 6.7 B parameters, a small number of activation channels have magnitudes 50–500× larger than the rest. A per-tensor scale chosen for the outliers would round the typical channels to zero. Solutions:

  • LLM.int8(): mixed precision — outlier channels in fp16, the rest in int8.
  • SmoothQuant (Xiao et al., 2023): migrate the difficulty from activations to weights: \[ Y = X \cdot W = (X \cdot \mathrm{diag}(s)^{-1})(\mathrm{diag}(s) \cdot W). \] Pick \(s_j = \max(|X_{:,j}|)^\alpha / \max(|W_{j,:}|)^{1-\alpha}\) with \(\alpha \approx 0.5\). Now both sides are well-conditioned.
  • AWQ (Lin et al., 2023): identify the 1 % “salient” weight channels by activation magnitude and scale them up before INT4 rounding — preserves their representation. Pure weight-only.

1.8.4 7.4 GPTQ — Frantar et al., 2022

GPTQ uses second-order information: layer-wise, it picks weights to quantise one at a time, updating the remaining weights to compensate, using the OBS (Optimal Brain Surgeon) approximation with a small calibration set (128 examples). Produces 3–4 bit per-channel quantised matrices with < 1 % perplexity loss on most LLMs.

QAT inserts fake-quant nodes (FakeQuantize) into the model and trains end-to-end with the quantisation simulation in the loss path. Used when post-training methods don’t recover enough accuracy.

For inference, the KV cache often dwarfs the model weights. Quantising it to fp8 or int8 can halve memory without measurable accuracy loss:

               cache_config={"backend": "HQQ", "nbits": 4, "axis_key": -1, "axis_value": -1})

vLLM, TensorRT-LLM, and HuggingFace all support fp8 KV cache.

1.8.8 7.8 FP8 — H100 / B200 native

FP8 (e4m3 for activations, e5m2 for gradients) is hardware-accelerated on H100. Use the Transformer Engine library:

Speed-ups: 1.5–2× over bf16 with no accuracy loss in most LLM pre-training runs.

1.8.9 7.9 Calibration data choice

The calibration dataset must mirror inference traffic:

  • For a chat VLM, calibrate with chat-formatted multimodal data.
  • For a video VLM, calibrate with video clips.
  • For a multilingual VLM, calibrate with samples from every language.

A 128–512 sample calibration set is usually enough; > 2000 brings diminishing returns and risks overfitting the quantisation parameters.

1.8.10 7.10 Validating a quantised model

Three layers of validation:

  1. Per-layer activation MSE vs reference — flag layers > 1 % MSE.
  2. End-to-end perplexity / loss on a held-out set.
  3. Downstream benchmarks (MMMU, MMBench). Anything worse than 1 % below the bf16 reference is a regression.

The dangerous failure: end-to-end perplexity looks fine but one specific capability (OCR? Chinese?) is broken. Always evaluate multiple axes.

1.9 8. Building Knowledge Distillation Systems

1.9.1 8.1 Why distillation matters

A 7 B-parameter teacher VLM running at 25 tok/s on an A100 is unusable on a phone. Knowledge distillation transfers the teacher’s competence into a 0.5–2 B-parameter student that runs at 50 tok/s on a Snapdragon 8 Gen 3.

1.9.2 8.2 Three flavours of distillation

Flavour What is transferred Loss
Logit distillation (Hinton) Soft probabilities over vocab KL divergence on softmaxed logits with temperature \(T\)
Feature distillation Intermediate hidden states MSE on student-projected features
Attention transfer Attention maps MSE / KL on attention weights

For VLMs you typically combine all three.

1.9.3 8.3 The Hinton 2015 loss

\[ \mathcal{L}_{\text{KD}} = \alpha \cdot T^2 \cdot \mathrm{KL}\!\big( p_\text{student}^{(T)} \,\|\, p_\text{teacher}^{(T)} \big) + (1-\alpha) \cdot \mathcal{L}_{\text{CE}}(p_\text{student}, y) \]

where \(p^{(T)}_i = \mathrm{softmax}(z_i / T)\) and \(T \in [1, 10]\) smooths the teacher’s distribution to reveal “dark knowledge” — relative probabilities among non-target classes that carry information beyond a one-hot label.

def kd_loss(student_logits, teacher_logits, labels, T=4.0, alpha=0.5):
    soft_s = F.log_softmax(student_logits / T, dim=-1)
    soft_t = F.softmax(teacher_logits / T, dim=-1)
    l_kl = F.kl_div(soft_s, soft_t, reduction="batchmean") * (T * T)
    l_ce = F.cross_entropy(student_logits, labels)
    return alpha * l_kl + (1 - alpha) * l_ce

For LLM distillation on a sequence:

def seq_kd(student_logits, teacher_logits, labels, T=2.0, alpha=0.7, ignore=-100):
    s = student_logits[:, :-1].reshape(-1, student_logits.size(-1))
    t = teacher_logits[:, :-1].reshape(-1, teacher_logits.size(-1))
    y = labels[:, 1:].reshape(-1)
    mask = (y != ignore)
    soft_s = F.log_softmax(s[mask] / T, dim=-1)
    soft_t = F.softmax(t[mask] / T, dim=-1)
    l_kl = F.kl_div(soft_s, soft_t, reduction="batchmean") * (T * T)
    l_ce = F.cross_entropy(s, y, ignore_index=ignore)
    return alpha * l_kl + (1 - alpha) * l_ce

1.9.4 8.4 Feature distillation

Match the student’s hidden state at layer \(l_s\) to the teacher’s at layer \(l_t\) via a learned projector \(g\):

\[ \mathcal{L}_{\text{feat}} = \frac{1}{N} \sum_i \| g(h^{(l_s)}_{\text{stu},i}) - h^{(l_t)}_{\text{tea},i} \|_2^2. \]

For VLM vision distillation, this is essentially what EVA does (CLIP teacher → masked-prediction student).

1.9.5 8.5 Attention transfer

Following Zagoruyko & Komodakis (2017), match a normalised attention map:

\[ \mathcal{L}_{\text{att}} = \sum_l \left\| \frac{A_{\text{stu}}^{(l)}}{\|A_{\text{stu}}^{(l)}\|_F} - \frac{A_{\text{tea}}^{(l)}}{\|A_{\text{tea}}^{(l)}\|_F} \right\|_F^2. \]

For transformers, attention maps have shape [B, H, T, T]; align heads either via a Hungarian assignment or by pooling over heads.

1.9.6 8.6 Reverse KL and MiniLLM

Hinton’s KL is \(\mathrm{KL}(p_\text{stu} \| p_\text{tea})\) — forward KL. Reverse KL \(\mathrm{KL}(p_\text{tea} \| p_\text{stu})\) is mode-seeking: the student focuses on the highest-probability modes of the teacher and ignores low-probability “long tails”. For long-form generation, reverse KL produces more decisive students. MiniLLM (Gu et al., 2023) implements policy-gradient optimisation of reverse KL for generative distillation.

1.9.7 8.7 On-policy vs off-policy distillation

  • Off-policy: teacher generates targets, student trains on them — fast but the distribution mismatch (teacher and student generate different sequences) hurts.
  • On-policy: at each step, student generates a token, teacher predicts the distribution at that exact context, loss matches them. Slower but better.

A typical recipe: off-policy for the first 10 k steps to bootstrap, then on-policy.

1.9.8 8.8 Distilling a VLM — a complete recipe

class DistillVLM(nn.Module):
    def __init__(self, teacher, student, T=2.0, alpha=0.7, feature_layers=(8, 16)):
        super().__init__()
        self.teacher = teacher.eval()
        for p in self.teacher.parameters(): p.requires_grad = False
        self.student = student
        self.T, self.alpha = T, alpha
        self.feat_proj = nn.ModuleList([
            nn.Linear(student.cfg.d, teacher.cfg.d) for _ in feature_layers
        ])
        self.feat_layers = feature_layers

    def forward(self, pixel_values, input_ids, labels):
        with torch.no_grad():
            t_out = self.teacher(pixel_values=pixel_values, input_ids=input_ids,
                                 output_hidden_states=True)
        s_out = self.student(pixel_values=pixel_values, input_ids=input_ids,
                             output_hidden_states=True)
        l_kd = seq_kd(s_out.logits, t_out.logits, labels, self.T, self.alpha)
        l_feat = 0
        for proj, l in zip(self.feat_proj, self.feat_layers):
            l_feat = l_feat + F.mse_loss(proj(s_out.hidden_states[l]),
                                         t_out.hidden_states[l])
        return l_kd + 0.1 * l_feat

1.9.9 8.9 Data — the secret sauce

The distillation data matters at least as much as the loss. Best practice:

  • Teacher-generated captions for raw images (LLaVA-1.5’s “CapFilt” trick).
  • Diverse instruction mix: VQA, OCR, charts, math, reasoning, dialog.
  • Self-distillation (student becomes the teacher of an even smaller student).

VLM-specific distillation papers: MiniLM-V (Wang et al., 2024), TinyLLaVA (2024), LLaVA-PoolDist, MobileVLM (Chu et al., 2023).

1.9.10 8.10 Evaluating a distilled model

  • Match teacher’s behaviour on a held-out agreement set (% same prediction).
  • Match teacher’s quality on standard benchmarks — gap should be < 5 pts for a 2× compression.
  • Verify what was lost — usually long-tail knowledge or rare languages. Re-fine-tune on those if necessary.

1.9.11 8.11 A unified loss for VLM distillation

\[ \mathcal{L} = \alpha \cdot \mathcal{L}_{\text{KD-logit}}^{T} + \beta \cdot \mathcal{L}_{\text{feat}} + \gamma \cdot \mathcal{L}_{\text{att}} + (1-\alpha-\beta-\gamma) \cdot \mathcal{L}_{\text{LM}} \]

with \(\alpha=0.5,\, \beta=0.1,\, \gamma=0.05\) as a sensible starting point. Anneal \(\beta, \gamma \to 0\) over training so the student isn’t forever tied to the teacher’s intermediate representations.

## 1. Managing Checkpoint Strategies

### 1.1 Why checkpointing is harder than `torch.save`

A naïve `torch.save(model.state_dict(), "ckpt.pt")` works for a 100 M model on one GPU. For a 70 B model on 64 GPUs it (a) takes 8 minutes, (b) requires rank-0 to hold the whole model in CPU RAM, (c) corrupts on crash, and (d) leaves your optimizer states orphaned across half the cluster. A real checkpoint strategy answers six questions:

1. **What** do we save? (weights, optimizer, LR scheduler, RNG state, dataloader cursor, scaler, EMA copy)
2. **When** do we save? (every N steps, every K minutes, best validation metric, at exit)
3. **Where** do we save? (local SSD, network FS, object store, sharded vs single file)
4. **How** do we save? (sync vs async, sharded vs gathered, atomic vs partial)
5. **How** do we resume? (bit-exact vs warm-start; same world size vs reshape)
6. **How** do we retire? (rolling window, by metric, manual pinning)

Get any of these wrong and a week of training disappears in seconds.

### 1.2 What is in a complete checkpoint

```python
state = {
    "model":          model.state_dict(),       # weights
    "optimizer":      optimizer.state_dict(),   # Adam m, v, step
    "scheduler":      scheduler.state_dict(),   # current step
    "scaler":         scaler.state_dict(),      # if using GradScaler
    "ema":            ema.state_dict(),         # exponential moving average
    "step":           global_step,
    "epoch":          epoch,
    "rng": {
        "python":      random.getstate(),
        "numpy":       np.random.get_state(),
        "torch":       torch.get_rng_state(),
        "cuda":        torch.cuda.get_rng_state_all(),
    },
    "sampler":        sampler.state_dict() if hasattr(sampler, "state_dict") else None,
    "config":         dataclasses.asdict(config),
    "git_sha":        get_git_sha(),
    "version":        "ckpt_v3",
}
```

Forgetting *any* of these creates a subtle resume bug. Forgetting the RNG state means a different shuffle order; forgetting the sampler state means re-doing seen samples; forgetting `scaler` means a different fp16 loss-scale.

### 1.3 Distributed Checkpoint (DCP) — the modern API

PyTorch 2.4+ ships `torch.distributed.checkpoint` (DCP). Every rank writes its own shard in parallel; resume gathers them back. Sharded checkpoints survive **world-size changes** (resharding-on-load) — invaluable for cluster failures.

```python
import torch.distributed.checkpoint as dcp
from torch.distributed.checkpoint.state_dict import get_state_dict, set_state_dict

state_dict, optim_state = get_state_dict(model, optimizer)
ckpt = {"model": state_dict, "optim": optim_state, "step": step}
dcp.save(ckpt, checkpoint_id=f"s3://bucket/ckpts/step_{step}")
```

Resume:

```python
state_dict, optim_state = get_state_dict(model, optimizer)
ckpt = {"model": state_dict, "optim": optim_state, "step": 0}
dcp.load(ckpt, checkpoint_id="s3://bucket/ckpts/step_5000")
set_state_dict(model, optimizer, model_state_dict=ckpt["model"], optim_state_dict=ckpt["optim"])
```

DCP integrates with FSDP, TP, PP. Don't roll your own.

### 1.4 Async checkpointing

A 30-second `dcp.save` once per 1000 steps wastes 3 % of training time. **Asynchronously** snapshot the state in pinned host memory, then offload the write to a background thread:

```python
from torch.distributed.checkpoint.state_dict_saver import async_save

future = async_save(ckpt, checkpoint_id=f"...{step}")   # returns immediately
# training continues
future.result()                                          # wait only at exit / next save
```

This makes checkpointing essentially free for any save period > 200 steps.

### 1.5 Atomicity

The checkpoint must be readable after a crash. Two strategies:

- **Write to a temp directory, then rename**: `dcp.save(..., "ckpt_TMP_{step}")` then `os.rename` to `ckpt_{step}`. POSIX rename is atomic.
- **Two-phase commit**: write all shards then a `DONE` marker; resume only loads if `DONE` exists.

### 1.6 Retention policy

Disk fills fast. A sensible default for a 7-day run:

- Keep the last $K=3$ recent checkpoints (rolling).
- Keep the **best** by validation metric (pinned).
- Keep one per epoch (sparse history).

```python
def retain_recent(folder, k=3):
    ckpts = sorted(p for p in os.listdir(folder) if p.startswith("step_"),
                   key=lambda p: int(p.split("_")[1]))
    for old in ckpts[:-k]:
        shutil.rmtree(os.path.join(folder, old))
```

### 1.7 Exponential Moving Average (EMA)

EMA produces a *parameter average* that is usually 1–3 points better than the raw model on validation. The update rule, applied every step:

$$
\theta^{\text{EMA}}_t = \beta \cdot \theta^{\text{EMA}}_{t-1} + (1-\beta) \cdot \theta_t, \qquad \beta \in [0.999, 0.9999].
$$

A common variant warms up $\beta$ from 0 over the first few thousand steps. Save *both* the live and EMA states in your checkpoint; serve from EMA.

```python
class EMA:
    def __init__(self, model, decay=0.9999):
        self.decay = decay
        self.shadow = {n: p.detach().clone() for n, p in model.named_parameters() if p.requires_grad}

    @torch.no_grad()
    def update(self, model):
        for n, p in model.named_parameters():
            if p.requires_grad:
                self.shadow[n].mul_(self.decay).add_(p.detach(), alpha=1 - self.decay)

    def state_dict(self):  return self.shadow
    def load_state_dict(self, sd): self.shadow = {k: v.clone() for k, v in sd.items()}
```

### 1.8 Best-metric checkpointing

```python
best_metric = float("-inf")
def maybe_save_best(val_metric, step):
    nonlocal best_metric
    if val_metric > best_metric:
        best_metric = val_metric
        dcp.save(ckpt, checkpoint_id=f"{folder}/best")
```

### 1.9 Object-store backends

For very large runs, save directly to S3 / GCS / Azure Blob. `dcp.save` accepts a custom `FsspecWriter`:

```python
from torch.distributed.checkpoint.fsspec import FsspecWriter
dcp.save(ckpt, storage_writer=FsspecWriter("s3://bucket/ckpts/step_5000"))
```

For Samsung / on-prem clusters, **Lustre** or **WekaIO** is typical; DCP works against any POSIX FS.

### 1.10 The cheat-sheet

| Decision | Default | Reason |
|---|---|---|
| Save period | 250 steps async, 2000 steps sync | overlap vs reliability |
| Format | DCP sharded | resharding & speed |
| Retention | last 3 + best + per-epoch | disk hygiene |
| EMA | β = 0.9999 from step 5000 | + ~1–3 pts |

## 2. Tuning Learning Rate Schedules

### 2.1 Why the schedule matters more than the optimizer

For transformers, learning-rate (LR) schedule and warmup are *single most-tuned hyperparameters*. The right schedule recovers 1–3 points; the wrong one diverges. The four pillars:

1. **Initial LR** — base learning rate at peak.
2. **Warmup** — linear ramp from 0 to the peak.
3. **Decay** — cosine, linear, inverse-square-root, or constant.
4. **Final LR** — value at the end of training (often 10 % of peak).

### 2.2 Warmup — why it exists

In the first few thousand steps, gradients are noisy and weights are far from a good solution. A large LR amplifies noise; a small LR makes effective progress impossible. Warmup linearly increases LR from 0 to peak over the first $W$ steps:

$$
\eta_t = \eta_{\max} \cdot \min(1, t/W).
$$

Typical $W = 500$ to 5000 steps; longer for tiny batch sizes; can be in *number of samples* rather than steps for stability across hardware.

### 2.3 The big four schedules

#### (a) Cosine

$$
\eta_t = \eta_{\min} + \tfrac{1}{2}(\eta_{\max} - \eta_{\min})\left(1 + \cos\!\left(\pi \cdot \frac{t-W}{T-W}\right)\right).
$$

The default for LLMs, ViTs, CLIP, BLIP-2, LLaVA. Smooth and forgiving.

#### (b) Linear

$$
\eta_t = \eta_{\max} \cdot \left(1 - \frac{t-W}{T-W}\right).
$$

Faster decay early on; used by RoBERTa, T5.

#### (c) Inverse square root

$$
\eta_t = \eta_{\max} \cdot \sqrt{W/t}\quad (t > W).
$$

The original Transformer schedule (Vaswani et al., 2017). Allows training to continue indefinitely; useful for never-ending pre-training.

#### (d) Constant with cooldown (D-Adaptation / Schedule-Free)

Keep LR constant during the bulk of training, then cool down at the end. Defazio & Mishchenko's *Schedule-Free* (2024) eliminates the schedule entirely with a momentum-style rule:

$$
\theta_t \leftarrow \theta_{t-1} - \eta \nabla L(\bar{\theta}_t), \quad \bar{\theta}_t = (1-c_t)\bar{\theta}_{t-1} + c_t \theta_t.
$$

Empirically matches cosine without needing to specify $T$.

### 2.4 PyTorch implementations

```python
from torch.optim.lr_scheduler import LambdaLR
import math

def cosine_with_warmup(optimizer, num_warmup, num_total, min_ratio=0.1):
    def lr_lambda(step):
        if step < num_warmup:
            return step / max(1, num_warmup)
        progress = (step - num_warmup) / max(1, num_total - num_warmup)
        return max(min_ratio, 0.5 * (1 + math.cos(math.pi * progress)))
    return LambdaLR(optimizer, lr_lambda)
```

HuggingFace `transformers.get_cosine_schedule_with_warmup` does the same; `optimum` exposes more exotic schedules.

### 2.5 Per-layer / discriminative LR

VLMs often fine-tune a **frozen ViT** at $10^{-6}$, a **fresh projector** at $10^{-3}$, and an **LLM with LoRA** at $10^{-4}$. Build parameter groups:

```python
optimizer = torch.optim.AdamW(
    [
        {"params": vision.parameters(),  "lr": 1e-6},
        {"params": projector.parameters(), "lr": 1e-3},
        {"params": [p for n,p in llm.named_parameters() if "lora_" in n], "lr": 1e-4},
    ],
    betas=(0.9, 0.95), weight_decay=0.05, eps=1e-8,
)
```

ULMFit (Howard & Ruder, 2018) introduced **discriminative fine-tuning** — lower LR for earlier layers, higher for later. For ViTs/LLMs this is implemented as a **layer-wise LR decay** factor $\gamma$ (commonly 0.65–0.85):

$$
\eta_{\ell} = \eta_{\max} \cdot \gamma^{L - \ell}
$$

where $\ell$ is the layer index and $L$ is the total layers.

### 2.6 Peak LR rules of thumb

| Setting | Suggested peak LR |
|---|---|
| ViT pre-training from scratch | $1\times 10^{-3}$ (AdamW, β2=0.95) |
| CLIP-style contrastive | $5\times 10^{-4}$ (linear) |
| LLM pre-training (decoder, 7 B) | $3\times 10^{-4}$ |
| LLM continued pre-training | $1\times 10^{-4}$ |
| VLM Stage-1 (projector only) | $1\times 10^{-3}$ |
| VLM Stage-2 (full instruction tune) | $2\times 10^{-5}$ |
| LoRA on a 7 B LLM | $1\times 10^{-4}$ to $5\times 10^{-4}$ |
| QLoRA | $2\times 10^{-4}$ |

These are *starting points*. A 30-minute LR-finder sweep (Smith, 2017) almost always pays for itself.

### 2.7 Batch size and LR — the linear / sqrt rules

For SGD-style optimisers, the **linear-scaling rule** (Goyal et al., 2017): when batch size goes $k \times$, multiply LR by $k$ and use linear warmup. For AdamW, the rule is **sub-linear** (often $\sqrt{k}$ or even less). For contrastive losses (CLIP, SigLIP) where larger batch directly improves the objective, the optimal LR rises slower than $\sqrt{k}$.

### 2.8 Schedule-Free and Lion

Two newer optimisers worth knowing:

- **Lion** (Chen et al., 2023) — uses only the sign of the momentum; needs LR ≈ 1/10 of AdamW; memory-efficient.
- **Schedule-Free AdamW** — eliminates the LR schedule; benchmark-competitive.

Both have first-class PyTorch implementations (`from lion_pytorch import Lion`, `from schedulefree import AdamWScheduleFree`).

### 2.9 Gradient clipping

Always clip gradients during VLM training:

```python
torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
```

The 1.0 norm is standard for LLMs; tighter (0.5) for ViTs pre-training; looser (5.0) for SGD-from-scratch. Clipping prevents the rare spike from blowing up Adam's moments.

### 2.10 Diagnosing a misbehaving schedule

- **Loss plateaus immediately**: warmup too short or LR too low.
- **Loss spikes after warmup**: peak LR too high; reduce or extend warmup.
- **Loss diverges late**: weight decay too low; learning rate too high.
- **Best validation arrives early**: training too long; reduce $T$.

## 3. Handling Pre-training Failures

### 3.1 The taxonomy of failures

| Symptom | Likely cause |
|---|---|
| **Loss = NaN** suddenly | fp16 overflow; exploding gradient; bad sample |
| **Loss diverges** monotonically | LR too high; weight init bad; missing LN |
| **Loss plateaus at high value** | LR too low; loss bug; label mismatch |
| **Loss spikes then recovers** | bad batch; well-behaved transformer thing |
| **Grad norm explodes** | dead ReLU / very deep block; precision issue |
| **GPU OOM partway** | dataloader memory leak; KV cache overflow |
| **Throughput collapses** | filesystem hot-spot; sync stragglers |
| **Validation diverges from train** | overfitting; eval data drift |

### 3.2 The first 48 hours of pre-training — a checklist

Most failures appear in the first day. Hit each item:

1. **Verify the first batch**: shapes, dtypes, image normalisation, tokens make sense; loss for one sample is reasonable.
2. **Overfit a tiny subset** (e.g., 64 samples) — loss should hit ~0 in a few hundred steps. If it doesn't, your model is broken before scaling matters.
3. **Initial loss == log(vocab)** for an untrained LM (e.g., ≈ 10.5 for 32k vocab).
4. **Warmup curve is smooth** (no sudden drops or spikes).
5. **Grad-norm is in [0.1, 10]** range; if it is $10^4$, you have a NaN-cascade in waiting.
6. **GPU utilisation > 60 %** on every rank.
7. **Per-rank loss differs by < 5 %** — bigger gaps mean data sharding bugs.
8. **EMA loss drifts only slowly** — large EMA-loss vs raw-loss gap means too-fast LR.

### 3.3 NaN — the killer bug

The most common cause is **fp16 overflow**. Possible loci:

- **Softmax denominator** — exponentiated logits > 65k overflow. Cast softmax to fp32 explicitly.
- **Loss scale too high** in `GradScaler` — the scaler should auto-adjust; watch its history.
- **Adam $\epsilon$** too small — bump to $1\times 10^{-6}$ for fp16, $1\times 10^{-8}$ for bf16.
- **Layer-norm divide-by-zero** — bf16 helps; ensure $\epsilon \geq 10^{-5}$.

**Switch to bf16** in any 2025+ training run. It has the same exponent range as fp32 and is supported on every GPU since A100. fp16 is a legacy choice you should only use on V100.

If you must keep fp16, instrument:

```python
def hook(module, input, output):
    if torch.is_tensor(output) and not torch.isfinite(output).all():
        print(f"non-finite at {module}; min/max={output.min().item()}/{output.max().item()}")
for name, m in model.named_modules(): m.register_forward_hook(hook)
```

### 3.4 Restart strategies

When a run NaNs at step 87 421:

1. **Roll back** to the most recent checkpoint (say step 86 000).
2. **Skip the bad batch**: advance the dataloader past the offending sample(s). If you don't track samples per rank, you cannot do this — another reason to log sampler state.
3. **Lower LR by 2×** temporarily and resume; usually fine after 1k steps.
4. **Clip more aggressively**: e.g., reduce `max_norm` from 1.0 to 0.5.
5. **If it NaNs again at the same step**: investigate the data. Often a corrupted JPEG, a single sample with insane CLIP score, or a near-empty text.

### 3.5 Loss spikes — sometimes you just wait

Transformers, especially LLM backbones at >1 B params, exhibit recoverable **loss spikes** during pre-training (Chowdhery et al., PaLM, 2022). The principled response is:

- Don't restart unless the spike is > 4 × the recent EMA loss *and* doesn't recover within 1k steps.
- Log gradient skew (kurtosis) — large skew predicts spikes.
- Add **embed-norm regularisation** (Dehghani et al., ViT-22B) for huge models.

### 3.6 Throughput regressions

Train throughput halving overnight is usually:

1. **Storage filling up** — checkpoint retention failed; SSD at 99 %.
2. **Stragglers** — one rank slow because of a hot CPU or NIC.
3. **Power throttling** — GPU drops from 700 W to 400 W when DC AC fails.
4. **NCCL congestion** — a noisy neighbour on the IB switch.

Always log `time_per_step` per rank and alert on a 30 % deviation.

### 3.7 Data pipeline disasters

A subtle but common bug: a **single corrupt shard** that gets sampled by every rank eventually. Symptoms:

- All ranks NaN around the same global step.
- Replaying from checkpoint reproduces the NaN at the same step.

Fix: hash every sample, fingerprint shards, and skip on read errors:

```python
import io, hashlib
def safe_decode(jpeg_bytes):
    try:
        img = Image.open(io.BytesIO(jpeg_bytes)).convert("RGB")
        img.load()
        return img
    except Exception:
        return None
```

### 3.8 Reproducibility for debugging

When investigating a failure, set `torch.use_deterministic_algorithms(True)`, fixed seeds, and run on a single GPU. The aim isn't reproducible *production* — it's a reliable repro of the bug.

### 3.9 Logging is your time machine

You cannot diagnose what you didn't log. Mandatory logs per step:

- step, epoch, sample count, wall time, throughput.
- raw loss, EMA loss, validation loss every $V$ steps.
- gradient norm (pre-clip), parameter norm.
- LR per group, optimizer step count.
- GPU memory, util, power.
- on-rank zero only: image-grids of a fixed sample, a generated caption, attention map.

Use **Weights & Biases**, **MLflow**, or plain TensorBoard.

### 3.10 The "post-mortem" template

Whenever a run crashes, write a one-page post-mortem with:

1. **Symptom** — what failed, when, on which rank.
2. **Trigger** — root cause if known; hypothesis if not.
3. **Diagnosis steps** — what you ran, what you saw.
4. **Resolution** — code, config, or hardware change.
5. **Prevention** — alert, test, or runbook entry.

A bug fixed and undocumented will recur.

## 4. Adapting Pretrained Models

### 4.1 The five adaptation strategies

| Strategy | What is updated? | Compute | Use case |
|---|---|---|---|
| **Linear probe** | A single linear classifier on frozen features | tiny | Diagnostic, retrieval |
| **Full fine-tune** | All parameters | huge | Domain shift, max accuracy |
| **Last-N-layers** | The top N layers only | medium | When data is scarce |
| **PEFT (LoRA / Adapters / Prompts)** | < 1 % of parameters | small | Personalisation, multi-tenant |
| **Continued pre-training** | All params, same objective | huge | New language / domain corpus |

For VLMs, the typical adaptation pipeline is:

- **Stage 1**: train projector only (alignment).
- **Stage 2**: instruction-tune projector + LLM (often LoRA on LLM, projector full).
- **Stage 3**: domain-tune on small specialised data (LoRA on a copy of LLM; switchable at inference).

### 4.2 Catastrophic forgetting

When fine-tuning, the model's original abilities decay. Mitigation:

- **Lower LR** for layers carrying broad knowledge (early ViT layers, base LLM blocks).
- **Mix the original pre-training data** into the fine-tune mix.
- **EWC** (Kirkpatrick et al., 2017) — penalise changes weighted by Fisher information.
- **LoRA / Adapters** — base weights frozen, so original abilities are mathematically preserved.

### 4.3 Selecting layers to unfreeze

For a ViT encoder, unfreezing the **last 4 blocks** typically wins; for an LLM, the **top 8 blocks** or **all attention layers**. A defensible heuristic:

- Probe by layer index — train a linear classifier on each layer's features. The "knee" of the curve identifies where task-specific information starts to dominate.
- Unfreeze from the knee upward.

### 4.4 Domain adaptation — medical, legal, satellite, robotics

The recipe:

1. **Continued pre-training** on a few B unlabelled images + captions (LM loss).
2. **Instruction tuning** on tens of thousands of domain instructions.
3. **Evaluation** on domain benchmarks (PathVQA, ChartQA, GeoChat, RoboVQA).

For medical VLMs, look at LLaVA-Med, RadFM, PathChat. For documents, LLaVA-NeXT-Doc, mPLUG-DocOwl, InternVL2-Doc.

### 4.5 Code template — staged fine-tune

```python
def stage1_alignment(vlm, loader, steps=5000):
    vlm.freeze_for_alignment()                  # vision + LLM frozen
    opt = torch.optim.AdamW(vlm.projector.parameters(), lr=1e-3)
    schd = cosine_with_warmup(opt, 500, steps)
    train(vlm, loader, opt, schd, steps)

def stage2_instruction(vlm, loader, steps=20000):
    add_lora(vlm.llm, r=64, alpha=16, target_modules=("q_proj","v_proj"))
    for p in vlm.vision.parameters(): p.requires_grad = False
    params = list(vlm.projector.parameters()) + [p for p in vlm.llm.parameters() if p.requires_grad]
    opt = torch.optim.AdamW(params, lr=2e-5)
    schd = cosine_with_warmup(opt, 1000, steps)
    train(vlm, loader, opt, schd, steps)

def stage3_domain(vlm, loader, steps=5000):
    # Train a new LoRA adapter named "medical" while keeping stage2 LoRA frozen
    add_lora(vlm.llm, r=8, alpha=16, name="medical")
    opt = torch.optim.AdamW(get_lora_params(vlm.llm, name="medical"), lr=1e-4)
    schd = cosine_with_warmup(opt, 200, steps)
    train(vlm, loader, opt, schd, steps)
```

### 4.6 Evaluating an adapted model

Always run **three suites** before declaring success:

- **Target benchmark** — domain-specific.
- **General VLM benchmarks** — MMMU, MMBench, MathVista. Drop > 2 pts = regression.
- **Smoke tests** — your own held-out prompts. Sanity-check qualitatively.

A common mistake is to celebrate a domain win that masks a 10-point general drop.

## 5. Implementing Parameter-Efficient Methods

### 5.1 The PEFT zoo

| Method | Trainable params | Memory | Notes |
|---|---|---|---|
| **LoRA** | $r(d_{\text{in}} + d_{\text{out}})$ per matrix | low | Default; merges at inference |
| **QLoRA** | Same as LoRA | 4-bit base | Fine-tune 33 B on a single 24 GB GPU |
| **DoRA** | LoRA + magnitude vector | slightly higher | Often beats LoRA |
| **VeRA** | Two shared random matrices + scaling | tiny | Multi-task heavy |
| **(IA)$^3$** | Three scaling vectors per block | tiny | Fast inference |
| **BitFit** | All biases | very low | Strong baseline |
| **Prompt tuning** | A small embedding matrix prepended | very low | Brittle for long context |
| **Prefix tuning** | Per-layer trainable prefixes | low | Powerful, harder to tune |
| **Adapters (Houlsby/Pfeiffer)** | Bottleneck MLPs inside blocks | low | Mature; serial vs parallel variants |
| **LoRA-FA** | LoRA + frozen down-projection | half of LoRA | Used in DoRA |

### 5.2 LoRA — the canonical method (Hu et al., 2021)

For a frozen linear layer $W \in \mathbb{R}^{d \times k}$, learn a low-rank update:

$$
W' = W + \Delta W, \qquad \Delta W = \tfrac{\alpha}{r} \cdot B A, \quad A \in \mathbb{R}^{r \times k}, \; B \in \mathbb{R}^{d \times r}.
$$

Typical $r \in \{4, 8, 16, 32, 64\}$, $\alpha = 2r$. At inference, **merge**: $W \leftarrow W + \tfrac{\alpha}{r} BA$ — zero overhead.

Pure-PyTorch implementation:

```python
class LoRALinear(nn.Module):
    def __init__(self, base: nn.Linear, r=8, alpha=16, dropout=0.0):
        super().__init__()
        self.base = base
        for p in self.base.parameters(): p.requires_grad = False
        d_in, d_out = base.in_features, base.out_features
        self.A = nn.Parameter(torch.empty(r, d_in))
        self.B = nn.Parameter(torch.zeros(d_out, r))
        nn.init.kaiming_uniform_(self.A, a=math.sqrt(5))   # Kaiming for A
        nn.init.zeros_(self.B)                              # zero for B -> ΔW=0 at init
        self.scale = alpha / r
        self.drop = nn.Dropout(dropout) if dropout > 0 else nn.Identity()

    def forward(self, x):
        return self.base(x) + self.drop(x) @ self.A.T @ self.B.T * self.scale
```

The standard target modules in a transformer block are the four attention projections (`q_proj`, `k_proj`, `v_proj`, `o_proj`); for LLaMA-style models add the MLP projections (`gate_proj`, `up_proj`, `down_proj`).

### 5.3 QLoRA — Dettmers et al., 2023

The breakthrough: **quantise the base model to 4-bit NormalFloat (NF4)**, keep LoRA in bf16, double-quantise the quantisation constants. Memory drops 4× while accuracy is preserved within 0.1 pts.

```python
from transformers import BitsAndBytesConfig, AutoModelForCausalLM
from peft import LoraConfig, get_peft_model, prepare_model_for_kbit_training

bnb = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",
    bnb_4bit_compute_dtype=torch.bfloat16,
    bnb_4bit_use_double_quant=True,
)

llm = AutoModelForCausalLM.from_pretrained(
    "Qwen/Qwen2.5-7B-Instruct", quantization_config=bnb,
    attn_implementation="flash_attention_2",
)
llm = prepare_model_for_kbit_training(llm, use_gradient_checkpointing=True)

lora_cfg = LoraConfig(
    r=64, lora_alpha=16, lora_dropout=0.05, bias="none",
    target_modules=["q_proj","k_proj","v_proj","o_proj",
                    "gate_proj","up_proj","down_proj"],
    task_type="CAUSAL_LM",
)
llm = get_peft_model(llm, lora_cfg)
llm.print_trainable_parameters()
```

Why NF4? The empirical distribution of pre-trained LLM weights is **roughly Gaussian**; NF4 is the quantile-based 4-bit code-book that is optimal for a Gaussian under squared error. Standard INT4 / FP4 codebooks distribute levels uniformly, which is sub-optimal.

QLoRA also introduces **paged optimisers** — Adam states overflow into CPU memory in pages, eliminating the OOM that used to plague long-context fine-tuning.

### 5.4 DoRA (Weight-Decomposed Low-Rank Adaptation, Liu et al., 2024)

DoRA decomposes the weight matrix into magnitude and direction:

$$
W = m \cdot \frac{V}{\|V\|_c}, \qquad m \in \mathbb{R}^{1 \times k}, \; V \in \mathbb{R}^{d \times k}.
$$

LoRA updates *direction*; DoRA additionally learns the per-column magnitude $m$. Typically +0.5 to +1.5 points over LoRA at similar parameter count. Supported in HuggingFace PEFT: `LoraConfig(use_dora=True)`.

### 5.5 Adapters (Houlsby et al., 2019 — Pfeiffer et al., 2020)

A bottleneck MLP inserted after each attention or FFN block:

$$
h \leftarrow h + W_{\text{up}} \cdot \sigma(W_{\text{down}} \cdot h), \quad W_{\text{down}} \in \mathbb{R}^{r \times d}, W_{\text{up}} \in \mathbb{R}^{d \times r}.
$$

The Pfeiffer variant keeps only one adapter per block (after FFN); Houlsby has two (after attention and FFN). Adapters are slightly slower than LoRA at inference because they're not mergeable; but they handle tasks where LoRA's low-rank assumption is too restrictive (multi-task switching).

### 5.6 (IA)$^3$ — Liu et al., 2022

Three learnable vectors per block: $l_k, l_v, l_{ff}$. They re-scale keys, values, and FFN activations:

$$
\text{Attn}(Q, K, V) = \text{softmax}\!\left(\frac{Q (l_k \odot K)^\top}{\sqrt d}\right)(l_v \odot V), \quad \text{FFN}(x) = W_2(l_{ff} \odot \text{GELU}(W_1 x)).
$$

Often 100× fewer parameters than LoRA. Competitive on many benchmarks.

### 5.7 Prefix and prompt tuning

**Prompt tuning** (Lester et al., 2021): prepend $K$ learnable token embeddings to the input. Only those embeddings train.

**Prefix tuning** (Li & Liang, 2021): per layer, prepend $K$ learnable key/value vectors to the attention. More expressive than prompt tuning.

Both work surprisingly well at very large model scales (≥ 11 B); at smaller scales they underperform LoRA.

### 5.8 PEFT for the visual branch

You can LoRA the ViT too. Target the attention projections plus the patch-embedding conv:

```python
target = ["q_proj","k_proj","v_proj","out_proj","fc1","fc2"]   # CLIP ViT
```

For best accuracy, also unfreeze the **last LayerNorm** and the **CLS token / positional embeddings** — those are tiny and frequently bottleneck adaptation.

### 5.9 Multi-adapter serving

A production VLM may host dozens of LoRA adapters (one per customer, per language, per domain). **S-LoRA** and **vLLM Multi-LoRA** schedule a single base model and swap adapter weights per request. Memory cost is proportional to total adapter size, not to base × adapters.

### 5.10 PEFT vs full fine-tune — when does it lose?

PEFT generally matches full fine-tune within 0.5–1 pt on most benchmarks. It loses when:

- The base model lacks the *capability* being fine-tuned (LoRA can't add what isn't there).
- The domain shift is huge (medical scans vs natural images) — consider continued pre-training first.
- The rank is too low for the task complexity — raise $r$ from 8 to 64 or 128.

## 6. Developing Loss Functions

### 6.1 The loss is the soul of the model

Two models with identical architecture but different losses behave entirely differently. A loss expresses *what counts as success*. For VLMs, success means a combination of contrastive alignment, image-text matching, language generation, masking-prediction, and (sometimes) explicit reward signals. We will walk every loss with formula + PyTorch.

### 6.2 Cross-entropy (the LM loss)

Given a sequence $w_1, \ldots, w_T$ and conditional images $\mathbf{V}$:

$$
\mathcal{L}_{\text{LM}} = -\sum_{t=1}^T \log p_\theta(w_t \mid w_{<t}, \mathbf{V}).
$$

Implementation detail: the LM loss is **only** computed over assistant tokens, never over system/user instruction tokens. Set `labels[mask_user_tokens] = -100` so PyTorch's `CrossEntropyLoss(ignore_index=-100)` skips them.

```python
def shift_and_mask(input_ids, labels):
    # next-token prediction
    return input_ids[:, :-1], labels[:, 1:]

logits = llm(input_ids).logits
shift_logits, shift_labels = shift_and_mask(logits, labels)
loss = F.cross_entropy(shift_logits.reshape(-1, shift_logits.size(-1)),
                       shift_labels.reshape(-1), ignore_index=-100)
```

### 6.3 Contrastive InfoNCE / CLIP loss

Already introduced in the Architecture doc. The PyTorch one-liner:

```python
def clip_loss(image_features, text_features, logit_scale):
    image_features = F.normalize(image_features, dim=-1)
    text_features  = F.normalize(text_features,  dim=-1)
    logits_per_image = logit_scale.exp() * image_features @ text_features.T
    labels = torch.arange(image_features.size(0), device=image_features.device)
    return 0.5 * (F.cross_entropy(logits_per_image, labels) +
                  F.cross_entropy(logits_per_image.T, labels))
```

The temperature $\tau = 1/\text{logit\_scale}$. Clip it to [0.01, 100] to avoid blow-ups:

```python
with torch.no_grad():
    logit_scale.clamp_(0, math.log(100))
```

### 6.4 SigLIP — pairwise sigmoid

$$
\mathcal{L}_{\text{SigLIP}} = -\frac{1}{B}\sum_{i,j} \log \sigma\!\big( z_{ij}(\tau s_{ij} + b) \big).
$$

PyTorch:

```python
def siglip_loss(img, txt, logit_scale, logit_bias):
    img = F.normalize(img, dim=-1); txt = F.normalize(txt, dim=-1)
    logits = logit_scale.exp() * img @ txt.T + logit_bias
    labels = 2 * torch.eye(img.size(0), device=img.device) - 1   # +1 on diag, -1 off-diag
    return -F.logsigmoid(labels * logits).mean()
```

### 6.5 Image-Text Matching (ITM)

A binary classifier on the fused [CLS] token:

```python
def itm_loss(model, img, txt, hard_negatives_idx):
    pos = model.fuse(img, txt).cls       # positive pair
    neg = model.fuse(img, txt[hard_negatives_idx]).cls
    logits = model.itm_head(torch.cat([pos, neg], dim=0))   # [2B, 2]
    targets = torch.cat([torch.ones(pos.size(0)),
                         torch.zeros(neg.size(0))]).long().to(logits.device)
    return F.cross_entropy(logits, targets)
```

Hard negatives are sampled with weights proportional to current cosine similarity — see ALBEF eq. (10).

### 6.6 Masked image modelling losses

**MAE** (He et al., 2022) reconstructs masked patches in pixel space:

$$
\mathcal{L}_{\text{MAE}} = \frac{1}{|\mathcal{M}|} \sum_{i \in \mathcal{M}} \| p_\theta(x_i) - x_i \|_2^2
$$

with per-patch L2 on the **normalised** target patches (mean-std per patch).

**BEiT** reconstructs **discrete visual tokens** from a frozen dVAE:

$$
\mathcal{L}_{\text{BEiT}} = \mathbb{E}_{x, \mathcal{M}}\sum_{i\in\mathcal{M}} -\log p_\theta(z_i \mid x_{\mathcal{M}^c})
$$

where $z_i$ are the dVAE's tokens.

**EVA** (Fang et al., 2023) predicts CLIP features at masked positions — feature distillation + masked modelling.

### 6.7 Captioning + LM mixture

Multi-task training combines several losses:

$$
\mathcal{L} = \lambda_1 \mathcal{L}_{\text{ITC}} + \lambda_2 \mathcal{L}_{\text{ITM}} + \lambda_3 \mathcal{L}_{\text{LM}} + \lambda_4 \mathcal{L}_{\text{MIM}}.
$$

Default weights: $1, 1, 1, 1$ for BLIP-2 representation stage; $0, 0, 1, 0$ for BLIP-2 generative stage; varied for FLAVA / BEiT-3.

### 6.8 DPO / KTO (preference losses)

For RLHF-style alignment of VLMs (LLaVA-RLHF, RLAIF-V):

$$
\mathcal{L}_{\text{DPO}} = -\mathbb{E}_{(x, y_w, y_l)} \log \sigma\!\Big(\beta \log \frac{\pi_\theta(y_w|x)}{\pi_{\text{ref}}(y_w|x)} - \beta \log \frac{\pi_\theta(y_l|x)}{\pi_{\text{ref}}(y_l|x)} \Big).
$$

where $(y_w, y_l)$ are chosen/rejected responses, $\pi_{\text{ref}}$ is the frozen reference model, $\beta \approx 0.1$. KTO (Ethayarajh et al., 2024) variant works with unpaired labels.

```python
def dpo_loss(pi_logp_chosen, pi_logp_rejected, ref_logp_chosen, ref_logp_rejected, beta=0.1):
    pi_diff  = pi_logp_chosen  - pi_logp_rejected
    ref_diff = ref_logp_chosen - ref_logp_rejected
    return -F.logsigmoid(beta * (pi_diff - ref_diff)).mean()
```

### 6.9 Auxiliary losses for stability

- **Z-loss** (Chowdhery et al., PaLM): penalise the log-sum-exp of the LM head logits — $\lambda_z \cdot \log^2 Z$ where $Z = \sum \exp(\text{logits})$. Keeps logits well-scaled, prevents NaN.
- **Router-z loss** for MoE (DeepSeek-MoE): balances expert load.
- **Sparsity / contrastive auxiliary** for image-text retrieval to encourage uniformity.

### 6.10 Designing a loss — a checklist

1. **Is the loss bounded below?** If not, expect divergence.
2. **Are gradients well-scaled when inputs are large?** Add log-sum-exp tricks.
3. **Does the loss reduce to a known good loss when constants degenerate?** Sanity.
4. **Is the loss invariant to your symmetry?** (translation, scale, etc.) If yes, good — if not, your data needs more augmentation.
5. **Does it scale linearly with batch size, or quadratically?** Contrastive scales $O(B^2)$; ITM scales $O(B)$.
6. **What does ChatGPT-as-a-judge think of the model trained on this loss?** A surprising sanity check.

### 6.11 Implementing a custom loss in PyTorch

```python
class VLMLoss(nn.Module):
    def __init__(self, lambda_itc=1.0, lambda_itm=1.0, lambda_lm=1.0):
        super().__init__()
        self.lambda_itc = lambda_itc
        self.lambda_itm = lambda_itm
        self.lambda_lm  = lambda_lm
        self.itm_head = nn.Linear(768, 2)

    def forward(self, vision_emb, text_emb, fused_cls, lm_logits, labels, hard_neg_idx, logit_scale):
        l_itc = clip_loss(vision_emb, text_emb, logit_scale)
        l_itm = F.cross_entropy(self.itm_head(fused_cls), labels)
        l_lm  = F.cross_entropy(lm_logits.reshape(-1, lm_logits.size(-1)),
                                labels.reshape(-1), ignore_index=-100)
        return self.lambda_itc * l_itc + self.lambda_itm * l_itm + self.lambda_lm * l_lm, {
            "itc": l_itc.detach(), "itm": l_itm.detach(), "lm": l_lm.detach(),
        }
```

## 7. Engineering Quantization Techniques

### 7.1 The vocabulary

| Term | Meaning |
|---|---|
| **PTQ** | Post-training quantisation; no re-training |
| **QAT** | Quantisation-aware training; simulate quantisation during fine-tuning |
| **Weight-only** | Activations stay in fp16/bf16, weights are int4/int8 |
| **W4A16** | 4-bit weights, fp16 activations (e.g., AWQ, GPTQ) |
| **W8A8** | 8-bit weights and activations (e.g., SmoothQuant) |
| **Activation outliers** | The tiny number of activation channels with huge magnitudes that ruin naive INT8 quantisation |
| **Per-tensor** | One scale for the whole tensor |
| **Per-channel** | One scale per output channel |
| **Per-group** | One scale per group of $g$ contiguous channels (typical $g=64$ or $128$) |
| **Static** | Activation scales precomputed from calibration data |
| **Dynamic** | Activation scales computed at runtime per batch |
| **Mixed-precision** | Different layers / channels use different bit widths |

### 7.2 The basic math

Symmetric uniform quantisation of a tensor $x$ to $b$-bit signed integers:

$$
s = \frac{\max(|x|)}{2^{b-1} - 1}, \quad q = \mathrm{clip}\!\left(\mathrm{round}(x/s),\; -2^{b-1},\; 2^{b-1}-1\right), \quad \hat{x} = s \cdot q.
$$

Asymmetric:

$$
s = \frac{x_{\max} - x_{\min}}{2^b - 1}, \quad z = \mathrm{round}(-x_{\min}/s), \quad q = \mathrm{clip}(\mathrm{round}(x/s) + z, 0, 2^b - 1).
$$

Quantisation error $\| x - \hat x \|$ is essentially uniform with variance $s^2/12$ → smaller $s$ means smaller error → group quantisation (one $s$ per small group) reduces error dramatically over per-tensor.

### 7.3 Outliers — why naive INT8 fails

Dettmers (LLM.int8(), 2022) discovered: in transformers > 6.7 B parameters, **a small number of activation channels have magnitudes 50–500× larger than the rest**. A per-tensor scale chosen for the outliers would round the typical channels to zero. Solutions:

- **LLM.int8()**: mixed precision — outlier channels in fp16, the rest in int8.
- **SmoothQuant** (Xiao et al., 2023): migrate the difficulty from activations to weights:
  $$
  Y = X \cdot W = (X \cdot \mathrm{diag}(s)^{-1})(\mathrm{diag}(s) \cdot W).
  $$
  Pick $s_j = \max(|X_{:,j}|)^\alpha / \max(|W_{j,:}|)^{1-\alpha}$ with $\alpha \approx 0.5$. Now both sides are well-conditioned.
- **AWQ** (Lin et al., 2023): identify the 1 % "salient" weight channels by activation magnitude and scale them up before INT4 rounding — preserves their representation. Pure weight-only.

### 7.4 GPTQ — Frantar et al., 2022

GPTQ uses second-order information: layer-wise, it picks weights to quantise one at a time, updating the remaining weights to compensate, using the OBS (Optimal Brain Surgeon) approximation with a small calibration set (128 examples). Produces 3–4 bit per-channel quantised matrices with < 1 % perplexity loss on most LLMs.

The competing landscape (early-2026 picture):

| Method | Bits | Salient features |
|---|---|---|
| **AWQ** | 4 W / 16 A | Best for serving; ~3× speedup |
| **GPTQ** | 3-4 W / 16 A | Best for low-bit |
| **GGUF (llama.cpp)** | 2-8 W / 16 A | Mature CPU/GPU ecosystem |
| **BitsAndBytes NF4** | 4 W / 16 A | Used in QLoRA; double quantisation |
| **SmoothQuant** | 8 W / 8 A | INT8 with no accuracy loss |
| **FP8 (Transformer Engine)** | 8 W / 8 A | H100 tensor cores |
| **HQQ** | 2-8 W / 16 A | Calibration-free |
| **SpQR** | 3 W / 16 A | Sparse-quantised |
| **OmniQuant** | 4-8 W / 16 A | Learnable equivalent transformations |

### 7.5 Quantising a VLM with HuggingFace + AWQ

```python
from awq import AutoAWQForCausalLM
from transformers import AutoTokenizer

tok = AutoTokenizer.from_pretrained("Qwen/Qwen2.5-7B-Instruct")
model = AutoAWQForCausalLM.from_pretrained(
    "Qwen/Qwen2.5-7B-Instruct", device_map="auto", torch_dtype="auto",
)
quant_config = {"zero_point": True, "q_group_size": 128, "w_bit": 4, "version": "GEMM"}
model.quantize(tok, quant_config=quant_config,
               calib_data="pileval", max_calib_samples=512, max_calib_seq_len=512)
model.save_quantized("./qwen2.5-7B-awq")
```

For VLMs, you must:

1. Quantise the LLM backbone (vision + projector remain in bf16).
2. **Calibrate with multimodal inputs** — pass image-text pairs through the model, not just text. Otherwise the activation statistics are wrong on visual-token positions.
3. Re-evaluate on multimodal benchmarks; sometimes you must rescale specific layers.

### 7.6 Quantisation-Aware Training (QAT)

QAT inserts **fake-quant** nodes (`FakeQuantize`) into the model and trains end-to-end with the quantisation simulation in the loss path. Used when post-training methods don't recover enough accuracy.

```python
import torch.ao.quantization as taq
qconfig = taq.QConfig(
    activation=taq.observer.MovingAverageMinMaxObserver.with_args(dtype=torch.qint8),
    weight=taq.observer.PerChannelMinMaxObserver.with_args(dtype=torch.qint8, qscheme=torch.per_channel_symmetric),
)
model.qconfig = qconfig
taq.prepare_qat(model, inplace=True)
# fine-tune for a few epochs
taq.convert(model, inplace=True)
```

For modern PyTorch (≥ 2.2) the `torch.ao.quantization.quantize_pt2e` API is the recommended path; it integrates with `torch.export`.

### 7.7 KV-cache quantisation

For inference, the KV cache often dwarfs the model weights. Quantising it to **fp8 or int8** can halve memory without measurable accuracy loss:

```python
model.generate(..., cache_implementation="quantized",
               cache_config={"backend": "HQQ", "nbits": 4, "axis_key": -1, "axis_value": -1})
```

vLLM, TensorRT-LLM, and HuggingFace all support fp8 KV cache.

### 7.8 FP8 — H100 / B200 native

FP8 (e4m3 for activations, e5m2 for gradients) is hardware-accelerated on H100. Use the Transformer Engine library:

```python
import transformer_engine.pytorch as te
from transformer_engine.common.recipe import Format, DelayedScaling

fp8_recipe = DelayedScaling(fp8_format=Format.HYBRID, amax_history_len=32)

with te.fp8_autocast(enabled=True, fp8_recipe=fp8_recipe):
    out = model(x)
```

Speed-ups: 1.5–2× over bf16 with no accuracy loss in most LLM pre-training runs.

### 7.9 Calibration data choice

The calibration dataset must mirror inference traffic:

- For a chat VLM, calibrate with chat-formatted multimodal data.
- For a video VLM, calibrate with video clips.
- For a multilingual VLM, calibrate with samples from every language.

A 128–512 sample calibration set is usually enough; > 2000 brings diminishing returns and risks overfitting the quantisation parameters.

### 7.10 Validating a quantised model

Three layers of validation:

1. **Per-layer activation MSE** vs reference — flag layers > 1 % MSE.
2. **End-to-end perplexity / loss** on a held-out set.
3. **Downstream benchmarks** (MMMU, MMBench). Anything worse than 1 % below the bf16 reference is a regression.

The dangerous failure: end-to-end perplexity looks fine but one specific capability (OCR? Chinese?) is broken. Always evaluate **multiple axes**.

## 8. Building Knowledge Distillation Systems

### 8.1 Why distillation matters

A 7 B-parameter teacher VLM running at 25 tok/s on an A100 is unusable on a phone. Knowledge distillation transfers the teacher's competence into a 0.5–2 B-parameter student that runs at 50 tok/s on a Snapdragon 8 Gen 3.

### 8.2 Three flavours of distillation

| Flavour | What is transferred | Loss |
|---|---|---|
| **Logit distillation (Hinton)** | Soft probabilities over vocab | KL divergence on softmaxed logits with temperature $T$ |
| **Feature distillation** | Intermediate hidden states | MSE on student-projected features |
| **Attention transfer** | Attention maps | MSE / KL on attention weights |

For VLMs you typically combine all three.

### 8.3 The Hinton 2015 loss

$$
\mathcal{L}_{\text{KD}} = \alpha \cdot T^2 \cdot \mathrm{KL}\!\big( p_\text{student}^{(T)} \,\|\, p_\text{teacher}^{(T)} \big) + (1-\alpha) \cdot \mathcal{L}_{\text{CE}}(p_\text{student}, y)
$$

where $p^{(T)}_i = \mathrm{softmax}(z_i / T)$ and $T \in [1, 10]$ smooths the teacher's distribution to reveal "dark knowledge" — relative probabilities among non-target classes that carry information beyond a one-hot label.

```python
def kd_loss(student_logits, teacher_logits, labels, T=4.0, alpha=0.5):
    soft_s = F.log_softmax(student_logits / T, dim=-1)
    soft_t = F.softmax(teacher_logits / T, dim=-1)
    l_kl = F.kl_div(soft_s, soft_t, reduction="batchmean") * (T * T)
    l_ce = F.cross_entropy(student_logits, labels)
    return alpha * l_kl + (1 - alpha) * l_ce
```

For LLM distillation on a sequence:

```python
def seq_kd(student_logits, teacher_logits, labels, T=2.0, alpha=0.7, ignore=-100):
    s = student_logits[:, :-1].reshape(-1, student_logits.size(-1))
    t = teacher_logits[:, :-1].reshape(-1, teacher_logits.size(-1))
    y = labels[:, 1:].reshape(-1)
    mask = (y != ignore)
    soft_s = F.log_softmax(s[mask] / T, dim=-1)
    soft_t = F.softmax(t[mask] / T, dim=-1)
    l_kl = F.kl_div(soft_s, soft_t, reduction="batchmean") * (T * T)
    l_ce = F.cross_entropy(s, y, ignore_index=ignore)
    return alpha * l_kl + (1 - alpha) * l_ce
```

### 8.4 Feature distillation

Match the student's hidden state at layer $l_s$ to the teacher's at layer $l_t$ via a learned projector $g$:

$$
\mathcal{L}_{\text{feat}} = \frac{1}{N} \sum_i \| g(h^{(l_s)}_{\text{stu},i}) - h^{(l_t)}_{\text{tea},i} \|_2^2.
$$

For VLM vision distillation, this is essentially what **EVA** does (CLIP teacher → masked-prediction student).

### 8.5 Attention transfer

Following Zagoruyko & Komodakis (2017), match a *normalised attention map*:

$$
\mathcal{L}_{\text{att}} = \sum_l \left\| \frac{A_{\text{stu}}^{(l)}}{\|A_{\text{stu}}^{(l)}\|_F} - \frac{A_{\text{tea}}^{(l)}}{\|A_{\text{tea}}^{(l)}\|_F} \right\|_F^2.
$$

For transformers, attention maps have shape `[B, H, T, T]`; align heads either via a Hungarian assignment or by pooling over heads.

### 8.6 Reverse KL and MiniLLM

Hinton's KL is $\mathrm{KL}(p_\text{stu} \| p_\text{tea})$ — *forward* KL. **Reverse KL** $\mathrm{KL}(p_\text{tea} \| p_\text{stu})$ is **mode-seeking**: the student focuses on the highest-probability modes of the teacher and ignores low-probability "long tails". For long-form generation, reverse KL produces more decisive students. **MiniLLM** (Gu et al., 2023) implements policy-gradient optimisation of reverse KL for generative distillation.

### 8.7 On-policy vs off-policy distillation

- **Off-policy**: teacher generates targets, student trains on them — fast but the distribution mismatch (teacher and student generate different sequences) hurts.
- **On-policy**: at each step, student generates a token, teacher predicts the distribution at that exact context, loss matches them. Slower but better.

A typical recipe: off-policy for the first 10 k steps to bootstrap, then on-policy.

### 8.8 Distilling a VLM — a complete recipe

```python
class DistillVLM(nn.Module):
    def __init__(self, teacher, student, T=2.0, alpha=0.7, feature_layers=(8, 16)):
        super().__init__()
        self.teacher = teacher.eval()
        for p in self.teacher.parameters(): p.requires_grad = False
        self.student = student
        self.T, self.alpha = T, alpha
        self.feat_proj = nn.ModuleList([
            nn.Linear(student.cfg.d, teacher.cfg.d) for _ in feature_layers
        ])
        self.feat_layers = feature_layers

    def forward(self, pixel_values, input_ids, labels):
        with torch.no_grad():
            t_out = self.teacher(pixel_values=pixel_values, input_ids=input_ids,
                                 output_hidden_states=True)
        s_out = self.student(pixel_values=pixel_values, input_ids=input_ids,
                             output_hidden_states=True)
        l_kd = seq_kd(s_out.logits, t_out.logits, labels, self.T, self.alpha)
        l_feat = 0
        for proj, l in zip(self.feat_proj, self.feat_layers):
            l_feat = l_feat + F.mse_loss(proj(s_out.hidden_states[l]),
                                         t_out.hidden_states[l])
        return l_kd + 0.1 * l_feat
```

### 8.9 Data — the secret sauce

The distillation data matters at least as much as the loss. Best practice:

- **Teacher-generated captions** for raw images (LLaVA-1.5's "CapFilt" trick).
- **Diverse instruction mix**: VQA, OCR, charts, math, reasoning, dialog.
- **Self-distillation** (student becomes the teacher of an even smaller student).

VLM-specific distillation papers: MiniLM-V (Wang et al., 2024), TinyLLaVA (2024), LLaVA-PoolDist, MobileVLM (Chu et al., 2023).

### 8.10 Evaluating a distilled model

- Match teacher's behaviour on a held-out **agreement set** (% same prediction).
- Match teacher's quality on **standard benchmarks** — gap should be < 5 pts for a 2× compression.
- Verify *what was lost* — usually long-tail knowledge or rare languages. Re-fine-tune on those if necessary.

### 8.11 A unified loss for VLM distillation

$$
\mathcal{L} = \alpha \cdot \mathcal{L}_{\text{KD-logit}}^{T} + \beta \cdot \mathcal{L}_{\text{feat}} + \gamma \cdot \mathcal{L}_{\text{att}} + (1-\alpha-\beta-\gamma) \cdot \mathcal{L}_{\text{LM}}
$$

with $\alpha=0.5,\, \beta=0.1,\, \gamma=0.05$ as a sensible starting point. Anneal $\beta, \gamma \to 0$ over training so the student isn't forever tied to the teacher's intermediate representations.
